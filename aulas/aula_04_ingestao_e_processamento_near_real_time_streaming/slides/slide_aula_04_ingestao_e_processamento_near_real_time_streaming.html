Tese da aula

Squad de IAs propõe; CloudFormation provisiona

Um único agente longo perde contexto, mistura papéis e alucina. Uma squad com papéis bem cortados acelera, audita e cruza pontos cegos. O console AWS some — entram CLI e templates declarativos.

Cinco regras desta aula
  1. Toda decisão de infra é arquivo YAML versionado — nada de cliques.
  2. Quatro papéis distintos na squad: Architect, DevOps, Tester, Reviewer.
  3. Cada recurso provisionado nasce com tag, log group e métrica.
  4. Re-rodar o deploy não pode mudar estado — idempotência ou nada.
  5. Recurso sem caminho de remediação no closed loop não entra no stack.
Teoria · Multi-agente

Por que sair do prompt único

Conforme o contexto cresce, o agente solitário começa a esquecer regras, misturar saídas e propagar alucinações. Dividir cognição é dividir risco.

Limites do prompt único

  • Contexto longo → perda de aderência.
  • Saídas genéricas misturadas (CFN+teste+ADR).
  • Difícil auditar onde a decisão nasceu.
  • Erro de uma etapa contamina as próximas.

Ganhos da squad

  • Cada papel tem prompt curto e auditável.
  • Saídas separadas, comparáveis, comitáveis.
  • Revisor age como devil's advocate.
  • Reduz alucinação cruzada entre artefatos.

Custos novos

  • Orquestração precisa estado mínimo.
  • Mais arquivos para versionar.
  • Latência soma se não houver paralelismo.
  • Conflito entre papéis exige árbitro humano.
Teoria · Padrões

Padrões de orquestração de agentes

Quatro arquétipos comuns. Hoje rodamos um híbrido planner+executor com revisão humana — é o que cabe na hora prática.

PadrãoComo funcionaQuando usarRisco
Planner + ExecutorUm agente planeja, outro(s) executam tarefas pequenas.Pipeline previsível com etapas claras.Plano ruim se propaga.
HierárquicoManager delega para sub-agentes especializados.Tarefas com dependências fortes entre subtarefas.Latência cumulativa.
BlackboardAgentes leem e escrevem em quadro comum.Problemas exploratórios, sem ordem fixa.Difícil auditar quem mudou o quê.
SwarmVários agentes paralelos, sem coordenador central.Geração de variantes, brainstorming.Pode divergir; precisa redutor.

Teoria · CloudFormation 101

Anatomia de um template CFN

Sete seções. Saber para que serve cada uma é pré-requisito para revisar o YAML que a IA gera.

SeçãoPara que serveObrigatória?
AWSTemplateFormatVersionVersão do schema (sempre 2010-09-09).Recomendada
DescriptionO que o template provisiona, em uma frase.Recomendada
ParametersEntradas do stack (env, iniciais, sizing).Não
MappingsTabelas estáticas (ex.: AMI por região).Não
ConditionsLiga/desliga recursos por env.Não
ResourcesO que vai existir na conta.Sim
OutputsValores expostos pelo stack (ARNs, URLs).Recomendada
Teoria · Funções intrínsecas

Funções intrínsecas que você vai ler hoje

A IA vai gerar template recheado disso. Saber decodificar é diferença entre revisar e copiar cego.

FunçãoPara quêExemplo
!RefReferencia parâmetro ou resource.!Ref BucketRaw
!SubInterpola variável em string.!Sub "${Iniciais}-raw"
!GetAttLê atributo de outro resource.!GetAtt MyEC2.PublicIp
!ImportValueLê Output exportado por outro stack.!ImportValue VpcId-lab
!JoinConcatena lista com separador.!Join ["/", [a, b]]
!IfCondicional baseado em Conditions.!If [IsProd, t3.large, t3.micro]
Teoria · Lifecycle

Lifecycle de um stack

CFN tem máquina de estados explícita. Conhecer os estados é condição para depurar deploy travado sem chutar.

OperaçãoEstados principaisQuando ocorre
CREATECREATE_IN_PROGRESS → CREATE_COMPLETEPrimeiro deploy com stack inexistente.
CREATE c/ falhaCREATE_FAILED → ROLLBACK_IN_PROGRESS → ROLLBACK_COMPLETERecurso falha; tudo é desfeito.
UPDATEUPDATE_IN_PROGRESS → UPDATE_COMPLETETemplate muda; aplica diff.
UPDATE c/ falhaUPDATE_ROLLBACK_IN_PROGRESS → UPDATE_ROLLBACK_COMPLETEReverte para estado anterior.
DELETEDELETE_IN_PROGRESS → DELETE_COMPLETEdelete-stack respeita DeletionPolicy.

Change set: simulação do diff antes do UPDATE. Recomendado em ambiente real, opcional aqui pelo escopo da aula.

Teoria · Guard rails

Capabilities, Stack Policy, DeletionPolicy

Três mecanismos que o CFN usa para impedir o "ops" silencioso. Entrar com elas no template ligadas é higiene de SRE.

Capabilities

  • CAPABILITY_IAM — aceita criar IAM Roles.
  • CAPABILITY_NAMED_IAM — aceita nome customizado.
  • Sem isso, deploy falha por segurança.

Stack Policy

  • JSON inline que protege resources contra UPDATE acidental.
  • Útil para banco de dados e buckets de raw.

DeletionPolicy

  • Delete · Retain · Snapshot.
  • Bucket S3 e RDS nunca devem ser Delete em prod.
  • No Lab, usamos Retain só para raw.
Teoria · Drift

Drift detection — IaC perde valor sem isso

Se alguém abre o console e edita um SG "só para ver", o template mente. Drift é a diferença entre o que está no template e o que está vivo na conta.

O que CFN considera drift
  • Propriedade alterada fora do CFN.
  • Recurso deletado manualmente.
  • Tag adicionada/removida no console.
  • Não detecta: dados dentro de S3/RDS.
Comandos da prática
  • aws cloudformation detect-stack-drift
  • ... describe-stack-drift-detection-status
  • ... describe-stack-resource-drifts
  • Resultado vai no 11_drift_runbook.md.
Teoria · Composição

Nested stacks e cross-stack references

Stack único e gigante é difícil de revisar e ainda mais difícil de atualizar. Quebrar em módulos por domínio é o padrão — do mesmo jeito que dividimos a squad.

Nested stacks
  • Pai inclui filhos via AWS::CloudFormation::Stack.
  • Atualização do pai propaga.
  • Bom quando o ciclo de vida é compartilhado.
Cross-stack via Outputs/Exports
  • Stack A exporta valor; Stack B faz !ImportValue.
  • Ciclos de vida independentes.
  • Cuidado: Export usado não pode mudar nome.

Nesta aula: usamos nested stacks com root.yaml incluindo network, storage, compute, observability e sre-loop. Ciclo de vida único: o stack do laboratório.

CloudFormation vs Terraform vs CDK

Por que CFN é o default da disciplina? Porque é o que cabe no Learner Lab sem instalar nada além do AWS CLI.

Teoria · SRE

SRE começa no template, não no alarme

Cada recurso provisionado já nasce com tag de SLO, log group e métrica. Observabilidade adicionada depois é dívida garantida.

Higiene mínima por recurso
  • Tags Project, Env, SLO, Owner.
  • Log group dedicado quando aplicável.
  • Métricas customizadas via CW Embedded Metric Format.
  • DeletionPolicy e UpdateReplacePolicy escolhidos.
Closed loop herdado da Aula 03
  • Sensor: CW alarm de frescor.
  • Comparador/Controlador: EventBridge rule.
  • Atuador: SSM Run Command (retry no ETL).
  • Audit: log group do controlador.
  • Tudo entra no sre-loop.yaml.
Teoria · Anti-padrões

Anti-padrões em template CFN

A IA vai acertar 80%. Os 20% restantes geralmente são variações destes erros. Reviewer existe para pegá-los.

Anti-padrãoPor que dóiComo evitar
Secret no templateVai parar no git e nos logs do stack.SSM Parameter Store + !Sub.
IAM com Action: '*'Privilege escalation pronta.Action mínimo + Resource mínimo.
Sem DeletionPolicyBucket de raw some no delete-stack.Definir explícito por resource crítico.
ARN hardcodedQuebra ao trocar de conta/região.!Sub com ${AWS::AccountId} e ${AWS::Region}.
Sem OutputsOutros stacks não conseguem consumir.Exportar IDs e ARNs relevantes.
Tags ausentesFinOps cego, SLO sem dono.Tags obrigatórias no padrão da turma.

Prática · Passo 4

Skills novas — gen_cfn_template

Mesmo padrão das skills da Aula 03: nome verbo+objeto, quando usar, entrada, passos, saída, critério de aceitação. Faça também lint_cfn e review_cfn análogas.

# documents/agents/skills/gen_cfn_template.md

## Quando usar
Quando existir iac/cloudformation/PLAN.md descrevendo modulos.

## Entrada
- iac/cloudformation/PLAN.md
- documents/03_architecture.md

## Passos
1. Para cada modulo do PLAN.md, gere um YAML em iac/cloudformation/.
2. Inclua AWSTemplateFormatVersion, Description, Parameters, Resources, Outputs.
3. Tags obrigatorias: Project=olist-sre, Env=!Ref Env, Owner=!Ref Iniciais, SLO=...
4. DeletionPolicy: Retain em S3 raw; Delete no resto do Lab.
5. Sem secrets. Sem Action:'*' em IAM. Sem ARN hardcoded.
6. root.yaml inclui modulos via AWS::CloudFormation::Stack.

## Saida
- iac/cloudformation/{root,network,storage,compute,observability,sre-loop}.yaml

## Aceitacao
- cfn-lint passa em todos os arquivos.
- Cada modulo tem ao menos 1 Output.
- Tags presentes em 100% dos resources tagueaveis.

Prática · Passo 9

Validar template no servidor antes do deploy

validate-template roda no lado da AWS. Pega erro de schema/intrinsic function que o lint local pode não cobrir.

for f in iac/cloudformation/*.yaml; do
  echo "==== $f ===="
  aws cloudformation validate-template \
    --template-body "file://$f" \
    --query "Description" --output text
done

# Empacotar nested stacks (sobe filhos para S3)
aws s3 mb s3://${INICIAIS}-cfn-artifacts-lab 2>/dev/null || true

aws cloudformation package \
  --template-file iac/cloudformation/root.yaml \
  --s3-bucket    ${INICIAIS}-cfn-artifacts-lab \
  --output-template-file iac/cloudformation/root.packaged.yaml
Prática · Passo 10

Deploy do stack — momento da verdade

Único comando que cria recursos hoje. Acompanhe pela CLI; se quiser ver eventos, use describe-stack-events — nada de console.

export INICIAIS=afn   # suas iniciais

aws cloudformation deploy \
  --stack-name olist-sre-pipeline-lab \
  --template-file iac/cloudformation/root.packaged.yaml \
  --capabilities CAPABILITY_IAM \
  --parameter-overrides \
      Env=lab \
      Iniciais=${INICIAIS} \
  --tags Project=olist-sre Env=lab Owner=${INICIAIS}

# Acompanhar (em outro shell)
aws cloudformation describe-stack-events \
  --stack-name olist-sre-pipeline-lab \
  --max-items 20 \
  --query "StackEvents[].[Timestamp,LogicalResourceId,ResourceStatus]" \
  --output table

Prática · Passo 13

Drift detection — foto vs realidade

Roda agora, antes que alguém edite um SG no console. O resultado vai para 11_drift_runbook.md como referência da próxima aula.

DRIFT_ID=$(aws cloudformation detect-stack-drift \
  --stack-name olist-sre-pipeline-lab \
  --query "StackDriftDetectionId" --output text)

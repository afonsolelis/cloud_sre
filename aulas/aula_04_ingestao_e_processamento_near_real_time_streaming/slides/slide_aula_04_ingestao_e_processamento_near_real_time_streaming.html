Tese da aula

Antes de pagar nuvem, sobe local

A interface é o contrato — não o fornecedor. Quem fala S3 API e SQL, fala com MinIO+ClickHouse local hoje e com S3+Redshift amanhã. A engenharia mora nas interfaces; a fatura, no provedor.

Quatro regras desta aula
  1. Stack inteira em docker compose up -d — nada de clique em console cloud hoje.
  2. Ingestão é append-only: cada execução marca unixtime e tag próprios.
  3. Modelagem é schema-on-read — landing como JSON, modelagem nas views.
  4. Healthcheck em todo serviço crítico — depends_on: service_healthy ou nada.
Teoria · Estratégia

Por que stack local antes da nuvem

A nuvem cobra por hora — e por erro. O laboratório local cobra por curiosidade. Mesmas interfaces, latência menor, custo zero.

Feedback loop

  • Subir/derrubar em segundos.
  • Erro caro vira erro barato.
  • Sem IAM no caminho do iniciante.
  • Tudo cabe no laptop do aluno.

Paridade dev/prod

  • MinIO fala S3 API — mesmo SDK boto3.
  • ClickHouse é o mesmo no laptop e no servidor.
  • O contrato é a interface, não o fornecedor.
  • Migrar para AWS muda só endpoint e credencial.

Custo & reprodutibilidade

  • Zero fatura mensal.
  • Stack inteira em um compose.yaml.
  • Aluno reproduz em casa sem session token.
  • Reset com docker compose down -v.
Teoria · Espectro

Batch, micro-batch, streaming

"Near real-time" é uma faixa do espectro — não um ponto. Hoje rodamos micro-batch frequente. Streaming verdadeiro vem com Kafka/Flink e tem custo de operação maior.

ModoLatência típicaComo funcionaQuando usar
BatchHoras / diaJanela longa, processa tudo de uma vez.Relatório diário, BI clássico.
Micro-batchSegundos / minutosLotes pequenos e frequentes — pandas, Spark Structured Streaming.Dashboards "vivos", agregações leves.
StreamingSub-segundoEvento a evento — Kafka, Flink, Kinesis.Detecção de fraude, telemetria crítica.

Hoje: micro-batch via ingest_minio.py rodando em loop manual ou cron. Sem Kafka, sem Flink — deliberadamente.

Teoria · Compose

Docker Compose como contrato de stack

Um arquivo YAML descreve todo o sistema — serviços, volumes, redes, healthchecks, dependências. Vira a fonte única de verdade do ambiente.

Conceitos centrais
  • services: cada container do sistema.
  • volumes: persistência fora do container.
  • healthcheck: SLO local — "estou pronto".
  • depends_on: service_healthy: orquestração de startup.
  • ports: expõe para o host.
  • environment: configuração injetada.
Os 4 serviços da nossa stack
  • clickhouse — warehouse colunar.
  • minio — object storage S3-compatible.
  • ingestor — worker Python idle.
  • dashboard — Streamlit servindo BI.

Teoria · MinIO

MinIO 101 — S3 API no laptop

Servidor de objetos open source que implementa a API S3. Mesmos SDKs, mesmos verbos, mesmas semânticas. O endpoint muda, o resto não.

Conceitos S3
  • Bucket: container nomeado de objetos.
  • Objeto: arquivo + metadados, identificado por key.
  • Endpoint URL: onde o SDK vai bater.
  • Path-style vs virtual-hosted: como o bucket aparece na URL.
  • API REST — PutObject, GetObject, ListObjectsV2.
O que muda do S3 real
  • Endpoint: http://minio:9000 (não AWS).
  • Credenciais: minioadmin/minioadmin — só local.
  • boto3 precisa de endpoint_url=... no client.
  • Console web em localhost:9003 — análogo ao AWS Console.
  • Sem IAM elaborado, sem KMS — volta a aparecer na nuvem.
Teoria · ClickHouse

ClickHouse 101 — OLAP colunar rápido

Banco colunar, vetorizado, SQL-friendly. Engine padrão é MergeTree: armazena ordenado pela chave, compacta por coluna, varre só o que precisa.

MergeTree

  • Engine de tabela base do CH.
  • ORDER BY define a chave de leitura.
  • Dados particionados em "parts" no disco.
  • Merges em background — nome da família.

Tipo JSON

  • Coluna semi-estruturada nativa.
  • Permite schema-on-read sem perder performance.
  • Indexa subcaminhos automaticamente.
  • Bom para landing de dados heterogêneos.

LowCardinality

  • Wrapper para colunas com poucos valores únicos.
  • Armazena dicionário + códigos.
  • Reduz drasticamente disco e CPU.
  • Perfeito para tag, region, status.
Teoria · Modelagem

Schema-on-read vs schema-on-write

Quem decide o schema — quem grava ou quem lê? A escolha define quão rápido o pipeline se adapta a novas fontes.

AspectoSchema-on-writeSchema-on-read
Onde valida o tipoNa ingestão.Na query.
Custo de mudar schemaAlto — ALTER, migration.Baixo — só muda a view.
RiscoQuebra cedo, perde dado.Quebra tarde, atrasa relatório.
Exemplo clássicoPostgres, Redshift.Data Lake, ClickHouse JSON, BigQuery STRUCT.
Nossa escolha hoje—Landing como JSON, modelagem nas views.
Modelagem · Raw

A tabela landing — uma só, para tudo

Três colunas só: unixtime, data, tag. Qualquer CSV que entre no bucket cai aqui. A "tabela" lógica do produto vira uma view mais tarde.

sql/raw/01_schema.sql 
CREATE DATABASE IF NOT EXISTS olist_raw;

CREATE TABLE IF NOT EXISTS olist_raw.ingestion
(
  unixtime Int64,
  data     JSON,
  tag      LowCardinality(String)
)
ENGINE = MergeTree
ORDER BY (tag, unixtime);

Padrão: append-only. Cada execução do ingestor gera um unixtime novo no início do run e tagueia cada linha com o nome do CSV de origem. Não existe UPDATE nem DELETE aqui — só INSERT.

Padrão · Idempotência

Idempotência sem MERGE nem UPSERT

Reingerir o mesmo CSV grava de novo — isso é bom. A view filtra max(unixtime) por tag e enxerga só a execução mais recente. Linhagem livre, sem efeito colateral.

Como funciona
  • Ingestor calcula unixtime = int(time.time()) uma vez no início.
  • Toda linha desse run leva esse unixtime.
  • tag = nome do CSV (ex.: olist_order_items_dataset.csv).
  • View consome WHERE unixtime = (SELECT max(unixtime) FROM ... WHERE tag = X).
O que isso compra
  • Reingestões sempre seguras — nunca corrompem o estado.
  • Histórico das execuções preservado — vira auditoria de graça.
  • Reverter para versão anterior = filtrar outro unixtime.
  • Zero locks, zero transações distribuídas.
Código · Ingestor

app/ingest_minio.py — o coração da ingestão

Lê CSV do MinIO via boto3, vira pandas, gera NDJSON, manda em batches de 50k linhas para o ClickHouse. Cada linha vira { unixtime, data, tag }.

ingest_minio.py (resumo) 
import time, json, boto3, pandas as pd, clickhouse_connect

UNIXTIME = int(time.time())  # uma marca por execucao

s3 = boto3.client(
    "s3",
    endpoint_url=os.getenv("S3_ENDPOINT_URL"),
    aws_access_key_id=os.getenv("AWS_ACCESS_KEY_ID"),
    aws_secret_access_key=os.getenv("AWS_SECRET_ACCESS_KEY"),
)

ch = clickhouse_connect.get_client(
    host=os.getenv("CLICKHOUSE_HOST"),
    username=os.getenv("CLICKHOUSE_USER"),
    password=os.getenv("CLICKHOUSE_PASSWORD"),
)

for obj in s3.list_objects_v2(Bucket=BUCKET)["Contents"]:
    key = obj["Key"]
    body = s3.get_object(Bucket=BUCKET, Key=key)["Body"]
    df = pd.read_csv(body, dtype=str).where(pd.notnull, None)

    rows = [
        (UNIXTIME, json.dumps(row.to_dict()), key)
        for _, row in df.iterrows()
    ]
    for i in range(0, len(rows), 50_000):
        ch.insert(
            "olist_raw.ingestion",
            rows[i : i + 50_000],
            column_names=["unixtime", "data", "tag"],
        )
Modelagem · View

v_top_products — junta três CSVs, sempre na execução mais recente

A view materializa "produto mais vendido" sem materializar tabela física. Cada CTE pega só o max(unixtime) da sua tag. Reingere sem medo.

sql/views/01_top_products.sql (resumo) 
CREATE OR REPLACE VIEW olist_raw.v_top_products AS
WITH
  order_items AS (
    SELECT data FROM olist_raw.ingestion
    WHERE tag = 'olist_order_items_dataset.csv'
      AND unixtime = (
        SELECT max(unixtime) FROM olist_raw.ingestion
        WHERE tag = 'olist_order_items_dataset.csv'
      )
  ),
  products AS (
    SELECT data FROM olist_raw.ingestion
    WHERE tag = 'olist_products_dataset.csv'
      AND unixtime = (SELECT max(unixtime) FROM olist_raw.ingestion
                      WHERE tag = 'olist_products_dataset.csv')
  ),
  translations AS (
    SELECT data FROM olist_raw.ingestion
    WHERE tag = 'product_category_name_translation.csv'
      AND unixtime = (SELECT max(unixtime) FROM olist_raw.ingestion
                      WHERE tag = 'product_category_name_translation.csv')
  )
SELECT
  oi.data.product_id           AS product_id,
  p.data.product_category_name AS category_pt,
  t.data.product_category_name_english AS category_en,
  count()                                AS units_sold,
  round(sum(toFloat64(oi.data.price)))   AS revenue,
  round(avg(toFloat64(oi.data.price)))   AS avg_price
FROM order_items oi
LEFT JOIN products     p ON p.data.product_id = oi.data.product_id
LEFT JOIN translations t ON t.data.product_category_name = p.data.product_category_name
GROUP BY product_id, category_pt, category_en;
Frontend · Streamlit

app/streamlit_app.py — dashboard de top produtos

Lê a view v_top_products via HTTP, cacheia 60s e renderiza filtros + tabela + gráfico de barras + breakdown por categoria. Indica a hora da última ingestão.

Estrutura da página
  • Sidebar: Top N (5 – 100), métrica, filtro de categoria.
  • Header: métricas resumo — total de produtos, receita.
  • Tabela: top-N ordenado.
  • Gráfico de barras: receita ou unidades por produto.
  • Breakdown: agregado por categoria.
  • Rodapé: timestamp UTC da última ingestão.
Decisões técnicas
  • @st.cache_data(ttl=60) — segura o ClickHouse.
  • Conexão como usuário dashboard read-only.
  • Sem joins na app — tudo já vem da view.
  • Headless: --server.headless=true para Docker.
  • Porta 8501 exposta no host.

Discussão · Espectro

Quando trocar micro-batch por streaming de verdade

Micro-batch resolve a maior parte dos dashboards. Streaming verdadeiro vira obrigatório quando a janela de decisão fica menor que o intervalo do lote.

Sinais de upgrade para streaming verdadeiro
  1. Decisão de negócio precisa rodar em < 5 segundos (fraude, autorização, matching).
  2. Volume torna lotes inviáveis — back-pressure constante, ingestor sempre atrasado.
  3. Múltiplos consumidores precisam dos mesmos eventos em paralelo — broker resolve melhor.
  4. Time já tem maturidade operacional para Kafka/MSK ou Kinesis — senão, o custo de operar mata o ganho.

Prática · Passos 1 – 3

Subir a stack

1. Clonar o repo

  • git clone do repositório da aula.
  • cp .env.example .env — defaults já funcionam.
  • Conferir docker-compose.yaml presente.

2. Subir os containers

  • docker compose up -d --build
  • docker compose ps → todos healthy.
  • Logs: docker compose logs -f clickhouse.

3. Criar bucket no MinIO

  • Abrir http://localhost:9003.
  • Login: minioadmin / minioadmin.
  • Buckets → Create → olist.
Prática · Passos 4 – 6

Popular landing e rodar ingestão

4. Upload dos CSVs

  • Pegar os CSVs da pasta data/ do repo.
  • Arrastar para o bucket olist no console.
  • Ou usar o cliente mc: mc cp data/*.csv local/olist/.

5. Aplicar o schema

  • docker exec -i olist_analytics clickhouse-client < sql/raw/01_schema.sql
  • Cria database olist_raw e tabela ingestion.
  • Reexecuta sem erro — IF NOT EXISTS.

6. Rodar a ingestão

  • docker exec olist_ingestor python /app/ingest_minio.py
  • Lê todos os CSVs do bucket.
  • Cada CSV gera N inserts no ingestion.
Prática · Passos 7 – 9

Validar landing e criar a view

7. Conferir a landing

  • SELECT tag, count() FROM olist_raw.ingestion GROUP BY tag
  • Deve aparecer uma linha por CSV ingerido.
  • Total deve bater com o número de linhas do CSV.

8. Aplicar a view

  • docker exec -i olist_analytics clickhouse-client < sql/views/01_top_products.sql
  • Cria olist_raw.v_top_products.
  • É só metadado — barata, refaz sem custo.

9. Validar a view

  • SELECT * FROM olist_raw.v_top_products LIMIT 10
  • Deve devolver produtos com categoria PT/EN e métricas.
  • Confere ordenação por revenue DESC.
Prática · Passos 10 – 12

Dashboard, idempotência e encerramento

10. Abrir o dashboard

  • Acessar http://localhost:8501.
  • Brincar com filtros (Top N, métrica, categoria).
  • Conferir o timestamp "última ingestão" no rodapé.

11. Reingerir — ver idempotência

  • Rodar o ingestor de novo: docker exec olist_ingestor python /app/ingest_minio.py.
  • SELECT tag, count(DISTINCT unixtime) FROM olist_raw.ingestion GROUP BY tag → aumenta.
  • Dashboard mostra os mesmos números — view filtra max(unixtime).

12. Encerrar

  • docker compose down — volumes ficam.
  • docker compose down -v — volumes vão embora (reset).
  • Commit do trabalho no fork pessoal.

Tese da aula

A interface como contrato de engenharia

A interface, e não o fornecedor, constitui o contrato técnico relevante. Um pipeline que se comunica por meio da API S3 e da linguagem SQL opera tanto sobre MinIO e ClickHouse em ambiente local quanto sobre S3 e Redshift em ambiente de nuvem, com alterações restritas a endpoint e credenciais. A engenharia reside nas interfaces; a fatura, no provedor.

Quatro regras desta aula
  1. A stack completa é orquestrada por meio de docker compose up -d, sem operações em consoles de provedores de nuvem ao longo desta aula.
  2. A ingestão segue o paradigma append-only, em que cada execução registra unixtime e tag próprios.
  3. A modelagem adota o regime schema-on-read, mantendo a landing como JSON bruto e concentrando a tipagem nas views.
  4. Healthchecks são obrigatórios em todo serviço crítico, sustentando a diretiva depends_on: service_healthy.
Teoria · Estratégia

Justificativa para a execução local previamente à nuvem

Provedores de nuvem cobram por tempo de uso e penalizam erros operacionais com fatura adicional, ao passo que o laboratório local permite experimentação irrestrita. Ao preservar as mesmas interfaces, obtém-se latência menor e custo nulo.

Ciclo de feedback

  • Provisão e desprovisão em segundos.
  • Erros de alto custo são convertidos em erros de baixo custo.
  • Ausência de IAM no caminho do iniciante.
  • Ambiente integralmente contido no equipamento do estudante.

Paridade entre desenvolvimento e produção

  • O MinIO implementa a API S3, preservando o mesmo SDK boto3.
  • O ClickHouse é idêntico no laptop e no servidor de produção.
  • O contrato é definido pela interface, não pelo fornecedor.
  • A migração para a AWS exige alteração apenas de endpoint e credenciais.

Custo e reprodutibilidade

  • Ausência de fatura mensal.
  • Stack inteira declarada em um único compose.yaml.
  • O estudante reproduz o ambiente sem dependência de session token.
  • Reset integral por meio de docker compose down -v.
Teoria · Espectro

Batch, micro-batch e streaming

A expressão "near real-time" designa uma faixa do espectro de latência, e não um ponto fixo. Nesta aula adota-se o regime de micro-batch frequente; o streaming propriamente dito, suportado por Kafka ou Flink, comporta custo operacional consideravelmente superior.

ModoLatência típicaMecânica de operaçãoCenários adequados
BatchHoras a diaJanela longa, processamento do conjunto integral em uma única execução.Relatório diário, BI clássico.
Micro-batchSegundos a minutosLotes pequenos e frequentes, conduzidos por pandas ou Spark Structured Streaming.Dashboards operacionais, agregações analíticas leves.
StreamingSub-segundoProcessamento evento a evento, suportado por Kafka, Flink ou Kinesis.Detecção de fraude, telemetria crítica.

Implementação adotada: micro-batch executado via ingest_minio.py, acionado por meio de loop manual ou cron, com exclusão deliberada de Kafka e Flink.

Teoria · Compose

Docker Compose como contrato declarativo da stack

Um único arquivo YAML descreve todo o sistema, abrangendo serviços, volumes, redes, healthchecks e dependências. Esse artefato constitui a fonte única de verdade do ambiente.

Conceitos centrais
  • services: cada container do sistema.
  • volumes: persistência externa ao ciclo de vida do container.
  • healthcheck: SLO local que sinaliza prontidão do serviço.
  • depends_on: service_healthy: orquestração condicionada do startup.
  • ports: exposição de portas para o host.
  • environment: configuração injetada no processo.
Os quatro serviços que compõem a stack
  • clickhouse — data warehouse colunar.
  • minio — object storage compatível com a API S3.
  • ingestor — worker Python em estado ocioso, executado sob demanda.
  • dashboard — aplicação Streamlit responsável pela camada de BI.

Teoria · MinIO

MinIO 101 — API S3 em ambiente local

O MinIO é um servidor de objetos de código aberto que implementa a API S3. Mantém-se a mesma família de SDKs, os mesmos verbos REST e a mesma semântica de operações; o endpoint constitui a única alteração substantiva.

Conceitos da API S3
  • Bucket: container nomeado de objetos.
  • Objeto: arquivo e metadados associados, identificado por key.
  • Endpoint URL: localização à qual o SDK direciona as requisições.
  • Path-style versus virtual-hosted: forma como o bucket é expresso na URL.
  • API REST com operações como PutObject, GetObject e ListObjectsV2.
Diferenças em relação ao S3 em produção
  • Endpoint: http://minio:9000, distinto do endereço AWS.
  • Credenciais: minioadmin/minioadmin, restritas ao ambiente local.
  • O boto3 exige a configuração explícita de endpoint_url=... no cliente.
  • Console web em localhost:9003, análogo funcional ao AWS Console.
  • Recursos como IAM elaborado e KMS são adicionados apenas no ambiente de nuvem.
Teoria · ClickHouse

ClickHouse 101 — banco colunar OLAP de alto desempenho

O ClickHouse é um banco colunar e vetorizado, com sintaxe SQL. Sua engine padrão é a MergeTree, que armazena os dados ordenados pela chave definida, compacta por coluna e realiza varreduras restritas às porções relevantes.

MergeTree

  • Engine fundamental de tabela do ClickHouse.
  • A cláusula ORDER BY define a chave de leitura.
  • Os dados são particionados em parts em disco.
  • Operações de merge ocorrem em background, originando o nome da família.

Tipo JSON

  • Coluna semi-estruturada de tipo nativo.
  • Viabiliza o regime schema-on-read sem degradação de performance.
  • Indexa subcaminhos de forma automática.
  • Adequado à landing de dados heterogêneos.

LowCardinality

  • Wrapper destinado a colunas com baixa cardinalidade.
  • Armazena dicionário e códigos correspondentes.
  • Reduz significativamente o consumo de disco e CPU.
  • Particularmente indicado para tag, region e status.
Teoria · Modelagem

Schema-on-read versus schema-on-write

A definição de quem estabelece o schema — a escrita ou a leitura — determina a velocidade com que o pipeline se adapta a novas fontes de dados.

AspectoSchema-on-writeSchema-on-read
Local de validação do tipoNa ingestão.Na query.
Custo de alteração do schemaAlto, exigindo ALTER e migração.Baixo, restrito à modificação da view.
Perfil de riscoFalha precoce, com perda potencial de dados.Falha tardia, com atraso na geração de relatórios.
Exemplos clássicosPostgres, Redshift.Data Lake, ClickHouse JSON, BigQuery STRUCT.
Opção adotada nesta aula—Landing como JSON, com modelagem concentrada nas views.
Modelagem · Raw

Tabela landing única para o conjunto de fontes

A tabela conta com apenas três colunas: unixtime, data e tag. Qualquer CSV depositado no bucket é gravado nessa estrutura, e a representação lógica das tabelas de produto emerge posteriormente, na forma de view.

sql/raw/01_schema.sql 
CREATE DATABASE IF NOT EXISTS olist_raw;

CREATE TABLE IF NOT EXISTS olist_raw.ingestion
(
  unixtime Int64,
  data     JSON,
  tag      LowCardinality(String)
)
ENGINE = MergeTree
ORDER BY (tag, unixtime);

Padrão adotado: append-only. Cada execução do ingestor calcula um novo unixtime no início da rotina e marca cada linha com o nome do CSV de origem. Não há operações de UPDATE ou DELETE; apenas INSERT.

Padrão · Idempotência

Idempotência sem recurso a MERGE ou UPSERT

A reingestão do mesmo CSV produz nova gravação, o que constitui comportamento desejado. A view aplica o filtro max(unixtime) por tag e expõe apenas a execução mais recente, preservando a linhagem sem qualquer efeito colateral.

Mecânica de operação
  • O ingestor calcula unixtime = int(time.time()) uma única vez ao iniciar.
  • Todas as linhas da execução recebem esse mesmo valor de unixtime.
  • A tag corresponde ao nome do CSV (por exemplo, olist_order_items_dataset.csv).
  • A view consulta WHERE unixtime = (SELECT max(unixtime) FROM ... WHERE tag = X).
Benefícios decorrentes
  • Reingestões são integralmente seguras e não comprometem o estado.
  • O histórico das execuções permanece preservado, configurando trilha de auditoria sem custo adicional.
  • A reversão para uma versão anterior reduz-se ao filtro por outro unixtime.
  • Não há necessidade de locks ou de transações distribuídas.
Código · Ingestor

app/ingest_minio.py — o núcleo da ingestão

O script lê os CSVs no MinIO por meio do boto3, converte os dados para pandas, gera NDJSON e os encaminha em lotes de 50 mil linhas ao ClickHouse. Cada linha resultante é representada como { unixtime, data, tag }.

ingest_minio.py (resumo) 
import time, json, boto3, pandas as pd, clickhouse_connect

UNIXTIME = int(time.time())  # uma marca por execucao

s3 = boto3.client(
    "s3",
    endpoint_url=os.getenv("S3_ENDPOINT_URL"),
    aws_access_key_id=os.getenv("AWS_ACCESS_KEY_ID"),
    aws_secret_access_key=os.getenv("AWS_SECRET_ACCESS_KEY"),
)

ch = clickhouse_connect.get_client(
    host=os.getenv("CLICKHOUSE_HOST"),
    username=os.getenv("CLICKHOUSE_USER"),
    password=os.getenv("CLICKHOUSE_PASSWORD"),
)

for obj in s3.list_objects_v2(Bucket=BUCKET)["Contents"]:
    key = obj["Key"]
    body = s3.get_object(Bucket=BUCKET, Key=key)["Body"]
    df = pd.read_csv(body, dtype=str).where(pd.notnull, None)

    rows = [
        (UNIXTIME, json.dumps(row.to_dict()), key)
        for _, row in df.iterrows()
    ]
    for i in range(0, len(rows), 50_000):
        ch.insert(
            "olist_raw.ingestion",
            rows[i : i + 50_000],
            column_names=["unixtime", "data", "tag"],
        )
Modelagem · View

v_top_products — junção de três CSVs sempre na execução mais recente

A view formaliza a métrica de "produto mais vendido" sem materializar uma tabela física. Cada CTE seleciona exclusivamente o max(unixtime) da respectiva tag, o que garante que as reingestões sejam seguras.

sql/views/01_top_products.sql (resumo) 
CREATE OR REPLACE VIEW olist_raw.v_top_products AS
WITH
  order_items AS (
    SELECT data FROM olist_raw.ingestion
    WHERE tag = 'olist_order_items_dataset.csv'
      AND unixtime = (
        SELECT max(unixtime) FROM olist_raw.ingestion
        WHERE tag = 'olist_order_items_dataset.csv'
      )
  ),
  products AS (
    SELECT data FROM olist_raw.ingestion
    WHERE tag = 'olist_products_dataset.csv'
      AND unixtime = (SELECT max(unixtime) FROM olist_raw.ingestion
                      WHERE tag = 'olist_products_dataset.csv')
  ),
  translations AS (
    SELECT data FROM olist_raw.ingestion
    WHERE tag = 'product_category_name_translation.csv'
      AND unixtime = (SELECT max(unixtime) FROM olist_raw.ingestion
                      WHERE tag = 'product_category_name_translation.csv')
  )
SELECT
  oi.data.product_id           AS product_id,
  p.data.product_category_name AS category_pt,
  t.data.product_category_name_english AS category_en,
  count()                                AS units_sold,
  round(sum(toFloat64(oi.data.price)))   AS revenue,
  round(avg(toFloat64(oi.data.price)))   AS avg_price
FROM order_items oi
LEFT JOIN products     p ON p.data.product_id = oi.data.product_id
LEFT JOIN translations t ON t.data.product_category_name = p.data.product_category_name
GROUP BY product_id, category_pt, category_en;
Frontend · Streamlit

app/streamlit_app.py — dashboard dos produtos com maior desempenho

A aplicação consulta a view v_top_products via HTTP, mantém cache de 60 segundos e renderiza filtros, tabela, gráfico de barras e breakdown por categoria, exibindo ainda o instante da última ingestão.

Estrutura da página
  • Sidebar: Top N (entre 5 e 100), métrica e filtro de categoria.
  • Cabeçalho: métricas-resumo, incluindo total de produtos e receita.
  • Tabela: top-N ordenado.
  • Gráfico de barras: receita ou unidades por produto.
  • Breakdown: agregação por categoria.
  • Rodapé: timestamp UTC da última ingestão.
Decisões técnicas
  • @st.cache_data(ttl=60) protege o ClickHouse de requisições redundantes.
  • Conexão estabelecida com o usuário dashboard em modo somente leitura.
  • Ausência de joins na aplicação, uma vez que a view já entrega o resultado consolidado.
  • Execução em modo headless: --server.headless=true para o Docker.
  • Porta 8501 exposta no host.

Discussão · Espectro

Critérios para a substituição do micro-batch por streaming propriamente dito

O regime de micro-batch atende à maioria absoluta dos dashboards analíticos. A adoção de streaming propriamente dito torna-se necessária quando a janela de decisão de negócio passa a ser menor que o intervalo do lote.

Indicadores que justificam a migração para streaming propriamente dito
  1. A decisão de negócio precisa ser executada em menos de cinco segundos (cenários de fraude, autorização ou matching).
  2. O volume de dados torna o processamento em lotes inviável, com backpressure constante e ingestor permanentemente em atraso.
  3. Múltiplos consumidores demandam os mesmos eventos em paralelo, situação para a qual um broker oferece resposta mais adequada.
  4. A equipe já dispõe da maturidade operacional necessária para operar Kafka/MSK ou Kinesis; sem ela, o custo operacional supera os ganhos.

Prática · Passos 1 a 3

Implantação da stack

1. Clonagem do repositório

  • Executar git clone do repositório da aula.
  • Aplicar cp .env.example .env; os valores-padrão são suficientes.
  • Confirmar a presença do docker-compose.yaml.

2. Inicialização dos containers

  • docker compose up -d --build
  • docker compose ps → todos os serviços em estado healthy.
  • Logs disponíveis em docker compose logs -f clickhouse.

3. Criação do bucket no MinIO

  • Acessar http://localhost:9003.
  • Login: minioadmin / minioadmin.
  • Em Buckets, selecionar Create e atribuir o nome olist.
Prática · Passos 4 a 6

Popular a landing e executar a ingestão

4. Carga dos CSVs

  • Localizar os CSVs na pasta data/ do repositório.
  • Arrastar os arquivos para o bucket olist no console.
  • Alternativamente, utilizar o cliente mc: mc cp data/*.csv local/olist/.

5. Aplicação do schema

  • docker exec -i olist_analytics clickhouse-client < sql/raw/01_schema.sql
  • Cria a database olist_raw e a tabela ingestion.
  • A reexecução é segura, dada a cláusula IF NOT EXISTS.

6. Execução da ingestão

  • docker exec olist_ingestor python /app/ingest_minio.py
  • O script percorre todos os CSVs presentes no bucket.
  • Cada CSV gera N inserts na tabela ingestion.
Prática · Passos 7 a 9

Validar a landing e criar a view

7. Verificação da landing

  • SELECT tag, count() FROM olist_raw.ingestion GROUP BY tag
  • Espera-se uma linha por CSV ingerido.
  • O total deve corresponder ao número de linhas do respectivo CSV.

8. Aplicação da view

  • docker exec -i olist_analytics clickhouse-client < sql/views/01_top_products.sql
  • Cria a view olist_raw.v_top_products.
  • Trata-se de metadado, de baixo custo e reexecutável sem prejuízo.

9. Validação da view

  • SELECT * FROM olist_raw.v_top_products LIMIT 10
  • Deve retornar produtos com as categorias em PT/EN e respectivas métricas.
  • Verificar a ordenação por revenue em ordem decrescente.
Prática · Passos 10 a 12

Dashboard, idempotência e encerramento

10. Acesso ao dashboard

  • Acessar http://localhost:8501.
  • Explorar os filtros disponíveis (Top N, métrica e categoria).
  • Verificar o timestamp da última ingestão exibido no rodapé.

11. Reingestão e observação da idempotência

  • Executar novamente o ingestor: docker exec olist_ingestor python /app/ingest_minio.py.
  • SELECT tag, count(DISTINCT unixtime) FROM olist_raw.ingestion GROUP BY tag; o valor é incrementado.
  • O dashboard exibe os mesmos números, pois a view filtra pelo max(unixtime).

12. Encerramento

  • docker compose down — os volumes são preservados.
  • docker compose down -v — os volumes são removidos, configurando reset completo.
  • Commit do trabalho no fork pessoal.

Motivação

Por que streaming importa para Olist

Decisões que dependem do evento recém-ocorrido não cabem em um job batch de 24h.

Casos de uso

  • Clickstream: cliente entra no produto
  • Carrinho abandonado em 10 minutos
  • Detecção de fraude em pagamento
  • Alerta de estoque e logística
  • Dashboards em tempo quase real

Valor de negócio

  • Agir antes do cliente sair do site
  • Recomendar agora, não amanhã
  • Operar com menos latência entre causa e resposta
  • Complementar o batch, não substituir

Teoria · Trade-offs

Latência vs throughput

Não existe sistema que maximize ambos ao mesmo tempo: você escolhe o ponto de operação.

Latência baixa
  • Processar o evento tão logo ele chega
  • Buffers pequenos, commits rápidos
  • Custo por evento mais alto
Throughput alto
  • Agregar eventos em lotes maiores
  • Menos overhead de rede e serialização
  • Custo por evento mais baixo
Metáfora

Dinâmica do avião aplicada a streaming

Lembra da esteira de bagagem? Ela volta como modelo mental de produtor, broker e consumidor.

Produtor · check-in

  • Aplicação web, app mobile, IoT
  • Gera o evento e entrega ao broker
  • Não espera o consumidor ler

Broker · esteira

  • Kinesis, Kafka, MSK
  • Mantém a ordem por partição
  • Guarda o evento por um tempo

Consumidor · balcão

  • Lambda, Firehose, Flink
  • Lê no próprio ritmo
  • Marca onde parou (checkpoint)

Separação de tempos

  • Produtor e consumidor não se conhecem
  • Broker absorve picos
  • Acoplamento fraco, escala independente

Teoria · Windowing

Tipos de janela

Streaming puro trata um evento por vez, mas agregações exigem agrupar eventos em janelas de tempo.

Tumbling

  • Janelas fixas e sem sobreposição
  • Ex.: vendas a cada 1 minuto
  • [0-60) [60-120) [120-180)

Sliding

  • Janelas com sobreposição
  • Ex.: média dos últimos 5 min a cada 1 min
  • Mais suave, mais cálculo

Session

  • Agrupa por inatividade
  • Fecha quando usuário fica ocioso
  • Ideal para navegação em site

Como escolher

  • Métricas regulares → tumbling
  • Alertas com suavização → sliding
  • Comportamento de usuário → session
Teoria · Tempo

Event time vs processing time

Event time
  • Quando o fato aconteceu
  • Está dentro do próprio evento
  • Invariante ao atraso de rede
  • Correto para relatórios de negócio
Processing time
  • Quando o sistema leu o evento
  • Depende da fila e do consumidor
  • Sensível a backlog e lentidão
  • Útil para SLAs operacionais
Watermark
Marca até que ponto no tempo o sistema confia que já viu todos os eventos. Janelas só fecham depois do watermark passar.
Teoria · Garantias

At-most-once, at-least-once, exactly-once

Garantia Pode perder? Pode duplicar? Custo
At-most-onceSimNãoBaixo
At-least-onceNãoSimMédio
Exactly-onceNãoNãoAlto (transacional)
Kinesis entrega at-least-once por padrão. Cabe ao consumidor ser idempotente.
Teoria · Robustez

Backpressure e idempotência

Backpressure

  • Consumidor mais lento que produtor
  • Fila cresce, latência cresce
  • Resposta: escalar shards ou consumidores
  • Métrica chave: IteratorAge

Idempotência

  • Processar duas vezes deve dar o mesmo resultado
  • Usar chave única (event_id)
  • Upsert em vez de insert cego
  • Essencial em at-least-once
AWS · Kinesis Data Streams

Kinesis Data Streams

Broker de eventos gerenciado pela AWS, com o conceito central de shard.

Capacidade por shard
  • Escrita: 1 MB/s ou 1000 registros/s
  • Leitura: 2 MB/s por shard
  • Escala aumentando shards
Retenção
  • Padrão: 24h
  • Máximo: até 365 dias
  • Permite reprocessar histórico recente
Kinesis · Internals

Shards e partition key

O shard é a unidade de paralelismo. A partition key determina em qual shard cada evento cai.

Regras práticas
  • Partition key com alta cardinalidade distribui melhor
  • Eventos com mesma key vão para o mesmo shard e preservam ordem
  • Hot shard: uma key concentra tráfego e satura o shard
  • Para Olist, customer_id costuma ser uma boa partition key
AWS · Firehose

Kinesis Data Firehose

Entrega gerenciada para S3, Redshift, OpenSearch e parceiros. Sem shards para gerenciar.

Recursos

  • Buffering por tempo ou tamanho
  • Conversão para Parquet / ORC
  • Compressão e criptografia
  • Transformação com Lambda

Quando usar

  • Aterrissar eventos no data lake
  • Near real-time, não real-time puro
  • Sem necessidade de replay fino
  • Custo menor que Streams puro
AWS · Lambda

Lambda como consumer de Kinesis

Event source mapping liga shards a funções Lambda automaticamente.

Como funciona
  • Lambda polla o shard em seu nome
  • Envia batches de até 10.000 registros
  • Commit de checkpoint após sucesso
  • Retry configurável, BisectBatchOnFunctionError, DLQ
  • Concorrência limitada pelo número de shards
Teoria · Serverless

Características serverless em streaming

Lambda e Firehose seguem o mesmo modelo mental: stateless, efêmero, event-driven e por consumo.

Stateless

  • Cada execução é independente
  • Estado vai para S3, DynamoDB ou ElastiCache
  • Fácil de escalar horizontalmente

Efêmero

  • O container vive minutos
  • Arquivos temporários em /tmp somem
  • Não confie em cache local

Event-driven

  • Dispara em resposta a eventos
  • Kinesis, S3, SQS, EventBridge
  • Integra naturalmente com broker

Cobrança por uso

  • Lambda: por GB-s e invocação
  • Firehose: por GB processado
  • Sem cobrança quando ocioso
Comparação

Kinesis vs Kafka / MSK

Mesmo modelo mental, ergonomias diferentes. No Learner Lab usamos Kinesis (MSK geralmente bloqueado).

Padrões de arquitetura

Lambda architecture e Kappa architecture

Lambda architecture
  • Duas trilhas: batch e streaming (speed)
  • Batch recalcula tudo periodicamente
  • Streaming complementa o último intervalo
  • Robusto, porém duplica código
Kappa architecture
  • Uma única trilha: streaming
  • Reprocessar é reler o log
  • Requer broker com retenção longa
  • Mais simples, exige disciplina

Lab · Passo 1

Criar o Kinesis Data Stream

Console
  1. Abra Kinesis na região us-east-1.
  2. Clique em Create data stream.
  3. Nome: olist-eventos.
  4. Capacity mode: Provisioned, 1 shard.
  5. Confirme e aguarde status Active.
Alternativa — AWS CLI 
aws kinesis create-stream \
  --stream-name olist-eventos \
  --shard-count 1 \
  --region us-east-1

Lab · Passo 3

Produtor Python

producer.py (trecho) 
import boto3, csv, json, random, time, uuid
from datetime import datetime, timezone

STREAM = "olist-eventos"
ACTIONS = ["view", "add_to_cart", "checkout", "purchase"]
client = boto3.client("kinesis", region_name="us-east-1")

with open("olist_order_items_dataset.csv") as f:
    rows = list(csv.DictReader(f))

while True:
    row = random.choice(rows)
    event = {
      "event_id": str(uuid.uuid4()),
      "event_time": datetime.now(timezone.utc).isoformat(),
      "customer_id": f"cust-{random.randint(1,500)}",
      "product_id": row["product_id"],
      "action": random.choice(ACTIONS),
      "price": float(row["price"]),
    }
    client.put_record(
      StreamName=STREAM,
      Data=json.dumps(event).encode("utf-8"),
      PartitionKey=event["customer_id"],
    )
    time.sleep(0.2)
Lab · Passo 4

Firehose com destino S3

Configuração
  1. Source: Amazon Kinesis Data Streams → olist-eventos.
  2. Destination: Amazon S3, bucket <aluno>-raw, prefix streaming/olist-eventos/.
  3. Buffer: 1 MB ou 60 s (o que vier primeiro).
  4. Formato: Parquet (exige tabela no Glue Catalog).
  5. IAM role: LabRole (Learner Lab).
Nome sugerido: olist-eventos-to-s3. Revise permissões do LabRole antes de criar.
Lab · Passo 5

Glue Catalog e Athena

Passo a passo
  1. No Glue, crie o database olist_streaming.
  2. Crie um Crawler apontando para s3://<aluno>-raw/streaming/olist-eventos/.
  3. Execute o Crawler e verifique a tabela criada.
  4. No Athena, selecione o database e rode a query de validação.
Query de validação 
SELECT action, COUNT(*) AS eventos
FROM olist_streaming.olist_eventos
GROUP BY action
ORDER BY eventos DESC;
Lab · Passo 6

Lambda consumer → DynamoDB

Preparativos
  1. Crie a tabela DynamoDB olist_eventos_agg, chave action (String).
  2. Crie a função Lambda olist-stream-consumer, runtime Python 3.x, role LabRole.
  3. Configure um Kinesis trigger apontando para olist-eventos, batch size 100.
lambda_function.py 
import base64, json, boto3
ddb = boto3.resource("dynamodb").Table("olist_eventos_agg")

def lambda_handler(event, context):
    for r in event["Records"]:
        data = json.loads(base64.b64decode(r["kinesis"]["data"]))
        ddb.update_item(
          Key={"action": data["action"]},
          UpdateExpression="ADD contagem :one",
          ExpressionAttributeValues={":one": 1},
        )
    return {"processed": len(event["Records"])}
Lab · Passo 7

Observabilidade com CloudWatch

Métricas do stream

  • IncomingRecords — volume entrando
  • IncomingBytes — vazão em bytes
  • IteratorAge — atraso do consumer

O que observar

  • IncomingRecords subindo durante o produtor
  • IteratorAge próximo de zero indica consumer saudável
  • Firehose DeliveryToS3.Success
  • Logs da Lambda em CloudWatch Logs

Lab · Custos

Limpeza obrigatória

Kinesis cobra por shard-hora. Esquecer o stream ligado consome crédito do AWS Student rapidamente.

Ordem de desligamento
  1. Parar o script produtor (Ctrl+C).
  2. Desabilitar o trigger da Lambda (ou deletar a Lambda).
  3. Deletar o Firehose delivery stream.
  4. Deletar o Kinesis Data Stream olist-eventos.
  5. (Opcional) Deletar a tabela DynamoDB de agregados.
  6. Manter os buckets S3 e o catalog para próximas aulas.

Ingestão e Processamento Near Real-time com Streaming

Este material aprofunda os conceitos apresentados nos slides da Aula 04. Cobre a arquitetura de ingestão contínua de eventos na AWS com Amazon Kinesis Data Streams e Kinesis Data Firehose, os trade-offs de latência e throughput, a entrega automatizada para o S3 com particionamento por tempo, e uma visão geral do Amazon EMR para processamento distribuído de grande escala.

Módulo 1 — Streaming versus Batch: quando cada um se aplica

O que é processamento em batch?

No modelo batch, os dados são acumulados por um período (hora, dia, semana) e processados em bloco ao final desse período. É o modelo mais simples e amplamente utilizado: jobs noturnos que transformam os dados do dia anterior, relatórios gerados semanalmente, snapshots diários de tabelas operacionais. O custo de infraestrutura tende a ser menor porque os recursos de computação são usados intensamente por períodos curtos e ficam ociosos o restante do tempo.

O batch tem uma limitação inerente: a latência mínima é igual ao tamanho da janela de acumulação. Se o job roda a cada hora, os dados têm no mínimo zero e no máximo 60 minutos de atraso. Para casos de uso que exigem reagir a eventos em segundos ou minutos, o batch não é adequado.

O que é streaming near real-time?

No modelo streaming, os dados são processados à medida que chegam, com latência que pode variar de milissegundos (true real-time) a alguns minutos (near real-time). Casos de uso típicos incluem: detecção de fraude em transações de cartão de crédito, monitoramento de métricas de aplicação, personalização de experiência do usuário em tempo real, e alertas operacionais baseados em séries temporais de logs.

O streaming exige infraestrutura persistente — os consumidores precisam estar rodando continuamente —, o que eleva o custo operacional em comparação com o batch. A escolha entre os dois modelos deve ser guiada pelo SLA de latência do caso de uso: se o negócio pode esperar algumas horas, use batch; se precisa reagir em minutos, use streaming.

Arquitetura Lambda e Kappa

A Arquitetura Lambda combina um caminho batch (para dados históricos e correção de erros) com um caminho de streaming (para dados recentes com baixa latência). Os consumidores finais consultam ambos os resultados e combinam as visões. É poderosa mas complexa de manter — dois pipelines, duas bases de código, dois conjuntos de testes.

A Arquitetura Kappa simplifica esse modelo: usa apenas o caminho de streaming, reprocessando dados históricos ao reiniciar o consumer desde o início do stream quando necessário. Requer que o sistema de streaming suporte retenção longa de dados (ex.: Kinesis com retenção de 7 dias, ou Kafka com retenção indefinida). É a abordagem preferida quando os dados chegam principalmente via eventos em tempo real.

Módulo 2 — Amazon Kinesis Data Streams

Conceitos fundamentais

O Kinesis Data Streams é um serviço de streaming gerenciado para ingestão de dados em tempo real. Os dados são organizados em shards: cada shard suporta até 1 MB/s ou 1.000 registros/s de escrita, e até 2 MB/s de leitura. O throughput total do stream é a soma dos shards.

Os dados ficam retidos no stream por até 365 dias (padrão: 24 horas), e múltiplos consumidores podem ler o mesmo stream independentemente, cada um mantendo sua própria posição de leitura (sequence number). Isso é fundamentalmente diferente de filas (como SQS), onde uma mensagem consumida desaparece.

A partition key determina em qual shard um registro vai parar: o Kinesis aplica um hash MD5 sobre a partition key e mapeia o resultado para um shard. Escolha uma partition key com alta cardinalidade para distribuir uniformemente os registros entre shards e evitar hot shards.

Produtores e consumidores

Produtores são qualquer sistema que envia dados ao stream. Exemplos: aplicações web enviando eventos de clique com o SDK da AWS, logs de aplicação via Kinesis Agent, dispositivos IoT via MQTT bridge, e outros serviços AWS como CloudWatch Logs.

Consumidores são os sistemas que leem do stream. Exemplos: funções AWS Lambda (trigger nativo do Kinesis), aplicações rodando no EC2 com a Kinesis Client Library (KCL), Apache Flink via Amazon Managed Service for Apache Flink, e o Kinesis Data Firehose.

Modos de capacidade

O Kinesis Data Streams opera em dois modos:

  • Provisioned: você define o número de shards manualmente. Custo por shard-hora. Use quando o volume de dados é previsível.
  • On-Demand: o Kinesis escala automaticamente com base no throughput observado. Custo por GB ingerido e lido. Use para cargas imprevisíveis ou durante fases de crescimento rápido.

Módulo 3 — Amazon Kinesis Data Firehose

Firehose versus Data Streams: qual a diferença?

O Kinesis Data Firehose é um serviço de entrega gerenciado: ele recebe dados (diretamente de produtores ou de um Kinesis Data Stream) e os entrega para um destino como S3, Redshift, OpenSearch ou um endpoint HTTP, sem que você precise gerenciar consumidores ou código de entrega.

A principal característica do Firehose é o buffering: ele acumula dados por um intervalo de tempo (1 a 900 segundos) ou até atingir um tamanho (1 MB a 128 MB) antes de escrever no destino. Isso transforma um fluxo contínuo de pequenos eventos em arquivos razoavelmente grandes no S3, evitando o problema de arquivos pequenos.

O Firehose não retém dados — após a entrega, os dados não ficam disponíveis para releitura como no Data Streams. Se você precisa que múltiplos consumidores independentes processem o mesmo evento, use Data Streams como fonte primária e conecte o Firehose como um dos consumidores.

Entrega no S3 com particionamento dinâmico

O Firehose suporta particionamento dinâmico: você define um prefixo usando expressões com campos extraídos dos registros. Por exemplo: raw/evento=!{partitionKeyFromQuery:tipo}/ano=!{timestamp:yyyy}/mes=!{timestamp:MM}/dia=!{timestamp:dd}/

Isso cria automaticamente a hierarquia de partições no S3 baseada no conteúdo dos eventos e no timestamp de entrega, compatível com o formato Hive que o Glue Catalog reconhece.

Transformação inline com Lambda

O Firehose permite associar uma função Lambda para transformar os registros antes da entrega. A Lambda recebe batches de registros em base64, processa cada um (filtra, converte, enriquece) e retorna os registros transformados. Casos de uso comuns: remover campos sensíveis (PII masking), converter JSON para Parquet via schema registry, e filtrar eventos irrelevantes para não armazenar no S3.

Módulo 4 — Amazon EMR: Processamento Distribuído de Grande Escala

O que é o EMR e quando usá-lo

O Amazon Elastic MapReduce (EMR) é um serviço gerenciado de cluster para executar frameworks de processamento distribuído como Apache Spark, Apache Hadoop, Apache Hive, Presto e outros. Ele provisiona e configura o cluster automaticamente, incluindo instalação dos frameworks, configuração de rede e integração com S3.

O EMR é a escolha certa quando você precisa de:

  • Processamento de volumes muito grandes (terabytes a petabytes) que excedem a capacidade de um único servidor.
  • Jobs Spark complexos com múltiplas transformações, shuffles intensivos e uso de ML libraries (MLlib, PyTorch distribuído).
  • Controle fino sobre a configuração do Spark (memória, paralelismo, serialização) que não é possível no AWS Glue.
  • Frameworks específicos não suportados pelo Glue (ex.: Presto para queries SQL interativas em escala, Apache Hudi para tabelas transacionais no lake).

Para pipelines ETL de complexidade média, o AWS Glue (serverless) é preferível ao EMR por não exigir gerenciamento de cluster. O EMR compensa quando a flexibilidade e o poder de computação superam a complexidade operacional.

EMR Serverless: o melhor dos dois mundos

O EMR Serverless elimina o gerenciamento de cluster: você define uma aplicação (Spark ou Hive), submete jobs, e a AWS provisiona e desprovisiona os workers automaticamente. Você paga pelo vCPU, memória e storage usados durante a execução do job. É a evolução natural para equipes que precisam da flexibilidade do Spark sem a complexidade operacional de clusters EMR tradicionais.

Módulo 5 — Laboratório: Configurando um Fluxo de Streaming com Kinesis Firehose

A prática utiliza o ambiente AWS Student. O objetivo é criar um delivery stream do Kinesis Firehose que entregue dados JSON para o S3 com particionamento por data.

Passo 1 — Criar um Kinesis Data Firehose Delivery Stream

  1. No console AWS, acesse Amazon Kinesis e clique em Kinesis Data Firehose.
  2. Clique em Create Firehose stream.
  3. Em Source, selecione Direct PUT (dados enviados diretamente para o Firehose, sem um Data Stream intermediário).
  4. Em Destination, selecione Amazon S3.
  5. Em Firehose stream name, insira: datalake-eventos-firehose.
  6. Em S3 bucket, selecione o bucket criado nas aulas anteriores.
  7. Em S3 bucket prefix, insira: raw/fonte=firehose/ano=!{timestamp:yyyy}/mes=!{timestamp:MM}/dia=!{timestamp:dd}/
  8. Em S3 bucket error output prefix, insira: raw/firehose-errors/ano=!{timestamp:yyyy}/mes=!{timestamp:MM}/
  9. Em Buffer hints, configure:
    • Buffer size: 1 MB (mínimo para o lab)
    • Buffer interval: 60 seconds
  10. Clique em Create Firehose stream.
  11. Validação: o delivery stream aparece com status Active.

Passo 2 — Enviar eventos de teste via AWS CLI ou console

  1. No console do Firehose, clique no stream criado.
  2. Clique em Test with demo data para enviar registros de exemplo.
  3. Aguarde o intervalo de buffer (60 segundos) expirar.
  4. Acesse o S3 e navegue até o prefixo raw/fonte=firehose/ para confirmar que os arquivos foram criados com a estrutura de particionamento por data.
  5. Validação: arquivos JSON aparecem no S3 nos prefixos corretos com o particionamento ano/mes/dia.

Passo 3 — Criar tabela no Athena para os dados do Firehose

  1. Acesse o Amazon Athena.
  2. Execute o DDL para criar a tabela sobre os dados entregues pelo Firehose:
CREATE EXTERNAL TABLE datalake_lab.raw_eventos_firehose (
  aproximateArrivalTimestamp BIGINT,
  kinesisRecordMetadata STRUCT<shardId: STRING>,
  data STRING
)
PARTITIONED BY (ano STRING, mes STRING, dia STRING)
ROW FORMAT SERDE 'org.openx.data.jsonserde.JsonSerDe'
LOCATION 's3://seu-bucket/raw/fonte=firehose/';
  1. Execute MSCK REPAIR TABLE datalake_lab.raw_eventos_firehose; para registrar as partições existentes.
  2. Execute uma query de validação: SELECT * FROM datalake_lab.raw_eventos_firehose LIMIT 5;
  3. Validação: os registros enviados pelo Firehose aparecem no resultado.

Exercícios de fixação

  1. Data Streams versus Firehose: uma fintech precisa processar transações de cartão de crédito para detectar fraudes em tempo real (latência de 2-3 segundos) E armazenar todas as transações no Data Lake para análise histórica (latência de 5 minutos é aceitável). Descreva a arquitetura usando Kinesis Data Streams e Firehose, explicando o papel de cada serviço e por que você não poderia usar apenas um deles neste cenário.
  2. Dimensionamento de shards: um sistema de monitoramento de aplicação gera 2.500 eventos por segundo, cada evento com média de 500 bytes. Calcule o número mínimo de shards necessários no Kinesis Data Streams justificando os cálculos com os limites por shard. Considere também uma margem de segurança de 20%.
  3. Particionamento no Firehose: você precisa entregar eventos de um aplicativo mobile (com campos platform: iOS/Android, event_type: click/purchase/view, e timestamp) para o S3 com particionamento que facilite queries no Athena por plataforma e data. Escreva o prefixo S3 com as expressões de particionamento dinâmico corretas para o Firehose.
  4. EMR versus Glue: um pipeline precisa processar 2 TB de logs de servidor por dia, aplicando transformações complexas com múltiplos joins e agregações, convertendo para Parquet particionado. Liste os critérios que fariam você escolher EMR em vez de AWS Glue para este caso, e vice-versa. Qual seria sua recomendação para este cenário específico?
  5. Arquitetura Lambda x Kappa: descreva com suas palavras a diferença entre as arquiteturas Lambda e Kappa para pipelines de dados. Identifique um caso de uso onde Lambda seria necessariamente superior a Kappa, e outro onde Kappa seria suficiente e mais simples. Justifique com base nos requisitos de latência, complexidade operacional e reprocessamento histórico.

Ingestão e Processamento Near Real-time com Streaming

Este material acompanha a Aula 04 da disciplina Cloud Computing e SRE — Visão Prática com AWS do MBA em Engenharia de Dados. A aula transita do plano conceitual para a implementação prática, mediante a implantação de uma stack completa de ingestão near real-time em ambiente local, com MinIO assumindo o papel de S3, ClickHouse como data warehouse colunar, um ingestor Python executado em Docker e um dashboard Streamlit responsável pelo consumo das métricas. O conjunto de dados utilizado é o já consolidado Olist Brazilian E-commerce Public Dataset.

Pré-requisitos Docker Desktop (ou Docker Engine acompanhado de Compose v2), Git, navegador web e aproximadamente 2 GB de RAM disponíveis. As portas 8123, 8501, 9000, 9002 e 9003 precisam estar livres no host.

Restrições da aula Não há operações em consoles AWS, tampouco utilização de Kafka, Flink ou Kinesis; nenhuma operação manual é realizada além do que está declarado no docker-compose.yaml; toda a pipeline é idempotente, de modo que as reingestões sejam seguras.

Repositório Todo o código executado em sala está disponível no GitHub. Recomenda-se realizar fork previamente a qualquer modificação.

1. Justificativa para a execução local previamente à nuvem

A primeira tentação ao se iniciar o estudo de uma plataforma de dados é acessar o console do provedor e operá-lo de modo exploratório. Essa rota apresenta três custos relevantes: cada erro consome tempo de recurso faturado, cada interação no console não deixa rastro reproduzível e cada participante torna-se dependente de um session token instável para sequer iniciar a atividade. A proposta desta aula inverte a ordem: implanta-se a stack completa em ambiente local, com as mesmas interfaces oferecidas pela nuvem, valida-se o pipeline de ponta a ponta e, somente em seguida, discute-se de que forma essa pipeline poderia ser migrada para S3, Redshift ou Kinesis.

A engenharia reside nas interfaces. O MinIO implementa a API S3 já suportada pelo boto3 e pelo AWS CLI. O ClickHouse é o mesmo binário no laptop, no servidor on-premise e no ClickHouse Cloud. O Streamlit executa de modo equivalente em qualquer plataforma. O contrato é a interface; o provedor constitui detalhe de implementação. Ao se utilizar a API S3 e a linguagem SQL, não se aprende propriamente "MinIO", e sim S3 e ClickHouse, com latência inferior e custo nulo.

Tese Recomenda-se executar o ambiente localmente previamente à implantação em nuvem. A interface, e não o fornecedor, constitui o contrato técnico relevante. Um pipeline que se comunica por meio da API S3 e da linguagem SQL opera tanto sobre MinIO e ClickHouse em ambiente local quanto sobre S3 e Redshift em ambiente de nuvem. A engenharia reside nas interfaces; a fatura, no provedor.

1.1 Quatro regras desta aula

  1. A stack completa é orquestrada por meio de docker compose up -d, sem operações em consoles de provedores de nuvem ao longo desta aula.
  2. A ingestão segue o paradigma append-only: cada execução registra unixtime e tag próprios e não sobrescreve dados preexistentes.
  3. A modelagem adota o regime schema-on-read: a landing é mantida como JSON bruto, e a modelagem é concentrada nas views.
  4. Healthchecks são obrigatórios em todo serviço crítico, sustentando a diretiva depends_on: condition: service_healthy que orquestra o startup.

2. Espectro batch, micro-batch e streaming

A expressão "near real-time" não designa um ponto fixo, mas uma faixa do espectro situada entre o lote diário tradicional e o evento processado em milissegundos. A engenharia de dados contemporânea seleciona o regime adequado para cada caso de uso. Adotar streaming propriamente dito em cenários nos quais o micro-batch é suficiente eleva o custo desnecessariamente; manter batch quando a janela de decisão é curta resulta em perda de oportunidades operacionais.

ModoLatência típicaMecânicaQuando aplicar
Batch Horas a dia Janela longa, com processamento integral do conjunto em uma única execução, disparada em horário fixo. Relatório diário, fechamento contábil, BI clássico.
Micro-batch Segundos a minutos Lotes pequenos e frequentes, executados em loop curto de leitura, transformação e escrita. Dashboards operacionais, agregações analíticas leves, análises operacionais.
Streaming propriamente dito Sub-segundo Processamento evento a evento, viabilizado por broker (Kafka, Kinesis) e processador (Flink, Spark Structured Streaming). Detecção de fraude, telemetria crítica, matching em tempo real.

Nesta aula adota-se o regime de micro-batch por meio de ingest_minio.py, script responsável por ler todos os CSVs presentes no bucket, transformá-los e escrevê-los no ClickHouse em lotes de 50 mil linhas. A execução pode ser acionada manualmente, por cron, por evento de upload no MinIO ou em loop contínuo. A utilização de Kafka e Flink é deliberadamente evitada, dado que o custo operacional de um broker associado a um processador stateful não se justifica frente ao volume e à janela de decisão de um dashboard analítico.

3. Docker Compose como contrato declarativo da stack

O Docker Compose consiste em um arquivo YAML que descreve a totalidade do sistema: cada serviço (container), suas portas, volumes, variáveis de ambiente, healthchecks e dependências. Trata-se de um único artefato versionado que substitui dezenas de comandos manuais de docker run. Para ambientes locais e de desenvolvimento, constitui o equivalente prático de um Kubernetes em escala reduzida, sem a sobrecarga conceitual associada.

3.1 Conceitos centrais

  • services: lista de containers do sistema, cada qual associado a uma imagem ou build próprio.
  • volumes: persistência externa ao ciclo de vida do container, garantindo que o estado sobreviva às operações de down e up.
  • ports: mapeamento host:container, responsável pela exposição de portas para acesso a partir do host.
  • environment: variáveis injetadas no processo do container.
  • healthcheck: comando que reporta a saúde do serviço, configurando o SLO local.
  • depends_on com condition: service_healthy: bloqueia o startup do dependente até a aprovação do healthcheck.

Os healthchecks merecem atenção particular. Em sua ausência, o Compose marca um serviço como "up" tão logo o processo seja iniciado, ainda que não esteja efetivamente respondendo. Com healthcheck, o status só evolui para healthy mediante confirmação do comando externo. Para o ClickHouse utiliza-se wget --spider http://localhost:8123/ping; para o MinIO, curl /minio/health/live. Como consequência, o ingestor só é inicializado quando seus dois backends estão genuinamente prontos, o que elimina uma classe inteira de race conditions de startup.

5. MinIO 101 — API S3 em ambiente local

O MinIO é um servidor de objetos de código aberto que implementa a API S3 da Amazon. A expressão "implementa" deve ser interpretada literalmente: os mesmos verbos REST, os mesmos cabeçalhos de assinatura V4 e os mesmos formatos de resposta XML. Como consequência, qualquer cliente que se comunica com S3 — boto3, AWS CLI, Spark, Trino ou DuckDB — opera com o MinIO mediante alteração restrita ao endpoint URL.

5.1 Conceitos relevantes da API S3

  • Bucket: container nomeado e único em todo o servidor. No MinIO corresponde a um diretório no disco.
  • Objeto: arquivo binário acompanhado de metadados (Content-Type, ETag, timestamps), identificado por key no escopo do bucket.
  • Endpoint URL: URL base do servidor. Para o AWS S3, o valor padrão é https://s3.amazonaws.com; para a instância MinIO desta aula é http://minio:9000, a partir da rede do Compose, ou http://localhost:9002, a partir do host.
  • Path-style versus virtual-hosted: no S3 moderno o bucket é convertido em subdomínio (bucket.s3.amazonaws.com); no MinIO local utiliza-se o padrão path-style (localhost:9000/olist/key). O boto3 detecta o regime de forma automática.
  • Credenciais: par access_key e secret_key. No MinIO local, o valor padrão é minioadmin / minioadmin, restrito ao contexto de laboratório.

6. ClickHouse 101 — banco colunar OLAP

O ClickHouse é um banco de dados analítico colunar, vetorizado e de código aberto. A natureza colunar implica que dados de uma mesma coluna são armazenados de forma contígua em disco, de modo que queries que envolvem poucas colunas realizem leituras reduzidas. A natureza vetorizada significa que os operadores processam vetores de valores em vez de linhas isoladas, ampliando substancialmente a vazão. Trata-se de uma classe de banco em que uma consulta do tipo SELECT count(*) FROM x, sobre bilhões de linhas, é respondida em segundos.

6.1 MergeTree, a engine padrão

As tabelas no ClickHouse são associadas a uma engine; a padrão da família OLAP é a MergeTree. Suas características essenciais são as seguintes:

  • ORDER BY define a chave de ordenação. Não se trata de uma chave primária no sentido SQL clássico, mas da ordenação física dos dados em disco e do índice primário esparso.
  • Os inserts geram novos "parts" no diretório da tabela. Em background, o ClickHouse executa merges entre os parts, origem do nome da engine.
  • A primeira coluna do ORDER BY tende a apresentar elevado poder de poda, de tal modo que queries que filtram por essa coluna leem apenas os parts relevantes.

6.2 Tipo JSON

A coluna JSON constitui um tipo semi-estruturado nativo do ClickHouse moderno (a partir da versão 24.x). Permite armazenar objetos JSON arbitrários sem definição antecipada de schema; a engine indexa subcaminhos de forma automática e expõe o acesso por meio de notação ponto, como em data.product_id. Esse recurso viabiliza, na prática, a adoção do padrão schema-on-read no ClickHouse.

6.3 LowCardinality

Trata-se de um wrapper de tipo destinado a colunas com poucos valores distintos. Internamente, armazena um dicionário (valor → código) e utiliza o código nas linhas. A compressão obtida é substancialmente superior, e operadores como GROUP BY e filtros tornam-se mais rápidos. Casos típicos incluem tag, status, region e category.

7. Schema-on-read versus schema-on-write

O regime schema-on-write é típico dos bancos OLTP: a estrutura é definida no DDL, e qualquer inserção incompatível com a tipagem é rejeitada na ingestão. Funciona adequadamente quando o schema é estável e quando poucos sistemas escrevem, como ocorre em aplicações web típicas. Apresenta dificuldades quando as fontes evoluem com frequência, dado que cada alteração demanda migração coordenada.

O regime schema-on-read inverte essa lógica: os dados ingressam no formato bruto em que se apresentam (JSON, NDJSON, Parquet semi-estruturado), e o schema é aplicado no momento da query, em geral por meio de views. O ganho é a absorção de mudanças nas fontes sem migrações; o custo associado é a transferência da validação para o momento da leitura, o que pode introduzir atrasos na geração de relatórios.

AspectoSchema-on-writeSchema-on-read
Momento de validação do tipoNa ingestão.Na query.
Custo de alteração do schemaAlto, exigindo ALTER e migração.Baixo, restrito à modificação da view.
Local de falha inicialPrecoce, com rejeição de dados na ingestão.Tardia, manifestando-se no momento da consulta.

Opção adotada nesta aula—Landing como JSON, com modelagem concentrada nas views.

A escolha entre as duas estratégias apresenta implicações organizacionais. O regime schema-on-read transfere a responsabilidade para a equipe de modelagem analítica, na qual o tipo "produto" emerge a partir de centenas de campos JSON. O regime schema-on-write, por sua vez, concentra a responsabilidade na equipe de produção, em que cada alteração de fonte precisa ser convertida em pull request sobre o DDL. Para um pipeline exploratório como o desta aula, o regime schema-on-read mostra-se substancialmente mais adequado.

8. Tabela landing única para o conjunto de fontes

Em vez de criar uma tabela por CSV do Olist, opta-se por uma única tabela landing genérica. Toda ingestão escreve nela, e a diferenciação entre arquivos é registrada na coluna tag, que armazena o nome do CSV de origem. 

CREATE DATABASE IF NOT EXISTS olist_raw;

CREATE TABLE IF NOT EXISTS olist_raw.ingestion
(
  unixtime Int64,
  data     JSON,
  tag      LowCardinality(String)
)
ENGINE = MergeTree
ORDER BY (tag, unixtime);

Análise sucinta das três colunas:

  • unixtime — valor Unix epoch (em segundos) registrado no momento de início da execução do ingestor. É único por run: todas as linhas inseridas em uma execução compartilham o mesmo valor.
  • data — registro original em formato JSON. Cada linha do CSV é convertida em um objeto cujas chaves correspondem às colunas.
  • tag — nome do CSV de origem, que cumpre função de discriminador. A escolha por LowCardinality deve-se ao fato de existirem, no máximo, cerca de uma dezena de tags possíveis.

A definição de ORDER BY (tag, unixtime) é deliberada: as queries quase sempre filtram primeiramente por tag (correspondente a uma "tabela lógica" por vez) e, em seguida, por unixtime (referente à execução mais recente). Essa ordem maximiza a poda de parts no MergeTree.

9. Idempotência sem recurso a MERGE ou UPSERT

A idempotência é a propriedade de uma operação cujo resultado, após N execuções (com N ≥ 1), permanece equivalente ao de uma execução única. Em pipelines de dados, é precisamente essa propriedade que viabiliza retries seguros, recuperação de falhas parciais e backfills sem corrupção de estado.

Bancos OLTP resolvem a idempotência por meio de MERGE ou UPSERT: a operação verifica a chave primária e decide entre inserir ou atualizar. O ClickHouse, otimizado para inserts em massa, não dispõe de MERGE com baixo custo, e o UPSERT via ReplacingMergeTree consolida os dados apenas em background. O pipeline desta aula contorna essa limitação por meio de um padrão mais simples:

Padrão adotado Append-only na tabela ingestion, com unixtime único por execução. As views aplicam o filtro WHERE unixtime = (SELECT max(unixtime) FROM ingestion WHERE tag = '<arquivo>'). A reingestão adiciona linhas com novo unixtime, fazendo com que a view passe a expor apenas a execução mais recente. O histórico permanece registrado sem custo adicional, configurando trilha de linhagem.

9.1 Benefícios decorrentes do padrão

  • Reingestões integralmente seguras: jamais corrompem o estado e, no máximo, ampliam o histórico.
  • Histórico preservado: cada execução constitui uma fotografia da fonte naquele instante.
  • Reversão para versão anterior: equivalente a filtrar outro unixtime na view, em operação puramente lógica.
  • Ausência de locks e de transações distribuídas: o ClickHouse processa em paralelo, sem necessidade de orquestração.

9.2 Custos associados

O consumo de disco apresenta crescimento monotônico. Em pipelines de produção, esse comportamento exige política de retenção, como execução periódica de OPTIMIZE TABLE ... FINAL ou definição de TTL na MergeTree (TTL toDateTime(unixtime) + INTERVAL 30 DAY). No contexto desta aula, o reset é realizado por meio de docker compose down -v.

10. O ingestor — app/ingest_minio.py

O ingestor consiste em um script Python único, com aproximadamente 120 linhas, que executa todo o fluxo: lista o bucket, lê cada CSV com pandas, converte os dados em registros e os insere em lotes no ClickHouse. Não emprega Kafka, Airflow ou Spark; configura-se como um exemplo introdutório consistente de pipeline near real-time, compacto e didaticamente acessível. 

import os, time, json, boto3, pandas as pd, clickhouse_connect

UNIXTIME = int(time.time())  # marca unica desta execucao

BUCKET = os.getenv("S3_BUCKET_NAME", "olist")

s3 = boto3.client(
    "s3",
    endpoint_url=os.getenv("S3_ENDPOINT_URL"),
    aws_access_key_id=os.getenv("AWS_ACCESS_KEY_ID"),
    aws_secret_access_key=os.getenv("AWS_SECRET_ACCESS_KEY"),
)

ch = clickhouse_connect.get_client(
    host=os.getenv("CLICKHOUSE_HOST"),
    port=int(os.getenv("CLICKHOUSE_PORT", "8123")),
    username=os.getenv("CLICKHOUSE_USER", "default"),
    password=os.getenv("CLICKHOUSE_PASSWORD", ""),
)

resp = s3.list_objects_v2(Bucket=BUCKET)
for obj in resp.get("Contents", []):
    key = obj["Key"]
    if not key.endswith(".csv"):
        continue

    body = s3.get_object(Bucket=BUCKET, Key=key)["Body"]
    df = pd.read_csv(body, dtype=str).where(pd.notnull, None)

    rows = [
        (UNIXTIME, json.dumps(row.to_dict()), key)
        for _, row in df.iterrows()
    ]

    BATCH = 50_000
    for i in range(0, len(rows), BATCH):
        ch.insert(
            "olist_raw.ingestion",
            rows[i : i + BATCH],
            column_names=["unixtime", "data", "tag"],
        )
    print(f"[ingest] {key}: {len(rows)} linhas em unixtime={UNIXTIME}")

10.1 Decisões de design

  • dtype=str no pandas: a inferência de tipo é deliberadamente postergada para a view; o CSV é tratado como texto. Valores NaN são convertidos em None para serialização como null no JSON.
  • Batch de 50 mil linhas: equilíbrio entre vazão (uma vez que o ClickHouse favorece inserts de grande volume) e consumo de memória (lotes maiores pressionam o processo Python no laptop).
  • UNIXTIME definido fora do loop: o valor é calculado uma única vez no início e reutilizado, garantindo que todo o run compartilhe a mesma marca temporal.
  • Ausência de retry sofisticado: na ocorrência de falha em meio à execução, basta reexecutar o script; o paradigma append-only torna a operação segura.

11. A view v_top_products

É na view que o schema-on-read se materializa. Selecionam-se três CSVs (order_items, products e product_category_name_translation), consideram-se apenas as execuções mais recentes de cada um deles, executam-se os joins e procede-se à agregação por produto. 

CREATE OR REPLACE VIEW olist_raw.v_top_products AS
WITH
  order_items AS (
    SELECT data FROM olist_raw.ingestion
    WHERE tag = 'olist_order_items_dataset.csv'
      AND unixtime = (
        SELECT max(unixtime) FROM olist_raw.ingestion
        WHERE tag = 'olist_order_items_dataset.csv'
      )
  ),
  products AS (
    SELECT data FROM olist_raw.ingestion
    WHERE tag = 'olist_products_dataset.csv'
      AND unixtime = (
        SELECT max(unixtime) FROM olist_raw.ingestion
        WHERE tag = 'olist_products_dataset.csv'
      )
  ),
  translations AS (
    SELECT data FROM olist_raw.ingestion
    WHERE tag = 'product_category_name_translation.csv'
      AND unixtime = (
        SELECT max(unixtime) FROM olist_raw.ingestion
        WHERE tag = 'product_category_name_translation.csv'
      )
  )
SELECT
  oi.data.product_id                          AS product_id,
  p.data.product_category_name                AS category_pt,
  t.data.product_category_name_english        AS category_en,
  count()                                     AS units_sold,
  round(sum(toFloat64(oi.data.price)))        AS revenue,
  round(avg(toFloat64(oi.data.price)))        AS avg_price
FROM order_items oi
LEFT JOIN products     p ON p.data.product_id            = oi.data.product_id
LEFT JOIN translations t ON t.data.product_category_name = p.data.product_category_name
GROUP BY product_id, category_pt, category_en;

Observações relevantes sobre essa view:

  • Cada CTE aplica filtro pelo max(unixtime) da respectiva tag, o que torna a view idempotente em relação a reingestões. Mesmo que o ingestor seja executado dez vezes, a view continua a retornar os mesmos resultados.
  • A view é constituída por metadado puro, sem materialização em disco. A reexecução da query realiza os CTEs integralmente. Para volumes elevados, a evolução natural seria a adoção de uma MATERIALIZED VIEW.
  • O acesso a campos JSON é realizado mediante notação ponto, como em data.product_id; a conversão é efetuada de forma transparente pelo ClickHouse.
  • A função toFloat64() é aplicada porque o campo price chega como string, em decorrência da ingestão com dtype=str. A conversão tardia constitui marca registrada do regime schema-on-read.

12. O dashboard — app/streamlit_app.py

Streamlit é um framework Python destinado à construção de dashboards interativos com baixa cerimônia. Cada interação dispara nova execução do script, e os widgets são representados como variáveis. No contexto desta aula, o dashboard consulta a view, aplica filtros e renderiza tabela e gráficos.

12.1 Estrutura da página

  • Sidebar: controles para Top N (entre 5 e 100), métrica (units sold, revenue ou avg price) e filtro de categoria.
  • Cabeçalho: métricas-resumo, incluindo o total de produtos distintos, a receita acumulada e o ticket médio.
  • Tabela top-N: ordenada conforme a métrica selecionada.
  • Gráfico de barras: visualização da mesma seleção.
  • Breakdown por categoria: agregação adicional.
  • Rodapé: timestamp UTC da última ingestão, obtido por meio de max(unixtime).

12.2 Decisões técnicas

  • @st.cache_data(ttl=60) aplicado à função que executa a query da view; na ausência desse recurso, cada interação com sliders desencadearia novo JOIN no ClickHouse.
  • Conexão com o ClickHouse mediante clickhouse-connect utilizando o usuário dashboard, com permissão restrita à leitura de olist_raw, em conformidade com o princípio do menor privilégio.
  • --server.headless=true, evitando que o Streamlit tente abrir browser no interior do container.
  • Porta 8501 exposta no host, de modo que o estudante acesse a aplicação em http://localhost:8501.

13. Near real-time pragmático — critérios para migração ao streaming propriamente dito

O regime de micro-batch atende à esmagadora maioria dos dashboards analíticos. A migração para streaming propriamente dito (Kafka/MSK acompanhado de Flink ou Spark Structured Streaming, ou Kinesis Data Streams com Lambda) somente se justifica quando alguma das condições a seguir se manifesta:

  1. Janela de decisão inferior ao intervalo do lote: a detecção de fraude que precisa ser executada em menos de cinco segundos não é compatível com micro-batch de cinco minutos.
  2. Volume que torna os lotes inviáveis: o ingestor passa a operar permanentemente em atraso, configurando situação de backpressure; os brokers oferecem buffer com semântica bem definida.
  3. Presença de múltiplos consumidores independentes: quando os mesmos eventos devem alimentar cinco ou mais destinos com lógicas distintas, um broker provê fan-out de forma nativa.
  4. Maturidade operacional preexistente na equipe: a operação de Kafka exige investimento contínuo em monitoramento, replicação e tuning de retenção. Na ausência dessa maturidade, o ganho de latência é absorvido pelo custo de operação.

A versão em nuvem desta pipeline seria estruturada da seguinte forma: MinIO → S3 (mesmo SDK boto3, alterando-se apenas o endpoint); ClickHouse local → ClickHouse Cloud ou Redshift Serverless; ingestor Python → AWS Lambda ou tarefa em Fargate via cron; Streamlit local → ECS ou App Runner. A topologia lógica permanece inalterada; modificam-se apenas os endereços e os modelos de cobrança.

14. Prática guiada — roteiro passo a passo

Esta seção apresenta o roteiro literal da segunda metade da aula. Cada passo é independente e idempotente: na ocorrência de falha, basta reexecutá-lo. O critério de conclusão é o seguinte: o dashboard, acessível em localhost:8501, deve exibir os produtos top-N com categoria em PT/EN e respectivo revenue.

Passo 1 — Clonagem do repositório

git clone https://github.com/afonsolelis/20261sre-primeironome-ultimonome.git
cd 20261sre-primeironome-ultimonome
cp .env.example .env   # defaults locais ja funcionam

No repositório deverão estar presentes: docker-compose.yaml, a pasta app/ (com ingest_minio.py e streamlit_app.py), a pasta sql/ (com raw/01_schema.sql e views/01_top_products.sql), a pasta data/ (contendo os CSVs do Olist) e o Dockerfile.

Passo 2 — Implantação da stack

docker compose up -d --build
docker compose ps

Aguarde até que os quatro serviços apresentem o estado healthy ou running. Caso algum serviço permaneça em starting por mais de 30 segundos, inspecione os logs: 

docker compose logs -f clickhouse
docker compose logs -f minio

Passo 3 — Criação do bucket no MinIO

Acesse http://localhost:9003 no navegador, autentique-se com minioadmin / minioadmin e, em Buckets → Create Bucket, crie um bucket com o nome exato olist. Não utilize letras maiúsculas, pois a regra do S3 se aplica neste contexto.

Passo 4 — Carga dos CSVs do Olist no bucket

Os CSVs encontram-se na pasta data/ do repositório. Há duas formas para o upload:

  1. Pelo console MinIO: arrastar todos os CSVs de data/ para o interior do bucket olist, na interface web.
  2. Pelo cliente mc (MinIO Client), mais expedito para múltiplos arquivos: 
    mc alias set local http://localhost:9002 minioadmin minioadmin
mc cp data/*.csv local/olist/

Confirme, por meio do console, a presença de nove arquivos: olist_orders_dataset.csv, olist_order_items_dataset.csv, olist_customers_dataset.csv, olist_products_dataset.csv, olist_sellers_dataset.csv, olist_order_reviews_dataset.csv, olist_order_payments_dataset.csv, olist_geolocation_dataset.csv e product_category_name_translation.csv.

Passo 5 — Aplicação do schema raw

docker exec -i olist_analytics clickhouse-client --multiquery < sql/raw/01_schema.sql

Cria a database olist_raw e a tabela ingestion. A reexecução do comando é segura, uma vez que o DDL utiliza a cláusula IF NOT EXISTS.

Passo 6 — Execução da ingestão

docker exec olist_ingestor python /app/ingest_minio.py

A execução produzirá uma linha por CSV processado, acompanhada da respectiva contagem de linhas. Em um laptop de configuração mediana, a ingestão completa do Olist concluiu-se em intervalo entre 30 segundos e 1 minuto.

Passo 7 — Verificação da landing

docker exec -it olist_analytics clickhouse-client --query \
  "SELECT tag, count() FROM olist_raw.ingestion GROUP BY tag ORDER BY tag"

Espera-se uma linha por CSV ingerido, acompanhada da contagem total. O número deve corresponder ao das linhas dos arquivos originais.

Passo 8 — Aplicação da view top-products

docker exec -i olist_analytics clickhouse-client --multiquery < sql/views/01_top_products.sql

Passo 9 — Validação da view

docker exec -it olist_analytics clickhouse-client --query \
  "SELECT * FROM olist_raw.v_top_products ORDER BY revenue DESC LIMIT 10"

A consulta deve retornar os dez produtos com maior receita, contendo product_id, categoria em PT, categoria em EN, unidades vendidas, receita e ticket médio.

Passo 10 — Acesso ao dashboard

Acesse http://localhost:8501 no navegador. Explore os controles da sidebar alterando Top N, alternando a métrica entre "units sold" e "revenue" e aplicando filtros por categoria. Observe o timestamp da última ingestão exibido no rodapé.

Passo 11 — Reingestão e observação da idempotência

docker exec olist_ingestor python /app/ingest_minio.py
docker exec -it olist_analytics clickhouse-client --query \
  "SELECT tag, count(DISTINCT unixtime) AS execucoes FROM olist_raw.ingestion GROUP BY tag"

A coluna execucoes aumentou de 1 para 2 em todas as tags. Volte ao dashboard e dê um refresh: os números devem permanecer iguais, porque a view filtra pelo max(unixtime). O timestamp da "última ingestão" no rodapé, porém, mudou. Esse é o coração do padrão.

Passo 12 — Encerrar

docker compose down       # mantem volumes (e os dados)
# ou
docker compose down -v    # apaga volumes (reset completo)

Faça commit do que mudou no seu fork. Esse é o seu artefato entregável da aula.

15. Troubleshooting

SintomaCausa provávelAção rápida
Porta 8123, 8501, 9002 ou 9003 já em uso Outro processo no host ocupando a porta. Editar o mapeamento host:container no docker-compose.yaml para uma porta livre.
ingestor ou dashboard não inicia Dependências (clickhouse, minio) ainda em starting. Esperar healthcheck completar ou rodar docker compose up -d novamente.
boto3 retorna NoSuchBucket: olist Bucket não foi criado no console MinIO. Criar manualmente em http://localhost:9003 ou via mc mb local/olist.
v_top_products retorna zero linhas Algum CSV foi ingerido com nome diferente do esperado pela view. Conferir as tags presentes: SELECT DISTINCT tag FROM olist_raw.ingestion. Renomear o arquivo no bucket se necessário.
Streamlit em loop de erro View ainda não foi criada quando a primeira request chegou. Aplicar 01_top_products.sql e dar refresh na página.
Disco cheio após várias reingestões ingestion acumulou todas as execuções. OPTIMIZE TABLE olist_raw.ingestion FINAL ou reset total com docker compose down -v.
boto3 erra SignatureDoesNotMatch Endpoint URL omitido ou credenciais erradas. Conferir S3_ENDPOINT_URL=http://minio:9000 no env do container ingestor.

Equipe de agentes IA + CloudFormation: do markdown ao stack provisionado

Este material acompanha a Aula 04 da disciplina Cloud Computing e SRE — Visão Prática com AWS do MBA em Engenharia de Dados. A aula tira a arquitetura do papel e a põe na conta AWS — mas com regras: nada manual, tudo declarativo via CloudFormation, e quem propõe o template é uma squad de agentes IA com papéis bem cortados, não um agente solitário. Entrega: stack olist-sre-pipeline-lab em CREATE_COMPLETE, ETL dummy escrevendo no S3 raw, drift detection rodada e dois novos arquivos em /documents: 10_deploy_log.md e 11_drift_runbook.md.

Pré-requisitos Aula 03 entregue: /documents/01..09 preenchidos, pasta documents/agents/ com AGENTS.md, GEMINI.md e seis skills, AWS CLI conectado pelo session token do Learner Lab, tag aula-03 no git.

Restrições duras Sem clicar no console AWS · sem secrets em template · sem Glue, Redshift ou SageMaker (Learner Lab não habilita) · tudo idempotente: re-rodar o deploy não pode mudar estado.

1. Da arquitetura ao stack

Na Aula 03 modelamos a arquitetura em RM-ODP, escrevemos a RTM e desenhamos o system design em Mermaid mapeando RF/RNF. Tudo isso vive em markdown na pasta /documents. O problema agora muda: como tirar a arquitetura do papel sem introduzir dívida operacional? A resposta da disciplina tem duas partes inseparáveis.

A primeira é Infrastructure as Code — toda decisão de infra é arquivo versionado, revisado em PR e auditável. Cliques no console são proibidos porque cliques não deixam rastro. A segunda parte é multi-agente — em vez de pedir para uma única IA "gerar tudo", separamos em quatro papéis com escopos curtos. Architect lê arquitetura, DevOps gera YAML, Tester valida, Reviewer veta. O humano é o árbitro e o único que aciona aws cloudformation deploy.

1.1 As cinco regras desta aula

Não-negociáveis
  1. Toda decisão de infra é arquivo YAML versionado — nada de cliques no console.
  2. Quatro papéis distintos na squad: Architect, DevOps, Tester, Reviewer.
  3. Cada recurso provisionado nasce com tag, log group e métrica.
  4. Re-rodar o deploy não pode mudar estado — idempotência ou nada.
  5. Recurso sem caminho de remediação no closed loop não entra no stack.

2. Por que sair do prompt único

Um agente sozinho funciona bem para tarefas curtas. Quando o pedido cresce — gerar templates, escrever testes, fazer ATAM, atualizar runbook — ele começa a perder aderência: esquece que IAM com Action: '*' está proibido, mistura saídas (joga ATAM dentro do YAML), e propaga alucinações entre artefatos. Um erro na primeira etapa contamina as próximas.

Multi-agente resolve isso dividindo cognição. Cada papel tem prompt curto, lê arquivos específicos e escreve em pasta específica. Saídas são separadas, comparáveis e comitáveis. O custo é orquestração: alguém precisa coordenar a sequência. Em troca, a auditoria fica trivial — você sabe quem produziu cada arquivo.

2.1 Padrões de orquestração

PadrãoComo funcionaQuando usarRisco
Planner + ExecutorUm agente planeja, outro(s) executam tarefas pequenas.Pipeline previsível com etapas claras.Plano ruim se propaga.
HierárquicoManager delega para sub-agentes especializados.Tarefas com dependências fortes entre subtarefas.Latência cumulativa.
BlackboardAgentes leem e escrevem em quadro comum.Problemas exploratórios, sem ordem fixa.Difícil auditar quem mudou o quê.
SwarmVários agentes paralelos, sem coordenador central.Geração de variantes, brainstorming.Pode divergir; precisa redutor.

Hoje rodamos um híbrido planner+executor com revisão humana. O Architect planeja, DevOps executa, Tester e Reviewer fecham o loop, e o humano dá o "go" antes do deploy. É o padrão que cabe na prática guiada sem perder rastreabilidade.

3. A squad canônica

Quatro papéis. Cada um com escopo de leitura, escopo de escrita e critério de aceitação. Nenhum papel pode invadir o outro — o Architect não escreve YAML, o DevOps não escreve ADR, o Tester não decide arquitetura, o Reviewer não fecha PR sozinho.

3.1 Architect

Lê documents/03_architecture.md da Aula 03 e devolve um plano de módulos CFN em iac/cloudformation/PLAN.md. O plano descreve cada módulo (network, storage, compute, observability, sre-loop) com parâmetros esperados, outputs e qual RF/RNF da Aula 03 ele atende. Nenhuma linha de YAML. A saída é texto em markdown, fácil de revisar e diffar.

3.2 DevOps

Recebe o PLAN.md do Architect e gera root.yaml + cinco módulos. Aplica a skill gen_cfn_template (ver seção 6). Tags obrigatórias, Outputs por módulo, DeletionPolicy escolhido por recurso. Saída entra como diff revisável; nada vai direto para a AWS.

3.3 Tester

Gera tests/iac/test_templates.py. Roda cfn-lint em todos os YAMLs e adiciona asserts pytest sobre Outputs esperados, ausência de Action: '*' em IAM, presença de DeletionPolicy em recursos críticos. Falha do Tester é blocker do deploy.

3.4 Reviewer

Faz ATAM enxuto sobre o template, reusando a skill review_architecture da Aula 03 e a nova review_cfn. Lista cinco riscos, três sensibilidades e dois trade-offs. Decide explicitamente: APROVADO PARA DEPLOY ou VETO + razão. A saída do Reviewer vai para a primeira seção do 10_deploy_log.md.

Regra de veto Se o Reviewer disser VETO, o ciclo volta para o DevOps com a lista de problemas. Não dá deploy "para ver no que dá". Custo de errar no Lab é tempo, mas a disciplina é hábito que precisa ser construído antes de chegar em produção.

4. AIOX e alternativas pragmáticas

AIOX é o orquestrador CLI multi-agente que adotamos como default da disciplina. A ideia é pequena: um binário Python que lê um squad.yml descrevendo papéis, skills e arquivos compartilhados, e roda cada papel chamando o LLM com o prompt certo. Logs por papel/turno ficam em arquivos auditáveis.

O nome do binário pode mudar — o que importa é o comportamento. Se a turma não conseguir instalar AIOX (cache do PyPI vazio, rede do Codespace bloqueando, versão Python inadequada), caímos para um plano B equivalente:

  • CrewAI — framework Python de squads, declarativo via YAML/Python.
  • AutoGen — multi-agent dialog da Microsoft, usado em pesquisa.
  • LangGraph CLI — grafo de estados para LLM workflows.
  • Caseiro — Makefile chamando Gemini CLI ou OpenCode com cada skill em sequência. Mais simples, audita do mesmo jeito.

Critério de escolha: roda no Codespace sem chave paga, lê markdown como input, escreve markdown e YAML como saída, deixa rastro no git. Nada além disso. A discussão sobre qual é "o melhor framework" fica fora desta aula.

6. CloudFormation 101

Antes de revisar o YAML que a IA gera, é preciso saber para que serve cada seção. CFN é declarativo: você descreve o estado desejado e o serviço da AWS cuida do diff. Não há comando create e comando delete separados — há um único objeto chamado stack que tem ciclo de vida.

6.1 Anatomia do template

SeçãoPara que serveObrigatória?
AWSTemplateFormatVersionVersão do schema (sempre 2010-09-09).Recomendada
DescriptionO que o template provisiona, em uma frase.Recomendada
ParametersEntradas do stack (env, iniciais, sizing).Não
MappingsTabelas estáticas (ex.: AMI por região).Não
ConditionsLiga/desliga recursos por env.Não
ResourcesO que vai existir na conta.Sim
OutputsValores expostos pelo stack (ARNs, URLs).Recomendada

6.2 Funções intrínsecas

A IA vai gerar template recheado de funções intrínsecas. Saber decodificar é a diferença entre revisar e copiar cego.

FunçãoPara quêExemplo
!RefReferencia parâmetro ou resource.!Ref BucketRaw
!SubInterpola variável em string.!Sub "${Iniciais}-raw"
!GetAttLê atributo de outro resource.!GetAtt MyEC2.PublicIp
!ImportValueLê Output exportado por outro stack.!ImportValue VpcId-lab
!JoinConcatena lista com separador.!Join ["/", [a, b]]
!IfCondicional baseado em Conditions.!If [IsProd, t3.large, t3.micro]

6.3 Lifecycle do stack

Stack tem máquina de estados explícita. Conhecer os estados é condição para depurar deploy travado sem chutar.

OperaçãoEstados principaisQuando ocorre
CREATECREATE_IN_PROGRESS → CREATE_COMPLETEPrimeiro deploy com stack inexistente.
CREATE c/ falhaCREATE_FAILED → ROLLBACK_IN_PROGRESS → ROLLBACK_COMPLETERecurso falha; tudo é desfeito.
UPDATEUPDATE_IN_PROGRESS → UPDATE_COMPLETETemplate muda; aplica diff.
UPDATE c/ falhaUPDATE_ROLLBACK_IN_PROGRESS → UPDATE_ROLLBACK_COMPLETEReverte para estado anterior.
DELETEDELETE_IN_PROGRESS → DELETE_COMPLETEdelete-stack respeita DeletionPolicy.

6.4 Capabilities, Stack Policy e DeletionPolicy

  • Capabilities — CAPABILITY_IAM autoriza CFN a criar IAM Roles. Sem isso, deploy falha por segurança. CAPABILITY_NAMED_IAM aceita nome customizado.
  • Stack Policy — JSON inline que protege resources críticos contra UPDATE acidental. Útil para banco de dados e buckets de raw.
  • DeletionPolicy — Delete (default) · Retain (mantém recurso após delete-stack) · Snapshot (faz snapshot antes de deletar). Bucket S3 e RDS nunca devem ser Delete em prod. No Lab, usamos Retain só para o bucket raw.

6.5 Drift detection

Drift é a diferença entre o que está no template e o que está vivo na conta. Se alguém abre o console e edita um Security Group "só para ver", o template mente. CFN consegue detectar drift em propriedades de resources (não consegue detectar drift em dados dentro de S3 ou RDS). Roda hoje, antes que algum colega edite no console e quebre o ciclo.

6.6 Composição: nested vs cross-stack

Stack único e gigante é difícil de revisar e ainda mais difícil de atualizar. O padrão é quebrar em módulos por domínio.

  • Nested stacks — pai inclui filhos via AWS::CloudFormation::Stack. Atualização do pai propaga. Bom quando o ciclo de vida é compartilhado.
  • Cross-stack via Outputs/Exports — stack A exporta valor; stack B faz !ImportValue. Ciclos de vida independentes. Cuidado: Export usado não pode mudar nome.

Nesta aula usamos nested stacks com root.yaml incluindo network, storage, compute, observability e sre-loop. Ciclo de vida único: o stack do laboratório.

7. CloudFormation vs Terraform vs CDK

Por que CFN é o default da disciplina? Porque é o que cabe no Learner Lab sem instalar nada além do AWS CLI já presente.

Decisão da disciplina: CFN puro nesta aula. CDK fica como leitura complementar para quem já programa em TS/Python e quer abstrair templates atrás de classes.

8. SRE começa no template

Cada recurso provisionado já nasce com tag de SLO, log group e métrica. Observabilidade adicionada depois é dívida garantida — o time esquece, o budget aperta, e o recurso fica sem dono. Por isso a Aula 04 adota um conjunto pequeno de regras que entram diretamente no template gerado pela squad.

8.1 Higiene mínima por recurso

  • Tags Project, Env, SLO, Owner em 100% dos resources tagueáveis.
  • Log group dedicado quando aplicável (Lambda, EC2 com agent, EventBridge target).
  • Métricas customizadas via CloudWatch Embedded Metric Format quando o caso justifica.
  • DeletionPolicy e UpdateReplacePolicy escolhidos explicitamente.

8.2 Closed loop herdado da Aula 03

O system design da Aula 03 já desenhou o closed loop CW → Controlador SRE → ETL/Postgres. Hoje ele vira YAML em sre-loop.yaml:

  • Sensor — CloudWatch alarm de frescor (idade do último arquivo no bucket raw).
  • Comparador/Controlador — EventBridge rule que escuta o alarm.
  • Atuador — SSM Run Command que tenta retry no ETL.
  • Audit — log group dedicado do controlador.

Recurso que não tem caminho de remediação no closed loop não entra no stack. Se alguém propõe um componente novo, a primeira pergunta é "o que acontece quando ele falha?".

9. Anti-padrões em template CFN

A IA vai acertar 80% do template na primeira tentativa. Os 20% restantes geralmente são variações desta lista. O Reviewer existe para pegá-los antes do deploy.

Anti-padrãoPor que dóiComo evitar
Secret no templateVai parar no git e nos logs do stack.SSM Parameter Store + !Sub.
IAM com Action: '*'Privilege escalation pronta.Action mínimo + Resource mínimo.
Sem DeletionPolicyBucket de raw some no delete-stack.Definir explícito por resource crítico.
ARN hardcodedQuebra ao trocar de conta/região.!Sub com ${AWS::AccountId} e ${AWS::Region}.
Sem OutputsOutros stacks não conseguem consumir.Exportar IDs e ARNs relevantes.
Tags ausentesFinOps cego, SLO sem dono.Tags obrigatórias no padrão da turma.

10. Prática guiada

Treze passos organizados em três blocos lógicos. Cada um tem um comando ou prompt pronto. Nada é feito clicando — tudo via CLI ou via skill da squad. Ao final, stack vivo, smoke test passando e drift detection rodada.

  • Bloco 1 · Bootstrap & squad — passos 1 a 4: token AWS renovado, orquestrador instalado, squad.yml e skills novas commitadas.
  • Bloco 2 · Squad gera — passos 5 a 8: Architect produz PLAN.md, DevOps gera os templates, Tester valida e Reviewer decide APROVADO ou VETO.
  • Bloco 3 · Deploy & validação — passos 9 a 13: validate-template, deploy, verificação por CLI, smoke test no S3 raw e drift detection.

10.A Bloco 1 — Bootstrap & squad

Antes de pedir IA para gerar qualquer coisa, deixar o Codespace conectado na AWS, instalar o orquestrador e formalizar a squad em arquivo versionado. Critério parcial para seguir adiante: token válido, --version retornando OK no orquestrador e no cfn-lint, e agents/squad.yml commitado com os 4 papéis.

Mesmo padrão das skills da Aula 03: nome verbo+objeto, quando usar, entrada, passos, saída, critério de aceitação. Esqueleto de uma delas: 

# documents/agents/skills/gen_cfn_template.md

## Quando usar
Quando existir iac/cloudformation/PLAN.md descrevendo modulos.

## Entrada
- iac/cloudformation/PLAN.md
- documents/03_architecture.md

10.C Bloco 3 — Deploy & validação

Aprovado pelo Reviewer, agora a infra entra na conta. Validate no servidor, deploy nested, verificação por CLI, smoke test no S3 raw e drift detection — tudo via CLI, nada de console. Critério final: stack em CREATE_COMPLETE, smoke test escreveu, drift = IN_SYNC, e tag aula-04 empurrada para o repositório.

10.9 Passo 9 — validar template no servidor

validate-template roda no lado da AWS. Pega erro de schema/intrinsic function que o lint local pode não cobrir. Em seguida, package sobe os módulos filhos para um bucket de artefatos e gera root.packaged.yaml. 

for f in iac/cloudformation/*.yaml; do
  echo "==== $f ===="
  aws cloudformation validate-template \
    --template-body "file://$f" \
    --query "Description" --output text
done

# Empacotar nested stacks (sobe filhos para S3)
aws s3 mb s3://${INICIAIS}-cfn-artifacts-lab 2>/dev/null || true

aws cloudformation package \
  --template-file iac/cloudformation/root.yaml \
  --s3-bucket    ${INICIAIS}-cfn-artifacts-lab \
  --output-template-file iac/cloudformation/root.packaged.yaml

10.10 Passo 10 — deploy

Único comando que cria recursos. Acompanhe pela CLI; se quiser ver eventos, use describe-stack-events. Nada de console. 

export INICIAIS=afn   # suas iniciais

aws cloudformation deploy \
  --stack-name olist-sre-pipeline-lab \
  --template-file iac/cloudformation/root.packaged.yaml \
  --capabilities CAPABILITY_IAM \
  --parameter-overrides \
      Env=lab \
      Iniciais=${INICIAIS} \
  --tags Project=olist-sre Env=lab Owner=${INICIAIS}

# Acompanhar (em outro shell)
aws cloudformation describe-stack-events \
  --stack-name olist-sre-pipeline-lab \
  --max-items 20 \
  --query "StackEvents[].[Timestamp,LogicalResourceId,ResourceStatus]" \
  --output table

10.11 Passo 11 — verificar pelo CLI

aws cloudformation describe-stacks \
  --stack-name olist-sre-pipeline-lab \
  --query "Stacks[0].[StackStatus,Outputs]" \
  --output json | tee documents/10_deploy_log.md.tmp

aws cloudformation describe-stack-resources \
  --stack-name olist-sre-pipeline-lab \
  --query "StackResources[].[LogicalResourceId,ResourceType,ResourceStatus]" \
  --output table

aws cloudformation list-exports \
  --query "Exports[?starts_with(Name, 'olist-')].[Name,Value]" \

10.13 Passo 13 — drift detection

Roda agora, antes que alguém edite um SG no console. Resultado vai para 11_drift_runbook.md como referência da próxima aula. 

DRIFT_ID=$(aws cloudformation detect-stack-drift \
  --stack-name olist-sre-pipeline-lab \
  --query "StackDriftDetectionId" --output text)

while true; do
  STATUS=$(aws cloudformation describe-stack-drift-detection-status \
    --stack-drift-detection-id $DRIFT_ID \
    --query "DetectionStatus" --output text)
  [ "$STATUS" = "DETECTION_COMPLETE" ] && break
  [ "$STATUS" = "DETECTION_FAILED" ]   && break
  sleep 5
done

aws cloudformation describe-stack-resource-drifts \
  --stack-name olist-sre-pipeline-lab \
  --query "StackResourceDrifts[].[LogicalResourceId,StackResourceDriftStatus]" \
  --output table

Ingestão e Processamento Near Real-time com Streaming

Este material acompanha a Aula 04 da disciplina Cloud Computing e SRE — Visão Prática com AWS do MBA em Engenharia de Dados. A aula desce do plano abstrato para o teclado: subimos uma stack inteira de ingestão near real-time no laptop, com MinIO no papel de S3, ClickHouse como warehouse colunar, um ingestor Python em Docker e um dashboard Streamlit consumindo as métricas. O dataset é o já clássico Olist Brazilian E-commerce Public Dataset.

Pré-requisitos Docker Desktop (ou Docker Engine + Compose v2), Git, navegador web e ~2 GB de RAM livre. Portas 8123, 8501, 9000, 9002 e 9003 precisam estar livres no host.

Restrições da aula Sem clicar em console AWS hoje · sem Kafka/Flink/Kinesis · nada manual fora do que está no docker-compose.yaml · tudo idempotente, reingestões precisam ser seguras.

Repositório Todo o código que rodamos em sala está no GitHub. Faça fork antes de modificar.

1. Por que stack local antes da nuvem

A primeira tentação ao aprender uma plataforma de dados é abrir o console do provedor e começar a clicar. O custo dessa rota é triplo: cada erro custa hora-recurso, cada clique não deixa rastro reproduzível, e cada participante depende de um session token instável para sequer começar. A proposta desta aula é inverter a ordem: subir a stack inteira no laptop, falar exatamente as mesmas interfaces que a nuvem oferece, validar o pipeline ponta a ponta, e só depois discutir como esse mesmo pipeline migraria para S3, Redshift ou Kinesis.

A engenharia mora nas interfaces. MinIO implementa a API S3 que o boto3 e o AWS CLI já falam. ClickHouse é o mesmo binário no laptop, no servidor on-premise e no ClickHouse Cloud. Streamlit roda igual em qualquer lugar. O contrato é a interface; o provedor é detalhe de implementação. Quando você fala S3 API e SQL, você não está aprendendo "MinIO" — está aprendendo S3 e ClickHouse com latência menor e custo zero.

Tese Antes de pagar nuvem, sobe local. A interface é o contrato — não o fornecedor. Quem fala S3 API e SQL fala com MinIO+ClickHouse hoje e com S3+Redshift amanhã. A engenharia mora nas interfaces; a fatura, no provedor.

1.1 Quatro regras desta aula

  1. A stack inteira sobe com docker compose up -d — nada de clique em console cloud hoje.
  2. Ingestão é append-only: cada execução marca unixtime e tag próprios e não sobrescreve nada.
  3. A modelagem é schema-on-read: landing como JSON bruto, modelagem nas views.
  4. Healthcheck obrigatório em todo serviço crítico — depends_on: condition: service_healthy orquestra startup.

2. Espectro batch — micro-batch — streaming

"Near real-time" não é um ponto fixo; é uma faixa do espectro entre o lote diário tradicional e o evento processado em milissegundos. A engenharia de dados moderna escolhe o regime certo para cada caso de uso. Forçar streaming verdadeiro onde basta micro-batch é caro; forçar batch onde a janela de decisão é curta deixa dinheiro na mesa.

ModoLatência típicaMecânicaQuando aplicar
Batch Horas a dia Janela longa, processa o conjunto inteiro de uma vez. Job dispara em horário fixo. Relatório diário, fechamento contábil, BI clássico.
Micro-batch Segundos a minutos Lotes pequenos e frequentes. Loop curto de leitura/transformação/escrita. Dashboards "vivos", agregações leves, análises operacionais.
Streaming verdadeiro Sub-segundo Evento a evento via broker (Kafka, Kinesis) + processador (Flink, Spark Structured Streaming). Detecção de fraude, telemetria crítica, matching em tempo real.

Nesta aula rodamos micro-batch via ingest_minio.py — lê todos os CSVs do bucket, transforma e escreve no ClickHouse em lotes de 50 mil linhas. Pode ser disparado manualmente, por cron, por evento de upload no MinIO, ou em loop infinito. Deliberadamente evitamos Kafka/Flink: o custo de operação de um broker mais um processador stateful não compensa para o volume e a janela de decisão de um dashboard analítico.

3. Docker Compose como contrato de stack

Docker Compose é um arquivo YAML que descreve todo o sistema: cada serviço (container), suas portas, volumes, variáveis de ambiente, healthchecks e dependências. É um único artefato versionado que substitui dezenas de comandos manuais de docker run. Para ambientes locais e de desenvolvimento, é o equivalente prático de um pequeno Kubernetes — sem a sobrecarga conceitual.

3.1 Conceitos centrais

  • services: lista de containers do sistema, cada um com imagem ou build próprio.
  • volumes: persistência fora do container — o estado sobrevive a down e up.
  • ports: mapeamento host:container; expõe portas para acesso a partir do host.
  • environment: variáveis injetadas no processo do container.
  • healthcheck: comando que reporta a saúde do serviço; é o SLO local.
  • depends_on com condition: service_healthy: bloqueia o startup do dependente até o healthcheck passar.

Healthchecks merecem destaque. Sem eles, o Compose marca um serviço como "up" assim que o processo inicia — mesmo que ainda não esteja respondendo. Com healthcheck, o status só vira healthy quando o comando externo confirma. Para ClickHouse usamos wget --spider http://localhost:8123/ping; para MinIO, curl /minio/health/live. O resultado é que o ingestor só sobe quando seus dois backends estão genuinamente prontos — eliminando uma classe inteira de race conditions do startup.

5. MinIO 101 — S3 API no laptop

MinIO é um servidor de objetos open source que implementa a API S3 da Amazon. "Implementa" aqui é literal: os mesmos verbos REST, os mesmos cabeçalhos de assinatura V4, os mesmos formatos de resposta XML. Como consequência, qualquer cliente que fala S3 — boto3, AWS CLI, Spark, Trino, DuckDB — fala com MinIO trocando apenas o endpoint URL.

5.1 Conceitos S3 que importam

  • Bucket: container nomeado, único em todo o servidor. No MinIO é só uma pasta no disco.
  • Objeto: arquivo binário + metadados (Content-Type, ETag, timestamps), identificado por key dentro do bucket.
  • Endpoint URL: a URL base do servidor. Para AWS S3 default é https://s3.amazonaws.com; para o nosso MinIO é http://minio:9000 de dentro da rede do Compose, ou http://localhost:9002 a partir do host.
  • Path-style vs virtual-hosted: na S3 moderna o bucket vira subdomínio (bucket.s3.amazonaws.com); em MinIO local usa-se path-style (localhost:9000/olist/key). O boto3 detecta automaticamente.
  • Credenciais: par access_key + secret_key. No MinIO local default é minioadmin / minioadmin — apenas para laboratório.

5.2 boto3 falando com MinIO

import boto3, os

s3 = boto3.client(

O console web do MinIO em http://localhost:9003 é o análogo direto do AWS Console — permite criar bucket, fazer upload de arquivos por drag-and-drop, configurar policies. Para a aula vamos usá-lo para criar o bucket olist e popular os CSVs.

6. ClickHouse 101 — OLAP colunar

ClickHouse é um banco de dados analítico colunar, vetorizado, open source. "Colunar" significa que dados da mesma coluna ficam contíguos no disco — queries que tocam poucas colunas leem pouco. "Vetorizado" significa que operadores processam vetores de valores ao invés de linhas uma a uma, multiplicando a vazão. É o tipo de banco onde SELECT count(*) FROM x sobre bilhões de linhas devolve em segundos.

6.1 MergeTree, a engine padrão

Tabelas em ClickHouse são criadas com uma engine; a padrão da família OLAP é MergeTree. Características essenciais:

  • ORDER BY define a chave de ordenação. Não é uma chave primária no sentido SQL clássico — é a ordem física dos dados em disco e o índice primário esparso.
  • Inserts criam "parts" novos no diretório da tabela. Em background, o ClickHouse faz merges entre parts — daí o nome.
  • A primeira coluna do ORDER BY tende a ter alto poder de poda — queries que filtram por ela só leem os parts relevantes.

6.2 Tipo JSON

Coluna JSON é um tipo semi-estruturado nativo do ClickHouse moderno (24.x+). Permite armazenar objetos JSON arbitrários sem definir schema antecipado — o engine indexa subcaminhos automaticamente e expõe acesso via notação ponto: data.product_id. É o que torna o padrão schema-on-read prático no ClickHouse.

6.3 LowCardinality

Wrapper de tipo para colunas com poucos valores distintos. Internamente armazena um dicionário (valor → código) e usa o código nas linhas. Compressão fica drasticamente melhor e operadores como GROUP BY e filtros ficam mais rápidos. Casos típicos: tag, status, region, category.

7. Schema-on-read vs schema-on-write

Schema-on-write é o regime clássico de bancos OLTP: a estrutura é definida no DDL, e qualquer inserção que não bata o tipo é rejeitada na hora. Funciona ótimo quando o schema é estável e poucos sistemas escrevem — aplicação web típica. Sofre quando fontes mudam de forma frequente, porque cada alteração vira migração coordenada.

Schema-on-read inverte o jogo: dados entram no formato bruto que apareceram (JSON, NDJSON, Parquet semi-estruturado), e o schema é aplicado na hora da query, geralmente em views. O ganho é absorver mudança de fonte sem migração; o custo é mover validação para a leitura, o que pode atrasar relatório.

AspectoSchema-on-writeSchema-on-read
Quando o tipo é validadoNa ingestão.Na query.
Custo de mudar schemaAlto — ALTER, migration.Baixo — muda só a view.
Onde quebra primeiroCedo — rejeita dado na ingestão.Tarde — quebra ao consultar.
Exemplos clássicosPostgres, Redshift.Data Lake, ClickHouse JSON, BigQuery STRUCT.
Nossa escolha hoje—Landing como JSON, modelagem nas views.

A escolha tem implicações organizacionais. Schema-on-read empurra responsabilidade para o time de modelagem analítica — é ali que o tipo "produto" emerge de centenas de campos JSON. Schema-on-write empurra para o time de produção — é ali que cada mudança de fonte precisa virar pull request no DDL. Para um pipeline de descoberta como o desta aula, schema-on-read ganha de longe.

8. A tabela landing — uma só, para tudo

Em vez de criar uma tabela por CSV do Olist, criamos uma única tabela landing genérica. Toda ingestão escreve nela. A diferenciação entre arquivos vem por uma coluna tag — o nome do CSV de origem. 

CREATE DATABASE IF NOT EXISTS olist_raw;

CREATE TABLE IF NOT EXISTS olist_raw.ingestion
(
  unixtime Int64,
  data     JSON,
  tag      LowCardinality(String)
)
ENGINE = MergeTree
ORDER BY (tag, unixtime);

Análise rápida das três colunas:

  • unixtime — Unix epoch (segundos) no momento de início da execução do ingestor. É único por run: todas as linhas inseridas em uma execução compartilham o mesmo valor.
  • data — o registro original, JSON. Cada linha do CSV vira um objeto com as colunas como chaves.
  • tag — o nome do CSV de origem. Funciona como discriminador. LowCardinality porque temos no máximo uma dúzia de tags possíveis.

ORDER BY (tag, unixtime) é deliberada: queries vão quase sempre filtrar por tag (uma "tabela lógica" por vez) e depois por unixtime (a execução mais recente). Essa ordem maximiza a poda de parts no MergeTree.

9. Idempotência sem MERGE nem UPSERT

Idempotência é a propriedade de uma operação cujo resultado, após N execuções (N ≥ 1), é o mesmo de uma única execução. Em pipelines de dados é o que permite retries seguros, recuperação após falha parcial e backfills sem corromper estado.

Bancos OLTP resolvem idempotência com MERGE/UPSERT: a operação verifica chave primária e decide inserir ou atualizar. ClickHouse, otimizado para inserts em massa, não tem MERGE barato — e UPSERT via ReplacingMergeTree só consolida em background. Nosso pipeline contorna isso com um padrão mais simples:

O padrão Append-only no ingestion, com unixtime único por execução. Views filtram WHERE unixtime = (SELECT max(unixtime) FROM ingestion WHERE tag = '<arquivo>'). Reingerir adiciona linhas com novo unixtime, e a view passa a enxergar apenas a execução mais recente. Histórico fica registrado de graça — vira auditoria de linhagem.

9.1 O que esse padrão compra

  • Reingestões sempre seguras — nunca corrompem o estado, no máximo aumentam o histórico.
  • Histórico preservado — cada execução fica como "fotografia" da fonte naquele instante.
  • Reverter para versão anterior = filtrar outro unixtime na view. Operação puramente lógica.
  • Zero locks, zero transações distribuídas — ClickHouse processa em paralelo sem orquestração.

9.2 O custo

Disco cresce monotonicamente. Para pipelines de produção isso vira política de retenção: OPTIMIZE TABLE ... FINAL periódico ou TTL no MergeTree (TTL toDateTime(unixtime) + INTERVAL 30 DAY). Para a aula, o reset é docker compose down -v.

10. O ingestor — app/ingest_minio.py

O ingestor é um script Python único, ~120 linhas, que faz todo o caminho: lista o bucket, lê cada CSV com pandas, converte para registros, e insere em lotes no ClickHouse. Não usa Kafka, não usa Airflow, não usa Spark — é o tipo de "Hello World" honesto de pipeline near real-time que cabe na cabeça. 

import os, time, json, boto3, pandas as pd, clickhouse_connect

UNIXTIME = int(time.time())  # marca unica desta execucao

BUCKET = os.getenv("S3_BUCKET_NAME", "olist")

s3 = boto3.client(
    "s3",
    endpoint_url=os.getenv("S3_ENDPOINT_URL"),
    aws_access_key_id=os.getenv("AWS_ACCESS_KEY_ID"),
    aws_secret_access_key=os.getenv("AWS_SECRET_ACCESS_KEY"),
)

ch = clickhouse_connect.get_client(
    host=os.getenv("CLICKHOUSE_HOST"),
    port=int(os.getenv("CLICKHOUSE_PORT", "8123")),
    username=os.getenv("CLICKHOUSE_USER", "default"),
    password=os.getenv("CLICKHOUSE_PASSWORD", ""),
)

resp = s3.list_objects_v2(Bucket=BUCKET)
for obj in resp.get("Contents", []):
    key = obj["Key"]
    if not key.endswith(".csv"):
        continue

    body = s3.get_object(Bucket=BUCKET, Key=key)["Body"]
    df = pd.read_csv(body, dtype=str).where(pd.notnull, None)

    rows = [
        (UNIXTIME, json.dumps(row.to_dict()), key)
        for _, row in df.iterrows()
    ]

    BATCH = 50_000
    for i in range(0, len(rows), BATCH):
        ch.insert(
            "olist_raw.ingestion",
            rows[i : i + BATCH],
            column_names=["unixtime", "data", "tag"],
        )
    print(f"[ingest] {key}: {len(rows)} linhas em unixtime={UNIXTIME}")

10.1 Decisões de design

  • dtype=str no pandas. Não tentamos inferir tipo na ingestão — isso é trabalho da view. CSV é texto; tratamos como texto. NaN vira None para serializar como null no JSON.
  • Batch de 50 mil linhas. Equilíbrio entre throughput (ClickHouse prefere inserts grandes) e memória (mais que isso pressiona o pod Python no laptop).
  • UNIXTIME fora do loop. Calculado uma vez no início e reusado — é o que garante que todo o run compartilha a mesma marca temporal.
  • Sem retry sofisticado. Se quebrar no meio, basta rodar de novo — o append-only torna isso seguro.

11. A view v_top_products

A view é onde o schema-on-read materializa. Pegamos três CSVs (order_items, products e product_category_name_translation), pegamos apenas a execução mais recente de cada um, fazemos os joins e agregamos por produto. 

CREATE OR REPLACE VIEW olist_raw.v_top_products AS
WITH
  order_items AS (
    SELECT data FROM olist_raw.ingestion
    WHERE tag = 'olist_order_items_dataset.csv'
      AND unixtime = (
        SELECT max(unixtime) FROM olist_raw.ingestion
        WHERE tag = 'olist_order_items_dataset.csv'
      )
  ),
  products AS (
    SELECT data FROM olist_raw.ingestion
    WHERE tag = 'olist_products_dataset.csv'
      AND unixtime = (
        SELECT max(unixtime) FROM olist_raw.ingestion
        WHERE tag = 'olist_products_dataset.csv'
      )
  ),
  translations AS (
    SELECT data FROM olist_raw.ingestion
    WHERE tag = 'product_category_name_translation.csv'
      AND unixtime = (
        SELECT max(unixtime) FROM olist_raw.ingestion
        WHERE tag = 'product_category_name_translation.csv'
      )
  )
SELECT
  oi.data.product_id                          AS product_id,
  p.data.product_category_name                AS category_pt,
  t.data.product_category_name_english        AS category_en,
  count()                                     AS units_sold,
  round(sum(toFloat64(oi.data.price)))        AS revenue,
  round(avg(toFloat64(oi.data.price)))        AS avg_price
FROM order_items oi
LEFT JOIN products     p ON p.data.product_id            = oi.data.product_id
LEFT JOIN translations t ON t.data.product_category_name = p.data.product_category_name
GROUP BY product_id, category_pt, category_en;

Observações importantes sobre essa view:

  • Cada CTE filtra pelo max(unixtime) da própria tag — isso é o que torna a view idempotente em relação a reingestões. Se rodarmos o ingestor 10 vezes, a view continua devolvendo os mesmos números.
  • A view é metadado puro — nada é materializado em disco. Reexecutar a query roda os CTEs do zero. Para volumes grandes evoluiríamos para uma MATERIALIZED VIEW.
  • O acesso a campos JSON é com notação ponto: data.product_id. O ClickHouse converte transparente.
  • toFloat64() aparece porque o campo price chega como string (ingerimos com dtype=str). Conversão tipo "tardia" é a marca registrada do schema-on-read.

12. O dashboard — app/streamlit_app.py

Streamlit é um framework Python para construir dashboards interativos com mínima cerimônia. Cada interação reexecuta o script; widgets viram variáveis. Para o nosso caso, lê a view, aplica filtros, renderiza tabela e gráficos.

12.1 Estrutura da página

  • Sidebar: controles — Top N (5 a 100), métrica (units sold / revenue / avg price), filtro de categoria.
  • Header: métricas resumo — total de produtos distintos, receita acumulada, pedido médio.
  • Tabela top-N: ordenada pela métrica escolhida.
  • Gráfico de barras: visualização da mesma seleção.
  • Breakdown por categoria: agregação adicional.
  • Rodapé: timestamp UTC da última ingestão (lido de max(unixtime)).

12.2 Decisões técnicas

  • @st.cache_data(ttl=60) na função que executa a query da view. Sem isso, cada slider mexido refaria o JOIN inteiro no ClickHouse.
  • Conexão com ClickHouse via clickhouse-connect usando um usuário dashboard com permissão apenas de leitura sobre olist_raw. Princípio do menor privilégio.
  • --server.headless=true para que o Streamlit não tente abrir browser dentro do container.
  • Porta 8501 exposta no host — o aluno acessa em http://localhost:8501.

13. Near real-time pragmático — quando trocar por streaming verdadeiro

Micro-batch resolve a esmagadora maioria dos dashboards analíticos. O upgrade para streaming verdadeiro (Kafka/MSK + Flink/Spark Structured Streaming, ou Kinesis Data Streams + Lambda) só compensa quando alguma das condições abaixo aparece:

  1. Janela de decisão menor que o intervalo do lote. Detecção de fraude que precisa rodar em < 5 segundos não cabe em micro-batch de 5 minutos.
  2. Volume torna lotes inviáveis. O ingestor passa a ficar permanentemente atrasado — sinal de backpressure. Brokers permitem buferizar com semântica conhecida.
  3. Múltiplos consumidores independentes. Quando os mesmos eventos precisam alimentar 5+ destinos com lógicas diferentes, um broker entrega fan-out nativo.
  4. Maturidade operacional já existe no time. Operar Kafka exige investimento em monitoramento, replicação, tuning de retenção. Sem isso, o ganho de latência é comido pelo custo de operação.

A versão na nuvem do nosso pipeline migraria assim: MinIO → S3 (mesmo SDK boto3, só muda endpoint); ClickHouse local → ClickHouse Cloud ou Redshift Serverless; ingestor Python → AWS Lambda ou Fargate task em cron; Streamlit local → ECS / App Runner. A topologia lógica não muda — só os endereços e os modelos de cobrança.

Esta seção é o roteiro literal da segunda metade da aula. Cada passo é independente e idempotente: se algo der errado, basta rodar de novo. O critério de pronto é: dashboard em localhost:8501 mostrando produtos top-N com categoria PT/EN e revenue.

Passo 1 — Clonar o repositório

git clone https://github.com/afonsolelis/20261sre-primeironome-ultimonome.git
cd 20261sre-primeironome-ultimonome
cp .env.example .env   # defaults locais ja funcionam

Você deve ver no repo: docker-compose.yaml, pasta app/ (com ingest_minio.py e streamlit_app.py), pasta sql/ (com raw/01_schema.sql e views/01_top_products.sql), pasta data/ (com os CSVs do Olist) e o Dockerfile.

Passo 2 — Subir a stack

docker compose up -d --build
docker compose ps

Espere até que os quatro serviços apareçam como healthy ou running. Se algo ficar em starting por mais de 30 segundos, inspecione os logs: 

docker compose logs -f clickhouse
docker compose logs -f minio

Passo 3 — Criar o bucket no MinIO

Abra http://localhost:9003 no navegador, logue com minioadmin / minioadmin, vá em Buckets → Create Bucket e crie um bucket chamado exatamente olist. Não use letras maiúsculas; a regra do S3 vale aqui.

Passo 4 — Subir os CSVs do Olist para o bucket

Os CSVs estão na pasta data/ do repo. Duas formas de fazer o upload:

  1. Via console MinIO: arrastar todos os CSVs do data/ para dentro do bucket olist na interface web.
  2. Via cliente mc (MinIO Client): mais rápido para muitos arquivos. 
    mc alias set local http://localhost:9002 minioadmin minioadmin
mc cp data/*.csv local/olist/

Confira pelo console que aparecem 9 arquivos — olist_orders_dataset.csv, olist_order_items_dataset.csv, olist_customers_dataset.csv, olist_products_dataset.csv, olist_sellers_dataset.csv, olist_order_reviews_dataset.csv, olist_order_payments_dataset.csv, olist_geolocation_dataset.csv e product_category_name_translation.csv.

Passo 5 — Aplicar o schema raw

docker exec -i olist_analytics clickhouse-client --multiquery < sql/raw/01_schema.sql

Cria a database olist_raw e a tabela ingestion. Reexecutar o comando é seguro — o DDL usa IF NOT EXISTS.

Passo 6 — Rodar a ingestão

docker exec olist_ingestor python /app/ingest_minio.py

Vai imprimir uma linha por CSV processado com a contagem de linhas. Em um laptop mediano, a ingestão completa do Olist demora em torno de 30 segundos a 1 minuto.

Passo 7 — Conferir a landing

docker exec -it olist_analytics clickhouse-client --query \
  "SELECT tag, count() FROM olist_raw.ingestion GROUP BY tag ORDER BY tag"

Deve aparecer uma linha por CSV ingerido com a contagem total. Compare com o número de linhas dos arquivos originais — precisa bater.

Passo 8 — Aplicar a view top-products

docker exec -i olist_analytics clickhouse-client --multiquery < sql/views/01_top_products.sql

Passo 9 — Validar a view

docker exec -it olist_analytics clickhouse-client --query \
  "SELECT * FROM olist_raw.v_top_products ORDER BY revenue DESC LIMIT 10"

Deve retornar os 10 produtos com maior receita, com product_id, categoria PT, categoria EN, unidades vendidas, receita e ticket médio.

Passo 10 — Abrir o dashboard

Acesse http://localhost:8501 no navegador. Brinque com os controles da sidebar: mude Top N, alterne a métrica entre "units sold" e "revenue", filtre por categoria. Observe o timestamp da última ingestão exibido no rodapé.

Passo 11 — Reingerir e observar a idempotência

docker exec olist_ingestor python /app/ingest_minio.py
docker exec -it olist_analytics clickhouse-client --query \
  "SELECT tag, count(DISTINCT unixtime) AS execucoes FROM olist_raw.ingestion GROUP BY tag"

A coluna execucoes aumentou de 1 para 2 em todas as tags. Volte ao dashboard e dê um refresh: os números devem permanecer iguais, porque a view filtra pelo max(unixtime). O timestamp da "última ingestão" no rodapé, porém, mudou. Esse é o coração do padrão.

Passo 12 — Encerrar

docker compose down       # mantem volumes (e os dados)
# ou
docker compose down -v    # apaga volumes (reset completo)

Faça commit do que mudou no seu fork. Esse é o seu artefato entregável da aula.

15. Troubleshooting

SintomaCausa provávelAção rápida
Porta 8123, 8501, 9002 ou 9003 já em uso Outro processo no host ocupando a porta. Editar o mapeamento host:container no docker-compose.yaml para uma porta livre.
ingestor ou dashboard não inicia Dependências (clickhouse, minio) ainda em starting. Esperar healthcheck completar ou rodar docker compose up -d novamente.
boto3 retorna NoSuchBucket: olist Bucket não foi criado no console MinIO. Criar manualmente em http://localhost:9003 ou via mc mb local/olist.
v_top_products retorna zero linhas Algum CSV foi ingerido com nome diferente do esperado pela view. Conferir as tags presentes: SELECT DISTINCT tag FROM olist_raw.ingestion. Renomear o arquivo no bucket se necessário.
Streamlit em loop de erro View ainda não foi criada quando a primeira request chegou. Aplicar 01_top_products.sql e dar refresh na página.
Disco cheio após várias reingestões ingestion acumulou todas as execuções. OPTIMIZE TABLE olist_raw.ingestion FINAL ou reset total com docker compose down -v.
boto3 erra SignatureDoesNotMatch Endpoint URL omitido ou credenciais erradas. Conferir S3_ENDPOINT_URL=http://minio:9000 no env do container ingestor.

16. Glossário

TermoDefinição
MinIOObject storage open source compatível com a API S3 da Amazon.
ClickHouseBanco analítico colunar vetorizado, foco em OLAP.
MergeTreeEngine padrão do ClickHouse, dados ordenados pelo ORDER BY e merges em background.
LowCardinalityWrapper de tipo no ClickHouse para colunas com poucos valores distintos — dicionário + códigos.
JSON (tipo)Coluna semi-estruturada nativa do ClickHouse moderno.
Schema-on-readTipagem aplicada na hora da query, não na ingestão.
Schema-on-writeTipagem validada no momento da inserção.
NDJSONNewline-Delimited JSON — um documento JSON por linha.
S3 APIConjunto de verbos REST do Amazon S3 (PutObject, GetObject, ListObjectsV2).
Endpoint URLURL base onde o SDK envia requisições.
Path-styleConvenção S3 em que o bucket aparece no path da URL.
Virtual-hostedConvenção S3 em que o bucket vira subdomínio.
HealthcheckComando que reporta saúde de um container para o Docker.
depends_on: service_healthyBloqueia o startup do dependente até o healthcheck do alvo passar.
VolumePersistência fora do container; sobrevive a down.
Micro-batchProcessamento em lotes pequenos e frequentes.
Streaming verdadeiroProcessamento evento a evento, sub-segundo, com broker + processador.
IdempotênciaPropriedade de uma operação cujo efeito após N execuções é o mesmo de 1.
Append-onlyInserção sem update nem delete; histórico fica preservado.
StreamlitFramework Python para dashboards de dados com mínima cerimônia.
boto3SDK oficial da AWS para Python.

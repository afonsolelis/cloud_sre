Data Reliability e SRE Aplicados a Pipelines de Dados

Este material aprofunda os conceitos apresentados nos slides da Aula 07. Cobre os princípios de Site Reliability Engineering (SRE) aplicados a pipelines de dados, a definição de SLOs e SLAs para entrega de dados, o uso do Amazon CloudWatch para observabilidade, e as ferramentas de orquestração resiliente — MWAA (Managed Workflows for Apache Airflow) e AWS Step Functions — com estratégias de retry e recuperação de falhas.

Módulo 1 — SRE Aplicado a Pipelines de Dados

O que é SRE e como se aplica a dados

Site Reliability Engineering (SRE) é uma disciplina que aplica práticas de engenharia de software a operações de infraestrutura, com o objetivo de criar sistemas escaláveis e altamente confiáveis. Originada no Google, a SRE define métricas objetivas de confiabilidade (SLOs) e usa automação para reduzir o trabalho manual repetitivo (toil).

Para pipelines de dados, a SRE se manifesta como Data Reliability Engineering: garantir que os dados cheguem ao destino correto, no tempo correto, com a qualidade correta. Assim como um serviço web tem um SLO de disponibilidade (ex.: 99,9% de uptime), um pipeline de dados pode ter um SLO de freshness (ex.: dados disponíveis com no máximo 4 horas de atraso) e de completeness (ex.: no máximo 0,01% de registros perdidos).

SLIs, SLOs e SLAs para pipelines de dados

SLI (Service Level Indicator): uma métrica quantitativa que mede um aspecto do comportamento do pipeline. Exemplos para pipelines de dados:

  • Freshness: tempo entre a chegada do dado na fonte e a disponibilidade na camada Trusted. Medido em minutos ou horas.
  • Completeness: percentual de registros esperados que efetivamente chegaram ao destino. Ex.: 99,98% dos eventos de um sistema foram ingeridos.
  • Accuracy: percentual de registros com valores corretos (sem erros de transformação ou corrupção).
  • Availability: percentual do tempo em que o pipeline está operacional e processando dados.

SLO (Service Level Objective): o alvo definido para um SLI. Exemplo: "o pipeline de vendas deve entregar dados com freshness menor que 60 minutos em 99,5% do tempo em qualquer janela de 30 dias."

SLA (Service Level Agreement): o acordo formal entre o provedor (equipe de dados) e o consumidor (time de BI, time de negócio), geralmente com consequências contratuais se o SLO não for atingido. O SLA é mais conservador que o SLO — se o SLO interno é 99,5%, o SLA externo pode ser 99%.

Error budgets: aceitando falhas de forma controlada

O error budget é a quantidade de falha permitida antes que o SLO seja violado. Se o SLO é 99,5% de freshness em 30 dias, o error budget é 0,5% de 30 dias = 3,6 horas de falha permitida por mês. Quando o error budget está esgotado, a prioridade da equipe deve mudar de novas features para estabilidade e redução de toil.

Módulo 2 — Amazon CloudWatch: Observabilidade para Pipelines

Os três pilares da observabilidade

Observabilidade é a capacidade de inferir o estado interno de um sistema a partir de suas saídas externas. Os três pilares são:

  • Métricas: valores numéricos agregados ao longo do tempo. Exemplos: número de registros processados por hora, latência média de execução de um Glue Job, número de falhas em 24h. No CloudWatch, as métricas são armazenadas com granularidade de 1 segundo (high resolution) ou 1 minuto (standard) e podem ser visualizadas em dashboards.
  • Logs: registros textuais de eventos que ocorreram no sistema. Exemplos: stack traces de erros em Glue Jobs, mensagens de progresso de pipelines, logs de acesso do Athena. No CloudWatch Logs, os logs são organizados em Log Groups (um por aplicação/serviço) e Log Streams (um por instância/execução).
  • Traces: registro do fluxo de uma requisição ou job através de múltiplos serviços. O AWS X-Ray fornece tracing distribuído para entender onde o tempo é gasto em pipelines com múltiplos componentes.

CloudWatch Alarms para pipelines de dados

Um CloudWatch Alarm monitora uma métrica e executa uma ação quando o valor cruza um limiar. Para pipelines de dados, alarmes essenciais incluem:

  • Glue Job failures: alarme quando o número de execuções com erro excede 1 em qualquer período de 1 hora.
  • Kinesis iterator age: alarme quando o GetRecords.IteratorAgeMilliseconds do Kinesis excede 60.000 ms (indicando que o consumidor está atrasado em relação ao produtor).
  • Athena query scan volume: alarme quando uma única query escaneia mais de 100 GB (indicando query problemática ou mal otimizada).
  • S3 bucket size: alarme quando o bucket cresce mais de X% ao dia em relação à média histórica (anomalia de volume que pode indicar problema de duplicação).

A ação típica de um alarme é publicar uma notificação no SNS (Simple Notification Service), que então pode enviar emails, mensagens Slack via Lambda, ou disparar uma função de remediação automática.

CloudWatch Logs Insights para análise de falhas

O Logs Insights é um motor de query para pesquisar e analisar logs no CloudWatch. Exemplo de query para encontrar erros em logs de Glue Jobs nas últimas 24h: 

fields @timestamp, @message
| filter @message like /ERROR/
| sort @timestamp desc
| limit 50

O Logs Insights suporta agregações, filtros com regex, e funções de estatística, tornando-o poderoso para análise de incidentes e identificação de padrões de falha recorrentes.

Módulo 3 — Orquestração Resiliente: MWAA e Step Functions

Por que orquestração é crítica

Um pipeline de dados típico é composto por múltiplas etapas com dependências: o job de transformação Trusted só deve rodar depois que o Crawler Raw terminar; o job de Refined só deve rodar depois que o Trusted estiver completo; a notificação ao time de BI só deve ser enviada após validações de qualidade. Executar essas etapas manualmente é insustentável e propenso a erros. Um orquestrador gerencia essas dependências automaticamente e define o que fazer quando uma etapa falha.

Amazon MWAA: Apache Airflow gerenciado

O MWAA (Managed Workflows for Apache Airflow) é o serviço gerenciado da AWS para executar o Apache Airflow. O Airflow usa DAGs (Directed Acyclic Graphs) escritas em Python para definir pipelines. Cada nó do DAG é uma Task (uma unidade de trabalho), e as arestas definem as dependências entre tasks.

Exemplo de DAG Airflow simples para o pipeline da disciplina: 

from airflow import DAG
from airflow.providers.amazon.aws.operators.glue import GlueJobOperator
from airflow.providers.amazon.aws.operators.glue_crawler import GlueCrawlerOperator
from datetime import datetime, timedelta

default_args = {
    'owner': 'data-engineering',
    'retries': 2,
    'retry_delay': timedelta(minutes=5),
    'email_on_failure': True,
    'email': ['alertas@empresa.com']
}

with DAG(
    dag_id='pipeline_vendas_diario',
    default_args=default_args,
    schedule_interval='0 6 * * *',  # Todo dia às 06h00
    start_date=datetime(2026, 4, 1),
    catchup=False
) as dag:

    crawl_raw = GlueCrawlerOperator(
        task_id='crawl_raw',
        config={'Name': 'crawler-raw-lab'}
    )

    etl_trusted = GlueJobOperator(
        task_id='etl_raw_to_trusted',
        job_name='etl-raw-to-trusted-pedidos'
    )

    crawl_trusted = GlueCrawlerOperator(
        task_id='crawl_trusted',
        config={'Name': 'crawler-trusted-lab'}
    )

    crawl_raw >> etl_trusted >> crawl_trusted

O parâmetro retries=2 faz o Airflow tentar reexecutar uma task que falhou até 2 vezes adicionais, com 5 minutos de intervalo entre tentativas. Isso cobre falhas transitórias (ex.: timeout de rede, throttling da AWS) sem intervenção manual.

AWS Step Functions: orquestração serverless

O Step Functions é uma alternativa serverless ao MWAA para orquestração de workflows. Em vez de Python, os workflows são definidos em Amazon States Language (JSON). Cada estado pode invocar uma Lambda, um Glue Job, um ECS Task, ou aguardar um evento.

Vantagens do Step Functions sobre MWAA:

  • Sem servidor para gerenciar — sem preocupação com escalabilidade do scheduler.
  • Integração nativa com praticamente todos os serviços AWS via SDK integrations.
  • Visualização do estado de execução em tempo real no console.
  • Custo baseado em transições de estado (paga por execução, não por hora de servidor).

Vantagens do MWAA sobre Step Functions:

  • Airflow é uma plataforma open-source com enorme ecossistema de providers e plugins.
  • Portabilidade: a mesma DAG pode rodar on-premises, em qualquer cloud, ou no MWAA.
  • Interface web rica para monitoramento, reexecução de tasks individuais e debugging.
  • Melhor para pipelines complexos com muitas dependências e lógica condicional sofisticada.

Retry patterns: estratégias de recuperação

Não basta configurar retries — é preciso configurar retries corretos. Alguns padrões essenciais:

  • Exponential backoff: aumenta o tempo de espera exponencialmente entre tentativas (1s, 2s, 4s, 8s). Reduz a sobrecarga em sistemas que estão sob pressão. Evita o "retry storm" onde múltiplos clientes tentam simultaneamente.
  • Jitter: adiciona um componente aleatório ao intervalo de retry. Evita que múltiplos jobs que falharam ao mesmo tempo façam retry exatamente ao mesmo instante, sobrecarregando o sistema novamente.
  • Dead letter queue (DLQ): após o número máximo de tentativas, o evento ou job que falhou é movido para uma fila DLQ para análise posterior. Evita perda silenciosa de dados e permite investigar e reprocessar manualmente os casos problemáticos.
  • Circuit breaker: interrompe tentativas quando o sistema detecta falhas sistemáticas (não apenas transitórias), evitando desperdício de recursos e cascata de falhas.

Módulo 4 — Laboratório: Configurando Alarmes no CloudWatch

A prática utiliza o ambiente AWS Student. O objetivo é criar um alarme no CloudWatch que notifique quando um Glue Job falha.

Passo 1 — Executar um Glue Job e explorar os logs

  1. No console AWS, acesse AWS Glue e execute o job criado na Aula 05.
  2. Aguarde a conclusão (sucesso ou falha).
  3. Clique no job e na aba Runs, clique na execução mais recente.
  4. Clique em Output logs para abrir o CloudWatch Logs.
  5. Explore os logs: procure por mensagens de progresso, erros (se houver) e o tempo de execução.
  6. Validação: você consegue localizar nos logs o início e o fim da execução do job.

Passo 2 — Criar um alarme de falha do Glue Job

  1. Acesse o Amazon CloudWatch no console AWS.
  2. No menu lateral, clique em Alarms e depois em Create alarm.
  3. Clique em Select metric.
  4. Navegue para Glue e depois Job Metrics.
  5. Procure pelo seu job e selecione a métrica glue.driver.aggregate.numFailedTasks.
  6. Configure: período = 5 minutos, estatística = Sum.
  7. Em Conditions: tipo = Static, quando a métrica for Greater than 0.
  8. Em Actions:
    • Em caso de alarme, enviar notificação para um tópico SNS.
    • Clique em Create new topic.
    • Nome do tópico: alertas-pipeline-dados.
    • Email de notificação: insira seu email.
    • Clique em Create topic.
  9. Nome do alarme: glue-job-failure-alert.
  10. Clique em Create alarm.
  11. Verifique seu email e confirme a inscrição no SNS topic.
  12. Validação: o alarme aparece no painel do CloudWatch com status OK (assumindo que o job não falhou recentemente).

Passo 3 — Criar um CloudWatch Dashboard

  1. No CloudWatch, clique em Dashboards e depois em Create dashboard.
  2. Nome: pipeline-monitoramento.
  3. Clique em Create dashboard.
  4. Adicione um widget de Alarm status mostrando o alarme criado.
  5. Adicione um widget de Metric mostrando o histórico de execuções do Glue Job.
  6. Salve o dashboard.
  7. Validação: o dashboard exibe o status do alarme e as métricas do job em uma única tela.

Exercícios de fixação

  1. Definindo SLOs para um pipeline: um pipeline de dados alimenta um dashboard de vendas que é consultado pelo time comercial toda manhã às 08h. Os dados vêm de um RDS via CDC, passam por um Glue Job de transformação, e são carregados no Redshift. Defina dois SLOs para esse pipeline — um de freshness e um de completeness — especificando: o SLI exato, o objetivo numérico, a janela de avaliação, e como você mediria cada SLI tecnicamente.
  2. MWAA versus Step Functions: uma empresa está decidindo entre MWAA e Step Functions para orquestrar um pipeline que: roda diariamente, tem 15 etapas sequenciais, usa Glue Jobs, Lambda e ECS Tasks, e precisa de reexecução manual de etapas individuais quando algo falha. Liste os critérios decisivos para cada opção e faça uma recomendação fundamentada.
  3. Retry storm e solução: um pipeline processa 500 jobs paralelos. Todos falham simultaneamente devido a um throttling da AWS. Os jobs são configurados para retry imediato após 1 segundo. Explique o que acontece a seguir (retry storm), por que isso piora o problema, e como configurar exponential backoff com jitter para evitar essa situação.
  4. Investigação de incidente com CloudWatch: às 03h, o alarme de falha do Glue Job disparou. Descreva o processo de investigação passo a passo: quais logs você consultaria, quais métricas você observaria, como distinguiria uma falha transitória de uma falha sistemática, e qual seria o critério para decidir entre reprocessar automaticamente ou escalar para a equipe.
  5. Data quality como SLO: além de freshness e completeness, proponha dois SLOs de qualidade de dados para um pipeline de ingestão de transações financeiras. Para cada SLO, especifique: o que é medido, como implementar a medição tecnicamente na AWS (serviço, métrica), o valor alvo, e a ação automática a ser tomada quando o SLO for violado.

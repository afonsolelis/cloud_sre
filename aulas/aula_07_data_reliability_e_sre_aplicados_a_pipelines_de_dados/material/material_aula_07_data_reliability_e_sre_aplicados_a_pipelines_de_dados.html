2. Tese da aula — o pipeline se observando e se reparando dentro do error budget

A tese da Aula 07 é simples e dura. SRE é o pipeline se observando e se reparando dentro do error budget. Todo trabalho que acontece abaixo da linha do error budget pertence à malha fechada — código que percebe a falha, código que aplica o conserto, código que registra o incidente. Pager humano só quando o budget esgota; abaixo disso, automação. E essa automação não pode ser cega: toda regra de auto-remediation passa por dry-run antes de ficar com State=ENABLED.

Cinco regras orientam toda a aula. Elas não são opcionais — são contrato. (1) Sem console AWS. Tudo via CloudFormation e CLI, revisável em diff. O console serve para ler; nunca para clicar configuração. (2) Alarm sem runbook é alarm-fadiga. Toda regra AWS::CloudWatch::Alarm carrega AlarmDescription com link para um runbook executável. (3) Runbook sempre tem dry-run antes do enable. Auto-remediação sem dry-run quebra mais do que conserta. (4) Toda métrica tem unidade e janela. Métrica sem unidade é número decorativo — ninguém sabe se é segundo, milissegundo ou minuto. (5) Todo incidente vira postmortem blameless em documents/17_postmortems/. Sem culpado — com contributing factors e action items com dono e prazo.

Essa tese não é original. Vem direto do livro Site Reliability Engineering (Beyer, Jones, Petoff, Murphy — Google, O'Reilly) e do Site Reliability Workbook. O que a Aula 07 faz é traduzir o livro azul para o vocabulário do AWS Learner Lab e do pipeline Olist. SLI vira EMF metric em namespace OlistETL. SLO vira CloudFormation alarm. Error budget vira tag em recurso. Runbook vira SSM Document. Postmortem vira arquivo markdown commitado no repo.

3. Pilares SRE — o que herdamos do livro azul

O Site Reliability Engineering do Google (2016) consolidou um vocabulário que vinha sendo usado internamente desde 2003. A Aula 07 importa seis pilares e mapeia cada um para uma prática operacional concreta no pipeline Olist. Nenhum deles é folclore; cada um existe como código em algum lugar do repositório no final da aula.

3.1 SLI / SLO / Error budget

SLI — Service Level Indicator é a medição. Um número, com unidade, dentro de uma janela temporal. Por exemplo: idade do registro mais recente carregado no Postgres, em minutos, calculada a cada 5 minutos. SLO — Service Level Objective é a meta interna sobre o SLI. Por exemplo: p95 da freshness ≤ 30 minutos, em janela de 24 horas. SLA — Service Level Agreement é o contrato externo, geralmente mais frouxo que o SLO — existe para o cliente, não para o time. Error budget é o complemento: 1 − SLO. Se o SLO é 99,5%, o error budget é 0,5%. Quando o budget esgota numa janela, o time freia mudanças até o budget se renovar — error-budget-driven freeze.

3.2 Toil — o trabalho que não escala

Toil tem definição operacional precisa: trabalho manual, repetitivo, automatizável, sem valor durável, que escala linearmente com o serviço. Não é "trabalho difícil" — trabalho difícil pode ser engenharia. Toil é o restart manual do ETL toda manhã, o SSH no Postgres para limpar quarentena, o copy-paste de comando vindo de uma página de Notion. A meta clássica do Google é que toil consuma menos de 50% do tempo do time SRE; o resto é engenharia.

3.3 Incident Response (IR) — papéis e cadência

Incidente é evento de alta severidade que precisa de coordenação. SRE separa papéis para evitar caos: Incident Commander (IC) coordena e decide; Operations Lead (Ops) põe a mão na massa investigando e aplicando fix; Communications (Comms) escreve atualizações para stakeholders a cada 15–30 minutos; Scribe registra a timeline factual em tempo real. Em time pequeno, a mesma pessoa pode acumular dois papéis — mas IC e Ops nunca são a mesma pessoa durante o pico do incidente.

3.4 Postmortem blameless

Toda vez que um incidente é fechado, o time escreve um postmortem. Blameless significa que o documento foca no sistema que permitiu o erro, não na pessoa que apertou o botão. Linguagem despessoal: "o deploy foi feito" no lugar de "o Fulano fez o deploy". O resultado é uma timeline factual, contributing factors honestos e action items com dono, prazo e link de ticket. O postmortem só está fechado quando o último action item está em status done.

3.5 Releases progressivos

SRE não acredita em deploy big-bang. Releases progressivos — canário, blue/green, feature flags — existem para reduzir o blast radius de uma mudança. O canário é a versão nova rodando para 1% do tráfego; se a métrica degradar, rollback automático. Blue/green é a versão nova rodando em paralelo, com switch no balanceador. Feature flag é a mudança guardada atrás de uma flag que pode ser desligada sem deploy.

3.6 Automação no lugar de heroísmo

SRE prefere runbook executável a procedimento em PDF. Heróis cansam, dormem, esquecem. Código não. Quando uma falha repete três vezes, o time não escreve um terceiro post no Slack — escreve um SSM Document. Quando uma resposta vira manual, o team a transforma em State Manager Association. Quando um patch vira tarefa recorrente, o time monta um Patch Manager baseline. Pager só para o que realmente exige humano.

4. Cinco SLIs canônicos para pipelines de dados

SLO genérico não funciona para dado. "Disponibilidade 99,9%" diz quase nada sobre um pipeline batch. Por isso a comunidade de Data Reliability convergiu para cinco indicadores que cobrem 90% das dores reais de pipelines de dados modernos. A Aula 07 os incorpora com IDs estáveis (SLO-01 a SLO-05) que valem do dashboard CW à RTM da Aula 03. Cada um tem fórmula, unidade e janela explícitas.

IDSLIFórmulaUnidadeJanela / SLO
SLO-01 Freshness — idade do dado now() − max(loaded_at) min p95 ≤ 30 min, 24 h
SLO-02 Completeness — linhas presentes rows_loaded / rows_expected % ≥ 99,5%, diária
SLO-03 Accuracy — linhas válidas 1 − (invalid_rows / total_rows) % ≥ 99,0%, diária
SLO-04 Availability — pipeline rodou successful_runs / scheduled_runs % ≥ 99,0%, mensal
SLO-05 Latency — tempo total da execução finish_ts − start_ts s p95 ≤ 30 min, 30 d

Freshness (SLO-01) mede quão atual está o dado consumido pelo cliente final — no caso Olist, o dashboard Grafana. É o SLI mais sensível a falhas de upstream: se a fonte parou, freshness sobe sozinha. Costuma ser medida em minutos. Completeness (SLO-02) mede quanto do que era esperado chegou de fato — calculada como razão entre linhas processadas e linhas esperadas, exige contagem de referência (cadastro de origem ou heurística calibrada). Accuracy (SLO-03) mede quanto do que chegou é válido segundo as regras do negócio — tipos certos, ranges plausíveis, relacionamentos íntegros. Availability (SLO-04) mede quão frequentemente o pipeline completou no horário planejado — é binária por execução, mas vira percentual ao longo da janela. Latency (SLO-05) mede quanto tempo a execução leva, do disparo do schedule até o último UPDATE no Postgres — importa porque latência alta corrói freshness rio abaixo.

Há ainda um sexto SLI implícito que aparece como black box: o canário Synthetics (ALARM-06 SyntheticsDown). Ele bate no /api/health do Grafana a cada 5 minutos e falha 2 datapoints quando o serviço cai. Não é um SLO de pipeline, é um SLO de experiência do consumidor.

5. Toil — definição operacional e como medir

Toil é o conceito do SRE que mais é mal entendido. Vale repetir a definição operacional: trabalho que é (a) manual, (b) repetitivo, (c) automatizável, (d) reativo em vez de proativo, (e) sem valor durável e (f) que escala linearmente com o serviço. Os seis critérios precisam estar presentes. Reunião de planejamento é repetitiva, mas não é toil — gera valor durável. Refactor difícil é manual, mas não é toil — não é automatizável trivialmente.

Como medir? A maneira mais simples é registrar 5 minutos no fim do dia: cada engenheiro anota quantas horas gastou em toil naquele dia. Em duas semanas, a média estabiliza. A meta é manter abaixo de 50%. Quando ultrapassa, o time prioriza tickets de automação na próxima sprint. Outra maneira é olhar para frequência de certas operações: quantos restarts manuais por semana, quantos SSH por semana, quantos copy-paste de comandos. Cada um vira candidato a virar runbook.

5.1 Sintomas de toil no pipeline Olist

  • Restart manual de ETL todas as manhãs — vira RetryEtlOnce SSM Document.
  • SSH no Postgres para limpar tabela de quarentena — vira CleanQuarantine SSM Document agendado.
  • Subir agente CloudWatch à mão em cada nova EC2 — vira State Manager Association idempotente.
  • Reabrir alarm que toca no fim de semana — vira composite alarm com lógica AND.
  • Copy-paste de comando vindo de página de Notion — vira documento SSM versionado no repo.
  • Aplicar patch de segurança manualmente em janela domingo 03h — vira Patch Manager baseline + maintenance window.

6. Incident Response — papéis, cadência e canais

Em incidente, o caos custa MTTR. Papéis explícitos previnem caos. A AWS, Google, Meta e empresas brasileiras maduras como Nubank e iFood operam com a mesma estrutura mínima. Mesmo em time pequeno, separar Incident Commander de quem está com a mão na massa muda o tom do incidente.

RoleResponsabilidadeNão faz
IC — Incident Commander Coordena, decide prioridades, garante que comunicação flua. Pode chamar mais gente. Não debuga código durante o incidente. Não é "o mais sênior".
Ops — Operations Lead Mão na massa: investiga logs, executa runbooks, aplica fix. Não comunica externamente, não toma decisão de escopo.
Comms — Communications Atualiza status page, escreve updates a cada 15–30 min, traduz tecnês para stakeholders. Não toma decisões técnicas; não inventa hipóteses sobre causa.
Scribe Registra timeline factual em tempo real. Base do postmortem. Não opera, só observa e anota com timestamp UTC.

A cadência típica é: a cada 15–30 minutos, o IC pede um sitrep (situational report) ao Ops e o Comms publica um update. Depois de 60 minutos sem progresso, escala para o próximo nível (gerente, diretor). O canal padrão é um Slack/Teams dedicado por incidente (ex.: #inc-2026-05-28-01) — nada de DM, nada de e-mail. Tudo com timestamp.

SEV-1, SEV-2, SEV-3, SEV-4. SEV-1 é o pior — serviço crítico fora do ar, error budget em queda livre, requer pager humano fora do horário. SEV-4 é o mais brando — degradação cosmética sem impacto em SLI. A política da Aula 07: chaos planejado vai como SEV-3 (incidente real, mas com kill switch e janela negociada).

7. Postmortem blameless — estrutura, contributing factors e action items

O postmortem blameless é a peça onde aprendizado vira código. O nome vem de John Allspaw e foi popularizado em 2012 no blog da Etsy. O ponto não é narrativa heróica nem caça às bruxas — é um documento curto, factual, acionável. Uma página é melhor que cinco. O leitor precisa entender o que houve, o que vai mudar, e quem é responsável pela mudança.

7.1 Estrutura mínima

  1. Cabeçalho — ID do incidente, severidade (SEV-N), datas, IC/Ops/Comms/Scribe, resumo de 3 linhas.
  2. Timeline (UTC) — eventos com timestamp absoluto: detect, ack, mitigate, resolve. Inclui MTTR e error budget consumido.
  3. Contributing factors — falhas técnicas plausíveis, falhas de processo, sinais que foram ignorados. Não é "causa raiz" — é uma teia.
  4. What went well — o que funcionou e por quê. Insumo para repetir, não elogio.
  5. Action items (AI-N) — uma frase, um dono, um prazo, um link de ticket. Sem AI ≡ sem aprendizado.
  6. Apêndices — queries de Logs Insights, capturas de gráficos, links de ADRs gerados.

7.2 Linguagem despessoal

Em vez de "o Fulano deployou a versão errada", escreva "a versão errada foi deployada". Em vez de "o time de dados deveria ter testado", escreva "não havia teste cobrindo o cenário X". Pessoas agem racionalmente dado o contexto que têm; quando o contexto é incompleto, o resultado é falha — e o sistema é responsável por garantir contexto suficiente. Esse é o ponto do blameless.

7.3 Contributing factors em pipeline de dados

Em pipeline batch, contributing factors costumam ser combinações: schema da fonte mudou + teste não cobria + alarm de accuracy não estava ativo. Cada um isoladamente seria absorvido; juntos, viram incidente. O postmortem precisa explicitar a teia, não escolher um culpado técnico.

8. Três pilares de observabilidade — logs, métricas, traces

Charity Majors, no livro Observability Engineering, define observabilidade como a capacidade de responder perguntas novas sobre o sistema sem precisar deployar código novo. Os três pilares clássicos — logs, métricas, traces — são as fontes de dado que tornam isso possível. Cada um responde uma pergunta diferente. Pipeline maduro tem os três e os correlaciona por request id ou execution id.

PilarPerguntaEstruturaCusto
Logs O que aconteceu aqui? Eventos discretos com contexto, JSON estruturado. Alto volume, custo proporcional ao GB ingerido.
Métricas Como está, agora e antes? Séries numéricas com dimensões, agregadas em janelas. Baixo — agregação reduz cardinalidade.
Traces Onde está o gargalo? Cadeia causal entre componentes (segment, subsegment). Alto sem sampling; sampling agressivo controla.

A correlação entre os três pilares é o que diferencia observabilidade de mera monitoração. Métrica indica que algo está fora do trilho. Logs explicam o quê. Traces mostram onde. Sem correlação por ID compartilhado, você tem três silos — e três silos não respondem perguntas novas.

9. CloudWatch Logs — log groups, retenção e structured logging

CloudWatch Logs é o serviço gerenciado de logs da AWS. Cobra por GB ingerido e GB armazenado. Tem três níveis hierárquicos: log group — container nomeado por aplicação ou por serviço; log stream — uma instância concreta dentro do grupo, geralmente uma execução ou instância; log event — o registro individual com timestamp e mensagem.

9.1 Anatomia operacional

  • Retention policy — em dias (1, 3, 5, 7, 14, 30, 60, 90, 120, 150, 180, 365, 400, 545, 731, 1827, 3653 ou nunca expira). No Learner Lab, 30 dias é razoável.
  • Metric filter — um pattern que extrai número do log e publica como métrica CW. Útil para contar erros, somar valores, calcular percentuais.
  • Subscription filter — envia eventos de log que casam um pattern para Lambda, Kinesis ou Firehose. Útil para processamento em tempo quase real.
  • Resource policy — controla quem pode escrever no log group (cross-account, cross-service).

9.3 Structured logging — JSON sempre

Texto solto no log obriga grep e regex. JSON estruturado libera Logs Insights, EMF e correlação. A regra do curso: nenhum print(), nenhum logger.info("texto livre"). Sempre logger.info(json.dumps({...})) com chaves estáveis: RunId, Stage, Env, RowsProcessed, etc. 

import json, logging, time

log = logging.getLogger("olist")

def log_event(event: str, **kwargs):
    payload = {
        "event": event,
        "timestamp_ms": int(time.time() * 1000),
        **kwargs,
    }
    log.info(json.dumps(payload))

# uso
log_event("etl.run.started", run_id="etl-2026-05-28T03:00", stage="raw")
log_event("etl.run.finished", run_id="etl-2026-05-28T03:00",
          rows_processed=102345, latency_s=1742.5, invalid=27)

10. CloudWatch Metrics — namespace, dimensão, estatística

Métrica é endereçada por namespace, identificada por dimensions e consumida por statistic. Errar qualquer dos três quebra o alarm silenciosamente — ele entra em INSUFFICIENT_DATA e ninguém percebe. Cada um dos seis conceitos a seguir precisa ser explícito em todo template CloudFormation.

ConceitoSignificadoOlist — exemplo
Namespace Bucket lógico (AWS gerenciados começam com AWS/) OlistETL, OlistGrafana
Metric name Nome do indicador, em PascalCase RowsProcessed, Latency, InvalidRows
Dimensions Pares chave-valor que segmentam (até 30 por métrica) Stage=raw|silver|gold, Env=lab|prod
Unit Unidade canônica do CW Count, Seconds, Bytes, Percent, None
Statistic Função sobre a janela Sum, Average, p95, p99, Maximum
Period Granularidade da agregação (alta resolução é múltiplo de 1s; padrão é múltiplo de 60s) 60s, 300s, 900s

10.1 Math expressions

Métricas podem ser combinadas com expressões matemáticas dentro do CloudWatch — sem precisar publicar uma nova métrica derivada. Por exemplo, completeness = m1 / m2 * 100 onde m1 = RowsProcessed e m2 = RowsExpected. A vantagem é que o alarm pode ser declarado sobre a expressão, garantindo versão única da verdade entre dashboard e alarm. Sempre usar math expression para SLIs derivados — nunca duas implementações para o mesmo cálculo.

11. Embedded Metric Format (EMF) — log JSON vira métrica

O Embedded Metric Format é o padrão atual da AWS para apps emitirem métricas customizadas sem chamar PutMetricData diretamente. Você emite um log estruturado em JSON; o CloudWatch reconhece o bloco especial _aws.CloudWatchMetrics e publica métricas automaticamente. O custo é uma única chamada PutLogEvents — um único request publica log e métrica.

11.1 Por que EMF importa

Antes do EMF, métrica customizada exigia boto3.client("cloudwatch").put_metric_data(...) — uma chamada de API por publicação, com cota e custo. Em pipeline com milhares de eventos por minuto, isso explodia em latência e cobrança. EMF resolve com três vantagens: (1) elimina chamadas extras de API; (2) mantém log e métrica sincronizados — o mesmo evento é fonte dos dois; (3) dimensões são definidas no próprio JSON, sem hardcoding. EMF é a maneira certa de instrumentar Lambda, ECS, EC2 e Glue Jobs em 2026.

11.2 Anatomia do JSON EMF

{
  "_aws": {
    "Timestamp": 1716902400000,
    "CloudWatchMetrics": [{
      "Namespace": "OlistETL",
      "Dimensions": [["Stage", "Env"]],
      "Metrics": [
        {"Name": "RowsProcessed", "Unit": "Count"},
        {"Name": "Latency",       "Unit": "Seconds"},
        {"Name": "InvalidRows",   "Unit": "Count"}
      ]
    }]
  },
  "Stage": "raw",
  "Env": "lab",
  "RowsProcessed": 102345,
  "Latency": 1742.5,
  "InvalidRows": 27,
  "RunId": "etl-2026-05-28T03:00:00Z"
}

O bloco _aws diz ao CloudWatch como interpretar o restante. Namespace é onde a métrica vai aparecer. Dimensions é uma lista de listas — cada lista interna é um conjunto de dimensões a serem combinadas para gerar uma série. Metrics declara cada métrica com nome e unidade. Os valores ficam fora do bloco _aws — o CloudWatch lê o JSON e correlaciona nome com valor. Tudo o que não está em Metrics fica como atributo de log, queryable em Logs Insights.

11.3 EMF em Python

import json, time, logging
log = logging.getLogger("olist")

def emit_emf(stage: str, env: str, rows: int, latency_s: float,
             invalid: int, run_id: str):
    payload = {
        "_aws": {
            "Timestamp": int(time.time() * 1000),
            "CloudWatchMetrics": [{
                "Namespace": "OlistETL",
                "Dimensions": [["Stage", "Env"]],
                "Metrics": [
                    {"Name": "RowsProcessed", "Unit": "Count"},
                    {"Name": "Latency",       "Unit": "Seconds"},
                    {"Name": "InvalidRows",   "Unit": "Count"},
                ],
            }],
        },
        "Stage": stage,
        "Env": env,
        "RowsProcessed": rows,
        "Latency": latency_s,
        "InvalidRows": invalid,
        "RunId": run_id,
    }
    log.info(json.dumps(payload))

Existe a biblioteca oficial aws-embedded-metrics (Python, Node, Java, .NET) que abstrai o JSON e oferece API mais ergonômica. Para o lab da Aula 07, a versão crua acima basta — é melhor entender a estrutura antes de adotar wrapper.

12. CloudWatch Alarms — threshold, anomaly e composite

Alarm é o gatilho da malha fechada. Tem três estados (OK, ALARM, INSUFFICIENT_DATA) e pode acionar SNS, Auto Scaling, EC2 actions ou EventBridge. Existem três variantes principais; a Aula 07 usa as três.

12.1 Threshold alarm

Compara estatística da métrica contra um valor fixo. Por exemplo: Freshness p95 maior que 30 min em 2 datapoints consecutivos de 5 min. É a variante mais usada — simples, defensável, auditável. Os parâmetros chave são MetricName, Statistic, Period, EvaluationPeriods, DatapointsToAlarm, Threshold, ComparisonOperator e TreatMissingData.

12.2 Anomaly detection alarm

Em vez de threshold fixo, o CloudWatch treina um modelo da métrica e dispara quando o valor sai do envelope esperado (configurável em desvios padrão). Útil para métricas com sazonalidade forte — volume de tráfego horário, contagem de pedidos por hora do dia. Cuidado: modelo aprende baseline ruim se você ligar durante incidente.

12.3 Composite alarm

Combina alarms via expressão booleana AND, OR, NOT. Reduz alarm-fadiga em incidentes correlacionados. Por exemplo, um composite que dispara só quando Freshness E Completeness violam ao mesmo tempo — sinal de problema sistêmico, não cosmético.

12.4 AlarmDescription — o link para o runbook

Toda regra AWS::CloudWatch::Alarm precisa carregar AlarmDescription com link para o runbook executável. Quando o alarm dispara fora de hora, o on-call lê a descrição e clica direto. Sem runbook, o alarm é apenas barulho — isso é alarm-fadiga. 

FreshnessAlarm:
  Type: AWS::CloudWatch::Alarm
  Properties:

13. Dashboards via CloudFormation — widget JSON e math expression

Dashboard CloudWatch é JSON. Vai versionado no repo, junto do template CloudFormation. Dashboard sem dono morre — cada um da turma fica com o seu, com tag Owner que aponta para o aluno responsável. O dashboard olist-sre tem 8 widgets, um por SLI mais canário e grade de alarms.

13.1 Tipos de widget

  • metric — gráfico de série temporal. Suporta múltiplas métricas, math expressions, anotações horizontais.
  • singleValue — KPI grande para o topo do dashboard. Ideal para "freshness atual" ou "última run".
  • log — tabela de Logs Insights embutida. Roda a query a cada refresh.
  • alarm — status visual de alarms. Verde, vermelho, cinza.
  • text — markdown estático. Lugar para link de runbook e contato do on-call.

13.2 Math expression no widget

14. CloudWatch Logs Insights — query estruturada em log group

Logs Insights é a linguagem de consulta do CloudWatch. SQL-like, com pipelines (sintaxe parecida com Splunk/Elastic). Cobra por GB escaneado — mantenha a janela curta. Bom para investigação durante incidente; queries salvas viram parte do runbook.

14.1 Operadores principais

  • fields — seleciona campos. @timestamp, @message e @logStream são built-in.
  • filter — filtra eventos. Aceita regex, comparação, like.
  • stats — agrega: count(*), sum(), avg(), pct(field, 95), by.
  • parse — extrai campos de string com glob ou regex.
  • sort — ordena pelo campo.
  • limit — trunca o resultado.

14.2 Query padrão para diagnóstico de run do ETL

fields @timestamp, @message, RunId, Stage, RowsProcessed, InvalidRows, Latency
| filter Stage = "raw"
| filter @timestamp > ago(1h)
| stats sum(RowsProcessed) as total_rows,
        sum(InvalidRows)   as total_invalid,
        max(Latency)       as worst_latency
        by RunId
| sort worst_latency desc
| limit 20

14.3 Query salva como widget

Toda query útil durante incidente vira widget de log no dashboard. Próxima vez, em vez de o on-call procurar a query no Notion, ele abre o dashboard e a query já está rodando — com janela ajustável.

15. AWS X-Ray — tracing distribuído

X-Ray é o serviço de tracing distribuído da AWS. Ele captura uma trace — árvore completa de uma request — composta de segments (trabalho em um componente) e subsegments (chamada interna). Você anota a trace com chaves indexadas (annotation) e não indexadas (metadata). No Learner Lab, X-Ray funciona via SDK em EC2 — o uso clássico é instrumentar o ETL para ver onde o tempo vai entre read S3 → transform → write Postgres.

15.1 Conceitos

TermoDefinição
TraceÁrvore completa de uma request, identificada por TraceId.
SegmentTrabalho em um componente (Lambda, EC2, ECS task).
SubsegmentChamada interna dentro do segment (ex.: query SQL, chamada HTTP).
AnnotationPar chave-valor indexado, queryable em filter expression.
MetadataPar chave-valor não indexado, só visível ao abrir a trace.
Sampling rulePolítica que decide quais traces são capturadas (controla custo).

15.2 Sampling no Learner Lab

Sampling 100% só vale em desenvolvimento curto. Em prod, padrão razoável é 5% + reservoir de 1 trace por segundo — garante visibilidade sem explodir custo. Sampling rule é declarada via CloudFormation (AWS::XRay::SamplingRule). Para o lab, manter RunId como annotation facilita correlação com EMF e Logs Insights pelo mesmo identificador.

15.3 Instrumentação Python

from aws_xray_sdk.core import xray_recorder, patch_all

patch_all()  # instrumenta boto3, requests, sqlalchemy

@xray_recorder.capture("etl.run")
def run(run_id: str):
    xray_recorder.put_annotation("RunId", run_id)
    xray_recorder.put_annotation("Stage", "raw")

    with xray_recorder.in_subsegment("read_s3"):
        df = read_s3(...)

    with xray_recorder.in_subsegment("transform"):
        df = transform(df)

    with xray_recorder.in_subsegment("write_pg"):
        write_pg(df)

16. CloudWatch Synthetics — canários sintéticos

Synthetics é um Lambda gerenciado pela AWS que executa um script (Node.js Puppeteer ou Python Selenium) a cada N minutos. A saída vira métrica (SuccessPercent, Duration) e screenshot no S3. É o primeiro SLI black box do dashboard — o ponto de vista do cliente, não do servidor. Quando o canário falha, o cliente está vendo o problema; o monitoramento white-box pode dizer que está tudo OK, mas o canário diz a verdade.

16.1 Tipos de canário

  • Heartbeat HTTP — bate em endpoint /api/health a cada 5 min e valida status code 200.
  • API canário — consulta um KPI do dashboard e valida shape do JSON retornado.
  • Visual canário — screenshot do dashboard, diff visual contra baseline registrado.
  • Broken link checker — navega na página e verifica que nenhum link 404.
  • Multi-step API — sequência de chamadas que simula fluxo do usuário.

16.2 Custo no Learner Lab

Cada execução de canário é um Lambda — cobra por invocação e duração. Em rate de 5 min, são 288 execuções por dia, ~8.640 por mês. No tier gratuito, isso costuma caber. Vale checar o saldo do Learner Lab antes de subir mais de 2 canários simultâneos.

17. EventBridge + SSM como closed loop

Aqui mora o coração da Aula 07. Alarm CloudWatch dispara um evento; EventBridge roteia esse evento para SSM; SSM executa um Document de remediação na EC2 alvo. Tudo em código, tudo auditável, tudo sem clique no console. É o closed loop que estava no markdown da Aula 03 virando recurso CloudFormation deployado.

17.1 EventBridge — o roteador

EventBridge é o bus de eventos serverless da AWS. Tem um bus default e suporta buses customizados. Cada bus aceita rules; cada rule combina um EventPattern (filtro sobre campos do evento) ou um ScheduleExpression (cron) com targets (Lambda, SSM, SNS, Step Functions, ECS, etc.).

Um InputTransformer formata o payload antes de mandar para o target — útil porque o SSM Document espera campos específicos. Dead Letter Queue por target (SQS) captura falhas; sem DLQ, falhas viajam para o vazio.

17.2 SSM — o atuador

AWS Systems Manager (SSM) é o canivete suíço para operação de fleet. Quatro capabilities cobrem o uso da Aula 07. Run Command — execução pontual de Document em uma ou várias EC2. State Manager — aplica Document de forma idempotente em janelas recorrentes. Automation — workflow multi-passo com aprovação humana opcional, branching, loops. Session Manager — acesso interativo seguro à EC2 sem SSH bastion.

Toda execução vai para o CloudTrail (audit) e os logs do Run Command vão para CloudWatch Logs em /olist/runbooks. Auditável por padrão.

18. SSM Documents — tipos e quando usar

Document é YAML (preferido) ou JSON declarativo. Vive no SSM como recurso versionado — toda alteração gera nova versão imutável. Cinco tipos cobrem 95% dos casos do Olist.

TipoQuando usarExemplo no Olist
Command Executar script ad-hoc em uma ou várias EC2 RetryEtlOnce — retry idempotente do ETL
Automation Workflow multi-passo com branching e aprovação FreezeDeployOnBudgetBurn — freeze quando budget esgota
Session Acesso interativo via Session Manager Debug de Postgres sem SSH público
Package Distribuir software para fleet Distribuir agente CW unificado
Policy State Manager — configuração contínua idempotente Garantir agent SSM sempre ligado

18.1 Estrutura de um Document Command

schemaVersion: "2.2"
description: |
  RUNBOOK-01 - Retry idempotente do ETL Olist quando ALARM-01
  (FreshnessBreach) entra em ALARM. Stop apos 1 tentativa: se
  ainda falhar, sobe para humano via SNS.
parameters:
  RunId:
    type: String
    default: "auto-{{global:DATE_TIME}}"
mainSteps:
  - action: aws:runShellScript
    name: retryEtl
    inputs:
      runCommand:
        - set -euo pipefail
        - cd /opt/olist
        - echo "[runbook] retry RunId={{ RunId }}"
        - python -m etl.main --idempotent --run-id "{{ RunId }}"
        - echo "[runbook] done"

18.2 AWS-RunShellScript — o coringa

Existe um Document built-in chamado AWS-RunShellScript que aceita um array de comandos como parâmetro. É a forma mais simples de mandar comandos arbitrários para EC2 — sem precisar criar Document customizado. Tem trade-off: comandos não ficam versionados como recurso, só na command history do Run Command. Bom para experimentação; ruim para runbook recorrente.

19. Padrões de auto-remediação

A auto-remediação não é improviso — tem padrões clássicos. Cada um tem trigger, ação, salvaguarda e critério de stop. Reconhecer o padrão certo evita reescrever runbook de cabeça toda vez.

PadrãoTriggerAçãoStop
Restart Heartbeat falhou systemctl restart 3 retries em 15 min
Scale out Latência subiu ASG +1 instância Cap absoluto + budget
Circuit breaker Erros 5xx > threshold Param flag etl_paused=true Cooldown e half-open
Freeze deploy Error budget esgotou Tag deploy=frozen bloqueia CI Humano remove tag após postmortem
Rollback SLO degradado pós-deploy Deploy versão anterior 1 vez automático, 2ª é humano
Quarentena Linhas inválidas > 0,5% Rota para tabela DLQ Backpressure no upstream

19.1 Idempotência é obrigatória

Toda ação de auto-remediação precisa ser idempotente. Se o EventBridge dispara duas vezes (acontece em retry transitório), o resultado precisa ser igual ao de uma execução só. No Olist, o --idempotent do etl.main garante isso por upsert em vez de insert, e por chave de idempotência baseada em RunId.

19.2 Salvaguardas

Toda ação carrega salvaguardas: cap absoluto (não escala além de N), cooldown (não repete em menos de M minutos), kill switch manual (humano consegue desligar a regra com 1 comando). Sem salvaguarda, a auto-remediação vira loop de pesadelo — cada conserto piora a situação.

20. Gestão distribuída de fleet — Inventory, Patch Manager, State Manager

SSM agente já vem em quase toda AMI moderna (Amazon Linux 2023, Amazon Linux 2, Ubuntu desde 18.04). Sem agente extra, você ganha visibilidade, patching e configuração contínua de toda a frota. As capabilities a seguir são as mais usadas em operação real.

CapabilityO que entregaOlist — uso
Inventory Visibilidade de software, network, processos por host Confirma que EC2 ETL roda Python 3.11 e não 3.9
Patch Manager Patching declarativo via baseline + maintenance window Aplica security patch domingo 03h sem clique
State Manager Associations idempotentes em schedule Garante CW agent sempre ligado, mesmo após reboot
Run Command Execução pontual em fleet selecionada por tag Executa RetryEtlOnce em EC2 com tag:Role=etl
Compliance Relatório agregado de drift de configuração e patch Alerta se EC2 perder encryption-at-rest

20.1 State Manager Association — por que importa

Sem State Manager, configuração que você aplicou na EC2 some no próximo reboot ou na próxima recriação por ASG. State Manager garante que toda EC2 com tag Role=etl tenha o CW agent rodando — checa a cada N minutos e re-aplica se sumir. É a versão SSM de "convergência" do Puppet/Chef, sem agente extra.

Passo 4 — skill wire_emf_logging — EMF no ETL

O ETL para de emitir log de texto solto. Passa a emitir EMF JSON. CloudWatch reconhece e publica métricas sem PutMetricData. RowsProcessed, Latency e InvalidRows ganham vida. 

# Em src/etl/main.py
import json, time, logging
log = logging.getLogger("olist")

def emit_emf(stage: str, rows: int, latency_s: float,
             invalid: int, run_id: str):
    payload = {
      "_aws": {
        "Timestamp": int(time.time() * 1000),
        "CloudWatchMetrics": [{
          "Namespace": "OlistETL",
          "Dimensions": [["Stage", "Env"]],
          "Metrics": [
            {"Name": "RowsProcessed", "Unit": "Count"},
            {"Name": "Latency",       "Unit": "Seconds"},
            {"Name": "InvalidRows",   "Unit": "Count"}
          ]
        }]
      },
      "Stage": stage, "Env": "lab",
      "RowsProcessed": rows, "Latency": latency_s, "InvalidRows": invalid,
      "RunId": run_id,
    }
    log.info(json.dumps(payload))

        exports.handler = async () => {
          await synthetics.executeHttpStep(
            'grafana-health',

Passo 6 — runbook RetryEtlOnce e dry-run

O Document é simples e idempotente. Roda no host do ETL via Run Command. Bem comentado — runbook mal escrito é runbook não rodado. Antes de habilitar a regra do EventBridge, executamos dry-run manualmente para validar parâmetros, role, conectividade e log. 

# runbooks/RetryEtlOnce.yaml
schemaVersion: "2.2"
description: |
  RUNBOOK-01 - Retry idempotente do ETL Olist quando ALARM-01
  (FreshnessBreach) entra em ALARM. Stop apos 1 tentativa: se
  ainda falhar, sobe para humano via SNS.
parameters:
  RunId:
    type: String
    default: "auto-{{global:DATE_TIME}}"
mainSteps:
  - action: aws:runShellScript
    name: retryEtl
    inputs:
      runCommand:
        - set -euo pipefail
        - cd /opt/olist
        - echo "[runbook] retry RunId={{ RunId }}"
        - python -m etl.main --idempotent --run-id "{{ RunId }}"
        - echo "[runbook] done"

Executar dry-run manualmente: 

INSTANCE=$(aws ec2 describe-instances \
  --filters "Name=tag:Role,Values=etl" \
  --query "Reservations[].Instances[].InstanceId" --output text)

aws ssm send-command \
  --document-name "RetryEtlOnce" \
  --instance-ids $INSTANCE \
  --parameters 'RunId=dryrun-2026-05-28' \
  --comment "Aula 07 - dry-run do runbook"

# Esperar status e ver log
CMDID=$(aws ssm list-commands --query "Commands[0].CommandId" --output text)
aws ssm get-command-invocation \
  --command-id $CMDID --instance-id $INSTANCE \
  --query "{Status:Status, Stdout:StandardOutputContent}"

# Soh DEPOIS, habilitar a regra do EventBridge
aws events enable-rule --name olist-alarm-retry-etl

Passo 7 — simulação de incidente (latência alta)

Reusamos o chaos da Aula 06 (Toxiproxy + SSM). Injetamos latência no Postgres, observamos o alarm subir, vemos o runbook executar e registramos a timeline. Kill switch sempre claro — janela máxima 10 min.

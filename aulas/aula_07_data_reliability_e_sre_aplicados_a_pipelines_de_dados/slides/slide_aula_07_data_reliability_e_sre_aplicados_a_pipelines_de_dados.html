Tese da aula

SRE é o pipeline se observando e se reparando dentro do error budget

Pager humano só quando o budget esgota. Tudo abaixo disso é responsabilidade da malha fechada — e da malha fechada em código, versionada via CloudFormation, com runbook validado em dry-run.

Cinco regras da Aula 07
  1. Sem console AWS. Tudo via CFN e CLI — revisável em diff.
  2. Alarm sem runbook é alarm-fadiga. Toda regra AWS::CloudWatch::Alarm aponta para um runbook.
  3. Runbook sempre tem dry-run antes do enable. Auto-remediação cega quebra mais do que conserta.
  4. Toda métrica tem unidade e janela. Métrica sem unidade é número decorativo.

SRE · Pilares

Pilares SRE — Beyer, Jones, Petoff, Murphy (Google)

Os mesmos pilares do livro azul, mapeados para um pipeline de dados. Nenhum é folclore: cada um é prática operacional concreta.

SLI / SLO / Error budget

  • SLI — indicador medido (freshness, completeness).
  • SLO — meta interna por janela.
  • Budget — 1 − SLO; quando esgota, freeze.

Toil

  • Trabalho manual repetitivo, sem valor durável.
  • Meta clássica do Google: < 50% do tempo do time.
  • Reduzir toil ≡ automatizar runbook.

Incident response

  • Roles definidos: IC, ops, comms.
  • Status page interna, escalation clara.
  • Comunicação assíncrona > reuniões de pânico.

Postmortem blameless

  • Timeline factual, não narrativa heróica.
  • Contributing factors, não culpado.
  • Action items com dono, prazo, status.

Releases progressivos

  • Canário, blue/green, feature flags.
  • Reduz blast radius de mudança.
  • Combina com error budget para gating.

Automação no lugar de heroísmo

  • Runbook executável bate procedimento em PDF.
  • SSM + EventBridge resolvem 80% dos casos rotineiros.
  • Pager só para o que realmente exige humano.

SRE · Toil

Toil — o inimigo silencioso

Toil não é "trabalho difícil". É trabalho manual, repetitivo, automatizável e sem valor durável. Quanto mais toil, menos engenharia. Quanto mais runbook automatizado, menos toil.

Sintomas de toil no pipeline
  • Restart manual de ETL todas as manhãs.
  • SSH no Postgres para limpar tabela de quarentena.
  • Subir agente de log à mão em cada nova EC2.
  • Reabrir alarm que toca no fim de semana.
  • Copiar-colar comando de uma página de Notion.
Cura
  • Cada toil identificado vira ticket de automação.
  • Runbook escrito em SSM Document, não em Notion.
  • State Manager garante configuração contínua.
  • Patch Manager aplica patches sem clicar.
  • Meta: < 50% do tempo do time em toil.
SRE · Incident Response

Roles de IR — quem faz o quê

Em incidente, papéis explícitos previnem caos. Mesmo em time pequeno, separar Incident Commander de quem está com a mão na massa.

RoleResponsabilidadeNão faz
IC — Incident CommanderCoordena, decide, garante comunicação fluindo.Não debuga código durante o incidente.
Ops — Operations LeadMão na massa: investiga logs, executa runbooks, aplica fix.Não comunica externamente.
Comms — CommunicationsAtualiza status page, escreve updates a cada 15–30 min.Não toma decisões técnicas.
ScribeRegistra timeline em tempo real, base do postmortem.Não opera, só observa e anota.
SRE · Postmortem

Postmortem blameless — estrutura mínima

Documento curto. 1 página é melhor que 5. O ponto não é narrativa — é aprendizado acionável.

Cabeçalho

  • ID do incidente, severidade, datas.
  • IC, Ops, Comms.
  • Resumo de 3 linhas.

Timeline

  • Eventos com timestamp UTC.
  • Detect · ack · mitigate · resolve.
  • MTTR e error budget consumido.

Contributing factors

  • Falhas técnicas plausíveis.
  • Falhas de processo.
  • Sinais que foram ignorados.

What went well

  • O que funcionou e por quê.
  • Não é elogio — é insumo para repetir.

Action items

  • 1 frase, 1 dono, 1 prazo, 1 link de ticket.
  • Sem AI ≡ sem ação.

Blameless

  • Pessoas agem racionalmente dado contexto.
  • Foco em sistema que permitiu o erro.
  • Linguagem despessoal: "o deploy foi feito".
Observabilidade

Três pilares — logs, métricas, traces

Cada um responde uma pergunta diferente. Pipeline maduro tem os três e os correlaciona por request id ou execution id.

Logs

  • Eventos discretos com contexto.
  • Pergunta: o que aconteceu aqui?
  • CW Logs: log groups, streams, retention.
  • JSON estruturado > texto solto.

Métricas

  • Séries numéricas com dimensões.
  • Pergunta: como está, agora e antes?
  • CW Metrics: namespace + dimensions.
  • Agregação 1 / 5 / 15 min — escolher conscientemente.

Traces

  • Cadeia causal entre componentes.
  • Pergunta: onde está o gargalo?
  • X-Ray: segments, subsegments, anotações.
  • Sampling agressivo para custo.
CloudWatch · Logs

CW Logs — pipeline padrão

Log group por componente, retenção definida, metric filter onde precisar de alarm e subscription filter onde precisar de processamento.

Anatomia
  • Log group — container nomeado por aplicação.
  • Log stream — uma instância ou execução.
  • Retention policy — em dias (1, 7, 30, 90, 365).
  • Metric filter — pattern → CW metric.
  • Subscription filter — pattern → Lambda/Kinesis.
Padrão Olist
  • /olist/etl — runs do ETL.
  • /olist/postgres — logs do Postgres.
  • /olist/grafana — access log do dashboard.
  • /olist/runbooks — execução de SSM Documents.
  • Retention 30 dias — budget do Learner Lab.

CloudWatch · Alarms

Alarms — threshold, anomaly, composite

Três variantes. Estados OK, ALARM, INSUFFICIENT_DATA. Cada alarm aponta para um runbook — sem exceção.

Threshold

  • Compara estatística contra valor fixo.
  • Ex.: Freshness p95 > 30min por 2 datapoints de 5min.
  • Simples, defensável, auditável.

Anomaly detection

  • Modelo aprende baseline da métrica.
  • Ex.: throughput cai 3σ abaixo do esperado.
  • Útil para sazonais; cuidado com falsos positivos.

Composite

  • Combina alarms via expressão booleana.
  • Ex.: Freshness E Completeness violados.
  • Reduz alarm-fadiga em incidentes correlacionados.
CloudWatch · Dashboards

Dashboards — widget JSON e mathExpression

Dashboard é JSON. Vai versionado no repo, junto do CFN. Dashboard sem dono morre — cada um da turma fica com o seu.

Widgets úteis
  • metric — gráfico de série temporal.
  • singleValue — KPI grande.
  • log — tabela de Logs Insights.
  • alarm — status visual.
  • text — markdown com link de runbook.
mathExpression
  • SLI derivado: m1 / m2 * 100.
  • Ex.: completeness = RowsProcessed / RowsExpected * 100.
  • Permite alarm em SLI calculado.
  • Versão única da verdade entre dashboard e alarm.
CloudWatch · Logs Insights

Logs Insights — query estruturada em logs

Linguagem SQL-like sobre log groups. Bom para investigação durante incidente. Queries salvas viram parte do runbook.

fields @timestamp, @message, RunId, Stage, RowsProcessed, InvalidRows
| filter Stage = "raw"
| filter @timestamp > ago(1h)
| stats sum(RowsProcessed) as total_rows,
        sum(InvalidRows)   as total_invalid,
        max(Latency)       as worst_latency
        by RunId
| sort worst_latency desc
| limit 20
Tracing · X-Ray

X-Ray — tracing distribuído

No Learner Lab funciona via SDK em EC2. Vamos instrumentar o ETL para enxergar onde o tempo é gasto entre read S3, transform, write Postgres.

Conceitos
  • Trace — árvore completa de uma request.
  • Segment — trabalho num componente.
  • Subsegment — chamada interna.
  • Annotation — chave indexada (RunId).
  • Metadata — chave não indexada.
Sampling no Learner Lab
  • Sampling 1.0 só em dev curto.
  • Default 5% + reservoir 1/s evita estouro de custo.
  • Sampling rule via CFN: AWS::XRay::SamplingRule.
  • Anotação RunId facilita correlacionar trace e log.
CloudWatch · Synthetics

Synthetics — canários sintéticos

Lambda gerenciado que executa script (Node ou Python) a cada N minutos. Saída vira métrica e screenshot. É o primeiro SLI black box do dashboard — o ponto de vista do cliente.

Canários úteis para o Olist
  • Heartbeat HTTP — bate em /api/health do Grafana a cada 5 min.
  • API canário — consulta um KPI do dashboard e valida shape do JSON.
  • Visual canário — screenshot do dashboard, diff visual contra baseline.
  • Falhar 2 datapoints → alarm composite junto com SLO de freshness.
Closed loop · Núcleo

EventBridge + SSM — o coração do closed loop

Alarm CW dispara evento no EventBridge. EventBridge chama SSM Run Command. SSM executa script de remediação na EC2. Tudo em código, tudo auditável, tudo sem clique.

EventBridge — o roteador
  • Bus default + buses customizados.
  • Rule com EventPattern ou ScheduleExpression.
  • Targets: SSM, Lambda, SNS, Step Functions, ECS.
  • InputTransformer formata payload para o target.
  • DLQ por target (SQS) para falha.
SSM — o atuador
  • Documents: AWS-RunShellScript, AWS-RunPowerShellScript, customizados.
  • Run Command: imperativo, uma vez.
  • State Manager: idempotente, recorrente.
  • Automation: workflow multi-passo com aprovação.
  • Audit em CloudTrail + log do Run Command em CW.
Closed loop · Sequência

Alarm → EventBridge → SSM → aplicação

A malha fechada na sequência real. Cinco atores, cada um faz pouco, ninguém faz nada que não esteja no template CFN versionado.

Closed loop · Padrões

Padrões de auto-remediation

Cinco padrões cobrem o repertório clássico. Cada um tem trigger, ação, salvaguarda e critério de stop.

Restart serviço

  • Trigger: heartbeat falhou.
  • Ação: systemctl restart.
  • Stop: 3 retries em 15 min.

Scale out

  • Trigger: latência subiu.
  • Ação: ASG +1 instância.
  • Stop: cap absoluto + budget.

Circuit breaker

  • Trigger: erros 5xx > threshold.
  • Ação: param flag etl_paused=true.
  • Stop: cooldown e half-open.

Freeze deploy

  • Trigger: error budget esgotou.
  • Ação: tag deploy=frozen bloqueia CI.
  • Stop: humano remove tag após postmortem.

Rollback

  • Trigger: SLO degradado pós-deploy.
  • Ação: deploy versão anterior.
  • Stop: 1 vez automático, 2ª é humano.

Quarentena

  • Trigger: linhas inválidas > 0.5%.
  • Ação: rota para tabela dlq.
  • Stop: backpressure no upstream.
SSM · Fleet

Gestão distribuída — Inventory, Patch, State Manager

SSM agente já vem em quase toda AMI moderna. Sem agente extra, você ganha visibilidade, patching e configuração contínua de toda a frota.

Capabilities
  • Inventory — pacotes, network, processos.
  • Patch Manager — baselines + maintenance window.
  • State Manager — associations idempotentes.
  • Run Command — execução pontual em fleet.
  • Compliance — relatório de drift agregado.
Aplicação no Olist
  • Inventory mostra que EC2 ETL está com Python certo.
  • Patch Manager aplica security patch domingo 03h.
  • State Manager garante CW agent rodando.
  • Run Command executa runbook RetryEtlOnce.
  • Config Rule alerta se EC2 perder encryption.

Prática · Passo 4

Skill wire_emf_logging — EMF no ETL

O ETL para de emitir log de texto solto. Passa a emitir EMF JSON. CW reconhece e publica métricas sem PutMetricData. RowsProcessed, Latency e InvalidRows ganham vida.

# Em src/etl/main.py
import json, time, logging
log = logging.getLogger("olist")

def emit_emf(stage: str, rows: int, latency_s: float, invalid: int, run_id: str):
    payload = {
      "_aws": {
        "Timestamp": int(time.time() * 1000),
        "CloudWatchMetrics": [{
          "Namespace": "OlistETL",
          "Dimensions": [["Stage", "Env"]],
          "Metrics": [
            {"Name": "RowsProcessed", "Unit": "Count"},
            {"Name": "Latency",       "Unit": "Seconds"},
            {"Name": "InvalidRows",   "Unit": "Count"}
          ]
        }]
      },
      "Stage": stage, "Env": "lab",
      "RowsProcessed": rows, "Latency": latency_s, "InvalidRows": invalid,

    }

Prática · Passo 6

Runbook RetryEtlOnce — SSM Document

Document Command simples e idempotente. Roda no host do ETL via Run Command. Bem comentado — runbook mal escrito é runbook não rodado.

# runbooks/RetryEtlOnce.yaml
schemaVersion: "2.2"
description: |
  RUNBOOK-01 - Retry idempotente do ETL Olist quando ALARM-01
  (FreshnessBreach) entra em ALARM. Stop apos 1 tentativa: se
  ainda falhar, sobe para humano via SNS.
parameters:
  RunId:
    type: String
    default: "auto-{{global:DATE_TIME}}"
mainSteps:
  - action: aws:runShellScript
    name: retryEtl
    inputs:
      runCommand:
        - set -euo pipefail
        - cd /opt/olist
        - echo "[runbook] retry RunId={{ RunId }}"
        - python -m etl.main --idempotent --run-id "{{ RunId }}"
        - echo "[runbook] done"
Prática · Passo 7

Dry-run do runbook antes do enable

Regra dura: nenhum runbook entra em produção sem dry-run. Validamos parâmetros, role, conectividade e log antes da regra do EventBridge ficar com State=ENABLED.

# Executar runbook MANUALMENTE primeiro
INSTANCE=$(aws ec2 describe-instances \
  --filters "Name=tag:Role,Values=etl" \
  --query "Reservations[].Instances[].InstanceId" --output text)

aws ssm send-command \
  --document-name "RetryEtlOnce" \
  --instance-ids $INSTANCE \
  --parameters 'RunId=dryrun-2026-05-28' \
  --comment "Aula 07 - dry-run do runbook"

# Esperar status e ver log
CMDID=$(aws ssm list-commands --query "Commands[0].CommandId" --output text)
aws ssm get-command-invocation \
  --command-id $CMDID --instance-id $INSTANCE \
  --query "{Status:Status, Stdout:StandardOutputContent}"

# Soh DEPOIS, habilitar a regra do EventBridge
aws events enable-rule --name olist-alarm-retry-etl

Contexto

O pipeline funciona… até não funcionar

Pipelines em produção falham silenciosamente. O time descobre pelo usuário de negócio — não pelo monitoramento.

Sintomas típicos
  • Dashboard com dados de ontem.
  • Tabela Gold com metade das linhas.
  • Job verde no orquestrador, dado errado no Athena.
  • Alerta que ninguém sabe resolver às 03h da manhã.
Raiz do problema
  • Sem SLO — ninguém sabe o que é “bom o suficiente”.
  • Sem SLI — não medimos a saúde real do dado.
  • Sem retry/DLQ — falha transitória vira incidente.
  • Sem runbook — cada incidente é resolvido do zero.
SRE · Origem

Site Reliability Engineering (Google, 2003)

Ben Treynor criou a disciplina de SRE no Google: “o que acontece quando você pede a um engenheiro de software para projetar uma equipe de operações”.

Princípios fundadores
  1. Embrace risk — 100% de disponibilidade não é meta, é desperdício.
  2. Service Level Objectives — metas mensuráveis negociadas com o negócio.
  3. Eliminate toil — automatizar trabalho manual e repetitivo.
  4. Monitoring — logs, métricas e traces como base da operação.
  5. Release engineering — deploys seguros, graduais e reversíveis.
  6. Blameless postmortem — aprender com falhas sem culpar pessoas.

Pilar 1 · SLI

Service Level Indicator

Métrica quantitativa de um aspecto do serviço. Sempre uma razão entre eventos bons e eventos totais.

Fórmula canônica 
SLI = eventos_bons / eventos_válidos_totais
  • Exemplo web: % requests com status < 500 em 5xx+2xx.
  • Exemplo dados: % execuções do job que terminaram antes das 06h.
  • Exemplo dados: % linhas com order_id não nulo.
Pilar 2 · SLO

Service Level Objective

Meta interna sobre um SLI, em uma janela de tempo. Negociada entre engenharia e produto.

Anatomia de um SLO
  • SLI: a métrica observada.
  • Alvo: 99%, 99.9%, 99.95%…
  • Janela: 30 dias corridos, trimestre.
  • Escopo: qual dataset ou pipeline.
Exemplos pipeline Olist
  • 99% das execuções do silver_to_gold terminam antes das 06h00.
  • 99.5% das linhas na gold.orders têm order_id não nulo.
  • 95% dos eventos Kinesis são processados em < 60s.

Error Budget

O orçamento de falhas

Se o SLO é 99%, existem 1% de falhas aceitáveis. Esse 1% é o error budget — e pode ser gasto.

Como usar na operação
  • Budget sobrando → libera deploy, experimentação, refactor agressivo.
  • Budget queimando rápido → congela features, foca em estabilidade.
  • Budget zerado → freeze total até recuperar o SLO.
  • Budget mensal de 30d com SLO 99% = 0.01 × 30d = 7.2 horas de indisponibilidade.
SLIs de pipeline · 1/5

Freshness — o dado está atual?

Tempo decorrido desde a última atualização bem-sucedida do dataset.

Como medir
  • NOW() - MAX(updated_at) no dataset.
  • Timestamp da última execução de sucesso do job.
  • Metadados do Glue Catalog ou Iceberg.
Exemplo Olist
  • SLO: 99% dos dias a gold.orders está atualizada antes das 07h.
  • Alarme: freshness > 8h dispara SNS.
  • Dashboard mostra hora da última atualização.
SLIs de pipeline · 2/5

Completeness — tudo chegou?

Razão entre linhas recebidas e linhas esperadas. Detecta perda silenciosa de dados.

Estratégias
  • Volume esperado: compara contagem com média dos últimos 7 dias (±20%).
  • Reconciliação de fonte: COUNT(raw) = COUNT(silver) = COUNT(gold).
  • Nulos em chave: % order_id IS NULL deve ser 0.
  • Particionamento: toda partição de hora/dia existe?
SLIs de pipeline · 3/5

Accuracy — os valores fazem sentido?

Percentual de valores que passam em regras de validação de negócio.

Regras típicas
  • price > 0 e price < 100_000.
  • order_status IN ('created','paid','shipped',...).
  • customer_city em lista de municípios válidos.
  • Timestamps no futuro = inválidos.
Onde checar
  • Em cada job silver → gold.
  • Via Glue Data Quality (DQDL).
  • Via Great Expectations expectations.
  • Via dbt tests (unique, not_null, accepted_values).
SLIs de pipeline · 4/5

Consistency — reconciliação entre fontes

O dado em sistemas diferentes conta a mesma história?

Exemplos práticos
  • SUM(gold.orders.price) == SUM(raw.orders.price) para o mesmo dia.
  • COUNT(DISTINCT customer_id) em CRM = em DW (tolerância < 0.1%).
  • Receita no Redshift bate com extrato no sistema transacional.
  • Cardinalidade entre tabelas fato e dimensão.
SLIs de pipeline · 5/5

Availability — o pipeline rodou?

Percentual de janelas agendadas em que o pipeline produziu resultado dentro do prazo.

Métrica
  • execuções_no_prazo / execuções_agendadas.
  • “No prazo” = sucesso antes do deadline do SLO.
  • Considera retries: se retry passou, conta como sucesso.
Fonte no AWS
  • Step Functions execution status.
  • EventBridge scheduled rule.
  • CloudWatch Logs do Glue Job.

Data Quality

Frameworks mais usados

Expressão declarativa de expectativas sobre o dado, com relatório de violações.

Great Expectations

  • Python, open-source, Data Docs.
  • Expectations em JSON/YAML.
  • Integra com Airflow, dbt, Spark.

Deequ / PyDeequ

  • Biblioteca da AWS sobre Spark.
  • Constraints e metrics repositories.
  • Escala para TB/PB.

dbt tests

  • Nativo no dbt (SQL).
  • unique, not_null, relationships.
  • Testes customizados via macros.

AWS Glue Data Quality

  • Serviço gerenciado (2023+).
  • DSL própria: DQDL.
  • Roda dentro de jobs Glue e no Catalog.
AWS Glue Data Quality

DQDL — Data Quality Definition Language

Linguagem declarativa para escrever rulesets. Roda como transform no Glue ETL ou como evaluator no Catalog.

Rules = [
  RowCount between 1000 and 5000000,
  IsPrimaryKey "order_id",
  IsComplete "order_id",
  IsComplete "customer_id",
  IsUnique "order_id",
  ColumnValues "order_status" in ["created","paid","shipped","delivered","canceled"],
  ColumnValues "price" > 0,
  ColumnValues "price" < 100000,
  Completeness "customer_city" > 0.98,
  ColumnCorrelation "price" "freight_value" > 0.1
]

Resiliência

Padrões essenciais

Falhas transitórias não devem virar incidentes. Desenhe o pipeline para absorver falha.

Retry + Backoff

  • Tentar de novo com espera crescente.
  • Exponencial: 1s → 2s → 4s → 8s.
  • Com jitter para evitar thundering herd.

Circuit Breaker

  • Se 5 falhas em 30s, abre o circuito.
  • Rejeita chamadas por N segundos.
  • Tenta half-open para recuperar.

Dead Letter Queue (DLQ)

  • Mensagens que falham N vezes vão para uma fila separada.
  • Evita reprocessamento infinito.
  • Alvo de análise manual / reprocesso assistido.

Idempotência + Checkpoint

  • Executar 2x produz o mesmo resultado.
  • Chave idempotente por registro.
  • Checkpoint de progresso para retomar.
Orquestração

DAG com políticas de resiliência

O orquestrador define a ordem, mas também o que fazer quando um passo falha.

Decisões por tarefa
  • Retry policy: quantas vezes e com qual intervalo.
  • Timeout: quando desistir de uma execução travada.
  • On failure: notificar? pular? abortar? fallback?
  • On success: qual caminho seguir na DAG.
  • Idempotência: retry não pode duplicar escrita.
AWS Step Functions

Estados principais

Linguagem ASL (Amazon States Language) — JSON declarativo para máquinas de estado.

EstadoUso
TaskExecuta uma ação (Lambda, Glue Job, Crawler, DynamoDB…)
ChoiceRamifica fluxo com base em condição.
ParallelExecuta branches simultaneamente.
MapItera sobre coleção (ex.: por partição).
WaitPausa por tempo fixo ou timestamp.
RetryPolítica de tentativa em erro (MaxAttempts, BackoffRate).
CatchDesvia para um estado alternativo em erro.
Pass / Succeed / FailEstados de controle e terminação.
Observabilidade na AWS

CloudWatch

Ferramenta central de métricas, logs e alarmes dos serviços AWS.

Alarms

  • Threshold ou anomaly detection.
  • Estado: OK, ALARM, INSUFFICIENT_DATA.
  • Ação: SNS, Auto Scaling, Lambda.

Dashboards

  • Painel visual de métricas.
  • Widgets: linha, número, log table.
  • Compartível entre contas.

Logs Insights

  • Query language sobre CloudWatch Logs.
  • Filtros, agregações, visualizações.
  • fields @timestamp, @message | filter @message like /ERROR/.

Métricas do Glue

  • glue.driver.aggregate.elapsedTime
  • glue.driver.aggregate.numFailedTasks
  • glue.driver.BlockManager.disk.diskSpaceUsed_MB
Cultura SRE

Runbook e postmortem blameless

Falhas são inevitáveis. Aprender com elas é o que diferencia o time.

Runbook
  • Passo a passo para tratar um alerta específico.
  • Comandos prontos, links para dashboards, dono.
  • Testado em simulações (game day).
Postmortem blameless
  • Timeline do incidente.
  • Root cause (5 whys).
  • Ações corretivas com dono e prazo.
  • Foco em sistema, não em pessoas.

Lab 1 · SNS

Tópico data-pipeline-alerts

Passo a passo
  1. Console AWS → Simple Notification Service → Create topic.
  2. Type: Standard. Name: data-pipeline-alerts. Create.
  3. Na tela do tópico → Create subscription.
  4. Protocol: Email. Endpoint: seu e-mail real (o AWS Student pode limitar e-mails não confirmados).
  5. Abra o e-mail e clique em Confirm subscription.
  6. No console, publique uma mensagem de teste para validar a recepção.
  7. Copie o Topic ARN: será usado nos alarmes e no Step Functions.
Lab 2 · CloudWatch Alarms

Alarmes no Glue Job da Aula 05

Alarm A — job lento

  • Métrica: glue.driver.aggregate.elapsedTime.
  • Namespace: AWS/Glue.
  • Statistic: Maximum, period 1 min.
  • Threshold: > 600000 ms (10 min).
  • Ação: notify data-pipeline-alerts.

Alarm B — task falhou

  • Métrica: glue.driver.aggregate.numFailedTasks.
  • Statistic: Sum, period 1 min.
  • Threshold: > 0.
  • Datapoints: 1/1.
  • Ação: notify data-pipeline-alerts.
Lab 3 · SQS + DLQ

Reforçando o consumer da Aula 04

Passo a passo
  1. Console → SQS → Create queue. Name: olist-events-dlq. Type: Standard.
  2. Criar outra fila: olist-events-queue. Em Dead-letter queue, habilite e selecione olist-events-dlq.
  3. Defina Maximum receives = 3 (redrive policy).
  4. Lambda da Aula 04 → Configuration → Triggers → Add: SQS — olist-events-queue.
  5. Ajuste a função para fazer raise quando o payload for inválido (simula falha).
  6. Envie 10 mensagens na olist-events-queue, sendo 2 inválidas.
  7. Após 3 tentativas, as 2 inválidas aparecem em olist-events-dlq.

Lab 5 · Step Functions

Workflow olist-pipeline

Orquestra Crawler, jobs Glue e check de DQ. Retry e Catch em cada Task crítica.

Fluxo de estados
  1. StartCrawler — Crawler olist_raw.
  2. RunRawToSilver — Glue Job, Retry 3× backoff 2.
  3. RunSilverToGold — Glue Job, Retry 3× backoff 2.
  4. CheckDataQuality — Evaluate Glue DQ ruleset.
  5. Choice — DQ pass → NotifySuccess. DQ fail → LogIncident + NotifyFailure.
Lab 5 · ASL

Retry e Catch em JSON

"RunRawToSilver": {
  "Type": "Task",
  "Resource": "arn:aws:states:::glue:startJobRun.sync",
  "Parameters": { "JobName": "olist_raw_to_silver" },
  "Retry": [{
    "ErrorEquals": [
      "Glue.ConcurrentRunsExceededException",
      "States.TaskFailed"
    ],
    "IntervalSeconds": 10,
    "MaxAttempts": 3,
    "BackoffRate": 2.0
  }],
  "Catch": [{
    "ErrorEquals": ["States.ALL"],
    "Next": "NotifyFailure",
    "ResultPath": "$.error"
  }],
  "Next": "RunSilverToGold"
}

ASL completo está no material desta aula, com todos os estados do olist-pipeline.

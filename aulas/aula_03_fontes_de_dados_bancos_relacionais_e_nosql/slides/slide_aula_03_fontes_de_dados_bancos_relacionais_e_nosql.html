Introducao

Fontes Transacionais

Nossos Data Lakes nascem da necessidade de analisar dados que servem nativamente a sistemas operacionais transacionais do dia-a-dia.

Bancos Relacionais (SQL)
  • Estruturados, normalizados, focados em integridade referencial.
  • O processamento analitico (OLAP) pesa em bancos focados em transacoes (OLTP). Precisamos leva-los para um Data Warehouse ou Data Lake!
Servico AWS

Amazon RDS

Caracteristicas
  • Servico de banco de dados relacional (PaaS) totalmente gerenciado.
  • Motores suportados: PostgreSQL, MySQL, MariaDB, Oracle, SQL Server.
  • Amazon Aurora: Motor Enterprise criado nativamente na nuvem.
  • Como os dados saem do RDS? Utilizamos pipelines de CDC (Change Data Capture) como o servico AWS Database Migration Service (DMS).
NoSQL

Por que Bancos NoSQL?

Quando escalar verticalmente (Adicionar RAM/CPU) um banco relacional nao e suficiente.

Vantagens
  • Performance previsivel em milissegundos, independente da quantidade de dados (O(1)).
  • Escalabilidade horizontal infinita (Sharding) tratada de forma transparente para nos da Engenharia de Dados.
  • Esquema de dados hiper flexivel.
Servico AWS

Amazon DynamoDB (Chave-Valor)

Caracteristicas
  • Banco de dados NoSQL chave-valor e documento totalmente gerenciado (Serverless).
  • Replicacao Multi-AZ automatica.
  • Modelagem foca na arquitetura Single-Table Design (Tabelas unicas contendo multiplos tipos de registros de sua aplicacao).
  • Completamente baseado em cobrancas por Requisicoes em massa (RCU/WCU).
Arquitetura

Componentes do DynamoDB

Partition Key (PK)

  • Como os dados serao agrupados e espalhados nas AZs. OBRIGATORIA.

Sort Key (SK)

  • Como ordenamos dados dentro da mesma PK. OPCIONAL.

GSI (Global Secondary Index)

  • A verdadeira magia do DynamoDB. Criar projecoes indexadas secundarias para consultar o dado de diferentes perspectivas de acesso.

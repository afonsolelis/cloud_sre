Introdução: Fontes Transacionais

Nossos Data Lakes nascem da necessidade de analisar dados que servem nativamente a sistemas operacionais transacionais do dia-a-dia.

  • Bancos Relacionais (SQL): Estruturados, normalizados, focados em integridade referencial.
  • O processamento analítico (OLAP) pesa em bancos focados em transações (OLTP). Precisamos levá-los para um Data Warehouse ou Data Lake!

Amazon RDS

  • Serviço de banco de dados relacional (PaaS) totalmente gerenciado.
  • Motores suportados: PostgreSQL, MySQL, MariaDB, Oracle, SQL Server.
  • Amazon Aurora: Motor Enterprise criado nativamente na nuvem.
  • Como os dados saem do RDS? Utilizamos pipelines de CDC (Change Data Capture) como o serviço AWS Database Migration Service (DMS).

Por que Bancos NoSQL?

Quando escalar verticalmente (Adicionar RAM/CPU) um banco relacional não é suficiente.

  • Performance previsível em milissegundos, independente da quantidade de dados (O(1)).
  • Escalabilidade horizontal infinita (Sharding) tratada de forma transparente para nós da Engenharia de Dados.
  • Esquema de dados hiper flexível.

Amazon DynamoDB (Chave-Valor)

  • Banco de dados NoSQL chave-valor e documento totalmente gerenciado (Serverless).
  • Replicação Multi-AZ automática.
  • Modelagem foca na arquitetura Single-Table Design (Tabelas únicas contendo múltiplos tipos de registros de sua aplicação).
  • Completamente baseado em cobranças por Requisições em massa (RCU/WCU).

Componentes do DynamoDB

  • Partition Key (PK): Como os dados serão agrupados e espalhados nas AZs. OBRIGATÓRIA.
  • Sort Key (SK): Como ordenamos dados dentro da mesma PK. OPCIONAL.
  • GSI (Global Secondary Index): A verdadeira magia do DynamoDB. Criar projeções indexadas secundárias para consultar o dado de diferentes perspectivas de acesso.

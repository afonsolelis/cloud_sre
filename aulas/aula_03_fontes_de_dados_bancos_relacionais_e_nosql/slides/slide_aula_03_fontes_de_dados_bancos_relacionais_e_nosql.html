Teoria · Contexto

Fontes operacionais no pipeline de dados

Toda analitica comeca em algum lugar. Na maioria das empresas esse lugar e um banco transacional.

O fluxo tipico

Fontes OLTP (RDS, DynamoDB, SaaS) → Ingestao (snapshot, incremental, CDC) → Data Lake (S3 raw/trusted/refined) → OLAP (Athena, Redshift, BI).

Nunca rode relatorios pesados na base de producao: voce derruba o sistema transacional.

Teoria · Modelo relacional

Entidades, atributos e relacionamentos

Codd, 1970. A base conceitual que continua viva em 2026.

Entidade

  • Objeto do mundo real: cliente, pedido, produto.
  • Vira uma tabela.

Atributo

  • Propriedade da entidade: nome, preco, cep.
  • Vira uma coluna.

Tupla / Linha

  • Uma ocorrencia concreta da entidade.
  • Ex.: pedido 1234 do cliente A em 30/04/2026.

Relacionamento

  • 1:1, 1:N, N:N.
  • Implementado com chave estrangeira.
Teoria · Normalizacao

1FN - Primeira Forma Normal

Elimina valores multiplos em uma mesma celula. Cada coluna deve conter um valor atomico.

Antes (viola 1FN) 
pedido_id | itens
1001      | camiseta, bone, meia
1002      | tenis
Depois (em 1FN) 
pedido_id | produto
1001      | camiseta
1001      | bone
1001      | meia
1002      | tenis
Teoria · Normalizacao

2FN - Segunda Forma Normal

Ja esta em 1FN e todo atributo nao-chave depende da chave inteira, nao apenas de parte dela.

Exemplo Olist

Chave composta (pedido_id, produto_id) com colunas quantidade e nome_produto.

quantidade depende da chave inteira - OK.
nome_produto depende apenas de produto_id - viola 2FN. Mova para a tabela products.

Teoria · Normalizacao

3FN - Terceira Forma Normal

Ja esta em 2FN e nao ha dependencia transitiva entre atributos nao-chave.

Exemplo Olist

Na tabela customers com colunas customer_id, cep e cidade.

cidade depende de cep, que depende de customer_id - dependencia transitiva.
Solucao: crie uma tabela geolocation (cep, cidade, uf).

Teoria · Chaves

Chaves primarias e estrangeiras

Primary Key (PK)

  • Identifica unicamente uma linha.
  • NOT NULL e UNIQUE.
  • Ex.: order_id em orders.

Foreign Key (FK)

  • Aponta para a PK de outra tabela.
  • Garante integridade referencial.
  • Ex.: customer_id em orders.

Chave natural

  • Valor do negocio (CPF, email, CEP).
  • Pode mudar - cuidado.

Surrogate key

  • Gerada pelo sistema (serial, UUID).
  • Estavel, sem significado de negocio.
Teoria · Transacoes

ACID - o contrato das transacoes

Propriedades que garantem que operacoes sobre um banco sejam confiaveis.

A - Atomicidade

  • Tudo ou nada. Se uma etapa falha, rollback da transacao inteira.

C - Consistencia

  • O banco passa de um estado valido para outro. Respeita constraints.

I - Isolamento

  • Transacoes concorrentes nao interferem umas nas outras. Niveis: READ COMMITTED, SERIALIZABLE.

D - Durabilidade

  • Uma vez commitada, a transacao persiste mesmo apos queda do servidor.
Teoria · Sistemas distribuidos

Teorema CAP - escolha 2 de 3

Em um sistema distribuido, diante de uma particao de rede, voce escolhe entre consistencia e disponibilidade.

C - Consistency

  • Toda leitura enxerga o dado mais recente.

A - Availability

  • Toda requisicao recebe uma resposta nao-erro.

P - Partition tolerance

  • O sistema continua funcionando mesmo com falhas de rede entre nos.

Na pratica

  • CP: RDS Multi-AZ, DynamoDB strongly consistent.
  • AP: DynamoDB eventually consistent, Cassandra.

Teoria · Decisao

Quando usar SQL e quando usar NoSQL

Prefira SQL quando
  • Dados altamente relacionais.
  • Transacoes com ACID forte.
  • Schema estavel e conhecido.
  • Queries ad hoc com JOIN.
  • Relatorios regulatorios e financeiros.
Prefira NoSQL quando
  • Escala horizontal massiva.
  • Baixa latencia consistente (< 10 ms).
  • Schema flexivel ou evolutivo.
  • Padroes de acesso conhecidos (key lookup).
  • Workloads com alta cardinalidade.
Teoria · Amazon RDS

Amazon RDS - SQL gerenciado

Motores suportados: PostgreSQL, MySQL, MariaDB, Oracle, SQL Server. Aurora e uma variante cloud-native.

Alta disponibilidade

  • Multi-AZ: standby sincrono em outra AZ.
  • Failover automatico em 60-120 s.

Escala de leitura

  • Read replicas assincronas.
  • Ate 15 replicas no Aurora.

Backup e restore

  • Snapshots automaticos diarios.
  • Point-in-time recovery.

Seguranca

  • VPC, Security Group, KMS at rest.
  • IAM database authentication (Postgres/MySQL).
Teoria · Amazon DynamoDB

DynamoDB - modelo de tabela

Serverless, chave-valor e documento. Latencia de milissegundos em qualquer escala.

Partition Key (PK)

  • Obrigatoria. Determina o node de armazenamento.
  • Deve ter alta cardinalidade para distribuir carga.

Sort Key (SK)

  • Opcional. Ordena itens dentro da mesma PK.
  • Habilita queries por range.

LSI

  • Local Secondary Index - mesma PK, outro SK.
  • Criado na criacao da tabela.

GSI

  • Global Secondary Index - nova PK e SK.
  • Replicado assincronamente.
Teoria · Amazon DynamoDB

Capacidade e cache

On-demand
  • Paga por request (RCU/WCU consumidos).
  • Escala automaticamente, sem planejamento.
  • Ideal para workloads imprevisiveis.
Provisioned
  • Voce reserva RCU/WCU por segundo.
  • Mais barato para workload previsivel.
  • Auto-scaling opcional.
DAX (DynamoDB Accelerator)

Cache in-memory compativel com a API DynamoDB. Reduz latencia de milissegundos para microssegundos em leituras repetidas.

Teoria · AWS DMS

AWS Database Migration Service

Servico gerenciado para replicar bancos com minimo downtime. Suporta full load, CDC e a combinacao dos dois.

Componentes
  • Replication instance - roda o processo de replicacao.
  • Source endpoint - RDS, Oracle, SQL Server, MongoDB.
  • Target endpoint - S3, Redshift, Kinesis, outro RDS.
  • Task - define tabelas, filtros e transformacoes.
Teoria · Decisao arquitetural

Replicar para o lake ou consultar a fonte?

Replicar para o lake
  • Analitica nao impacta OLTP.
  • Historico ilimitado.
  • Custo de armazenar e pipeline.
  • Latencia de horas ou minutos.
Consultar a fonte viva
  • Dados sempre atualizados.
  • Risco de prejudicar producao.
  • Sem historico alem da retencao da fonte.
  • Uso aceitavel: read replica dedicada a analitica.

Pratica · Passo 1

Criar RDS PostgreSQL

Console AWS → RDS → Create database. Use Easy create + Free tier quando disponivel.

Parametros
  • Engine: PostgreSQL 15.x
  • Template: Free tier / Dev-Test
  • DB instance identifier: olist-oltp
  • Master username: postgres
  • Master password: anote em local seguro
  • Instance class: db.t3.micro
  • Storage: 20 GB gp3
  • Public access: Yes (para conectar do laptop)
  • VPC security group: novo, permitindo TCP 5432 do seu IP
  • Initial database name: (deixe vazio, criaremos manualmente)

Pratica · Passo 3

DDL - tabelas Olist

CREATE TABLE customers (
  customer_id    VARCHAR(40) PRIMARY KEY,
  customer_city  VARCHAR(80),
  customer_state CHAR(2)
);

CREATE TABLE orders (
  order_id     VARCHAR(40) PRIMARY KEY,
  customer_id  VARCHAR(40) REFERENCES customers(customer_id),
  order_status VARCHAR(20),
  order_purchase_timestamp TIMESTAMP
);

CREATE TABLE products (
  product_id         VARCHAR(40) PRIMARY KEY,
  product_category   VARCHAR(60),
  product_weight_g   INT
);

CREATE TABLE order_items (
  order_id    VARCHAR(40) REFERENCES orders(order_id),
  order_item_id INT,
  product_id  VARCHAR(40) REFERENCES products(product_id),
  price       NUMERIC(10,2),
  freight_value NUMERIC(10,2),
  PRIMARY KEY (order_id, order_item_id)
);

Pratica · Passo 5

Queries analiticas em SQL

-- top 5 categorias por receita
SELECT p.product_category,
       SUM(oi.price) AS receita
  FROM order_items oi
  JOIN products p ON oi.product_id = p.product_id
 GROUP BY p.product_category
 ORDER BY receita DESC
 LIMIT 5;

-- ticket medio por estado
SELECT c.customer_state,
       ROUND(AVG(total.valor),2) AS ticket_medio
  FROM (SELECT order_id, SUM(price + freight_value) AS valor
          FROM order_items GROUP BY order_id) total
  JOIN orders o    ON total.order_id   = o.order_id
  JOIN customers c ON o.customer_id    = c.customer_id
 GROUP BY c.customer_state
 ORDER BY ticket_medio DESC;
Pratica · Passo 6

Criar tabela DynamoDB olist_events

Eventos de sessao do site Olist. PK por cliente, SK por timestamp do evento.

aws dynamodb create-table \
  --table-name olist_events \
  --attribute-definitions \
      AttributeName=customer_id,AttributeType=S \
      AttributeName=event_timestamp,AttributeType=S \
  --key-schema \
      AttributeName=customer_id,KeyType=HASH \
      AttributeName=event_timestamp,KeyType=RANGE \
  --billing-mode PAY_PER_REQUEST \
  --region us-east-1

Pratica · Passo 8

Consultar: query e PartiQL

# query classico (todos os eventos do cliente c1)
aws dynamodb query \
  --table-name olist_events \
  --key-condition-expression "customer_id = :c" \
  --expression-attribute-values '{":c":{"S":"c1"}}'

# PartiQL (SQL-like)
aws dynamodb execute-statement \
  --statement "SELECT * FROM olist_events WHERE customer_id = 'c1'"
Pratica · Passo 9

Snapshot logico: pg_dump para S3

Simula ingestao batch da fonte OLTP para a camada raw do data lake.

# exportar o schema olist para arquivo SQL
pg_dump -h olist-oltp.abc123.us-east-1.rds.amazonaws.com \
        -U postgres -d olist \
        -f olist_2026-04-30.sql

# subir para o bucket raw criado na Aula 01
aws s3 cp olist_2026-04-30.sql \
  s3://<seu-login>-raw/rds_snapshot/olist/dt=2026-04-30/olist.sql
Proximo passo conceitual

Na Aula 05 substituimos pg_dump por AWS DMS ou AWS Glue JDBC para fazer ingestao incremental automatica.

Pratica · Passo 10 (obrigatorio)

Parar ou deletar o RDS

RDS consome budget mesmo ocioso. Sem esse passo o Learner Lab pode ficar sem saldo antes do fim do curso.

Opcoes
  1. Stop: Console RDS → selecionar instancia → Actions → Stop temporarily (ate 7 dias).
  2. Delete: Console RDS → Actions → Delete. Desmarque "Create final snapshot" para ir mais rapido.
  3. DynamoDB on-demand: cobrado por request, pode deixar. Se preferir, delete a tabela olist_events.
Checklist de encerramento
  • RDS parado ou deletado.
  • Arquivo SQL enviado para s3://...-raw/rds_snapshot/.
  • Tabela DynamoDB criada (ou removida).

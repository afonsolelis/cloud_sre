Fontes de Dados: Bancos Relacionais e NoSQL

Este material aprofunda os conceitos apresentados nos slides da Aula 03. Cobre o ecossistema de bancos de dados transacionais na AWS — Amazon RDS e Amazon DynamoDB —, as estratégias de extração de dados para o Data Lake, o mecanismo de Change Data Capture (CDC) com AWS DMS, e a modelagem Single-Table Design no DynamoDB para alto throughput.

Módulo 1 — Bancos de Dados Relacionais: Amazon RDS

OLTP versus OLAP: por que separar?

Sistemas transacionais (OLTP — Online Transaction Processing) são otimizados para operações de leitura e escrita de baixa latência em poucos registros por vez: criar um pedido, atualizar o saldo de uma conta, registrar um login. Esses bancos usam índices otimizados para busca por chave primária e normalização para evitar redundância.

Sistemas analíticos (OLAP — Online Analytical Processing) exigem o oposto: leituras de grandes volumes de dados, agregações, junções entre tabelas de fato e dimensão, e queries que varrem milhões de linhas. Executar queries analíticas diretamente no banco de produção OLTP degrada a performance das aplicações que dependem dele, pois as queries competem pelos mesmos recursos (CPU, I/O) com as transações dos usuários.

A solução é extrair os dados do OLTP periodicamente (batch) ou em tempo quase real (CDC) e carregá-los em um Data Lake ou Data Warehouse dedicado para análise.

Amazon RDS: banco relacional gerenciado

O Amazon Relational Database Service (RDS) é um serviço PaaS que gerencia automaticamente patching do sistema operacional e do banco de dados, backups automáticos com retenção configurável, failover automático entre Zonas de Disponibilidade (Multi-AZ), e monitoramento de métricas via CloudWatch.

Motores suportados: MySQL, PostgreSQL, MariaDB, Oracle Database, SQL Server, e Amazon Aurora (MySQL-compatível e PostgreSQL-compatível). Aurora é o motor recomendado para novas aplicações na AWS — ele usa armazenamento distribuído que separa computação de storage, permitindo escalabilidade de leitura com até 15 Read Replicas e tolerância a falhas superior.

Do ponto de vista do engenheiro de dados, o RDS é uma fonte de dados transacionais que precisa ser extraída com cuidado. As estratégias de extração são:

  • Full extract (extração completa): copia a tabela inteira a cada execução. Simples de implementar, mas impraticável para tabelas grandes (centenas de GB). Causa carga significativa no banco de produção.
  • Incremental extract por timestamp: usa uma coluna de updated_at para extrair apenas registros modificados desde a última execução. Requer que todas as tabelas tenham essa coluna populada corretamente — um requisito que nem sempre é atendido pelos sistemas legados.
  • Change Data Capture (CDC): captura mudanças no nível do transaction log do banco de dados, sem precisar modificar as tabelas de origem. A abordagem mais robusta e com menor impacto no banco de produção.

AWS Database Migration Service (DMS) e CDC

O AWS DMS é o serviço gerenciado da AWS para migração e replicação contínua de dados entre bancos. No contexto de Data Lake, ele é usado principalmente para CDC: o DMS lê o binary log (MySQL/MariaDB) ou o WAL — Write-Ahead Log (PostgreSQL) do banco de origem e entrega as mudanças (INSERT, UPDATE, DELETE) em formato JSON para um destino como S3 ou Kinesis.

Cada evento de mudança no CDC contém: o tipo da operação (I/U/D), o timestamp da mudança, os valores antes e depois (para UPDATE), e todos os campos do registro. Isso permite reconstruir o estado exato do banco em qualquer ponto no tempo, simplesmente replaying os eventos em ordem.

Para configurar CDC no DMS com PostgreSQL, é necessário habilitar logical replication no RDS: rds.logical_replication = 1 no parameter group. O DMS então cria um replication slot no PostgreSQL para consumir o WAL.

Módulo 2 — Amazon DynamoDB: NoSQL de Alto Throughput

Quando usar NoSQL em vez de SQL

Bancos relacionais escalam verticalmente (adicionar CPU e RAM ao servidor), o que tem limites físicos e custos exponenciais. Para aplicações que precisam servir milhões de requisições por segundo com latência de um dígito de milissegundos — independentemente do volume total de dados —, o modelo relacional não é adequado.

O DynamoDB foi projetado para escala horizontal: os dados são particionados automaticamente entre centenas ou milhares de servidores, e o throughput escala linearmente adicionando capacidade sem downtime. A latência de leitura e escrita é garantida em microssegundos para respostas em cache (DAX) e poucos milissegundos para operações diretas na tabela.

Componentes fundamentais do DynamoDB

Partition Key (PK): é o atributo que determina em qual partição física um item será armazenado. O DynamoDB aplica uma função hash sobre o valor da PK para distribuir os itens entre partições. Uma boa PK tem alta cardinalidade e distribuição uniforme — valores que se concentram em poucos valores (ex.: status = 'ativo') criam hot partitions que limitam o throughput.

Sort Key (SK): quando definida junto com a PK, forma uma chave composta. Todos os itens com a mesma PK são armazenados juntos, ordenados pela SK. Isso permite queries eficientes do tipo "busque todos os pedidos do cliente X entre as datas Y e Z" usando operadores de range (BETWEEN, BEGINS_WITH, ><).

Global Secondary Index (GSI): é uma projeção da tabela com uma PK e SK diferentes da tabela principal. Um GSI permite consultar a tabela de uma perspectiva diferente sem duplicar o dado principal. Por exemplo, se a tabela usa userID como PK, um GSI com emailAddress como PK permite buscar usuários por email em O(1). Cada GSI tem sua própria capacidade de throughput e cobra separadamente.

Capacidade (RCU/WCU): o DynamoDB opera em dois modos. No modo On-Demand, você paga por requisição sem pré-provisionar capacidade — ideal para cargas imprevisíveis. No modo Provisioned, você define Read Capacity Units (RCU) e Write Capacity Units (WCU) antecipadamente e paga por capacidade provisionada — ideal para cargas previsíveis com possibilidade de usar Auto Scaling.

Single-Table Design

O Single-Table Design é a abordagem de modelagem recomendada para DynamoDB. Em vez de criar uma tabela por entidade (como no modelo relacional), você armazena múltiplos tipos de entidades em uma única tabela, usando valores prefixados na PK e SK para distingui-los.

Por exemplo, em um sistema de e-commerce, a mesma tabela pode armazenar:

  • Usuários: PK = USER#U001, SK = PROFILE
  • Pedidos do usuário: PK = USER#U001, SK = ORDER#2026-04-01#P001
  • Itens do pedido: PK = ORDER#P001, SK = ITEM#SKU-123

Esse design permite buscar "o perfil do usuário U001 e todos os seus pedidos" com uma única operação Query (sem JOIN), já que todos os itens com a mesma PK ficam na mesma partição. A regra é: modele com base nos padrões de acesso (access patterns) da aplicação, não no schema lógico dos dados.

Exportando DynamoDB para o Data Lake (S3)

O DynamoDB oferece dois mecanismos nativos de integração com S3:

  • DynamoDB Export to S3: exporta um snapshot completo da tabela para S3 em formato JSON ou DynamoDB JSON. Funciona em segundo plano sem impactar o throughput da tabela, usando o recurso Point-in-Time Recovery (PITR). Ideal para carga inicial no Data Lake.
  • DynamoDB Streams + Lambda/Kinesis: captura cada mudança (INSERT, UPDATE, DELETE) em tempo real e permite processá-las com Lambda ou entregá-las via Kinesis Firehose para o S3. Implementa o padrão CDC para DynamoDB.

Módulo 3 — Laboratório: DynamoDB para o Data Lake

A prática ocorre entre 20h00 e 22h00 utilizando o ambiente AWS Student. Siga cada passo na ordem indicada.

Passo 1 — Criar a tabela DynamoDB

  1. No console AWS, acesse o Amazon DynamoDB.
  2. Clique em Create table.
  3. Em Table name, insira: Ecommerce_Transactions_MBA.
  4. Em Partition key, insira userID com tipo String.
  5. Em Sort key, insira timestamp com tipo String.
  6. Em Table settings, selecione Customize settings.
  7. Em Read/write capacity settings, selecione On-demand.
  8. Clique em Create table e aguarde o status Active.
  9. Validação: a tabela aparece na lista com status Active e as chaves corretas.

Passo 2 — Inserir itens de exemplo com schema variável

  1. Clique na tabela e depois em Explore table items.
  2. Clique em Create item e insira o seguinte item (modo JSON):
{
  "userID": {"S": "U001"},
  "timestamp": {"S": "2026-04-01T10:00:00Z"},
  "tipo": {"S": "compra"},
  "produto": {"S": "Notebook Dell"},
  "valor": {"N": "3500.00"},
  "parcelas": {"N": "12"}
}
  1. Clique em Create item.
  2. Crie mais dois itens com schemas diferentes para simular a flexibilidade NoSQL:
{
  "userID": {"S": "U002"},
  "timestamp": {"S": "2026-04-01T11:30:00Z"},
  "tipo": {"S": "visualizacao"},
  "produto": {"S": "Smartphone Samsung"},
  "duracao_segundos": {"N": "45"}
}
{
  "userID": {"S": "U001"},
  "timestamp": {"S": "2026-04-02T09:15:00Z"},
  "tipo": {"S": "devolucao"},
  "produto": {"S": "Notebook Dell"},
  "motivo": {"S": "defeito_bateria"},
  "valor_reembolso": {"N": "3500.00"}
}
  1. Validação: os três itens aparecem na tabela com seus campos específicos. Note que cada item tem atributos diferentes — o DynamoDB aceita schema livre.

Passo 3 — Habilitar PITR e exportar para S3

  1. Na tela da tabela, clique na aba Backups.
  2. Em Point-in-time recovery (PITR), clique em Enable.
  3. Confirme a ativação. O PITR mantém um histórico contínuo das últimas 35 dias de mudanças.
  4. Clique na aba Exports and streams.
  5. Em Amazon S3 export, clique em Export to S3.
  6. Em S3 bucket, selecione o bucket criado na Aula 02 ou crie um novo.
  7. Em S3 bucket prefix, insira raw/fonte=dynamodb/tabela=ecommerce/.
  8. Em Export format, selecione DynamoDB JSON.
  9. Clique em Export.
  10. Aguarde o status da exportação mudar para Completed.
  11. Validação: acesse o S3 e confirme que os arquivos JSON foram criados no prefixo especificado. Inspecione um arquivo para verificar o formato dos dados exportados.

Passo 4 — Consultar os dados exportados com Athena

  1. Acesse o Amazon Athena.
  2. Execute o DDL para criar uma tabela apontando para os dados exportados do DynamoDB:
CREATE EXTERNAL TABLE datalake_lab.raw_dynamodb_ecommerce (
  Item STRUCT<
    userID: STRUCT<S: STRING>,
    timestamp: STRUCT<S: STRING>,
    tipo: STRUCT<S: STRING>,
    produto: STRUCT<S: STRING>,
    valor: STRUCT<N: STRING>
  >
)
ROW FORMAT SERDE 'org.openx.data.jsonserde.JsonSerDe'
LOCATION 's3://seu-bucket/raw/fonte=dynamodb/tabela=ecommerce/AWSDynamoDB/';
  1. Execute uma query para validar os dados:
SELECT Item.userID.S AS user_id,
       Item.tipo.S AS tipo_evento,
       Item.produto.S AS produto
FROM datalake_lab.raw_dynamodb_ecommerce;
  1. Validação: os registros exportados do DynamoDB aparecem na query do Athena. O Data Lake agora contém dados tanto do RDS/CSV (Aula 02) quanto do DynamoDB.

Exercícios de fixação

  1. RDS versus DynamoDB: uma startup está construindo um sistema de agendamento de consultas médicas. O sistema precisa: registrar pacientes com dados pessoais estruturados, armazenar consultas vinculadas a pacientes e médicos, e responder a queries como "todos os agendamentos do médico X na semana Y". Justifique qual banco (RDS ou DynamoDB) você usaria para cada entidade e descreva um possível Single-Table Design para DynamoDB com exemplos de PK e SK.
  2. CDC e implicações no Data Lake: explique a diferença entre extração incremental por timestamp e CDC via transaction log. Descreva um cenário onde a extração por timestamp falha silenciosamente (retorna dados incorretos sem gerar erro), e como o CDC via DMS resolveria esse problema.
  3. Hot partition no DynamoDB: um engenheiro modelou uma tabela de logs de acesso usando PK = data (ex.: "2026-04-23") para facilitar consultas por dia. Explique por que essa escolha de PK causa um hot partition, quais são as consequências práticas, e proponha uma PK alternativa que distribua melhor a carga.
  4. Fluxo de dados transacional para analítico: descreva o fluxo completo de dados desde uma transação de compra registrada no RDS MySQL de produção até estar disponível para consulta no Athena na camada Trusted do Data Lake. Inclua: o mecanismo de extração, o serviço de entrega, o formato de armazenamento em cada camada, e a frequência de atualização.
  5. Limpeza de ambiente: após o laboratório, quais recursos da AWS você deve deletar para evitar cobranças desnecessárias no ambiente de lab? Liste os recursos na ordem correta de exclusão, justificando por que a ordem importa (por exemplo, dependências entre recursos).

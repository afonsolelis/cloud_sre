Modelagem da arquitetura com IA: RF, RNF, RTM e planos de teste

Este material acompanha a Aula 03 da disciplina Cloud Computing e SRE — Visão Prática com AWS do MBA em Engenharia de Dados. A aula é um dia de modelagem — nada será provisionado na AWS hoje. A entrega é a arquitetura desenhada, validada e rastreada em /documents, com a IA como copiloto. Ao final, o AWS CLI é conectado ao Codespace via session token do Student Lab, deixando o ambiente armado para a Aula 04.

Pré-requisitos Aula 02 entregue: spec/00_problem.md commitado, Codespace com OpenCode e Gemini CLI funcionando, repo olist-sre-pipeline sincronizado.

1. Do problema à arquitetura

Na Aula 02 modelamos o problema. Tínhamos um pipeline imaginário do Olist e listamos stakeholders, fluxos críticos, modos de falha conhecidos e propriedades emergentes desejadas. Esse arquivo — spec/00_problem.md — é o input desta aula. A pergunta agora muda: com qual estrutura responderemos a esse problema?

A modelagem da arquitetura tem uma armadilha clássica: pular para o desenho de caixinhas antes de ter requisitos escritos. O resultado é uma arquitetura sedutora visualmente, porém sem critério de aceitação. Para escapar da armadilha, seguimos uma cadeia rígida: problema → RF → RNF → arquitetura → testes → RTM → system design. A IA acelera cada etapa, mas a coerência entre etapas é responsabilidade humana.

1.1 As cinco regras da modelagem com IA

Não-negociáveis nesta aula
  1. Nada de código antes de ter RF, RNF e arquitetura escritos.
  2. Nada conectado à AWS antes do system design estar completo no markdown.
  3. A IA propõe; você decide. Toda saída entra como diff revisável no git.
  4. Cada componente da arquitetura aponta para ao menos um RF ou RNF — senão sai do desenho.
  5. Cada RF/RNF aponta para ao menos um teste — senão é decoração.

2. Requisitos funcionais como contrato

Um requisito funcional descreve o que o sistema faz, observável de fora pelo ator que o aciona. Ele não descreve implementação. Ele não descreve "como". Ele descreve o comportamento contratado.

2.1 Anatomia de um RF

Cada requisito funcional precisa de:

  • Identificador estável — RF-NN, nunca renumerado.
  • Título curto — verbo + objeto.
  • Descrição em EARS — sintaxe controlada (ver 2.2).
  • Atores — quem dispara o comportamento.
  • Pré e pós-condições — estado antes e depois.
  • Critério de aceitação em Gherkin — Given / When / Then.
  • Prioridade MoSCoW — Must, Should, Could, Won't.
  • Origem — stakeholder ou doc anterior que motivou o RF.
  • Vinculado a — referência cruzada para componentes (Aula 04+) e casos de teste.

2.2 EARS — cinco padrões pragmáticos

EARS (Easy Approach to Requirements Syntax) foi criado por Alistair Mavin na Rolls-Royce para escrever requisitos em linguagem natural controlada, sem cair em ambiguidade. Cada requisito é uma das cinco formas:

  • Ubíquo: "O sistema deve…" — comportamento sempre verdadeiro.
  • Estado: "Enquanto <estado>, o sistema deve…" — vale só num modo.
  • Evento: "Quando <gatilho>, o sistema deve…" — reage a um evento.
  • Opcional: "Onde <feature>, o sistema deve…" — depende de configuração.
  • Indesejado: "Se <condição>, então o sistema deve…" — reage a anomalia.

Para o pipeline Olist, o uso é direto: ingestão é evento (quando o agendador disparar…), idempotência é opcional (onde houver re-execução…) e quarentena é indesejado (se uma linha falhar a validação…).

3. RNF mensuráveis — ISO/IEC 25010

RNF não é "ser confiável". RNF é número, unidade, janela e fonte de medição. Sem essas quatro coisas, ele vira aspiração e ninguém é capaz de testar. A norma ISO/IEC 25010 organiza qualidade em oito atributos — os mesmos oito eixos que conduzem o curso desde a Aula 01.

3.2 SLI, SLO, SLA e error budget

Confiabilidade vira contrato operacional quando atravessa três camadas: SLI (Service Level Indicator, o que medimos), SLO (Service Level Objective, a meta interna) e SLA (Service Level Agreement, o compromisso externo). Confundir as três é perder a referência do que é negociável.

  • SLI — é a métrica. Ex.: % de dias com dashboard pronto até 06h. Coletado pela observabilidade.
  • SLO — é a meta. Ex.: SLI ≥ 99% por trimestre. Define o budget de erro.
  • SLA — é o contrato. Ex.: diretoria recebe crédito se SLI < 95%. Sempre < SLO.
  • Error budget — é o complemento de 1. Se SLO = 99%, error budget = 1% no trimestre.

O error budget é a régua que define quando o time pode experimentar e quando precisa parar para corrigir. Quando o budget esgota, o pipeline congela mudanças. Esse "freeze" é uma das peças do closed loop que veremos em 7.

3.3 Utility tree — o que merece atenção primeiro

Utility tree é a ferramenta clássica do método ATAM para priorizar atributos de qualidade. Cada folha recebe um par (impacto no negócio, dificuldade técnica) em escala alta/média/baixa. O canto superior-esquerdo — alto impacto e alta dificuldade — é onde a arquitetura ganha ou perde.

AtributoCenário concreto(Negócio, Técnico)
ReliabilityFonte cai por 1h durante a janela; pipeline retoma sem dado perdido.(Alto, Alto)
PerformanceVolume dobra para 200k linhas; ETL ainda termina em < 30 min.(Alto, Médio)
SecurityCredenciais não vivem em código nem em buckets públicos.(Alto, Médio)
MaintainabilityNovo dev entrega ajuste em ETL em ≤ 1 dia.(Médio, Médio)
CostCusto mensal ≤ teto definido pelo Learner Lab.(Alto, Baixo)

4. RTM — rastreabilidade ponta a ponta

A Requirements Traceability Matrix é a costura entre todos os artefatos. Ela liga cada RF/RNF a um componente da arquitetura e a um caso de teste. Se a coluna "Componente" estiver vazia, o requisito não tem onde ser realizado — é arquitetura faltando. Se a coluna "Caso de teste" estiver vazia, ninguém vai validar — é teste faltando.

5. Plano de testes — pirâmide do pipeline

A pirâmide de testes tradicional foi pensada para aplicações OLTP. Para um pipeline de dados, ela ganha camadas adicionais: contrato (validação de schema), carga (performance e escala), segurança (STRIDE/OWASP), caos (resiliência sistêmica) e modelagem (coerência arquitetural). 5.1 Carga — Load, Soak, Spike, Stress (LISA)

Quatro perfis distintos respondem a perguntas diferentes. Modelar os quatro evita a armadilha do "passou no benchmark e morreu na produção".

  • Load — volume típico esperado. Pergunta: aguenta o dia normal? Olist: 100k linhas, p95 < 30 min.
  • Soak — carga estendida por horas. Pergunta: vaza memória? cresce backlog? Olist: 6h consecutivas no Codespace.
  • Spike — subida súbita. Pergunta: recupera ou trava? Olist: rajada 5x em 60s no Postgres.
  • Stress — vai até quebrar. Pergunta: onde é o teto? Olist: dobra volume até falhar.

Cada cenário precisa de hipótese, ferramenta, volume, duração, métrica de sucesso e RNF coberto. A última coluna é a costura com a RTM — sem ela, o teste de carga fica órfão de propósito.

5.2 Segurança — STRIDE + OWASP + camadas

Modelar segurança é saber quais ameaças importam. STRIDE classifica por categoria; OWASP fornece cenários concretos; SAST/DAST/SCA automatizam a verificação. As seis categorias STRIDE são:

  • Spoofing — identidade falsa (ex.: falsificar conexão ao Postgres).
  • Tampering — alteração não autorizada (ex.: modificar CSV antes da ingestão).
  • Repudiation — ação sem rastro (ex.: ETL sem log estruturado).
  • Information disclosure — vazamento (ex.: bucket S3 público).
  • Denial of service — ex.: rajada que derruba Grafana.
  • Elevation of privilege — ex.: ETL com role excessiva.

Por camada, o pipeline Olist é coberto por:

CamadaFerramentaO que verifica

Secret scangitleaksTokens em commit ou histórico
DASTOWASP ZAPEndpoints expostos do Grafana

5.3 Modelagem — o desenho passa em revisão?

Antes de qualquer linha de código, o desenho passa por dois testes de modelagem: ATAM (atributos vs cenários) e FMEA leve (modos de falha por componente). É barato falhar no markdown.

  • ATAM por cenários — os cenários da utility tree são "exercitados" no desenho. Cada cenário recebe um rótulo: risco, sensibilidade ou trade-off. Saída: lista de pontos sensíveis a monitorar.
  • FMEA leve — para cada componente, lista-se modo de falha, efeito e detecção. Severidade × ocorrência × detecção = RPN. Componentes com RPN alto puxam mitigação na arquitetura.
  • Walkthrough do papel — ler em voz alta o fluxo ponta a ponta, pedir à IA que faça o devil's advocate, anotar cada "e se?" como caso de teste novo.

6. RM-ODP — cinco viewpoints da arquitetura

Para evitar arquiteturas que viram diagramas de caixinhas sem direção, separamos a descrição em cinco viewpoints, padrão ITU-T X.901 / ISO/IEC 10746. Cada um olha o mesmo sistema sob uma lente diferente, e cada um vira uma seção em documents/03_architecture.md.

ViewpointPerguntaSaída esperada
EnterpriseQuem se beneficia e por quê?Stakeholders, objetivos, políticas
InformationQue informação flui?Modelo de dados, entidades, eventos
ComputationalQue serviços expomos?Componentes, interfaces, contratos
EngineeringOnde roda? Como se distribui?Topologia, deploy, rede, segurança
TechnologyQuais tecnologias concretas?Postgres, EC2, Grafana, AWS CLI, Mermaid

Para cada componente proposto em qualquer viewpoint, anotamos quais RF e RNF ele atende. Isso já alimenta a RTM. Decisões controversas viram ADRs (Architecture Decision Records) curtos: contexto, decisão, consequências.

7. Closed loops — malha fechada como sistema de segurança

Em malha aberta, o pipeline executa, gera log e segue. Em malha fechada, a observação volta a influenciar a próxima execução. Confiabilidade séria nasce de loops fechados — é controle clássico aplicado à confiabilidade de software. Cada SLO importa nada se o sistema não age sobre o desvio.

7.1 Anatomia de um closed loop em SRE

  • Sensor — SLI medido continuamente (frescor, lag, contagem, taxa de erro).
  • Comparador — compara SLI com SLO; a diferença é o erro de controle.
  • Controlador — decide a ação corretiva (retry, scale, freeze, quarentena, breaker).
  • Atuador — aplica a ação no componente correspondente (ETL, Postgres, scheduler).
  • Reobservação — mede de novo após a ação e fecha o ciclo.

7.2 Por que isso é segurança operacional

Sistemas em malha fechada falham de forma contida. A perturbação não se propaga porque o controlador intervém antes do dano agregado. Em SRE, isso se traduz em quatro ganhos diretos:

  • Bloqueia propagação de falha — falha em um nó vira sinal, não cascata.
  • Reduz dependência humana — MTTR cai porque a primeira reação é automática.
  • Transforma alerta passivo em ação garantida — "Slack pingando às 3h" deixa de ser único recurso.
  • Exige idempotência — reforça RF-06; se o atuador é idempotente, retries são seguros.
  • Fornece evidência auditável — logs do controlador mostram o que foi decidido e quando.

7.3 Cinco closed loops do Olist a modelar hoje

  1. Backpressure de ingestão — se a fila do ETL passar do limiar, o scheduler trava o próximo run até drenar.
  2. Circuit breaker do Postgres — 3 falhas consecutivas → ETL pausa; após health-check ok, retoma.
  3. Quarentena auto-reentrante — linhas em quarentena entram no próximo ciclo após correção do schema.
  4. Freeze por error budget — SLO de frescor abaixo de 95% → pipeline congela mudanças até budget se recuperar.
  5. Drift de dados — contagem fora de banda histórica → alerta + dashboard marca dado como suspeito.
Cuidado · loops mal projetados

Closed loops sem limites se transformam em "fugas térmicas": o controlador agita o sistema mais do que estabiliza. Por isso, todo loop precisa de limite de atuação (tentativas máx., janela mínima, kill switch manual) e backoff exponencial. A IA tende a propor loops agressivos — questione os limites em cada review.

8. A IA na modelagem — copiloto, nunca piloto

A IA acelera redação técnica, encontra ângulos esquecidos e revisa decisões. Mas autoridade arquitetural continua no engenheiro — é ele quem assina o desenho. Para que a IA renda no projeto, ela precisa de instruções persistentes (não prompts gigantes a cada sessão) e de skills focadas.

8.1 Arquivos canônicos de instrução

  • AGENTS.md — padrão emergente, lido por OpenCode e similares na raiz do repo.
  • GEMINI.md ou .gemini/system.md — carregado pelo Gemini CLI.
  • CLAUDE.md — equivalente para o Claude Code.
  • documents/agents/skills/*.md — skills focadas, chamadas por nome.

8.2 O que entra em AGENTS.md

  • Contexto do projeto (Olist, SRE, AWS Student Lab).
  • Restrições duras (sem Glue, sem Redshift, sem secrets em código).
  • Padrões de saída (markdown, EARS, tabelas RTM).
  • Ordem de leitura dos artefatos.
  • Comportamento esperado: criticar, autocriticar, listar premissas.

8.3 Skills — comportamentos focados e composáveis

Skill é uma instrução pequena, com escopo único, que a IA carrega quando o contexto bate. Em vez de prompts gigantes que carregam todo o projeto a cada chamada, mantemos um catálogo curto de skills nomeadas. A IA é instruída a "rodar a skill X usando o arquivo Y como entrada".

Cada skill tem seis seções obrigatórias:

  1. Quando usar — gatilho de aplicação.
  2. Entrada esperada — arquivos lidos.
  3. Passos rígidos — sequência determinística.
  4. Critérios de aceitação — o que torna a saída válida.
  5. Formato de saída — markdown, tabela, ou seção específica.
  6. Anti-padrões — o que a skill nunca deve fazer.

8.4 Catálogo das skills da Aula 03

  • elicit_rf — extrai RFs do problema, em EARS.
  • elicit_rnf — extrai RNFs mensuráveis cobrindo as 8 dimensões da ISO 25010.
  • build_rtm — gera a tabela de rastreabilidade.
  • review_architecture — aplica ATAM enxuto e devolve riscos/sensibilidades/trade-offs.
  • plan_load_test — gera Load/Soak/Spike/Stress com hipótese, ferramenta, volume, duração e RNF coberto.
  • plan_security_test — aplica STRIDE em cada componente e mapeia camadas SAST/DAST/SCA.

8.5 Requisitos não-negociáveis para a IA na modelagem

Regras duras — entram em AGENTS.md
  1. Toda saída em markdown válido, com cabeçalhos hierárquicos e identificadores estáveis (RF-NN, RNF-NN, TC-NN, ADR-NN).
  2. Toda afirmação tem fonte — cita stakeholder, doc anterior ou marca explicitamente como "premissa".
  3. Toda decisão técnica vira ADR curto: contexto, decisão, consequências.
  4. Toda RNF é mensurável — valor, unidade, janela, fonte de medição.
  5. Sem código pronto nesta aula. Apenas descrição estrutural e diagramas Mermaid.
  6. "Premissas e questões em aberto" obrigatório ao final de cada arquivo.
  7. Sem inventar serviços AWS bloqueados (Glue, Redshift, SageMaker estão fora).

2. Mapear cada fluxo aos 8 atributos da ISO 25010.
3. Para cada atributo, propor 1 a 3 RNFs com SLI mensuravel.
4. Marcar prioridade MoSCoW.
5. Listar premissas e fontes de medicao.

## Saida
Arquivo documents/02_non_functional_requirements.md com:
- secao por atributo ISO 25010
- IDs RNF-NN unicos
- tabela final com (ID, atributo, SLI, SLO, fonte, prioridade)

## Criterios de aceitacao
- 8 atributos cobertos.
- Todo RNF tem unidade e janela.
- Nenhum RNF aspiracional ("ser confiavel" e proibido).

## Anti-padroes
- RNF sem unidade ou sem janela.
- Atributo ISO 25010 ausente.
- Resposta com codigo Python pronto.

  EC2APP -->|RF-02 normaliza| EC2DB

  EC2APP -.->|RF-07 logs| CW
  EC2DB -.->|RNF-REL-02 metrics| CW

  EC2OBS -->|RF-05 dashboard| DIR

As arestas grossas formam o closed loop: sensor (CW) → controlador (EventBridge + SSM Run) → atuadores (ETL e Postgres) → nova medição (CW). Esse loop é o sistema de segurança operacional do pipeline.

12. Glossário de siglas

Toda sigla usada na aula, organizada por tema. Use como referência rápida durante a leitura ou ao escrever os artefatos em /documents. Quando uma sigla aparece em prompts da IA, mantenha a forma escrita (preferir "Service Level Indicator" a "SLI") na primeira ocorrência de cada arquivo.

12.1 Requisitos e modelagem

SiglaSignificado
RFRequisito Funcional — o que o sistema faz, observável de fora.
RNFRequisito Não-Funcional — com qual qualidade o sistema faz (performance, segurança, etc.).
RTMRequirements Traceability Matrix — matriz que liga RF/RNF a componentes e testes.
TCTest Case — identificador estável de caso de teste (TC-LOAD-01, TC-SEC-04).
ADRArchitecture Decision Record — documento curto com contexto, decisão e consequências.
EARSEasy Approach to Requirements Syntax — sintaxe controlada para escrever RFs.
BDDBehavior-Driven Development — cenários no formato Given/When/Then.
MoSCoWMust · Should · Could · Won't — escala de prioridade.
RM-ODPReference Model for Open Distributed Processing (ITU-T X.901) — cinco viewpoints de arquitetura.
arc42Template aberto e amplamente adotado para descrição de arquitetura de software.
ATAMArchitecture Tradeoff Analysis Method — análise por cenários (utility tree).
FMEAFailure Mode and Effects Analysis — modo de falha · efeito · detecção, por componente.
RPNRisk Priority Number — severidade × ocorrência × detecção.
SDDSpec-Driven Development — loop semanal proposto na Aula 02 (spec antes de código).
SoSSystem of Systems — agregação de sistemas com governança e ciclos de vida independentes.
ETLExtract, Transform, Load — processo de ingestão e transformação de dados.
ELTExtract, Load, Transform — variante que carrega antes de transformar.

12.2 SRE e confiabilidade

SiglaSignificado
SRESite Reliability Engineering — disciplina que aplica engenharia ao problema de operação.
SLIService Level Indicator — métrica medida continuamente (ex.: % de dias com dashboard pronto até 06h).
SLOService Level Objective — meta interna para o SLI (ex.: SLI ≥ 99% por trimestre).
SLAService Level Agreement — compromisso externo, sempre menor que o SLO.
Error budget1 − SLO em uma janela. Quando esgota, freeze de mudanças; quando sobra, dá para experimentar.
MTTRMean Time To Recovery — tempo médio até recuperação após falha.
MTBFMean Time Between Failures — tempo médio entre falhas consecutivas.
KPIKey Performance Indicator — indicador-chave de negócio (não confundir com SLI técnico).
USLUniversal Scalability Law — modelo matemático de escalabilidade (Gunther).
p95 / p99Percentil 95 / 99 da distribuição (ex.: latência p95 = valor abaixo do qual cabem 95% das medições).
BIBusiness Intelligence — consumidor analítico do pipeline (dashboards, relatórios).
MLMachine Learning — consumidor preditivo do pipeline.
CI / CDContinuous Integration / Continuous Delivery — integração e entrega contínuas via GitHub Actions ou similar.

12.3 Segurança e testes

SiglaSignificado
STRIDESpoofing · Tampering · Repudiation · Information disclosure · Denial of service · Elevation of privilege — categorias de ameaças (Microsoft).
OWASPOpen Worldwide Application Security Project — comunidade que mantém o Top 10 e o ASVS.
ASVSApplication Security Verification Standard — checklist OWASP por nível de maturidade.
SASTStatic Application Security Testing — análise estática do código (ex.: Bandit).
DASTDynamic Application Security Testing — análise dinâmica via tráfego HTTP (ex.: OWASP ZAP).
SCASoftware Composition Analysis — verificação de dependências vulneráveis (ex.: Trivy).
DoSDenial of Service — categoria do STRIDE: tornar serviço indisponível.
LGPDLei Geral de Proteção de Dados — legislação brasileira de privacidade.
GDPRGeneral Data Protection Regulation — regulamento europeu equivalente à LGPD.

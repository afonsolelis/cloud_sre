Fontes de Dados: Bancos Relacionais e NoSQL

Este material aprofunda todos os tópicos tratados nos slides da Aula 03. O objetivo é apresentar, em profundidade, o ecossistema de fontes operacionais que alimentam um pipeline de dados moderno: bancos relacionais (OLTP) e bancos NoSQL, com foco em Amazon RDS PostgreSQL e Amazon DynamoDB. O material explora a teoria (OLTP vs OLAP, modelagem, normalização, ACID, CAP, famílias NoSQL, decisão SQL vs NoSQL) e em seguida guia o aluno passo a passo pelo laboratório prático no AWS Learner Lab, onde subiremos um banco PostgreSQL, popularemos com dados do dataset Olist, rodaremos consultas analíticas, criaremos uma tabela DynamoDB de eventos de sessão e exportaremos um snapshot do RDS para o bucket raw do data lake criado na Aula 01. A aula segue o contrato obrigatório da disciplina: das 19h00 às 20h30 é teoria, e das 20h30 às 22h00 é prática no AWS Student.

Módulo 1 — Fontes operacionais no pipeline de dados

Recapitulando onde estamos

Nas duas primeiras aulas construímos a fundação do nosso data lake: na Aula 01 configuramos o AWS Student, o usuário IAM e os três buckets S3 (raw, trusted, refined). Na Aula 02 colocamos dados Olist em formato bruto no S3, criamos um banco no AWS Glue Data Catalog e aprendemos a consultar esses dados com Amazon Athena. O lake, por enquanto, está sendo alimentado manualmente: alguém baixou os CSVs do Kaggle e os fez upload para o S3.

Esta aula responde a uma pergunta fundamental: na vida real, de onde vêm os dados que são colocados no data lake? A resposta, na maioria esmagadora dos casos corporativos, são as fontes operacionais — os bancos de dados usados pelos sistemas transacionais das empresas. São eles que guardam o estado corrente do negócio: pedidos, clientes, produtos, pagamentos, inventário. Uma engenharia de dados séria precisa entender profundamente essas fontes, porque é delas que depende toda a confiabilidade downstream.

O fluxo típico de um pipeline moderno

O fluxo que usaremos como referência durante toda a disciplina é:

  1. Fontes OLTP — RDS, DynamoDB, sistemas SaaS, ERPs, CRMs.
  2. Camada de ingestão — snapshot batch, ingestão incremental, CDC.
  3. Data Lake — S3 com camadas raw, trusted e refined.
  4. Catalogação e processamento — Glue Data Catalog, Glue ETL, EMR.
  5. OLAP / consumo — Athena, Redshift, QuickSight, ML.
Regra de ouro da engenharia de dados. Nunca rode consultas analíticas pesadas diretamente no banco transacional de produção. Um SELECT ... GROUP BY ... em uma tabela com 500 milhões de linhas pode derrubar o sistema de checkout. A solução é replicar (batch ou CDC) para uma plataforma otimizada para análise — no nosso caso, o data lake.

Módulo 2 — OLTP vs OLAP

OLTP (Online Transaction Processing) e OLAP (Online Analytical Processing) são os dois grandes paradigmas de cargas de banco de dados. O conceito é antigo (Codd, 1993) mas continua extremamente atual. Entender a diferença entre eles é a base para escolher a tecnologia certa e projetar pipelines que não se sabotem.

Aspecto OLTP (transacional) OLAP (analítico)
ObjetivoRegistrar operações do negócio em tempo realResponder perguntas sobre o histórico do negócio
Volume por operaçãoPoucas linhas por transaçãoMilhões de linhas por consulta
Operações dominantesINSERT, UPDATE, DELETE e SELECT pontuaisSELECT com GROUP BY, JOIN, agregações
ModelagemNormalizada, tipicamente 3FNDesnormalizada: star schema, snowflake, tabelas wide
Latência alvoMilissegundosSegundos a minutos
ConcorrênciaMilhares de conexões simultâneas curtasDezenas de conexões longas
Armazenamento preferidoRow-basedColunar (Parquet, ORC, Redshift)
UsuáriosAplicações, APIs, clientes finaisAnalistas, cientistas de dados, BI, ML
AWSRDS, Aurora, DynamoDBAthena, Redshift, EMR

É comum ouvir que "o DynamoDB é NoSQL, então é analítico". Falso: DynamoDB é OLTP — é otimizado para operações de baixa latência em poucos itens por vez. Analítica em DynamoDB (scans longos, agregações) é um anti-padrão que destrói capacidade e custo. O caminho correto é exportar DynamoDB para S3 e analisar lá.

Módulo 3 — Modelagem relacional

O modelo relacional foi proposto por Edgar F. Codd em 1970 no paper seminal "A Relational Model of Data for Large Shared Data Banks". A ideia central é representar dados como relações (tabelas) com linhas (tuplas) e colunas (atributos), e permitir operar sobre eles com uma álgebra formal (a álgebra relacional) que, na prática, se traduz em SQL.

Entidade, atributo, tupla, relacionamento

  • Entidade: objeto do mundo real que queremos representar. No Olist, entidades são cliente, pedido, produto. Cada entidade vira uma tabela.
  • Atributo: propriedade de uma entidade. Para pedido temos id do pedido, data da compra, status. Cada atributo vira uma coluna.
  • Tupla / linha: uma ocorrência concreta da entidade, por exemplo o pedido 1234 feito pelo cliente A em 30/04/2026 e entregue em 05/05/2026.
  • Relacionamento: associação entre entidades. Pode ser 1:1 (um cliente tem um perfil), 1:N (um cliente faz muitos pedidos) ou N:N (um pedido contém muitos produtos, um produto aparece em muitos pedidos). O modelo relacional implementa relacionamentos com chaves estrangeiras; relações N:N exigem uma tabela de associação (no Olist, order_items).

Normalização — 1FN, 2FN, 3FN

Normalizar significa reorganizar o schema para eliminar redundância e anomalias de atualização. As três primeiras formas normais cobrem 95% dos casos práticos em sistemas transacionais.

Primeira Forma Normal (1FN)

Uma tabela está em 1FN quando cada coluna contém apenas valores atômicos — isto é, indivisíveis — e não há grupos repetitivos. Listas, arrays e "campos separados por vírgula" violam 1FN.

Exemplo de violação:

pedido_id | itens
1001      | camiseta, bone, meia
1002      | tenis

Corrigindo para 1FN:

pedido_id | produto
1001      | camiseta
1001      | bone
1001      | meia
1002      | tenis

Segunda Forma Normal (2FN)

Uma tabela está em 2FN quando está em 1FN e todo atributo não-chave depende da chave primária inteira, não apenas de uma parte dela. A 2FN só é problema quando a chave primária é composta.

Exemplo Olist: considere uma tabela order_items (order_id, product_id, quantidade, nome_produto) com chave primária (order_id, product_id). O atributo quantidade depende da chave inteira — OK. Mas nome_produto depende apenas de product_id. Isso viola 2FN. A correção é mover nome_produto para a tabela products.

Terceira Forma Normal (3FN)

Uma tabela está em 3FN quando está em 2FN e nenhum atributo não-chave depende transitivamente da chave primária — ou seja, nenhum atributo não-chave depende de outro atributo não-chave.

Exemplo Olist: em customers (customer_id, cep, cidade, uf), temos que cidade e uf dependem de cep, que por sua vez depende de customer_id. Isso é uma dependência transitiva. A correção é criar uma tabela geolocation (cep, cidade, uf) e deixar apenas cep em customers.

Quando não normalizar. Em OLAP (data warehouse, star schema) a desnormalização é a norma, não a exceção: queremos reduzir JOINs e acelerar queries analíticas. A regra prática é: normalize em OLTP, desnormalize em OLAP. O processo de ETL da Aula 05 justamente traduz um schema normalizado em schema dimensional.

Chaves: primária, estrangeira, natural e surrogate

  • Primary Key (PK): identifica univocamente cada linha. Deve ser NOT NULL e UNIQUE. Toda tabela bem projetada tem uma PK.
  • Foreign Key (FK): atributo (ou conjunto de atributos) que aponta para a PK de outra tabela. Garante integridade referencial: não é possível criar um orders.customer_id que não exista em customers.customer_id.
  • Chave natural: valor do próprio negócio usado como chave — CPF, email, CEP, ISBN. Vantagem: significativa. Desvantagem: pode mudar (pessoa troca de email), pode ser sensível (LGPD), e muda regras de unicidade.
  • Surrogate key: chave artificial, gerada pelo sistema (SERIAL, UUID). Vantagem: estável, sem semântica de negócio, sempre única. Desvantagem: precisa de um JOIN extra para recuperar o significado. Em sistemas modernos, o padrão é preferir surrogate keys.

Módulo 4 — ACID: o contrato das transações

Uma transação é uma sequência de operações tratada como uma unidade lógica. ACID é o acrônimo das quatro propriedades que um SGBD transacional deve garantir.

  • A — Atomicidade. Todas as operações de uma transação ou acontecem juntas, ou nenhuma acontece. Se uma etapa falha no meio, o banco faz rollback de tudo que foi feito até então. Exemplo: transferir R$ 100 entre contas exige UPDATE em duas linhas; se a segunda falhar, a primeira precisa ser revertida.
  • C — Consistência. Uma transação leva o banco de um estado válido para outro estado válido. Constraints (PK, FK, CHECK, NOT NULL) continuam satisfeitas após o commit.
  • I — Isolamento. Transações concorrentes não interferem umas nas outras. Há diferentes níveis de isolamento (READ UNCOMMITTED, READ COMMITTED, REPEATABLE READ, SERIALIZABLE) que equilibram performance e garantias. PostgreSQL usa READ COMMITTED por padrão.
  • D — Durabilidade. Uma vez que o banco confirmou o commit, a mudança sobrevive a quedas de energia, crashes do processo e até falhas de disco (se o storage replicar dados, como o RDS).

Bancos relacionais tradicionais (PostgreSQL, MySQL, Oracle, SQL Server) oferecem ACID completo. Muitos NoSQLs oferecem ACID relaxado ou apenas transações por item. DynamoDB suporta transações ACID em múltiplos itens desde 2018, mas com restrições de tamanho e custo adicional.

Módulo 5 — Teorema CAP

O teorema CAP, formulado por Eric Brewer em 2000 e provado formalmente por Gilbert e Lynch em 2002, afirma que, em um sistema de dados distribuído, é impossível garantir simultaneamente as três propriedades: Consistência, Disponibilidade e Tolerância a partições de rede. Na prática, partições de rede acontecem (cabos são cortados, switches falham), então o projetista escolhe entre CP (consistência + tolerância) e AP (disponibilidade + tolerância).

  • C — Consistency. Toda leitura enxerga o dado mais recente já escrito, ou um erro.
  • A — Availability. Toda requisição recebe uma resposta não-erro, mas sem garantia de ser o dado mais recente.
  • P — Partition tolerance. O sistema continua funcionando mesmo quando há perda arbitrária de mensagens entre nós.

Exemplos concretos na AWS

  • RDS Multi-AZ: sistema CP. Em caso de falha da AZ primária, há um failover que pode causar indisponibilidade breve — mas não leitura de dado stale.
  • DynamoDB com leitura strongly consistent: comportamento CP. Se a partição não puder ser lida consistente, retorna erro.
  • DynamoDB com leitura eventually consistent (padrão): comportamento AP. Retorna sempre, mas pode devolver um valor antigo por até um segundo.
  • Cassandra / Amazon Keyspaces: AP configurável; o nível de consistência é escolhido por operação (QUORUM, ONE, ALL).

O artigo "CAP Twelve Years Later", do próprio Brewer, refina essa visão: o trade-off não é tão binário. Sistemas modernos oferecem diferentes níveis de consistência por operação, o que os permite ser CP em algumas chamadas e AP em outras.

Módulo 6 — Famílias NoSQL

NoSQL ("Not Only SQL") é um termo guarda-chuva que cobre quatro grandes famílias de bancos, cada uma otimizada para um tipo diferente de carga. Entender essa taxonomia é essencial para escolher a ferramenta correta.

Família Modelo de dados Caso de uso típico Produto AWS
DocumentoJSON/BSON hierárquico por documentoCatálogos de produtos, perfis de usuário, CMSAmazon DocumentDB (MongoDB-compatible)
Chave-valorMapa key -> value (value pode ser documento)Sessões, carrinhos, cache, feature storeAmazon DynamoDB, Amazon ElastiCache (Redis/Memcached)
Colunar (wide-column)Linhas com número variável de colunas agrupadas em famíliasSéries temporais, IoT, logs, mensagensAmazon Keyspaces (Cassandra-compatible), Amazon Timestream
GrafoVértices e arestas com propriedadesDetecção de fraude, redes sociais, recomendaçãoAmazon Neptune

Documento vs chave-valor — uma distinção sutil

DynamoDB costuma ser descrito como "chave-valor", mas seus valores são documentos ricos (mapas aninhados, listas, sets, números, strings, booleanos). Por isso muitos autores o classificam também como documento. A distinção prática é o foco: bancos chave-valor priorizam lookup por chave; bancos documento priorizam consultas por campos internos e índices secundários flexíveis. DynamoDB se encaixa mais como chave-valor com valor documento.

Módulo 7 — Quando usar SQL e quando usar NoSQL

Não existe uma resposta universal: a escolha depende do padrão de acesso, do volume, da evolução do schema e dos requisitos de consistência. A seguir, heurísticas que costumam funcionar bem em engenharia de dados.

Prefira SQL quando

  • Seus dados são altamente relacionais (muitas entidades, muitos JOINs).
  • Você precisa de transações ACID envolvendo múltiplas linhas/tabelas.
  • O schema é estável e bem compreendido (domínios maduros: financeiro, contábil).
  • Você precisa fazer queries ad hoc, com agregações, GROUP BY e JOIN não previstos.
  • Há requisitos regulatórios ou de auditoria que se beneficiam da rigidez do schema.

Prefira NoSQL quando

  • Você precisa de escala horizontal massiva (milhões de ops/s).
  • Requisito de latência muito baixa e consistente (< 10 ms p99).
  • O schema é flexível, evolutivo ou varia por documento.
  • Os padrões de acesso são bem conhecidos e giram em torno de chave (key lookup, key + range).
  • Você está modelando séries temporais, eventos, sessões, feature store.
Padrão comum em produção. Muitas empresas usam polyglot persistence: RDS para dados transacionais relacionais, DynamoDB para sessões e eventos, ElastiCache para cache, Redshift ou Athena para analítica. A decisão não é "ou SQL ou NoSQL" — é "que ferramenta para que problema".

Módulo 8 — Amazon RDS

Amazon Relational Database Service (RDS) é o serviço gerenciado da AWS para bancos relacionais. A AWS cuida de provisioning, patching, backup, alta disponibilidade e monitoramento; você cuida de schema, queries e otimização. RDS suporta os motores PostgreSQL, MySQL, MariaDB, Oracle e SQL Server. Há ainda o Amazon Aurora, uma reimplementação cloud-native dos protocolos PostgreSQL e MySQL, com storage distribuído em 6 réplicas e escalabilidade superior.

Recursos principais

  • Multi-AZ. Provisiona uma réplica standby síncrona em outra Availability Zone. Em caso de falha, o failover é automático em 60–120 segundos. Multi-AZ não serve para escalar leitura — a standby não aceita conexões de aplicação. Serve para alta disponibilidade.
  • Read Replicas. Réplicas assíncronas que aceitam apenas SELECTs. Servem para descarregar leitura analítica da instância principal. RDS permite até 5 read replicas por instância; Aurora permite até 15.
  • Backups automáticos. Snapshots diários incrementais, com retenção configurável (até 35 dias). Suporte a point-in-time recovery (PITR) com granularidade de segundos, dentro da janela de retenção.
  • Segurança. RDS roda dentro de uma VPC; o acesso é controlado por Security Groups. Dados em repouso podem ser criptografados com AWS KMS; em trânsito, com TLS. Autenticação pode ser por senha, IAM database authentication (para Postgres e MySQL) ou AWS Secrets Manager integrado.
  • Performance Insights. Ferramenta de observabilidade que mostra queries lentas, wait events e recursos consumidos.
Custo. RDS cobra por instância rodando (mesmo ociosa), storage alocado e transferência de dados para fora da região. No AWS Learner Lab o budget é limitado. Nesta aula usaremos db.t3.micro, que é a instância mais barata e cabe no free-tier quando disponível. Ao final do lab, pare ou delete o RDS para preservar o saldo.

Módulo 9 — Amazon DynamoDB

DynamoDB é o banco NoSQL fully managed da AWS. É serverless: você não provisiona servidores nem escolhe tamanho de instância. Apenas define a tabela, as chaves e (opcionalmente) a capacidade. DynamoDB entrega latência de milissegundos em qualquer escala, replicado automaticamente em três AZs por região.

Modelo de tabela

  • Partition Key (PK). Atributo obrigatório. O valor da PK é hasheado e determina em qual partição física o item será armazenado. Uma boa PK tem alta cardinalidade (muitos valores distintos) e distribuição uniforme, para evitar hot partitions.
  • Sort Key (SK). Atributo opcional. Permite ordenar itens dentro da mesma PK e consultar por range (begins_with, between). A combinação (PK, SK) deve ser única.
  • Atributos adicionais. Cada item pode ter qualquer quantidade de atributos extras, com tipos variados (string, number, binary, list, map, set, boolean, null). Um item não precisa ter os mesmos atributos que outros.
  • LSI (Local Secondary Index). Mesmo PK da tabela principal, mas outro SK. Precisa ser criado na criação da tabela e não pode ser removido. Compartilha a capacidade da tabela base.
  • GSI (Global Secondary Index). PK e SK totalmente diferentes da tabela base. Pode ser criado a qualquer momento. Replicado assincronamente (eventually consistent). Tem capacidade própria.

Modos de capacidade

  • On-demand (PAY_PER_REQUEST). Você paga por request (RCU para leitura, WCU para escrita). A tabela escala automaticamente. Recomendado quando a carga é imprevisível ou está começando.
  • Provisioned. Você reserva RCU e WCU por segundo. Mais barato quando a carga é previsível. Auto-scaling pode ajustar os limites dinamicamente dentro de bandas configuradas.

DAX — DynamoDB Accelerator

DAX é um cache write-through em memória, compatível com a API DynamoDB. Não altera o código da aplicação: você redireciona o client DynamoDB para o endpoint DAX e passa a ter latências de microssegundos em reads repetidos. Útil em cenários com leitura pesada do mesmo conjunto pequeno de itens (catálogos, configurações).

Design por padrões de acesso. Em DynamoDB, a modelagem começa identificando todas as queries que a aplicação precisa rodar. Diferentemente de SQL, você não pode "improvisar" uma query depois: ou ela se encaixa em uma PK/SK/GSI, ou é um scan caríssimo. Leitura recomendada: The DynamoDB Book, Alex DeBrie.

Módulo 10 — Padrões de ingestão OLTP para data lake

Há três padrões clássicos para levar dados de um banco transacional ao data lake. A escolha depende do volume da tabela, da existência de colunas de auditoria e do requisito de latência.

Full snapshot

Exporta a tabela inteira periodicamente (diariamente, por exemplo) e escreve no lake com particionamento por data. Simples, mas inviável para tabelas grandes. Ideal para tabelas pequenas de referência (ex.: products, categories).

Ingestão incremental

Depende da existência de uma coluna updated_at (ou equivalente) na tabela fonte. A cada execução, o pipeline lê apenas linhas com updated_at > last_run e mescla no destino. Reduz volume, mas não captura deletes a menos que exista um esquema de soft delete.

CDC — Change Data Capture

O pipeline lê o próprio log de replicação do banco (WAL no PostgreSQL, binlog no MySQL) e replica cada INSERT, UPDATE e DELETE para o destino, near real-time. É o padrão mais robusto para tabelas grandes e captura todas as mudanças, inclusive deletes. O AWS DMS implementa CDC com poucos cliques.

AWS Database Migration Service (DMS)

DMS é o serviço gerenciado da AWS para migrar e replicar bancos. Os componentes principais são:

  • Replication Instance — uma instância EC2 gerenciada que roda o processo de replicação.
  • Source Endpoint — RDS, Aurora, Oracle, SQL Server, MongoDB, DocumentDB, entre outros.
  • Target Endpoint — S3, Redshift, Kinesis, outro RDS, DynamoDB etc.
  • Migration Task — define quais tabelas replicar, com quais filtros e transformações, em que modo (full load, CDC only, full load + CDC).

Trade-off: replicar ou consultar a fonte viva?

Há cenários em que consultar o banco transacional diretamente é aceitável — especialmente se existir uma read replica dedicada à analítica. Mas, para qualquer workload sério, replicar para o data lake é a escolha padrão. Os motivos:

  • Protege o OLTP de queries pesadas.
  • Permite manter histórico ilimitado no lake, mesmo que a fonte faça purge.
  • Desacopla consumidores analíticos da disponibilidade do OLTP.
  • Torna viável combinar múltiplas fontes (RDS + DynamoDB + SaaS) num único lugar.

Módulo 11 — Laboratório prático (20h30 – 22h00)

O laboratório desta aula usa o AWS Learner Lab. Abra o lab, clique em Start Lab, aguarde o ponto verde e então acesse o console AWS. Região padrão: us-east-1. Use a role LabRole sempre que precisar escolher um role. Todos os recursos criados devem ficar na mesma conta e região.

Passo 1 — Criar o RDS PostgreSQL

  1. No console, pesquise e abra RDS.
  2. Clique em Create database.
  3. Escolha Standard create.
  4. Em Engine options, selecione PostgreSQL versão 15.x.
  5. Em Templates, escolha Free tier (ou Dev/Test, se o free tier não estiver disponível).
  6. Em Settings:
    • DB instance identifier: olist-oltp
    • Master username: postgres
    • Master password: defina uma senha forte e anote em local seguro
  7. Em Instance configuration, selecione db.t3.micro.
  8. Em Storage: tipo gp3, 20 GB alocados, sem auto-scaling.
  9. Em Connectivity:
    • VPC: default
    • Public access: Yes
    • VPC security group: crie um novo chamado olist-oltp-sg
    • Availability zone: qualquer
  10. Em Additional configuration: deixe o initial database em branco (criaremos o banco olist manualmente via psql).
  11. Clique em Create database. Aguarde alguns minutos até o status ficar Available.
  12. Depois de criado, abra o security group olist-oltp-sg e adicione uma inbound rule: TCP 5432, source: seu IP público (My IP).

  freight_value NUMERIC(10,2),
  PRIMARY KEY (order_id, order_item_id)
);

Passo 5 — Queries analíticas

-- top 5 categorias por receita
SELECT p.product_category,
       SUM(oi.price) AS receita
  FROM order_items oi
  JOIN products p ON oi.product_id = p.product_id
 GROUP BY p.product_category
 ORDER BY receita DESC
 LIMIT 5;

-- ticket medio por estado
SELECT c.customer_state,
       ROUND(AVG(total.valor),2) AS ticket_medio
  FROM (SELECT order_id, SUM(price + freight_value) AS valor
          FROM order_items GROUP BY order_id) total
  JOIN orders    o ON total.order_id  = o.order_id
  JOIN customers c ON o.customer_id   = c.customer_id
 GROUP BY c.customer_state
 ORDER BY ticket_medio DESC;

  --table-name olist_events \
  --attribute-definitions \
      AttributeName=customer_id,AttributeType=S \
      AttributeName=event_timestamp,AttributeType=S \
  --key-schema \
      AttributeName=customer_id,KeyType=HASH \
      AttributeName=event_timestamp,KeyType=RANGE \
  --billing-mode PAY_PER_REQUEST \
  --region us-east-1

Crie o arquivo events.json:

{

Passo 8 — Consultar com query e PartiQL

# query classico
aws dynamodb query \
  --table-name olist_events \
  --key-condition-expression "customer_id = :c" \
  --expression-attribute-values '{":c":{"S":"c1"}}'

# PartiQL (SQL-like)
aws dynamodb execute-statement \
  --statement "SELECT * FROM olist_events WHERE customer_id = 'c1'"

Passo 9 — pg_dump para S3

Esta etapa simula a ingestão batch do OLTP para o data lake. Substitua <seu-login> pelo prefixo do bucket raw que você criou na Aula 01. 

# exportar o banco olist
pg_dump -h olist-oltp.abc123xyz.us-east-1.rds.amazonaws.com \
        -U postgres -d olist \
        -f olist_2026-04-30.sql

# subir para o bucket raw com particionamento por data
aws s3 cp olist_2026-04-30.sql \
  s3://<seu-login>-raw/rds_snapshot/olist/dt=2026-04-30/olist.sql

Passo 10 — Encerrar o RDS (obrigatório)

Não pule este passo. RDS cobra enquanto a instância existe, mesmo que você não esteja usando. No Learner Lab, esquecer uma instância pode esgotar o budget antes do fim do curso.
  1. No console RDS, selecione a instância olist-oltp.
  2. Clique em Actions → Stop temporarily (para estado parado por até 7 dias) ou Delete para remover tudo.
  3. Se escolher Delete, desmarque Create final snapshot e Retain automated backups para finalizar mais rápido.
  4. DynamoDB em on-demand não cobra por ociosidade (cobra apenas por request). Você pode manter a tabela olist_events ou executar aws dynamodb delete-table --table-name olist_events.

Checklist final do lab

  • RDS olist-oltp criado, populado e (ao final) parado ou deletado.
  • Queries analíticas executadas com sucesso no banco olist.
  • Tabela DynamoDB olist_events criada com PK/SK corretas.
  • Eventos inseridos via batch-write-item e consultados via query e PartiQL.
  • Arquivo olist_2026-04-30.sql publicado em s3://<seu-login>-raw/rds_snapshot/olist/dt=2026-04-30/.

Exercícios de fixação

  1. OLTP vs OLAP na prática. Você é engenheiro de dados em uma varejista que recebe 20.000 pedidos por dia em um PostgreSQL RDS. O time de BI começou a rodar, a cada 15 minutos, uma query do tipo SELECT categoria, SUM(valor) FROM order_items GROUP BY categoria direto na instância principal, e o aplicativo de checkout começou a apresentar timeouts. Explique detalhadamente o que está acontecendo, por que isso é um anti-padrão, e proponha duas soluções diferentes usando serviços AWS, justificando os trade-offs.
  2. Normalização. Dada a tabela a seguir, identifique todas as violações de 1FN, 2FN e 3FN, e redesenhe o schema normalizado em 3FN. Mostre o DDL SQL resultante com PKs e FKs. 
    pedidos (
  pedido_id, data_compra,
  cliente_cpf, cliente_nome, cliente_cep, cliente_cidade, cliente_uf,
  itens_comprados   -- lista de "produto_id:quantidade" separada por virgula
)
  3. ACID na vida real. Descreva um cenário concreto do Olist em que a ausência de cada uma das propriedades ACID (Atomicidade, Consistência, Isolamento, Durabilidade) causaria um bug visível para o cliente final. Seja específico sobre que operação falha e que dado fica inconsistente.
  4. Escolha CAP. Para cada sistema abaixo, decida se ele deve ser projetado como CP ou AP, justificando: (a) sistema de autenticação bancária (login + saldo), (b) feed de notícias de uma rede social, (c) controle de estoque em um e-commerce, (d) leitura de métricas de IoT para dashboard operacional, (e) transferência internacional entre bancos.
  5. Modelagem DynamoDB. Projete a tabela DynamoDB para guardar todos os eventos de sessão do site Olist, considerando as seguintes consultas de acesso: (a) listar os últimos 50 eventos de um cliente ordenados por timestamp decrescente; (b) contar os eventos do tipo checkout por dia; (c) recuperar um evento específico por seu UUID. Defina PK, SK, índices secundários necessários e justifique cada escolha.
  6. SQL vs NoSQL. Uma startup de logística precisa armazenar: (a) um cadastro de 10 milhões de clientes com CPF, nome, endereço; (b) o rastreio em tempo real de 500.000 entregas simultâneas (posição GPS a cada 30 s); (c) um catálogo de 200.000 produtos com descrição e fotos; (d) a contabilidade financeira com débito/crédito entre 3 contas internas. Indique, para cada caso, qual produto AWS você escolheria (RDS, DynamoDB, DocumentDB, Keyspaces, Timestream etc.) e por quê.
  7. Ingestão OLTP para data lake. Uma empresa tem três tabelas importantes no RDS: customers com 2 milhões de linhas e updated_at, products com 50 mil linhas sem coluna de auditoria, e order_events com 500 milhões de linhas e mudanças constantes. Para cada tabela, escolha entre full snapshot, incremental ou CDC via DMS, justificando.
  8. Custos e operação RDS. Em um laboratório do MBA, um aluno criou uma instância db.m5.large com Multi-AZ, storage gp3 de 100 GB e retenção de backup de 35 dias na região us-east-1. Ele esqueceu a instância ligada por 30 dias. Explique, com base na documentação pública da AWS: (a) que componentes geram cobrança, (b) qual ordem de grandeza do custo mensal estimado, (c) quais práticas o aluno deveria ter adotado para evitar esse cenário e (d) como o AWS Budgets e o AWS Cost Anomaly Detection ajudariam.

Integração, ETL Serverless e Catálogo de Dados

Este material aprofunda os conceitos apresentados nos slides da Aula 05. Cobre o AWS Glue Data Catalog como repositório central de metadados, o uso de Crawlers para descoberta automática de schemas, a criação de Glue Jobs com PySpark para transformações ETL serverless, e as boas práticas de pipeline de Raw para Trusted com conversão para Parquet.

Módulo 1 — AWS Glue Data Catalog: O Metastore Central

O que é um catálogo de dados e por que é essencial

Em um Data Lake maduro, dezenas de fontes diferentes geram centenas de tabelas armazenadas em múltiplos formatos e prefixos no S3. Sem um catálogo central, cada equipe precisa conhecer manualmente onde os dados estão, qual é o schema de cada arquivo, quais colunas existem e o que cada uma significa. Esse conhecimento fica disperso em planilhas, wikis e na cabeça das pessoas, gerando silos de informação e retrabalho constante.

O AWS Glue Data Catalog resolve esse problema sendo o repositório central de metadados do ecossistema analítico AWS. Ele armazena:

  • Databases: agrupamentos lógicos de tabelas (equivalente a schemas no SQL).
  • Tables: definições de schema, formato, localização no S3 e particionamento.
  • Partitions: metadados de cada partição individual de uma tabela.
  • Connections: configurações de conexão JDBC para bancos externos (RDS, Redshift).

O Glue Catalog é compatível com o Apache Hive Metastore, o que significa que todos os serviços AWS que entendem Hive — Athena, Redshift Spectrum, EMR, Glue Jobs — compartilham as mesmas definições de tabela sem duplicação. Alterar o schema no Catalog reflete imediatamente em todos os serviços.

Glue Crawlers: descoberta automática de schema

Um Crawler é um agente do Glue que escaneia uma fonte de dados (S3, banco JDBC, DynamoDB) e automaticamente infere o schema, cria ou atualiza tabelas no Catalog, e detecta novas partições. Sem Crawlers, você precisaria escrever e executar manualmente os DDLs de criação de tabela sempre que novos dados chegassem ou o schema mudasse.

O Crawler usa classificadores para detectar o formato dos arquivos (CSV, JSON, Parquet, ORC, Avro) e inferir tipos de colunas. Para CSV, ele usa heurísticas para detectar o delimitador e amostras de valores para inferir tipos. Para Parquet, ele lê o schema embutido nos metadados do arquivo.

Boas práticas para Crawlers:

  • Configure um crawler separado por camada (Raw, Trusted, Refined) e por domínio.
  • Agende crawlers para rodar após os jobs que populam cada camada.
  • Use o modo Crawl new sub-folders only para incrementalizar a descoberta de partições sem re-escanear dados já catalogados.
  • Defina um database de destino diferente para cada ambiente (dev, staging, prod).

Módulo 2 — AWS Glue Jobs: ETL Serverless com PySpark

O que é um Glue Job

Um Glue Job é um script ETL executado em um ambiente serverless gerenciado pela AWS. Você escreve o código (Python com PySpark ou Scala com Spark), define o tipo de worker e o número de DPUs (Data Processing Units), e o Glue provisiona o cluster Spark, executa o job e desprovisiona os recursos automaticamente. Você paga apenas pelo tempo de execução em DPU-segundos.

O Glue oferece três tipos de worker:

  • Standard: 2 vCPUs, 8 GB de RAM. Uso geral.
  • G.1X: 4 vCPUs, 16 GB de RAM. Recomendado para transformações com uso intensivo de memória.
  • G.2X: 8 vCPUs, 32 GB de RAM. Para workloads pesados com shuffles frequentes e datasets muito grandes.

Estrutura de um Glue Job em PySpark

Um Glue Job típico para converter dados da camada Raw (CSV) para Trusted (Parquet) tem a seguinte estrutura: 

import sys
from awsglue.transforms import *
from awsglue.utils import getResolvedOptions
from pyspark.context import SparkContext
from awsglue.context import GlueContext
from awsglue.job import Job
from pyspark.sql.functions import col, to_date, trim

## @params: [JOB_NAME]
args = getResolvedOptions(sys.argv, ['JOB_NAME'])

sc = SparkContext()
glueContext = GlueContext(sc)
spark = glueContext.spark_session
job = Job(glueContext)
job.init(args['JOB_NAME'], args)

# Ler da camada Raw via Glue Data Catalog
raw_df = glueContext.create_dynamic_frame.from_catalog(
    database="datalake_lab",
    table_name="raw_pedidos"
).toDF()

# Transformações de qualidade
trusted_df = (
    raw_df
    .withColumn("pedido_id", trim(col("pedido_id")))
    .withColumn("data_pedido", to_date(col("data_pedido"), "yyyy-MM-dd"))
    .withColumn("valor", col("valor").cast("decimal(10,2)"))
    .dropDuplicates(["pedido_id"])
    .filter(col("valor") > 0)
)

# Gravar na camada Trusted em Parquet com particionamento
trusted_df.write \
    .mode("overwrite") \
    .partitionBy("status") \
    .parquet("s3://seu-bucket/trusted/dominio=vendas/entidade=pedidos/")

job.commit()

Glue Studio: interface visual para ETL

O Glue Studio oferece uma interface drag-and-drop para construir pipelines ETL visualmente, sem escrever código. Você arrasta nós de Source (Glue Catalog, S3, JDBC), Transform (Filter, Join, Map, Drop Fields, Change Schema) e Target (S3, Glue Catalog, JDBC), conecta-os visualmente, e o Glue Studio gera o código PySpark automaticamente. Para equipes que precisam de agilidade sem experiência profunda em Spark, o Studio é um ponto de entrada poderoso.

Bookmarks: processamento incremental

O Glue Job Bookmark é um mecanismo que registra quais dados já foram processados em execuções anteriores. Ao reexecutar o job, o Glue lê apenas os novos dados adicionados desde a última execução bem-sucedida, evitando reprocessar tudo. É essencial para pipelines que rodam periodicamente sobre dados que crescem continuamente.

Módulo 3 — Laboratório: Catálogo com Crawler e ETL com Glue Job

A prática utiliza o ambiente AWS Student. Usaremos os dados criados nas aulas anteriores.

Passo 1 — Criar um Glue Crawler para a camada Raw

  1. No console AWS, acesse AWS Glue e clique em Crawlers.
  2. Clique em Create crawler.
  3. Nome: crawler-raw-lab. Clique em Next.
  4. Em Data source, clique em Add a data source.
  5. Tipo: S3. Path: s3://seu-bucket/raw/. Clique em Add an S3 data source.
  6. Clique em Next.
  7. Em IAM role, selecione uma role com permissões de Glue e S3 (crie uma com AWSGlueServiceRole + AmazonS3FullAccess se necessário).
  8. Clique em Next.
  9. Em Output configuration, em Target database, clique em Add database, insira datalake_lab e confirme.
  10. Prefix para tabelas criadas: raw_.
  11. Clique em Next, revise e clique em Create crawler.
  12. Selecione o crawler e clique em Run. Aguarde o status Succeeded.
  13. Validação: acesse Databases no Glue e confira que as tabelas foram criadas automaticamente com schemas inferidos.

Passo 2 — Explorar o Glue Data Catalog

  1. No menu do Glue, acesse Databases e clique em datalake_lab.
  2. Clique em Tables e inspecione as tabelas criadas pelo crawler.
  3. Clique em uma tabela e explore: schema detectado, localização no S3, formato, e partições detectadas.
  4. Acesse o Athena e execute SHOW TABLES IN datalake_lab; para confirmar que as tabelas catalogadas estão disponíveis para query.
  5. Validação: as tabelas criadas pelo Crawler aparecem no Athena sem necessidade de DDL manual.

Passo 3 — Criar um Glue Job ETL no Glue Studio

  1. No menu do Glue, acesse ETL Jobs e clique em Visual ETL.
  2. Clique em Add nodes e adicione um nó AWS Glue Data Catalog como fonte. Configure: Database = datalake_lab, Table = a tabela raw_pedidos criada pelo crawler.
  3. Adicione um nó Change Schema e configure os tipos de colunas:
    • valor: altere de string para double
    • data_pedido: altere de string para date
  4. Adicione um nó Filter e configure: valor > 0 para remover registros inválidos.
  5. Adicione um nó Amazon S3 como destino. Configure: Format = Parquet, Compression = Snappy, Path = s3://seu-bucket/trusted/dominio=vendas/entidade=pedidos/.
  6. No painel de Job details:
    • Nome: etl-raw-to-trusted-pedidos
    • IAM Role: a mesma role com acesso a Glue e S3
    • Number of workers: 2 (mínimo para o lab)
    • Worker type: G.1X
  7. Clique em Save e depois em Run.
  8. Acompanhe o job na aba Runs. Aguarde o status Succeeded.
  9. Validação: acesse o S3 e confirme que arquivos Parquet foram criados no prefixo trusted/dominio=vendas/entidade=pedidos/.

Passo 4 — Validar os dados Trusted no Athena

  1. Execute um novo Crawler apontando para s3://seu-bucket/trusted/ para catalogar a camada Trusted, ou crie a tabela manualmente com DDL no Athena.
  2. Execute uma query comparando Raw e Trusted para validar a transformação:
-- Raw: tipos originais
SELECT pedido_id, data_pedido, valor FROM datalake_lab.raw_pedidos;

-- Trusted: tipos corrigidos e dados filtrados
SELECT pedido_id, data_pedido, valor FROM datalake_lab.trusted_pedidos;
  1. Validação: os tipos de dados na camada Trusted estão corretos (DATE para data_pedido, DOUBLE para valor) e apenas registros com valor > 0 aparecem.

Exercícios de fixação

  1. Catálogo versus DDL manual: explique o papel do AWS Glue Data Catalog em uma plataforma analítica com 10 equipes diferentes consumindo dados de 50 tabelas distintas. Quais problemas surgiriam se cada equipe mantivesse sua própria definição de tabela local, e como o Catalog centralizado resolve cada um desses problemas?
  2. Pipeline Raw para Trusted: desenhe (em texto, com bullet points) o fluxo completo de um Glue Job que transforma dados de pedidos da camada Raw (CSV com schema inconsistente) para a camada Trusted (Parquet particionado). Especifique cada transformação necessária, o tipo de output, o particionamento escolhido, e como o Job Bookmark seria configurado para processamento incremental.
  3. Classificação de qualidade de dados: um Glue Crawler detectou uma tabela com as seguintes características: coluna preco inferida como STRING (deveria ser DOUBLE), coluna data_criacao inferida como STRING (deveria ser TIMESTAMP), e 15% dos registros com pedido_id nulo. Escreva as transformações PySpark necessárias para corrigir cada problema em um Glue Job.
  4. Glue Studio versus código PySpark manual: compare as vantagens e desvantagens do Glue Studio (interface visual) versus escrever o código PySpark manualmente. Em quais cenários cada abordagem é mais adequada? Considere manutenibilidade, controle, curva de aprendizado e capacidade de versionar o código.
  5. Estratégia de catálogo para múltiplos ambientes: uma empresa tem três ambientes: dev, staging e prod. Os dados de cada ambiente estão em buckets S3 separados. Descreva como você organizaria o Glue Data Catalog (databases, tabelas, prefixos) para garantir que jobs do ambiente dev não leiam dados de prod, e que seja fácil promover a definição de uma tabela de staging para prod.

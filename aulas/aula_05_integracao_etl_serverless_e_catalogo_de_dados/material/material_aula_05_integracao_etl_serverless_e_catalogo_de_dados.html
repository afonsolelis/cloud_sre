Pipeline de testes white box + black box: GitHub Actions com EC2 distribuído

Este material acompanha a Aula 05 da disciplina Cloud Computing e SRE — Visão Prática com AWS do MBA em Engenharia de Dados. A aula constrói o primeiro closed loop humano do projeto Olist: um pipeline de Continuous Integration que quebra o pull request quando o código viola um RF, um RNF ou um contrato de dashboard. O pipeline é distribuído entre runner público da GitHub (white box, determinístico) e o EC2 marcado Role=runner da Aula 04 (black box, dentro da VPC). A conexão entre o EC2 e o GitHub é feita por SSM Run Command — sem console, sem SSH, sem IP exposto.

Pré-requisitos Aula 04 entregue: stack olist-sre-pipeline-lab ativo, templates CFN versionados em iac/cloudformation/, EC2 candidato com tag Role=runner, squad de agentes de IA configurada (Architect, DevOps, Tester, Reviewer), tag aula-04 no git.

1.2 A tese desta aula

Sistemas distribuídos exigem closed loops — sensor mede, comparador avalia, controlador decide, atuador executa. A Aula 03 mapeou cinco loops automáticos (backpressure, circuit breaker, quarentena reentrante, freeze por error budget, drift de dados). A Aula 04 acendeu o primeiro deles (drift detection). Hoje montamos o sexto loop, que tem uma diferença importante: o atuador é humano. O CI é a primeira malha fechada onde a saída do controlador (status check) age sobre a decisão de uma pessoa (apertar Merge ou não).

Cinco regras inegociáveis do pipeline desta aula
  1. Nenhum PR vai para main sem checks verdes nos dois workflows.
  2. Toda mudança em iac/ ou em etl/ roda wb.yml e bb.yml.
  3. Self-hosted runner é ephemeral — um job, uma instância de runner, um descarte.
  4. Nenhuma access key permanente em workflow. Apenas session token rotacionado a cada aula.
  5. CI vermelho em main dispara cancel-update-stack automático no CloudFormation.

2. White box — conhecendo a implementação

Teste de caixa branca é aquele em que quem escreve o teste tem acesso ao código. A pergunta que ele responde é "a implementação está correta dado o caminho que escolhi?". A unidade de medida é a função, a branch, o ramo do if. Se você ainda não viu o código, não é caixa branca.

2.1 Unit tests

Unit é o piso da pirâmide. Função pura, dependências mockadas, milissegundos por teste. O critério para reconhecer um bom unit test é o tempo de execução: se um teste demora mais de 50ms, ele provavelmente está fazendo I/O real e deveria ser integration. A vantagem do unit é o ciclo de feedback — o engenheiro roda pytest -m unit antes mesmo de abrir o PR e descobre erros enquanto a intenção ainda está fresca.

Para guiar a quantidade de testes por função, usamos a complexidade ciclomática (CC). CC é o número de caminhos linearmente independentes da função: cada if, for, and/or, try/except soma um. Funções com CC ≤ 5 normalmente precisam de um único teste; CC entre 6 e 10 pede um teste por branch importante; CC > 10 é sinal de refactor — testar primeiro só fixa a complexidade no lugar.

2.2 Integration tests

Integration sobe os colaboradores reais e roda o ETL contra eles. O ganho aqui é capturar tudo que o mock esconde: serialização de tipo nativo, timezone do Postgres, encoding do CSV, transação aninhada, constraint do banco. O custo é tempo — um teste de integração demora segundos, não milissegundos.

Hoje (Aula 05) ainda usamos um Postgres mockado com SQLite em memória. É portável e roda no runner GitHub-hosted, mas mente sobre tipos. Na Aula 06 substituímos por Testcontainers com Postgres 14 real em Docker. Os identificadores TC-WB-INT-NN permanecem — só a infraestrutura por trás muda.

2.3 Coverage — que tipo importa

Tipo de coverageO que medeVeredito
Statement / line% de linhas executadas pelo menos uma vezMínimo aceitável — engana fácil
Branch% de if/else exercitados nos dois ladosIndispensável para lógica de validação

Function% de funções com pelo menos 1 chamadaÚtil para flagrar dead code

A política do projeto é deliberadamente conservadora: 80% line + 70% branch nas funções marcadas com decorator @critical. O restante do projeto roda sem gate de coverage. Coverage de 100% no projeto inteiro é um número que enche os olhos do gestor e cria testes ruins; coverage focado nas funções que governam idempotência, normalização e validação é onde a métrica vira sinal e não ruído.

2.4 Mutation testing

Mutation testing inverte a relação: em vez de medir se o teste roda o código, mede se o teste detecta mudanças no código. A ferramenta gera mutantes (troca > por >=, and por or, retorna None) e roda a suíte. Se a suíte continua verde, o mutante "sobreviveu" — havia uma cobertura falsa.

Em Python usamos mutmut ou cosmic-ray. O custo é alto: cada mutante exige rodar a suíte inteira. Por isso não roda em todo PR — roda agendado (cron noturno) ou em pré-release. E o gate é suave: warning, não falha. Mutation score abaixo de 70% nas funções críticas vira ticket de débito técnico, não bloqueio de merge.

3. Black box — defendendo o contrato

Teste de caixa preta é o que não conhece a implementação. A pergunta é "a promessa para quem consome continua sendo cumprida?". É escrito por quem usa, ou por QA, ou por uma ferramenta que conhece só a interface. Quando um refactor interno passa em todos os black box, está garantido que nenhum consumidor vai notar a mudança.

3.1 Contract tests

Contrato é a promessa explícita de uma interface. JSON Schema descreve um payload esperado; Great Expectations descreve uma suíte de propriedades sobre uma tabela inteira (linhas, ranges, nulls); OpenAPI snapshot tira foto da especificação atual e diff entre versões denuncia breaking change.

No projeto Olist, o contrato mais sensível é o do dashboard da diretoria: as colunas kpi_receita_mensal, kpi_ticket_medio_estado, kpi_top10_categoria. Se o ETL mudar nome de coluna ou tipo, o contract test fica vermelho antes do dashboard quebrar para a diretoria. Esse é o ROI do black box.

3.2 API tests

O ETL hoje expõe um endpoint pequeno (POST /load, GET /healthz, GET /metrics). Os API tests usam pytest + httpx para bater nesses endpoints e validar status, headers, payload e idempotência. Identificadores: TC-BB-API-NN. O caso TC-BB-API-04 verifica RF-06 — chama POST /load duas vezes com o mesmo payload e checa que o Postgres tem a quantidade certa de linhas, não o dobro.

3.3 Property-based tests

Property-based testing (com a biblioteca hypothesis) gera entradas aleatórias dentro de um domínio descrito e procura contraexemplo. Em vez de "para esta entrada x, espere y", a propriedade é "para qualquer x do domínio D, a saída deve ter a propriedade P". Funciona muito bem para a quarentena (RF-03): a propriedade é "se a linha não passa na validação, o sistema deve gravá-la em quarentena com motivo, e jamais deve crescer a tabela final".

3.4 End-to-end tests com Schemathesis

Schemathesis é uma ferramenta de fuzz baseada em OpenAPI. Você aponta para a spec gerada do ETL, e ela cria centenas de chamadas válidas e inválidas, verificando se o servidor responde dentro do contrato. É property-based aplicado a APIs HTTP. Roda no runner self-hosted porque precisa atravessar a VPC para alcançar o ETL.

4. A pirâmide de testes do projeto Olist

A pirâmide de testes (Mike Cohn, 2009) é o modelo mental: muitos testes rápidos baratos na base, poucos testes lentos caros no topo. Há variantes (diamante, troféu) para sistemas com forte componente UI; para nosso pipeline de dados, a pirâmide clássica funciona bem. A diferença é que cada camada também carrega uma decisão de onde roda — runner público ou self-hosted.

5. GitHub Actions — o motor do pipeline

GitHub Actions é a plataforma de CI/CD nativa do GitHub. Um workflow é um arquivo YAML em .github/workflows/ com triggers (on:), jobs e steps. Cada job roda em um runner (máquina virtual provisionada pela GitHub ou self-hosted). Steps de um job rodam em sequência, mas jobs paralelizam por padrão — needs: introduz dependência.

5.1 Triggers que importam

  • push — commit chega numa branch.
  • pull_request — PR aberto, atualizado ou reaberto.
  • workflow_dispatch — botão manual; útil para suite de carga.
  • schedule — cron noturno; útil para mutation.
  • workflow_run — reage a outro workflow terminar; usamos no closed loop CI ↔ CFN.

5.2 Filtros indispensáveis

Sem filtro, todo commit roda todo workflow — quota da GitHub queima rápido. Os dois filtros que sempre vão num workflow real: 

on:
  pull_request:
    paths: ["etl/**", "tests/**", ".github/workflows/wb.yml"]
  push:
    branches: [main]

concurrency:
  group: wb-${{ github.ref }}
  cancel-in-progress: true

paths: diz "só rode se tocou em arquivos relevantes". concurrency: diz "se eu já estou rodando para essa branch, cancela o anterior" — evita pilhas de jobs presas atrás de uma série de commits no mesmo PR.

5.3 Padrões avançados — matrix, reusable, composite

Matrix roda a mesma lógica com várias dimensões (Python 3.11/3.12/3.13, ou unit/integration/contract). fail-fast: false é o padrão recomendado — se um item da matrix quebra, os outros continuam para dar visibilidade total. Já reusable workflows (workflow_call:) deixam um workflow chamar outro, com inputs e secrets passados explicitamente — centraliza lógica que se repete entre repos.

5.4 Secrets, environments e a etiqueta de fork

MecanismoQuem lêQuando usar
Secrets do repoWorkflows do mesmo repoToken genérico de teste
Secrets da orgRepos selecionadosToken compartilhado entre projetos

Detalhe importante de segurança: PR aberto a partir de um fork não recebe os secrets. Isso protege contra workflow malicioso introduzido por contribuidor externo. Se sua suíte precisa de secret e o repo aceita PR de fork, separe em workflow disparado por workflow_run da branch principal — nunca dispare por pull_request de fork com secrets.

6. OIDC entre GitHub e AWS — o ideal e o realista

OpenID Connect (OIDC) é a forma moderna de federar identidade entre GitHub Actions e AWS. Em vez de guardar uma access key permanente como secret, o workflow apresenta um JWT assinado pelo GitHub, e a AWS — configurada com aquele issuer como IdP federado — emite uma credencial temporária para a role configurada. Auditável, curto, sem rotação manual.

6.1 Por que OIDC não roda no Learner Lab

Para configurar federation OIDC, a conta AWS precisa criar um recurso iam:CreateOpenIDConnectProvider e uma role com sts:AssumeRoleWithWebIdentity apontando para o issuer do GitHub. Ambas as ações estão bloqueadas pela política do AWS Academy Learner Lab. Para o curso, então, ensinamos OIDC como conceito — é o que se usa em produção — e adotamos um caminho mais simples no laboratório: o session token rotacionado a cada início de aula.

6.2 ADR-05 — session token no Learner Lab

ADR-05 · usar session token rotacionado em vez de OIDC
  • Contexto: Learner Lab bloqueia iam:CreateOpenIDConnectProvider; OIDC indisponível.
  • Decisão: usar AWS_ACCESS_KEY_ID, AWS_SECRET_ACCESS_KEY e AWS_SESSION_TOKEN como secrets do repo, atualizados a cada início de aula.
  • Consequências: CI quebra quando o token expira (~3h); necessário renovar manualmente.
  • Mitigação: primeiro step de todo workflow chama aws sts get-caller-identity — falha cedo, mensagem clara, evita erro confuso lá no meio do job.
  • Validade: apenas neste laboratório. Em produção, sempre OIDC ou Roles Anywhere.

7. Self-hosted runners no EC2

GitHub-hosted runner é grátis até quota e ótimo para trabalho determinístico (lint, unit). Mas roda fora da VPC. Para tocar em recursos privados (Postgres atrás de SG fechado, S3 com endpoint policy restritiva, ETL escutando em porta interna) o runner público obriga abrir caminho pela internet ou ter NAT — aumenta superfície e custo. Self-hosted dentro da VPC mata o problema.

7.1 Por que vale o esforço

  • Custo: 2.000 minutos grátis por mês acabam rápido com matrix; EC2 t3.micro está coberto pelo Free Tier durante o curso.
  • Latência: EC2 us-east-1 falando com S3 us-east-1 é praticamente LAN; runner GitHub-hosted dá um salto pela internet pública. Suite de integração observada cai para um quarto do tempo.
  • Acesso privado: Postgres já em SG fechado — o runner já está dentro da VPC, sem precisar abrir 0.0.0.0/0 no firewall.

7.2 Os riscos — e por que não usar em produção

Self-hosted runner executa o que o workflow mandar com a identidade da máquina (instance role). Isso significa que um PR malicioso de fork, se conseguir disparar workflow self-hosted, roda código arbitrário dentro da sua VPC. As ameaças clássicas:

  • Workflow lê instance metadata (IMDS) e exfiltra credencial.
  • Persistência entre jobs — cache envenenado para o próximo PR.
  • Privilege escalation no host se o runner roda como root.
  • Mineração silenciosa em runner ocioso.
  • Movimento lateral pela VPC para serviços vizinhos.

7.3 Mitigações obrigatórias

  • --ephemeral: o runner roda um job e some. Sem persistência, sem cache envenenado.
  • runs-on: [self-hosted, lab]: label específico, evita workflow alheio pegar o runner.
  • Branch protection bloqueia PR de fork executar workflow no self-hosted (configuração no repo: Require approval for first-time contributors).
  • IMDSv2 obrigatório no EC2 — protege contra SSRF que tentaria ler credencial.
  • Instance role com policy mínima: read no S3 do projeto, logs:Put* no CloudWatch, e nada mais.
  • Nunca usar self-hosted runner em conta de produção da empresa, especialmente em repos públicos.

8. Conectar o EC2 sem console — SSM Run Command

A regra do projeto (Aula 04) é: nada por console. O que vai ao código vai pelo CLI; o que vai ao EC2 vai pelo SSM. SSM Run Command executa um script remotamente usando a instance role — sem SSH, sem porta 22 aberta, sem IP exposto. Cada execução é registrada no CloudTrail com quem mandou, o quê, quando e qual foi a saída.

8.1 Pré-requisitos no EC2

  • SSM Agent instalado (Amazon Linux 2 e 2023 já vêm com ele).
  • Instance profile com AmazonSSMManagedInstanceCore anexado.
  • Conectividade com endpoint do SSM — via internet ou VPC endpoint.
  • Tag Role=runner — deixa o filtro CLI inequívoco.

8.2 Comando completo

OWNER=$(gh repo view --json owner -q .owner.login)
REPO=$(gh repo view --json name -q .name)

# 1) Localiza a instancia candidata
INSTANCE_ID=$(aws ec2 describe-instances \
  --filters "Name=tag:Role,Values=runner" \
            "Name=instance-state-name,Values=running" \
  --query "Reservations[0].Instances[0].InstanceId" --output text)

# 2) Token de registro emitido pelo GitHub (vale 1 hora)
RUNNER_TOKEN=$(gh api -X POST \
  /repos/$OWNER/$REPO/actions/runners/registration-token \
  --jq .token)

# 3) Manda o EC2 instalar e registrar como ephemeral
aws ssm send-command \
  --instance-ids "$INSTANCE_ID" \
  --document-name AWS-RunShellScript \
  --comment "register self-hosted runner ephemeral lab" \
  --parameters commands="cd /home/ec2-user; sudo -u ec2-user bash scripts/register_runner.sh $OWNER $REPO $RUNNER_TOKEN"

# 4) Confirmacao
gh api /repos/$OWNER/$REPO/actions/runners \
  --jq '.runners[] | {name, status, labels: [.labels[].name]}'

O scripts/register_runner.sh (no repo) é simplesmente um wrapper sobre o tarball oficial: baixa actions-runner-linux-x64, descompacta, chama config.sh com flag --ephemeral e dispara run.sh em background. A flag --ephemeral garante que o runner desregistra e morre depois de um job — o que é perfeito para o caso lab, onde a instância é descartável.

10. Closed loop — CI vermelho dispara rollback do change set

Aqui é onde a aula encontra o coração SRE do projeto. A Aula 04 deixou o stack vivo com drift detection. Hoje liga o pipeline ao stack: se um cloudformation update-stack for aplicado em main e o workflow falhar logo depois, o EventBridge captura o evento e dispara cancel-update-stack automaticamente. O change set ruim é revertido sem precisar de ninguém olhar o tela.

  end

10.2 Anatomia do loop

  • Sensor: evento workflow_run com conclusion=failure emitido pelo GitHub para o EventBridge da AWS (via integração GitHub-AWS ou via webhook).
  • Comparador: rule no EventBridge filtra por workflow alvo + branch main.
  • Controlador: SSM Document CancelOlistStackUpdate chama aws cloudformation cancel-update-stack.
  • Atuador: CloudFormation reverte para o último estado consistente.
  • Reobservação: drift detection já rodando reporta o estado pós-rollback.

12.8 Passo 8 — closed loop CI ↔ CFN

# Rule no EventBridge filtrando workflow_run com falha em main
aws events put-rule \
  --name olist-ci-failure-rollback \
  --event-pattern '{
    "source": ["github.actions"],
    "detail-type": ["workflow_run"],
    "detail": {
      "workflow_run": {
        "conclusion": ["failure"],
        "head_branch": ["main"]
      }
    }
  }'

# SSM Document que cancela update-stack
aws ssm create-document \
  --name CancelOlistStackUpdate \
  --document-type Automation \
  --content file://iac/ssm/cancel_update_stack.yaml

# Target da rule -> Automation
aws events put-targets \
  --rule olist-ci-failure-rollback \
  --targets "Id=1,Arn=arn:aws:ssm:us-east-1:$ACCOUNT:automation-definition/CancelOlistStackUpdate,RoleArn=arn:aws:iam::$ACCOUNT:role/LabRole"

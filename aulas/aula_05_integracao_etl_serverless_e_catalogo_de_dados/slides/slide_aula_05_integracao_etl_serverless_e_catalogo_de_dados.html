Tese da aula

CI é o primeiro closed loop humano do projeto

A Aula 03 mapeou cinco closed loops automáticos. Hoje montamos o sexto: o pipeline de PR. Sensor é o conjunto de testes; comparador é o checks status; controlador é a branch protection; atuador é o merge bloqueado e o rollback do change set.

Cinco regras do pipeline desta aula
  1. Nenhum PR vai para main sem checks verdes nos dois workflows.
  2. Toda mudança em iac/ ou em etl/ roda wb.yml e bb.yml.
  3. Self-hosted runner é ephemeral — um job, uma instância de runner, um descarte.
  4. Nenhuma access key permanente em workflow. Apenas session token rotacionado (ADR explícito).
  5. CI vermelho fora da branch ⇒ cancel-update-stack automático no CFN.

White box · Unit

Unit — a unidade está isolada e correta?

Função pura, dependências mockadas, milissegundos por teste. É o piso da pirâmide e o lugar onde mais se ganha tempo de feedback.

O que cabe num unit test
  • Uma função · uma asserção principal
  • Sem rede, sem I/O real, sem clock real
  • fixtures de pytest para reuso
  • mocker.patch em colaboradores
  • Identificador TC-WB-NN ligado à RTM
Complexidade ciclomática como guia
  • CC = número de caminhos independentes da função
  • CC ≤ 5 — 1 unit basta
  • CC 6–10 — cobre cada branch
  • CC > 10 — refatorar antes de testar
  • Ferramenta: radon cc -s etl/

Integration — os componentes se entendem?

Sobe Postgres real em container e roda o ETL contra ele. Mais lento que unit, mas captura aquilo que mock esconde: serialização, timezone, encoding, transação.

Hoje — Postgres mock

  • Testa SQL portável
  • Identificador TC-WB-INT-NN
  • Roda no runner GitHub-hosted

Aula 06 — Testcontainers

  • Captura erro de tipo nativo
  • Roda no runner EC2 com Docker
  • Mesmo identificador TC-WB-INT-NN

Antipadrões

  • Compartilhar Postgres entre testes
  • Fixture sem rollback
  • Esperar sleep() em vez de healthcheck
  • Mockar o que devia ser real
White box · Coverage

Coverage — mas qual?

Line coverage é o mais barato e o mais enganoso. Branch e path são onde a verdade aparece. Meta realista: 80% nas funções críticas, não no projeto inteiro.

TipoO que medeVeredito
Statement / line% de linhas executadasMínimo aceitável — não basta sozinho
Branch% de if/else exercitados nos dois ladosIndispensável para lógica de validação
Path% de combinações de branchesCaro — cresce exponencial
Mutation% de mutantes mortos pelos testesPadrão-ouro — mas devagar
Function% de funções com pelo menos 1 chamadaÚtil para flagrar dead code

Política do projeto: 80% line + 70% branch nas funções marcadas @critical. Resto sem gate — cobertura por moral, não por número.

White box · Mutation

Mutation testing — quem testa o teste?

A ferramenta muta seu código (troca > por >=, and por or, retorna None). Se a suíte continua verde, o mutante "sobreviveu" e há cobertura falsa.

Quando vale o custo
  • Funções de validação ou normalização
  • Cálculos de pricing, tax, dedup
  • Idempotência (RF-06) e quarentena (RF-03)
  • Antes de release — não em todo PR
Como rodar · mutmut
  • mutmut run --paths-to-mutate=etl/critical
  • mutmut results — lista sobreviventes
  • Gate suave: warning, não falha
  • CI dedicado: agendado, não em PR
  • Alternativa: cosmic-ray distribuído
Black box · Contract

Contract — o schema do dashboard ainda é o mesmo?

Contract test não conhece código. Recebe entrada, vê saída, compara com schema. Se o ETL muda nome de coluna, o contract test vermelho antes do dashboard quebrar para a diretoria.

JSON Schema

  • Contrato leve para payloads
  • Valida tipo, required, enum
  • Vive em tests/contract/schemas/
  • Identificador TC-BB-CT-NN

Great Expectations

  • Contrato pesado para tabela inteira
  • Suite de expectativas em YAML
  • Conta linhas, ranges, nulls
  • Roda contra Postgres real

OpenAPI snapshot

  • Spec gerada do código
  • Diff entre versões = breaking change
  • Gate em PR: spec mudou + sem bump = vermelho
  • Ferramenta: openapi-diff

CI · GitHub Actions 101

Anatomia de um workflow

Workflow é YAML em .github/workflows/. Tem trigger, jobs, steps. Job paraleliza por padrão; needs: introduz dependência.

# .github/workflows/wb.yml
name: white-box
on:
  pull_request:
    paths: ["etl/**", "tests/**"]
  push:
    branches: [main]

jobs:
  lint:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-python@v5
        with: { python-version: "3.12" }
      - run: pip install -r requirements-dev.txt
      - run: ruff check etl/
      - run: mypy etl/

  unit:
    needs: lint
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: pytest -m unit --cov=etl --cov-branch --cov-fail-under=80
CI · Padrões avançados

Matrix, reusable e composite

Três ferramentas para evitar workflow gigante e copy-paste entre repos.

matrix — mesma lógica, dimensões diferentes
  • Versão de Python: 3.11, 3.12, 3.13
  • Categoria de teste: unit, integration, contract
  • Combinações geradas automaticamente
  • fail-fast: false — um quebra, outros seguem
reusable — workflow chamado por outro
  • workflow_call: trigger
  • Inputs e secrets passados explicitamente
  • Centraliza lógica de teste em .github/workflows/_test.yml
  • uses: ./.github/workflows/_test.yml

PR de fork não vê secret da org — isso protege contra workflow malicioso roubando token. Environment com required reviewers transforma deploy em ato deliberado.

CI · OIDC

OIDC entre GitHub e AWS — o ideal

Sem access key estática. O workflow apresenta um JWT assinado pelo GitHub; a AWS confia naquele issuer e emite credencial temporária para a role configurada. Auditável, curto, sem rotação manual.

sequenceDiagram
  autonumber
  participant W as GitHub Workflow
  participant T as token.actions.githubusercontent.com
  participant I as AWS IAM (IdP federation)
  participant R as AWS API

  W->>T: 1. solicita JWT (id-token write)
  T-->>W: 2. JWT assinado
  W->>I: 3. AssumeRoleWithWebIdentity
  I-->>W: 4. credencial temporaria (sts)
  W->>R: 5. chama AWS API com credencial temporaria
  Note over W,R: nenhuma secret estatica
credencial expira em ate 1h
ADR-05 · Decisão

No Learner Lab: session token rotacionado

Learner Lab não permite criar IdP federation. Ensinamos OIDC como conceito e usamos session token na prática. ADR explícito no repo para que ninguém leve esse atalho para produção.

ADR-05 · resumo
  • Contexto: Learner Lab bloqueia iam:CreateOpenIDConnectProvider.
  • Decisão: usar AWS_SESSION_TOKEN em secrets.AWS_SESSION_TOKEN, rotacionado a cada início de aula.
  • Consequências: CI quebra quando o token expira (~3h); aluno precisa renovar; aceitável no laboratório, inaceitável em produção.
  • Mitigação: step de check em todo workflow chama aws sts get-caller-identity primeiro — falha cedo, mensagem clara.
Runner · Por quê

Self-hosted runner — por que pagar essa complexidade

GitHub-hosted runner é grátis até quota, mas roda fora da VPC. Para tocar em recursos privados (Postgres, S3 com endpoint policy, EC2 interno) precisa atravessar a internet ou ter NAT. Self-hosted dentro da VPC mata o problema.

Custo

  • Quota grátis: 2.000 min/mês
  • Lab queima rápido com matrix
  • EC2 t3.micro — coberto pelo Free Tier
  • Runner pago seria USD por minuto

Latência

  • EC2 us-east-1 + S3 us-east-1 = LAN
  • GitHub-hosted — salto pela internet
  • Suite de integração 4× mais rápida
  • Custo de transferência cai a zero

Acesso privado

  • Postgres em SG fechado — runner já está dentro
  • S3 com endpoint policy aws:SourceVpc
  • SSM Run Command via instance role
  • Sem precisar abrir 0.0.0.0/0
Runner · Riscos

Self-hosted — o lado sombrio

Runner self-hosted executa o que o workflow mandar com a identidade da máquina. PR malicioso de fork = código arbitrário rodando dentro da sua VPC. Mitigação não é opcional.

Ameaças
  • Workflow lê instance metadata e exfiltra credencial
  • Persistência entre jobs — cache envenenado
  • Privilege escalation no host
  • Mineração silenciosa
  • Movimento lateral pela VPC
Mitigações desta aula
  • --ephemeral — runner some após o job
  • runs-on: [self-hosted, lab] — label restritivo
  • Branch protection bloqueia PR de fork executar workflow self-hosted
  • IMDSv2 obrigatório no EC2
  • Instance role com policy mínima (read-only no S3 do projeto)
  • Nunca usar self-hosted em conta de produção da empresa
SSM · Run Command

Conectar o EC2 sem console

A regra do projeto: nada por console. SSM Run Command executa script remotamente usando a instance role — sem SSH, sem porta aberta, sem IP exposto. Cada execução fica auditada no CloudTrail.

# 1) Achar a instancia candidata (tag Role=runner da Aula 04)
aws ec2 describe-instances \
  --filters "Name=tag:Role,Values=runner" \
            "Name=instance-state-name,Values=running" \
  --query "Reservations[].Instances[].InstanceId" --output text
# saida: i-0abc123...

# 2) Pegar token de registro do runner (Codespace usa gh CLI)
RUNNER_TOKEN=$(gh api -X POST \
  /repos/$OWNER/$REPO/actions/runners/registration-token \
  --jq .token)

# 3) Mandar o EC2 instalar e registrar como ephemeral
aws ssm send-command \
  --instance-ids i-0abc123 \
  --document-name AWS-RunShellScript \
  --comment "register self-hosted runner ephemeral" \
  --parameters commands='[
    "cd /home/ec2-user && mkdir -p actions-runner && cd actions-runner",
    "curl -L -o run.tgz https://github.com/actions/runner/releases/download/v2.319.1/actions-runner-linux-x64-2.319.1.tar.gz",
    "tar xzf run.tgz",
    "./config.sh --url https://github.com/owner/repo --token TOKEN --labels lab --ephemeral --unattended",
    "nohup ./run.sh > runner.log 2>&1 &"
  ]'

Desde o push do aluno até o status check. Em verde: merge libera. Em vermelho: rollback automático do change set ativo da Aula 04.

SRE · Closed loop

CI vermelho ⇒ rollback do change set

A Aula 04 deixou drift detection. Hoje liga o pipeline ao stack: se o workflow falhar depois de um cloudformation update-stack, o EventBridge dispara cancel-update-stack. Sem esse loop, change ruim fica em produção esperando alguém perceber.

Anatomia do loop
  • Sensor: evento workflow_run com conclusion=failure emitido pelo GitHub.
  • Comparador: rule no EventBridge filtra workflow alvo + branch main.
  • Controlador: SSM Run Command com aws cloudformation cancel-update-stack.
  • Atuador: CloudFormation reverte para o último estado consistente.
  • Reobservação: drift detection rodando — reporta o stack pós-rollback.

Prática · Passo 1

Renovar o session token e setar como secret

O token de hoje precisa estar no ~/.aws/credentials do Codespace e no secret AWS_SESSION_TOKEN do repo. Os dois espelham — CI usa o do GitHub, terminal usa o local.

# 1. AWS Academy Learner Lab -> Start Lab -> AWS Details -> Show
#    Copiar bloco "AWS CLI" (3 linhas com tokens).

# 2. Atualizar credenciais locais do Codespace
mkdir -p ~/.aws
cat > ~/.aws/credentials <<'EOF'
[default]
aws_access_key_id=ASIA...
aws_secret_access_key=...
aws_session_token=...
EOF
chmod 600 ~/.aws/credentials

# 3. Espelhar nos secrets do GitHub via gh
gh secret set AWS_ACCESS_KEY_ID
gh secret set AWS_SECRET_ACCESS_KEY
gh secret set AWS_SESSION_TOKEN
gh secret set AWS_REGION --body "us-east-1"

# 4. Validar handshake
aws sts get-caller-identity

Prática · Passo 3

.github/workflows/wb.yml — white box completo

Lint → unit (com coverage) → mutation suave. Roda em ubuntu-latest — tudo que é determinístico e sem VPC fica grátis no runner público.

name: wb
on:
  pull_request:
    paths: ["etl/**", "tests/**", ".github/workflows/wb.yml"]
  push:
    branches: [main]

concurrency:
  group: wb-${{ github.ref }}
  cancel-in-progress: true

jobs:
  lint:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-python@v5
        with: { python-version: "3.12" }
      - run: pip install -r requirements-dev.txt
      - run: ruff check etl/ tests/
      - run: mypy etl/

  unit:
    needs: lint
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-python@v5
        with: { python-version: "3.12" }
      - run: pip install -r requirements-dev.txt
      - run: pytest -m unit --cov=etl --cov-branch --cov-report=xml --cov-fail-under=80
      - uses: actions/upload-artifact@v4
        with: { name: coverage-xml, path: coverage.xml }

  mutation:
    needs: unit
    if: github.event_name == 'push'
    runs-on: ubuntu-latest
    continue-on-error: true   # gate suave
    steps:
      - uses: actions/checkout@v4
      - run: pip install mutmut
      - run: mutmut run --paths-to-mutate=etl/critical || true
      - run: mutmut results
Prática · Passo 4

Plugar o EC2 como self-hosted runner via SSM

A instância já existe desde a Aula 04 com tag Role=runner. Hoje ela ganha o agente actions-runner em modo --ephemeral, sem console, sem SSH.

OWNER=$(gh repo view --json owner -q .owner.login)
REPO=$(gh repo view --json name -q .name)

# 1) Localizar a instancia runner candidata
INSTANCE_ID=$(aws ec2 describe-instances \
  --filters "Name=tag:Role,Values=runner" \
            "Name=instance-state-name,Values=running" \
  --query "Reservations[0].Instances[0].InstanceId" --output text)
echo "Instance: $INSTANCE_ID"

# 2) Token de registro do runner
RUNNER_TOKEN=$(gh api -X POST \
  /repos/$OWNER/$REPO/actions/runners/registration-token \
  --jq .token)

# 3) Mandar instalar via SSM Run Command
aws ssm send-command \
  --instance-ids "$INSTANCE_ID" \
  --document-name AWS-RunShellScript \
  --comment "register self-hosted runner ephemeral lab" \
  --parameters commands="cd /home/ec2-user; sudo -u ec2-user bash scripts/register_runner.sh $OWNER $REPO $RUNNER_TOKEN"

# 4) Confirmar online
gh api /repos/$OWNER/$REPO/actions/runners \
  --jq '.runners[] | {name, status, labels: [.labels[].name]}'

Branch protection — CI vira gate de merge

Sem branch protection, todo o pipeline é decoração. Configurar via API garante que está versionado e auditável.

OWNER=$(gh repo view --json owner -q .owner.login)
REPO=$(gh repo view --json name -q .name)

gh api -X PUT /repos/$OWNER/$REPO/branches/main/protection \
  --input - <<'EOF'
{
  "required_status_checks": {
    "strict": true,
    "contexts": [
      "wb / lint",
      "wb / unit",
      "bb / api",
      "bb / contract",
      "bb / e2e"

  },
  "enforce_admins": false,
  "required_pull_request_reviews": {
    "required_approving_review_count": 1,
    "dismiss_stale_reviews": true
  },
  "restrictions": null,
  "required_linear_history": true,
  "allow_force_pushes": false
}

Prática · Passo 8

Closed loop — CI vermelho dispara rollback

Adicionamos rule no EventBridge que escuta workflow_run com conclusion=failure em main. Target é SSM Document que chama cancel-update-stack.

# Rule no EventBridge filtrando workflow_run com falha em main
aws events put-rule \
  --name olist-ci-failure-rollback \
  --event-pattern '{
    "source": ["github.actions"],
    "detail-type": ["workflow_run"],
    "detail": {
      "workflow_run": {
        "conclusion": ["failure"],
        "head_branch": ["main"]
      }
    }
  }'

# SSM Document que cancela update-stack
aws ssm create-document \
  --name CancelOlistStackUpdate \
  --document-type Automation \
  --content file://iac/ssm/cancel_update_stack.yaml

# Target da rule -> Automation
aws events put-targets \
  --rule olist-ci-failure-rollback \
  --targets "Id=1,Arn=arn:aws:ssm:us-east-1:$ACCOUNT:automation-definition/CancelOlistStackUpdate,RoleArn=arn:aws:iam::$ACCOUNT:role/LabRole"

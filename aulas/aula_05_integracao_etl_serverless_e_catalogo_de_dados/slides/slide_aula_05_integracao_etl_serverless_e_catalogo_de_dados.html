Teoria · Fundamentos do ATAM

Architecture Tradeoff Analysis Method

O método foi proposto por Kazman, Klein e Clements no SEI/CMU e avalia uma arquitetura em face de cenários de quality attribute, com o propósito de evidenciar sensitivity points, tradeoffs, riscos e non-risks. Distingue-se de checklists e revisões de código por seu foco estrutural sobre as decisões arquiteturais.

Caracterização do ATAM
  • Constitui um método formal de revisão arquitetural.
  • É conduzido em formato de workshop com a participação dos stakeholders.
  • Tem como saída central uma tabela analítica.
  • Concentra-se na identificação de tradeoffs, e não na emissão de juízos absolutos sobre correção.
  • É aplicável em diferentes escalas, de equipes reduzidas a sistemas críticos.
Delimitações do método
  • Não consiste em revisão de código.
  • Não atribui pontuações nem produz veredictos de aprovação ou reprovação.
  • Não substitui testes automatizados ou benchmarks.
  • Não impõe soluções; sua função é tornar explícitas as escolhas arquiteturais.
  • Não se encerra em uma única sessão, configurando-se como exercício iterativo.

Teoria · Fundamentos de Bass

Bass — distinção entre tática e padrão arquitetural

Em Software Architecture in Practice (4ª edição), Len Bass define a tática como uma decisão de design que afeta diretamente a resposta de um quality attribute. O padrão é compreendido como a combinação coerente de múltiplas táticas voltadas à resolução de um problema recorrente.

Estrutura conceitual proposta por Bass
  1. Quality Attribute — availability, performance, security, modifiability, entre outros.
  2. Cenário — estímulo, ambiente, resposta e medida.
  3. Tática — a menor unidade de decisão arquitetural capaz de alterar a resposta do QA. Exemplos típicos incluem retry, cache, circuit breaker e ping/echo.
  4. Padrão — conjunto coerente de táticas (por exemplo, o padrão resilient client, que articula timeout, retry, circuit breaker e fallback).

Availability Tática · SP-01

Ping/Echo e Heartbeat — healthcheck robusto

A tática Ping/Echo é ativa: um componente formula a verificação e o outro responde. O Heartbeat, por sua vez, opera em sentido inverso: o componente comunica seu estado periodicamente, sem requisição prévia. O Compose já implementa Ping/Echo nativamente; cabe aqui o ajuste fino do timing e a adição de um endpoint customizado.

docker-compose.yaml (patch) 
clickhouse:
  healthcheck:
    test: ["CMD", "wget", "--spider", "-q", "http://localhost:8123/ping"]
    interval: 5s        # mais frequente
    timeout: 3s         # corta cedo
    retries: 6          # tolera startup lento de mergetree
    start_period: 20s   # nao reporta unhealthy nos primeiros 20s

dashboard:
  depends_on:
    clickhouse: { condition: service_healthy }
  healthcheck:
    test: ["CMD", "curl", "-f", "http://localhost:8501/_stcore/health"]
    interval: 10s
    timeout: 3s
    retries: 3
Availability Tática · SP-02

Retry, Exception Handling e Idempotency

No cenário formulado por Bass, o ingestor depara-se com throttling ou desconexão temporária e procede a novas tentativas com backoff exponencial. A propriedade de idempotência preexistente assegura que a reexecução não compromete a integridade do estado.

app/ingest_minio.py (refactor) 
from tenacity import retry, stop_after_attempt, wait_exponential, retry_if_exception_type
import botocore.exceptions, clickhouse_connect.driver.exceptions as cherr

@retry(
    stop=stop_after_attempt(5),
    wait=wait_exponential(multiplier=1, min=2, max=30),
    retry=retry_if_exception_type((
        botocore.exceptions.ClientError,
        botocore.exceptions.EndpointConnectionError,
        cherr.OperationalError,
    )),
    reraise=True,
)
def insert_batch(ch, rows):
    ch.insert("olist_raw.ingestion", rows,
              column_names=["unixtime", "data", "tag"])

A idempotência já estava assegurada por max(unixtime) e tag desde a Aula 04. Nesta sessão, a propriedade é apenas nomeada formalmente e validada por testes automatizados.

Availability + Performance Tática · SP-03

Circuit Breaker no acesso ao S3

Após um número predefinido de falhas consecutivas, a tática interrompe novas tentativas. Sem esse mecanismo, o ingestor sobrecarrega continuamente o downstream indisponível, amplificando o problema. A implementação adotada consiste em um wrapper apoiado em tenacity com estado persistido em memória.

app/breakers.py 
import time
from dataclasses import dataclass

@dataclass
class CircuitBreaker:
    failure_threshold: int = 5
    recovery_seconds: int = 30
    fail_count: int = 0
    opened_at: float | None = None

    def call(self, fn, *args, **kwargs):
        if self.opened_at and time.time() - self.opened_at < self.recovery_seconds:
            raise RuntimeError("Circuit open — downstream provavelmente fora")
        try:
            result = fn(*args, **kwargs)
            self.fail_count = 0; self.opened_at = None
            return result
        except Exception:
            self.fail_count += 1
            if self.fail_count >= self.failure_threshold:
                self.opened_at = time.time()
            raise

Circuit breaker aplicado ao boto3

Branch tactic/availability-circuit-breaker. A tarefa consiste em implementar a classe CircuitBreaker apresentada no slide 13 em app/breakers.py, encapsular as chamadas a S3 do ingest_minio.py e configurar os parâmetros threshold em 5 falhas e recovery em 30 segundos.

app/ingest_minio.py (uso do breaker) 
from breakers import CircuitBreaker
from tenacity import retry, stop_after_attempt, wait_exponential

cb = CircuitBreaker(failure_threshold=5, recovery_seconds=30)

@retry(stop=stop_after_attempt(5),
       wait=wait_exponential(multiplier=1, min=2, max=30),
       reraise=True)
def s3_get(key):
    return cb.call(lambda: s3.get_object(Bucket=BUCKET, Key=key))
Prática · Passo 4

Testes unitários com pytest

Branch tactic/testability-pytest. A atividade consiste em criar o arquivo tests/unit/test_ingestor.py com cobertura sobre quatro aspectos: (a) o retry efetua o número esperado de tentativas; (b) a idempotência é mantida via max(unixtime); (c) o circuit breaker abre após cinco falhas; (d) o endpoint de healthcheck retorna código HTTP 200.

tests/unit/test_ingestor.py 
import pytest
from unittest.mock import MagicMock
from app.breakers import CircuitBreaker

def test_ingestor_retries_on_s3_throttle():
    s3 = MagicMock(side_effect=[
        Exception("SlowDown"), Exception("SlowDown"),
        {"Body": "ok"}
    ])
    from app.ingest_minio import s3_get
    assert s3_get.retry.statistics["attempt_number"] == 3

def test_circuit_breaker_opens_after_threshold():
    cb = CircuitBreaker(failure_threshold=3, recovery_seconds=1)
    def boom(): raise RuntimeError("falhou")
    for _ in range(3):
        with pytest.raises(RuntimeError): cb.call(boom)
    with pytest.raises(RuntimeError, match="Circuit open"):
        cb.call(boom)

Prática · Passo 9 · Estresse

Cenário de estresse 4 — exaustão de disco

Branch stress/disk-full. O procedimento consiste em preencher artificialmente o volume do ClickHouse por meio do utilitário dd, tentar realizar a ingestão, observar o comportamento do MergeTree diante da indisponibilidade de espaço, registrar a mensagem de erro retornada e, ao final, restaurar o estado original do volume.

stress/04_disk_full.sh 
#!/usr/bin/env bash
# atencao: simule em um tmpfs ou volume separado; nao no /
docker exec olist_analytics bash -c '
  df -h /var/lib/clickhouse
  dd if=/dev/zero of=/var/lib/clickhouse/_fill bs=1M count=500
  df -h /var/lib/clickhouse
'
docker exec olist_ingestor python /app/ingest_minio.py || true
docker exec olist_analytics clickhouse-client \
  -q "SELECT count() FROM olist_raw.ingestion"
# restaurar
docker exec olist_analytics rm -f /var/lib/clickhouse/_fill

Saída esperada: o ClickHouse retorna a mensagem Cannot reserve ... bytes; o tenacity efetua novas tentativas; ao final, opera-se a estratégia de fail-fast. Recomenda-se documentar integralmente o ocorrido.

Coleta de evidências e revisão da tabela ATAM

Cada cenário de estresse produz um conjunto de artefatos, incluindo logs, capturas de tela do dashboard, métricas de p95 e p99, taxa de erro e MTTR. Esses artefatos são documentados em docs/stress-evidence.md e, em seguida, a tabela ATAM construída no bloco teórico é revisitada, com a adição de uma nova coluna destinada a registrar as alterações observadas após a aplicação das táticas.

Evidências por sensitivity point
  • SP-01 healthcheck: o tempo médio até o estado healthy reduziu-se de 35 para 12 segundos.
  • SP-02 ingestor: a ingestão passa a tolerar o encerramento abrupto do ClickHouse durante a execução, ao contrário do comportamento anterior, que resultava em aborto.
  • SP-03 circuit breaker: após cinco falhas consecutivas, o ingestor interrompe novas tentativas, deixando de sobrecarregar o downstream.
  • SP-04 cache: o p95 do dashboard regrediu de 1,8s para 230ms sob carga de cinquenta sessões concorrentes.
  • SP-05 credenciais: o usuário dashboard deixa de dispor de permissão para executar DDL no ClickHouse.

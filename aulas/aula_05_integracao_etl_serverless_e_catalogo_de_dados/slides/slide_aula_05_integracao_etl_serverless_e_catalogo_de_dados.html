Teoria · ATAM 101

Architecture Tradeoff Analysis Method

Metodo criado por Kazman, Klein e Clements no SEI/CMU. Avalia uma arquitetura contra cenarios de quality attribute para revelar sensitivity points, tradeoffs, riscos e non-risks. Diferente de checklist ou code review — o foco e estrutural.

O que ATAM e
  • Metodo formal de revisao arquitetural.
  • Conduzido em workshop com stakeholders.
  • Produz uma tabela como saida central.
  • Foca em tradeoffs — nao em "esta certo".
  • Cabe em qualquer escala: equipe pequena, sistema critico.
O que ATAM nao e
  • Nao e revisao de codigo.
  • Nao da nota nem aprova/reprova.
  • Nao substitui testes ou benchmarks.
  • Nao impoe solucoes — expoe escolhas.
  • Nao termina em um dia — e iterativa.

Teoria · Bass 101

Bass — tatica vs padrao arquitetural

Len Bass, em Software Architecture in Practice (4a ed.), define tatica como uma decisao de design que afeta a resposta de um quality attribute. Padrao e a combinacao coerente de varias taticas para um problema recorrente.

Mapa mental Bass
  1. Quality Attribute — availability, performance, security, modifiability, etc.
  2. Cenario — estimulo + ambiente + resposta + medida.
  3. Tatica — a menor unidade de decisao arquitetural que move o ponteiro do QA. Exemplos: retry, cache, circuit breaker, ping/echo.
  4. Padrao — conjunto coerente de taticas (ex.: resilient client = timeout + retry + circuit breaker + fallback).

Availability Tatica · SP-01

Ping/Echo + Heartbeat — healthcheck robusto

A tatica e ativa: alguem pergunta, o componente responde. Heartbeat e o oposto: o componente fala sem perguntarem. Compose ja faz ping/echo; vamos tunar timing e adicionar um endpoint custom.

docker-compose.yaml (patch) 
clickhouse:
  healthcheck:
    test: ["CMD", "wget", "--spider", "-q", "http://localhost:8123/ping"]

    timeout: 3s         # corta cedo
    retries: 6          # tolera startup lento de mergetree

Availability Tatica · SP-02

Retry + Exception Handling + Idempotency

Cenario Bass: ingestor encontra throttle ou desconexao temporaria; retenta com backoff exponencial; idempotencia ja existente garante que reexecucao nao corrompe estado.

app/ingest_minio.py (refactor) 
from tenacity import retry, stop_after_attempt, wait_exponential, retry_if_exception_type
import botocore.exceptions, clickhouse_connect.driver.exceptions as cherr

@retry(
    stop=stop_after_attempt(5),
    wait=wait_exponential(multiplier=1, min=2, max=30),
    retry=retry_if_exception_type((
        botocore.exceptions.ClientError,
        botocore.exceptions.EndpointConnectionError,
        cherr.OperationalError,
    )),
    reraise=True,
)
def insert_batch(ch, rows):
    ch.insert("olist_raw.ingestion", rows,
              column_names=["unixtime", "data", "tag"])

A idempotencia ja vinha por max(unixtime) + tag da Aula 04. Aqui apenas nomeamos e testamos.

Availability + Performance Tatica · SP-03

Circuit Breaker no acesso ao S3

Apos N falhas consecutivas, parar de tentar. Sem isso, o ingestor martela o downstream caido e amplifica o problema. Usaremos um wrapper baseado em tenacity com estado externo persistido em memoria.

app/breakers.py 
import time
from dataclasses import dataclass

@dataclass
class CircuitBreaker:
    failure_threshold: int = 5
    recovery_seconds: int = 30
    fail_count: int = 0
    opened_at: float | None = None

    def call(self, fn, *args, **kwargs):
        if self.opened_at and time.time() - self.opened_at < self.recovery_seconds:
            raise RuntimeError("Circuit open — downstream provavelmente fora")
        try:
            result = fn(*args, **kwargs)
            self.fail_count = 0; self.opened_at = None
            return result
        except Exception:
            self.fail_count += 1
            if self.fail_count >= self.failure_threshold:
                self.opened_at = time.time()
            raise

Circuit breaker no boto3

Branch tactic/availability-circuit-breaker. Implementar a classe CircuitBreaker do slide 13 em app/breakers.py; envolver as chamadas S3 do ingest_minio.py; configurar threshold = 5, recovery = 30s.

app/ingest_minio.py (uso do breaker) 
from breakers import CircuitBreaker
from tenacity import retry, stop_after_attempt, wait_exponential

cb = CircuitBreaker(failure_threshold=5, recovery_seconds=30)

@retry(stop=stop_after_attempt(5),
       wait=wait_exponential(multiplier=1, min=2, max=30),
       reraise=True)
def s3_get(key):
    return cb.call(lambda: s3.get_object(Bucket=BUCKET, Key=key))
Pratica · Passo 4

Testes unitarios com pytest

Branch tactic/testability-pytest. Criar tests/unit/test_ingestor.py cobrindo: (a) retry tenta N vezes; (b) idempotencia via max(unixtime); (c) circuit breaker abre apos 5 falhas; (d) healthcheck endpoint responde 200.

tests/unit/test_ingestor.py 
import pytest
from unittest.mock import MagicMock
from app.breakers import CircuitBreaker

def test_ingestor_retries_on_s3_throttle():
    s3 = MagicMock(side_effect=[
        Exception("SlowDown"), Exception("SlowDown"),
        {"Body": "ok"}
    ])
    from app.ingest_minio import s3_get
    assert s3_get.retry.statistics["attempt_number"] == 3

def test_circuit_breaker_opens_after_threshold():
    cb = CircuitBreaker(failure_threshold=3, recovery_seconds=1)
    def boom(): raise RuntimeError("falhou")
    for _ in range(3):
        with pytest.raises(RuntimeError): cb.call(boom)
    with pytest.raises(RuntimeError, match="Circuit open"):
        cb.call(boom)

Stress 1 — kill no meio da ingestao

Branch stress/kill-clickhouse. Iniciar a ingestao em uma janela; em outra, docker kill olist_clickhouse; observar o retry; subir o CH de novo; ver a ingestao retomar; dashboard volta sem perda de dados.

set -euo pipefail

docker kill olist_clickhouse

echo "Subindo ClickHouse..."

echo "Aguardando healthy..."

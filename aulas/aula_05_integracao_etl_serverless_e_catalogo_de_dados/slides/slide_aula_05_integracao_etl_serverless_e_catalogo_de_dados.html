Teoria · Padrões de integração

ETL vs ELT

A ordem das letras muda tudo: onde a transformação acontece redefine a arquitetura.

AspectoETLELT
OrdemExtract → Transform → LoadExtract → Load → Transform
Onde transformaFora do destino (engine ETL)Dentro do destino (DW/Lake)
Típico emDW on-premiseCloud data lake / lakehouse
VolumesMédiosGrandes / massivos
FerramentasInformatica, SSIS, DataStageGlue, dbt, Spark, Redshift
Teoria · Decisão

Quando usar cada padrão?

Use ETL quando...
  • Dados sensíveis precisam ser mascarados antes de pousar.
  • DW destino é pequeno ou caro por compute.
  • Regras de negócio complexas precisam validar antes do load.
  • Compliance exige auditoria na zona de staging.
Use ELT quando...
  • Storage é barato (S3) e compute é elástico.
  • Dados brutos precisam ficar disponíveis para reprocessar.
  • Cientistas de dados querem o dado cru.
  • Volumes são grandes e variam ao longo do dia.
Teoria · Modelo operacional

Serverless para dados

Você não gerencia o cluster. A AWS aloca recursos sob demanda, você paga pelo tempo de execução.

4 características centrais

  • Stateless — sem estado entre execuções.
  • Efêmero — recursos vivem só durante o job.
  • Event-driven — disparado por evento, cron, API.
  • Pay-per-use — cobraça por segundo de execução.

Modelo de cobrança do Glue

  • Unidade: DPU (Data Processing Unit).
  • 1 DPU = 4 vCPU + 16 GB RAM.
  • Cobraça por DPU-hora, faturada por segundo.
  • Mínimo 2 DPUs por job Spark.
Teoria · AWS Glue

AWS Glue — visão arquitetural

Glue é a plataforma serverless de integração de dados da AWS. Reúne catalogar, descobrir, transformar e orquestrar em um único serviço.

Componentes

  • Data Catalog — metastore central.
  • Crawler — descobre schema.
  • Job — executa Spark ou Python.
  • Studio — ETL visual.
  • Workflow — orquestração.

Integrações

  • Lê e escreve em S3, RDS, DynamoDB, Redshift.
  • Catalog consumido por Athena, EMR, Redshift Spectrum.
  • Disparado por EventBridge, Lambda, cron.
  • Logs em CloudWatch.
Teoria · Metastore

Glue Data Catalog

Repositório central de metadados, compatível com Apache Hive Metastore. Torna dados no S3 consultáveis como tabelas.

O que vive no Catalog
  • Database — agrupamento lógico (ex.: olist_raw, olist_trusted, olist_refined).
  • Tables — schema + localização S3 + formato.
  • Partitions — subdivisões físicas (ex.: year_month=2018-05).
  • Connections — credenciais para fontes externas.
  • Classifiers — regras para reconhecer formatos customizados.
Teoria · Descoberta

Glue Crawler

Varre um caminho no S3, infere schema, detecta partições e popula o Catalog automaticamente.

O que ele faz
  • Detecta formato (CSV, JSON, Parquet).
  • Infere tipos a partir de amostras.
  • Cria ou atualiza tabelas no Catalog.
  • Descobre partições estilo Hive.
Cuidados
  • Tipagem inferida nem sempre acerta (ex.: datas viram string).
  • Rodar crawler com excesso é caro (cobra por DPU).
  • Em produção, prefira criar tabelas via código.

Teoria · ETL visual

Glue Studio

Interface visual para desenhar pipelines arrastando blocos. O Studio gera código PySpark automaticamente.

Vantagens e limitações
  • Acelera prototipagem e democratiza ETL.
  • Gera código que você pode editar depois.
  • Transformações avançadas ainda precisam de código.
  • Versionamento do fluxo visual é limitado.
Teoria · Orquestração

Glue Workflow

Encadeia crawlers e jobs em um DAG, com triggers condicionais e rerun em falhas.

Elementos
  • Triggers — iniciam execução (schedule, on-demand, event).
  • Jobs — unidades de trabalho encadeadas.
  • Crawlers — podem rodar entre jobs.
  • Condições — success, failure, timeout.
Teoria · Spark

Conceitos de Apache Spark

Motor de processamento distribuído em memória que roda por baixo do Glue.

Abstrações

  • RDD — coleção distribuída imutável (low-level).
  • DataFrame — tabela distribuída com schema.
  • Dataset — DataFrame tipado (Scala/Java).

Execução

  • Driver — programa principal.
  • Executors — nós que fazem o trabalho.
  • Lazy evaluation — transformações só rodam ao ver uma ação.
Teoria · Spark API

Transformações vs Ações

Transformações constroem o plano lógico. Ações disparam execução.

Transformações (lazy)Ações (eager)
select, filter, wherecount, collect
map, flatMapshow, take
groupBy, aggwrite.parquet, saveAsTable
join, unionforeach
withColumn, droptoPandas
Teoria · PySpark

PySpark básico para ETL

from pyspark.sql import SparkSession
from pyspark.sql.functions import col, to_timestamp

spark = SparkSession.builder.getOrCreate()

df = spark.read.option("header", True).csv("s3://bucket/raw/orders/")

clean = (df
  .withColumn("purchased_at", to_timestamp(col("order_purchase_timestamp")))
  .filter(col("customer_id").isNotNull())
  .dropDuplicates(["order_id"])
  .select("order_id", "customer_id", "purchased_at"))

clean.write.mode("overwrite").parquet("s3://bucket/silver/orders/")
Teoria · Qualidade

Qualidade no ETL

Dado sem qualidade é passivo. Toda camada silver deve aplicar garantias mínimas.

Checks essenciais

  • Dedup — dropDuplicates por chave.
  • Casting — tipos corretos (timestamp, decimal).
  • Nulls — filtrar ou preencher.
  • Schema enforcement — StructType explícito.

Ferramentas AWS

  • Glue Data Quality — regras DQDL.
  • Deequ — lib open source integrável.
  • Great Expectations — expectations em Python.
Teoria · Modelagem

Star schema — fato e dimensões

Modelagem dimensional clássica (Kimball): uma tabela fato central cercada por dimensões.

No nosso Olist
  • fato_vendas — 1 linha por item de pedido.
  • dim_cliente — customer_id, estado, cidade.
  • dim_produto — product_id, categoria, peso.
  • dim_tempo — ano, mês, dia, dia da semana.
  • dim_local — CEP, estado, região.
Por que star schema?
  • Queries analíticas mais rápidas (menos joins).
  • Modelo intuitivo para usuários de BI.
  • Denormalização é aceitável em DW.

Lab 2 · Código

Script raw_to_silver.py

import sys
from awsglue.context import GlueContext
from pyspark.context import SparkContext
from pyspark.sql.functions import col, to_timestamp, date_format, trim

sc = SparkContext.getOrCreate()
glue = GlueContext(sc)
spark = glue.spark_session

ALUNO = "<aluno>"
src = f"s3://{ALUNO}-raw/olist/orders/"
dst = f"s3://{ALUNO}-trusted/olist/orders/"

orders = (spark.read.option("header", True).csv(src)
  .withColumn("order_purchase_timestamp",
              to_timestamp(col("order_purchase_timestamp")))
  .withColumn("customer_id", trim(col("customer_id")))
  .filter(col("customer_id").isNotNull())
  .dropDuplicates(["order_id"])
  .withColumn("year_month",
              date_format(col("order_purchase_timestamp"), "yyyy-MM")))

(orders.write.mode("overwrite")
  .partitionBy("year_month")
  .parquet(dst))

Lab 3 · Código (1/2)

Script silver_to_gold.py — joins

from pyspark.context import SparkContext
from awsglue.context import GlueContext
from pyspark.sql.functions import col, year, month, dayofmonth

sc = SparkContext.getOrCreate()
glue = GlueContext(sc)
spark = glue.spark_session

ALUNO  = "<aluno>"
silver = f"s3://{ALUNO}-trusted/olist"
gold   = f"s3://{ALUNO}-refined/olist/gold"

orders    = spark.read.parquet(f"{silver}/orders/")
items     = spark.read.parquet(f"{silver}/order_items/")
customers = spark.read.parquet(f"{silver}/customers/")
products  = spark.read.parquet(f"{silver}/products/")

base = (items
  .join(orders,    "order_id")
  .join(customers, "customer_id")
  .join(products,  "product_id"))
Lab 3 · Código (2/2)

Script silver_to_gold.py — fato e dims

fato = base.select(
    col("order_id"),
    col("customer_id").alias("customer_sk"),
    col("product_id").alias("product_sk"),
    col("order_purchase_timestamp").alias("time_sk"),
    col("price"),
    col("freight_value").alias("freight"),
    (col("price") + col("freight_value")).alias("total"))

dim_cliente = customers.select(
    "customer_id","customer_city","customer_state").distinct()
dim_produto = products.select(
    "product_id","product_category_name").distinct()
dim_tempo   = (orders.select("order_purchase_timestamp")
    .withColumn("year",  year(col("order_purchase_timestamp")))
    .withColumn("month", month(col("order_purchase_timestamp")))
    .withColumn("day",   dayofmonth(col("order_purchase_timestamp")))
    .distinct())

fato.write.mode("overwrite").parquet(f"{gold}/fato_vendas/")
dim_cliente.write.mode("overwrite").parquet(f"{gold}/dim_cliente/")
dim_produto.write.mode("overwrite").parquet(f"{gold}/dim_produto/")
dim_tempo.write.mode("overwrite").parquet(f"{gold}/dim_tempo/")

Lab 5 · Validação no Athena

Queries analíticas

-- Receita mensal por categoria
SELECT  p.product_category_name AS categoria,
        date_format(t.order_purchase_timestamp, '%Y-%m') AS mes,
        ROUND(SUM(f.total), 2) AS receita
FROM    olist_refined.fato_vendas f
JOIN    olist_refined.dim_produto p ON f.product_sk = p.product_id
JOIN    olist_refined.dim_tempo   t ON f.time_sk    = t.order_purchase_timestamp
GROUP BY 1, 2
ORDER BY receita DESC
LIMIT 20;

-- Top 10 clientes por ticket
SELECT  customer_sk, ROUND(SUM(total), 2) AS ticket
FROM    olist_refined.fato_vendas
GROUP BY customer_sk
ORDER BY ticket DESC
LIMIT 10;

Teoria · 19h00 – 21h00

O que e Cloud Computing?

Entrega de recursos computacionais sob demanda pela internet, com pagamento por uso. Definição NIST (SP 800-145).

5 Características essenciais (NIST)
  1. On-demand self-service — provisionar recursos sem intervenção humana.
  2. Broad network access — acesso via rede por qualquer dispositivo.
  3. Resource pooling — recursos compartilhados entre múltiplos clientes.
  4. Rapid elasticity — escalar para cima ou para baixo conforme demanda.
  5. Measured service — uso monitorado e cobrado por consumo.

Teoria · Arquiteturas de Dados

Data Lake

Repositório centralizado para armazenar dados brutos em qualquer formato. Princípio: schema-on-read.

3 camadas clássicas
  • Raw / Bronze — dados originais, sem transformação.
  • Trusted / Silver — dados limpos, validados, deduplicados.
  • Refined / Gold — dados prontos para consumo analítico.
Stack AWS
  • S3 — armazenamento das 3 camadas.
  • Glue — catalogação e ETL.
  • Athena — consultas SQL diretamente no S3.

Teoria · Arquiteturas de Dados

Data Mesh

Paradigma de descentralização proposto por Zhamak Dehghani. Quatro princípios fundamentais.

1. Domain Ownership

  • Cada domínio de negocio e responsável pelos seus dados.

2. Data as a Product

  • Dados tratados com qualidade, SLA e documentação como um produto.

3. Self-Serve Platform

  • Infraestrutura compartilhada que reduz fricção para os domínios.

4. Federated Governance

  • Padrões globais com autonomia local. AWS: Lake Formation, EventBridge.

Teoria · AWS

AWS Infraestrutura Global

Entender a infraestrutura física da AWS e essencial para decidir onde hospedar dados e serviços.

Componentes
  • Regions — areas geograficas isoladas (ex.: us-east-1).
  • Availability Zones — data centers independentes dentro de uma região.
  • Edge Locations — pontos de presenca para CDN e DNS.
Serviços da disciplina

  • Kinesis, Lambda
  • RDS, DynamoDB, Redshift
  • IAM, Lake Formation

Teoria · Segurança

Modelo de Responsabilidade Compartilhada

A AWS protege a infraestrutura DA nuvem. Você protege o que coloca NA nuvem.

AWS e responsável por (OF the cloud)
  • Hardware físico e data centers
  • Rede global e hypervisors
  • Patches do sistema operacional gerenciado
  • Disponibilidade da infraestrutura
Você e responsável por (IN the cloud)
  • Configuração de IAM e permissões
  • Criptografia de dados (em transito e em repouso)
  • Configuração de security groups e ACLs
  • Proteção e classificação dos seus dados
Teoria · Governança

Por que Governança de Acessos e crítica?

Riscos reais
  • Buckets S3 públicos por configuração errada
  • Credenciais expostas em repositórios Git
  • Crypto mining em contas comprometidas
  • Usuários com permissão de administrador sem necessidade
Consequências
  • Vazamento de dados sensíveis de clientes
  • Violação de LGPD e GDPR
  • Custos inesperados de milhares de dólares
  • Perda de confiança e danos reputacionais
Princípio fundamental: Menor privilégio — conceda apenas o acesso estritamente necessário para a função. Revise e revogue regularmente.
Teoria · AWS IAM

IAM: Identidades e Entidades

Identity and Access Management — controla quem pode chamar quais APIs da AWS e em quais condições.

Users (Usuários)

  • Credenciais de longo prazo (senha, access key).
  • Representam pessoas ou sistemas.
  • Nunca use a conta root no dia a dia.

Groups (Grupos)

  • Coleção de usuários com permissões compartilhadas.
  • Simplifica gestao de equipes.
  • Ex.: DataEngineers, DataAnalysts.

Roles (Papéis)

  • Credenciais temporárias, rotacionadas automaticamente.
  • Usadas por serviços AWS (Glue, Lambda).
  • Preferência sobre access keys.

Policies (Políticas)

  • Documento JSON que define permissões.
  • Attached a users, groups ou roles.
  • Gerenciadas (AWS) ou customizadas.
Teoria · IAM Policies

Anatomia de uma Política IAM

Toda decisão de acesso e avaliada contra documentos JSON. Entender a estrutura e obrigatório.

Estrutura do Statement
  • Effect — Allow ou Deny.
  • Action — operações permitidas (ex.: s3:GetObject).
  • Resource — ARN do recurso alvo.
  • Condition — restrições adicionais (IP, MFA, tags).
Regras de avaliação
  • Deny explicito sempre prevalece.
  • Sem Allow = negado por padrão (implicit deny).
  • Todas as políticas ativas são avaliadas em conjunto.
  • Evite wildcards (*) em Action e Resource.

Setup do ambiente AWS Student

Em 1 hora vamos deixar sua conta AWS Student pronta para todas as próximas aulas: identidade, grupo e buckets do Data Lake.

O que vamos configurar

  • 3 buckets S3 (raw, trusted, refined)

  • Bloco de notas para Account ID e URL de login
Lab · Passo 1

Acesso ao AWS Student e região

Entrar no console, confirmar a região e localizar o Account ID.

Passo a passo
  1. Acesse o portal AWS Academy / Student e inicie o laboratório.
  2. Clique em AWS Details > AWS Console > Open Console.
  3. No canto superior direito, confirme a região: N. Virginia (us-east-1).
  4. Navegue até IAM > Dashboard e localize o Account ID.
  5. Anote o Account ID — será usado nos próximos passos.
Lab · Passo 2A

Criar o grupo DataEngineers

Grupo IAM com as políticas base para engenharia de dados.

Passo a passo
  1. No console, acesse IAM > User groups.
  2. Clique em Create group.
  3. Nome do grupo: DataEngineers.
  4. Anexe as políticas: AmazonS3FullAccess e CloudWatchLogsFullAccess.
  5. Clique em Create group.
Lab · Passo 2B

Criar o usuário eng-dados-01

Usuário IAM pessoal, dentro do grupo DataEngineers.

Passo a passo
  1. Acesse IAM > Users > Create user.
  2. Nome: eng-dados-01.
  3. Marque Provide user access to the AWS Management Console e defina uma senha.
  4. Na etapa de permissões, selecione Add user to group e escolha DataEngineers.
  5. Revise e clique em Create user.
  6. Anote a Console sign-in URL exibida ao final.
Lab · Passo 3

Buckets S3 do Data Lake

Criar 3 buckets, um por camada. Substitua {account-id} pelo ID anotado no Passo 1.

Passo a passo
  1. Abra o serviço S3 > Create bucket.
  2. Crie os 3 buckets abaixo, na região us-east-1:
  • data-lake-raw-{account-id} — dados brutos
  • data-lake-trusted-{account-id} — dados validados
  • data-lake-refined-{account-id} — dados prontos para análise

Mantenha Block Public Access habilitado (padrão). Não altere as demais opções.

Lab · Teste final

Validando o login do usuário

Garanta que o usuário recém-criado consegue usar os recursos permitidos e é barrado nos demais.

Passo a passo
  1. Abra uma janela anônima no navegador.
  2. Acesse a Console sign-in URL salva no Passo 2B.
  3. Faça login com eng-dados-01 e a senha temporária.
  4. Abra S3 — você deve ver os 3 buckets do Data Lake.
  5. Tente abrir IAM — acesso negado (esperado).

Contexto · Papel profissional

Quem é o engenheiro de dados em 2026

A função expandiu nos últimos anos. Hoje o engenheiro de dados cobre quatro frentes simultâneas, bem além de escrever ETL.

Pipeline

  • Ingere, transforma e entrega dados com qualidade.
  • Escolhe entre batch e streaming.
  • Define contratos entre produtor e consumidor.

Governança

  • Catálogo, linhagem e classificação de dados.
  • Controle de acesso por domínio.
  • Conformidade com LGPD e políticas internas.

Reliability

  • SLOs de frescor e qualidade.
  • Observabilidade, alertas e runbooks.
  • Postmortems e cultura de incidente.

Custo

  • Quanto custa cada pipeline por mês.
  • Escolha de storage class e formato.
  • FinOps e previsibilidade de gasto.

Contexto · Impacto na prática

Por que a cloud mudou engenharia de dados

Quatro ganhos concretos que redefinem o que dá para construir e em quanto tempo.

Elasticidade

  • Escala sob demanda, sem comprar hardware.
  • Ex.: Black Friday consome mais cluster e depois encolhe.

OpEx em vez de CapEx

  • Paga-se pelo consumo, não por capacidade ociosa.
  • Ex.: Athena cobra por TB escaneado, sem servidor fixo.

Serviços gerenciados

  • Menos operação, mais foco em lógica de negócio.
  • Ex.: RDS cuida do backup e patch do Postgres.

Time-to-market

  • Do zero ao MVP de pipeline em dias, não meses.
  • Ex.: S3 + Athena geram BI básico no mesmo dia.

Contexto · Fronteiras do papel

O que NÃO é engenharia de dados

Desmistificar ajuda a calibrar expectativa. Engenharia de dados conversa com vários papéis, mas não é nenhum deles isoladamente.

Não é DBA clássico

  • DBA cuida de tuning, backup e alta disponibilidade de um banco.
  • Engenheiro de dados move dado entre sistemas e prepara para análise.

Não é cientista de dados

  • Cientista constrói modelos e investiga hipóteses.
  • Engenheiro de dados entrega o dataset confiável que o cientista consome.

Não é analytics / BI

  • Analista de dados faz dashboards e respostas de negócio.
  • Engenheiro de dados garante que o dado chegue limpo ao BI.

Não é DevOps puro

  • DevOps entrega aplicações e infraestrutura genérica.
  • Engenheiro de dados aplica as mesmas práticas ao mundo de dados.

Teoria · Cloud

O que é Cloud Computing?

Entrega de recursos computacionais sob demanda pela internet, com pagamento por uso. Definição do NIST (SP 800-145).

5 características essenciais (NIST)
  1. On-demand self-service — provisionar recursos sem intervenção humana.
  2. Broad network access — acesso via rede por qualquer dispositivo.
  3. Resource pooling — recursos compartilhados entre múltiplos clientes.
  4. Rapid elasticity — escalar para cima ou para baixo conforme demanda.
  5. Measured service — uso monitorado e cobrado por consumo.
Teoria · Contexto

Como chegamos aqui: do mainframe à cloud

A cloud não nasceu do nada. Ela é a culminação de décadas de mudança cultural e técnica, com o DevOps como ponte essencial.

EraPeríodoModelo dominanteLegado para dados

DevOps como precursor: sem automação de infra e cultura de colaboração entre dev e ops, a cloud seria só um data center alugado. É DevOps que destrava o self-service e a elasticidade reais.
Teoria · Trade-offs

Benefícios e desafios da cloud para dados

Benefícios
  • Elasticidade real: picos de ingestão sem replanejar hardware.
  • OpEx: paga-se pelo consumido, sem CapEx de data center.
  • Tempo de provisionamento em minutos.
  • Serviços gerenciados (Athena, Lambda) reduzem operação.
  • Acesso a tecnologias de ponta (GPUs, ML, streaming).
Desafios
  • Custo variável exige disciplina de FinOps.
  • Segurança passa a ser responsabilidade do cliente.
  • Lock-in de serviços proprietários.
  • Latência de rede para cargas híbridas.
  • Curva de aprendizado constante (serviços novos todo mês).

Teoria · Arquiteturas

Data Lake

Repositório centralizado para armazenar dados brutos em qualquer formato. Princípio: schema-on-read.

3 camadas clássicas
  • Raw / Bronze — dados originais, sem transformação.
  • Trusted / Silver — dados limpos, validados, deduplicados.
  • Refined / Gold — dados prontos para consumo analítico.
Stack da disciplina (AWS Learner Lab)
  • S3 — armazenamento das 3 camadas.
  • Athena — consultas SQL e catálogo de tabelas externas (substitui o Glue Catalog no lab).
  • PySpark em EC2 — ETL que promove Raw → Trusted → Refined.

Teoria · Arquiteturas

Data Mesh

Paradigma de descentralização proposto por Zhamak Dehghani. Quatro princípios fundamentais.

1. Domain Ownership

  • Cada domínio de negócio é responsável pelos seus dados.

2. Data as a Product

  • Dados tratados com qualidade, SLA e documentação como um produto.

3. Self-Serve Platform

  • Infraestrutura compartilhada que reduz fricção para os domínios.

4. Federated Governance

  • Padrões globais com autonomia local. AWS: Lake Formation, EventBridge.

Teoria · AWS

AWS: Infraestrutura Global

Entender a infraestrutura física da AWS é essencial para decidir onde hospedar dados e serviços.

Componentes
  • Regions — áreas geográficas isoladas (ex.: us-east-1).
  • Availability Zones — data centers independentes dentro de uma região.
  • Edge Locations — pontos de presença para CDN e DNS.
Serviços da disciplina

  • EC2 (PySpark, ClickHouse)

  • RDS, DynamoDB

Teoria · Segurança

Modelo de Responsabilidade Compartilhada

A AWS protege a infraestrutura DA nuvem. Você protege o que coloca NA nuvem.

AWS é responsável por (OF the cloud)
  • Hardware físico e data centers
  • Rede global e hypervisors
  • Patches do sistema operacional gerenciado
  • Disponibilidade da infraestrutura
Você é responsável por (IN the cloud)
  • Configuração de IAM e permissões
  • Criptografia de dados (em trânsito e em repouso)
  • Configuração de security groups e ACLs
  • Proteção e classificação dos seus dados
Teoria · Governança

Por que governança de acessos é crítica?

Riscos reais
  • Buckets S3 públicos por configuração errada
  • Credenciais expostas em repositórios Git
  • Crypto mining em contas comprometidas
  • Usuários com permissão de administrador sem necessidade
Consequências
  • Vazamento de dados sensíveis de clientes
  • Violação de LGPD e GDPR
  • Custos inesperados de milhares de dólares
  • Perda de confiança e danos reputacionais
Princípio fundamental: menor privilégio — conceda apenas o acesso estritamente necessário para a função. Revise e revogue regularmente.
Teoria · AWS IAM

IAM: identidades e entidades

Identity and Access Management controla quem pode chamar quais APIs da AWS e em quais condições.

Users (usuários)

  • Credenciais de longo prazo (senha, access key).
  • Representam pessoas ou sistemas.
  • Nunca use a conta root no dia a dia.

Groups (grupos)

  • Coleção de usuários com permissões compartilhadas.
  • Simplifica gestão de equipes.
  • Ex.: DataEngineers, DataAnalysts.

Roles (papéis)

  • Credenciais temporárias, rotacionadas automaticamente.
  • Usadas por serviços AWS (EC2, Lambda).
  • Preferência sobre access keys.

Policies (políticas)

  • Documento JSON que define permissões.
  • Atachadas a users, groups ou roles.
  • Gerenciadas (AWS) ou customizadas.
Teoria · IAM Policies

Anatomia de uma política IAM

Toda decisão de acesso é avaliada contra documentos JSON. Entender a estrutura é obrigatório.

Estrutura do Statement
  • Effect — Allow ou Deny.
  • Action — operações permitidas (ex.: s3:GetObject).
  • Resource — ARN do recurso alvo.
  • Condition — restrições adicionais (IP, MFA, tags).
Regras de avaliação
  • Deny explícito sempre prevalece.
  • Sem Allow = negado por padrão (implicit deny).
  • Todas as políticas ativas são avaliadas em conjunto.
  • Evite wildcards (*) em Action e Resource.
Teoria · AWS Student

O LabRole do AWS Academy Learner Lab

No ambiente estudantil, quase todos os serviços assumem uma role pré-configurada chamada LabRole. Entender suas limitações agora evita surpresas nas próximas aulas.

O que o LabRole entrega
  • Permissões amplas em EC2, S3, Athena, Lambda, Kinesis, IAM (read), CloudWatch.
  • É passada automaticamente a serviços gerenciados no lab.
  • Evita que você gaste tempo criando roles de serviço.
O que o LabRole não permite
  • Acesso a AWS Glue e Amazon Redshift (bloqueados no Learner Lab).
  • Criar usuários IAM persistentes entre sessões.
  • Operações de billing e organizações.
  • Instâncias de alta classe (GPUs, r5.24xlarge).
  • O ambiente é reinicializado ao expirar a sessão.
Em produção não faríamos assim: em vez do LabRole com permissões amplas, criaríamos uma role dedicada por serviço com menor privilégio. Aqui usamos o LabRole só para reduzir atrito didático.

Lab · Passo 1

Acesso ao AWS Student e região

Entrar no Learner Lab, iniciar o laboratório, abrir o console e confirmar a região correta.

Passo a passo
  1. Acesse o portal AWS Academy · Learner Lab.
  2. Clique em Start Lab e aguarde o indicador ficar verde.
  3. Clique em AWS para abrir o AWS Management Console.
  4. No canto superior direito, confirme a região: US East (N. Virginia) us-east-1.
  5. Na barra de busca, digite IAM e abra o dashboard.
  6. Anote o Account ID exibido no topo — será usado nos próximos passos.
Lab · Passo 2

Reconhecendo o IAM do Learner Lab

Antes de criar recursos, entenda o que já existe na conta de laboratório.

Passo a passo
  1. Em IAM > Users, observe os usuários pré-criados pelo lab.
  2. Em IAM > Roles, localize a LabRole e abra a aba Permissions.
  3. Anote as políticas atachadas: ela combina acessos a EC2, S3, Athena, Lambda, Kinesis, IAM read e CloudWatch.
  4. Abra a aba Trust relationships e identifique quais serviços podem assumi-la.
  5. Em IAM > Policies, observe a diferença entre políticas AWS (gerenciadas) e customizadas.
Observação: o lab não permite criar usuários IAM adicionais nem habilita Glue ou Redshift — por isso eles não aparecem na LabRole. Usaremos a LabRole e a conta principal do Learner Lab durante toda a disciplina.
Lab · Passo 3

Criar os 3 buckets do Data Lake

Cada aluno cria suas 3 camadas. Substitua <iniciais> por suas iniciais em minúsculo (ex.: jps).

Passo a passo
  1. No console, acesse S3 > Create bucket.
  2. Região: us-east-1. Crie os 3 buckets abaixo:
  • olist-raw-<iniciais> — dados brutos (CSV, JSON)
  • olist-trusted-<iniciais> — dados validados (Parquet)
  • olist-refined-<iniciais> — dados prontos para BI/ML

Mantenha Block Public Access habilitado e habilite Bucket Versioning. Demais opções no padrão.

Lab · Passo 4

Upload de arquivo de teste

Valide que você consegue escrever no Raw. Ainda não é o Olist — é apenas um smoke test.

Passo a passo
  1. Abra o bucket olist-raw-<iniciais>.
  2. Clique em Upload > Add files.
  3. Selecione qualquer arquivo pequeno (um .txt ou .csv).
  4. Clique em Upload e aguarde o sucesso.
  5. Abra o objeto carregado e confira tamanho, storage class (Standard) e versão.
  6. Em Permissions do bucket, confirme que Block all public access está ativo.

Módulo 1 — Cloud para Dados

O que é Cloud Computing?

Entrega de recursos computacionais sob demanda pela internet, com pagamento por uso.

IaaS

  • Infraestrutura como serviço.
  • Você gerencia SO, runtime e aplicação.
  • Ex.: EC2, EBS, VPC.

PaaS

  • Plataforma gerenciada.
  • Você gerencia apenas a aplicação e dados.
  • Ex.: RDS, Elastic Beanstalk.

SaaS

  • Software completo como serviço.
  • Você consome a funcionalidade.
  • Ex.: QuickSight, Athena.

Por que importa para dados?

  • Elasticidade para cargas variáveis.
  • Pagamento apenas pelo processado.
  • Serviços gerenciados reduzem operação.

Módulo 1 — Arquiteturas de Dados

Data Lake, Data Warehouse e Data Mesh

Três abordagens diferentes para organizar dados em larga escala. Cada uma resolve um problema diferente.

Data Lake

  • Armazena dados brutos em qualquer formato.
  • Schema on read — flexibilidade total.
  • Ideal para exploração e ML.
  • AWS: S3 + Glue + Athena.

Data Warehouse

  • Dados estruturados e transformados.
  • Schema on write — performance de consulta.
  • Ideal para BI e relatórios.
  • AWS: Redshift.

Data Mesh

  • Dados como produto, por domínio.
  • Descentralização com governança federada.
  • Ideal para organizações grandes.
  • AWS: Lake Formation + Event Bridge.

Camadas do Data Lake

  • Raw / Bronze: dados originais.
  • Trusted / Silver: dados limpos.
  • Refined / Gold: dados prontos.

Módulo 1 — AWS

AWS como plataforma de dados

Regiões, zonas de disponibilidade e os serviços centrais que usaremos ao longo da disciplina.

Infraestrutura Global

  • Regiões: conjuntos geográficos isolados.
  • AZs: data centers independentes dentro da região.
  • Edge Locations: pontos de presença para latência baixa.

Serviços usados na disciplina

  • S3, Athena, Glue, Kinesis.
  • RDS, DynamoDB, Redshift.
  • IAM, Lake Formation, Step Functions.
  • CloudWatch, MWAA.

Modelo de responsabilidade

  • AWS: segurança da nuvem (hardware, rede, hypervisor).
  • Você: segurança na nuvem (dados, IAM, criptografia).

Região recomendada no lab

  • us-east-1 (N. Virginia).
  • Maior cobertura de serviços.
  • Custo ligeiramente menor.

Módulo 2 — Governança de Acessos

Por que o controle de acesso é crítico em dados?

Dados sensíveis expostos por configuração incorreta de IAM são uma das causas mais comuns de vazamentos em cloud.

Riscos sem governança

  • Usuários com permissão excessiva ("God Mode").
  • Chaves de acesso expostas em código.
  • Buckets S3 públicos por acidente.
  • Ausência de trilha de auditoria.

Consequências reais

  • Vazamento de dados de clientes.
  • Custos inesperados por mineração de cripto.
  • Violação de LGPD / GDPR.
  • Incidentes de ransomware em S3.

O que a governança resolve

  • Define quem pode fazer o quê e em qual recurso.
  • Garante rastreabilidade de todas as ações.
  • Isola ambientes de dev, staging e produção.

Princípio base

  • Menor privilégio: conceda apenas o acesso estritamente necessário para a função.
  • Revise e revogue regularmente.

Módulo 2 — AWS IAM

IAM: Identidades e Entidades

IAM (Identity and Access Management) controla quem pode chamar quais APIs da AWS e em quais condições.

Usuários (Users)

  • Representa uma pessoa ou sistema.
  • Credenciais de longo prazo (senha / access key).
  • Evite usar a conta root para operações do dia a dia.

Grupos (Groups)

  • Coleção de usuários com as mesmas permissões.
  • Simplifica gerenciamento em equipes.
  • Ex.: grupo DataEngineers, grupo DataAnalysts.

Roles (Papéis)

  • Identidade assumida temporariamente.
  • Usada por serviços AWS (ex.: Glue, Lambda).
  • Credenciais rotacionam automaticamente.

Políticas (Policies)

  • Documento JSON que define permissões.
  • Attached a usuários, grupos ou roles.
  • Pode ser gerenciada (AWS) ou customizada.

Módulo 2 — IAM Policies

Anatomia de uma Política IAM

Toda decisão de acesso na AWS é avaliada contra uma ou mais políticas JSON. Entender sua estrutura é fundamental.

Estrutura básica

  • Effect: Allow ou Deny.
  • Action: quais operações (ex.: s3:GetObject).
  • Resource: ARN do recurso alvo.
  • Condition: restrições adicionais (opcional).

Exemplo — leitura em S3

  • Effect: Allow
  • Action: s3:GetObject, s3:ListBucket
  • Resource: arn:aws:s3:::meu-data-lake/*

Regras de avaliação

  • Deny explícito sempre prevalece.
  • Sem Allow = negado por padrão.
  • Permissões são avaliadas no contexto de todas as políticas ativas.

Boas práticas

  • Prefira políticas gerenciadas pela AWS quando adequadas.
  • Use condições para restringir por IP ou MFA.
  • Evite wildcards (*) em Actions e Resource.

Módulo 2 — Design de Permissões

Mapeando acessos para o Data Lake

Cada perfil de time de dados precisa de um conjunto de permissões diferente. Mapear esses perfis antes de criar roles evita retrabalho.

Engenheiro de Dados

  • S3: leitura e escrita em todas as camadas.
  • Glue: criar e executar jobs.
  • Kinesis: produzir e consumir streams.
  • CloudWatch: visualizar logs.

Cientista de Dados

  • S3: leitura em Silver e Gold.
  • Athena: executar queries.
  • SageMaker: criar notebooks e treinamentos.

Analista de BI

  • Athena: leitura somente.
  • Redshift: conexão somente leitura.
  • QuickSight: acesso ao dashboard.

Serviços AWS (Roles)

  • Glue Job Role: S3 + Glue + CloudWatch.
  • Lambda Role: S3 leitura + DynamoDB.
  • Step Functions Role: invocar Lambda + Glue.

O que vamos fazer no lab?

Prática com AWS Student. Configuração segura da conta e criação das primeiras estruturas de acesso para a disciplina.

Passo 1 — Acesso inicial

  • Acessar o ambiente AWS Student.
  • Confirmar região: us-east-1.
  • Verificar que a conta root nunca deve ser usada para trabalho direto.

Passo 2 — Usuário e MFA

  • Criar usuário IAM com permissão de admin inicial.
  • Habilitar MFA (Multi-Factor Authentication).
  • Criar access key para uso no CLI (opcional).

Passo 3 — Grupos

  • Criar grupo DataEngineers.
  • Associar políticas: AmazonS3FullAccess, AWSGlueConsoleFullAccess.
  • Adicionar o usuário ao grupo.

Passo 4 — Role para serviços

  • Criar role GlueJobRole.
  • Trusted entity: Glue.
  • Políticas: AmazonS3FullAccess + AWSGlueServiceRole + CloudWatchLogsFullAccess.

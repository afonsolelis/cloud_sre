Teoria · 19h00 – 20h00

O que e Cloud Computing?

Entrega de recursos computacionais sob demanda pela internet, com pagamento por uso. Definicao NIST (SP 800-145).

5 Caracteristicas essenciais (NIST)
  1. On-demand self-service — provisionar recursos sem intervencao humana.
  2. Broad network access — acesso via rede por qualquer dispositivo.
  3. Resource pooling — recursos compartilhados entre multiplos clientes.
  4. Rapid elasticity — escalar para cima ou para baixo conforme demanda.
  5. Measured service — uso monitorado e cobrado por consumo.

Teoria · Arquiteturas de Dados

Data Lake

Repositorio centralizado para armazenar dados brutos em qualquer formato. Principio: schema-on-read.

3 camadas classicas
  • Raw / Bronze — dados originais, sem transformacao.
  • Trusted / Silver — dados limpos, validados, deduplicados.
  • Refined / Gold — dados prontos para consumo analitico.
Stack AWS
  • S3 — armazenamento das 3 camadas.
  • Glue — catalogacao e ETL.
  • Athena — consultas SQL diretamente no S3.

Teoria · Arquiteturas de Dados

Data Mesh

Paradigma de descentralizacao proposto por Zhamak Dehghani. Quatro principios fundamentais.

1. Domain Ownership

  • Cada dominio de negocio e responsavel pelos seus dados.

2. Data as a Product

  • Dados tratados com qualidade, SLA e documentacao como um produto.

3. Self-Serve Platform

  • Infraestrutura compartilhada que reduz fricao para os dominios.

4. Federated Governance

  • Padroes globais com autonomia local. AWS: Lake Formation, EventBridge.
Teoria · AWS

AWS Infraestrutura Global

Entender a infraestrutura fisica da AWS e essencial para decidir onde hospedar dados e servicos.

Componentes
  • Regions — areas geograficas isoladas (ex.: us-east-1).
  • Availability Zones — data centers independentes dentro de uma regiao.
  • Edge Locations — pontos de presenca para CDN e DNS.
Servicos da disciplina
  • S3, Athena, Glue
  • Kinesis, Lambda
  • RDS, DynamoDB, Redshift
  • IAM, Lake Formation
  • CloudWatch, Step Functions

Regiao recomendada: us-east-1

Teoria · Seguranca

Modelo de Responsabilidade Compartilhada

A AWS protege a infraestrutura DA nuvem. Voce protege o que coloca NA nuvem.

AWS e responsavel por (OF the cloud)
  • Hardware fisico e data centers
  • Rede global e hypervisors
  • Patches do sistema operacional gerenciado
  • Disponibilidade da infraestrutura
Voce e responsavel por (IN the cloud)
  • Configuracao de IAM e permissoes
  • Criptografia de dados (em transito e em repouso)
  • Configuracao de security groups e ACLs
  • Protecao e classificacao dos seus dados
Teoria · Governanca

Por que Governanca de Acessos e critica?

Riscos reais
  • Buckets S3 publicos por configuracao errada
  • Credenciais expostas em repositorios Git
  • Crypto mining em contas comprometidas
  • Usuarios com permissao de administrador sem necessidade
Consequencias
  • Vazamento de dados sensiveis de clientes
  • Violacao de LGPD e GDPR
  • Custos inesperados de milhares de dolares
  • Perda de confianca e danos reputacionais
Principio fundamental: Menor privilegio — conceda apenas o acesso estritamente necessario para a funcao. Revise e revogue regularmente.
Teoria · AWS IAM

IAM: Identidades e Entidades

Identity and Access Management — controla quem pode chamar quais APIs da AWS e em quais condicoes.

Users (Usuarios)

  • Credenciais de longo prazo (senha, access key).
  • Representam pessoas ou sistemas.
  • Nunca use a conta root no dia a dia.

Groups (Grupos)

  • Colecao de usuarios com permissoes compartilhadas.
  • Simplifica gestao de equipes.
  • Ex.: DataEngineers, DataAnalysts.

Roles (Papeis)

  • Credenciais temporarias, rotacionadas automaticamente.
  • Usadas por servicos AWS (Glue, Lambda).
  • Preferencia sobre access keys.

Policies (Politicas)

  • Documento JSON que define permissoes.
  • Attached a users, groups ou roles.
  • Gerenciadas (AWS) ou customizadas.
Teoria · IAM Policies

Anatomia de uma Politica IAM

Toda decisao de acesso e avaliada contra documentos JSON. Entender a estrutura e obrigatorio.

Estrutura do Statement
  • Effect — Allow ou Deny.
  • Action — operacoes permitidas (ex.: s3:GetObject).
  • Resource — ARN do recurso alvo.
  • Condition — restricoes adicionais (IP, MFA, tags).
Regras de avaliacao
  • Deny explicito sempre prevalece.
  • Sem Allow = negado por padrao (implicit deny).
  • Todas as politicas ativas sao avaliadas em conjunto.
  • Evite wildcards (*) em Action e Resource.

Lab · Passo 1

Acesso ao AWS Student e exploracao do console

Objetivo: Entrar no ambiente AWS Student, confirmar a regiao us-east-1 e explorar o IAM Dashboard para entender o estado inicial da conta.
Passos
  1. Acesse o portal AWS Student e inicie o laboratorio.
  2. No console AWS, verifique no canto superior direito: a regiao deve ser N. Virginia (us-east-1).
  3. Navegue ate IAM > Dashboard.
  4. Observe: quantos usuarios, grupos e roles ja existem? Anote.
  5. Localize o Account ID — voce vai precisar dele.
Lab · Passo 2

Criacao de grupos IAM

Grupo Politicas gerenciadas Proposito
DataEngineers AmazonS3FullAccess, AWSGlueConsoleFullAccess, CloudWatchLogsFullAccess Ingestao, ETL, monitoramento
DataAnalysts AmazonAthenaFullAccess, AmazonS3ReadOnlyAccess Consultas SQL, leitura de dados
Caminho: IAM > User groups > Create group > Selecione as politicas > Create group.
Lab · Passo 3

Criacao de usuario IAM

Passos
  1. IAM > Users > Create user.
  2. Nome: eng-dados-01.
  3. Marque Provide user access to the AWS Management Console.
  4. Defina uma senha temporaria (Custom password).
  5. Na etapa de permissoes, selecione Add user to group > DataEngineers.
  6. Revise e crie o usuario.
  7. Anote a Console sign-in URL — ela usa o Account ID.
Teste: Abra uma janela anonima, acesse a sign-in URL e faca login com o usuario criado. Voce deve conseguir acessar S3 e Glue, mas nao IAM.
Lab · Passo 4

Criacao de roles para servicos AWS

Role Trusted Entity Politicas
GlueJobRole AWS Service > Glue AmazonS3FullAccess, AWSGlueServiceRole, CloudWatchLogsFullAccess
LambdaExecutionRole AWS Service > Lambda AmazonS3ReadOnlyAccess, AWSLambdaBasicExecutionRole
Caminho: IAM > Roles > Create role > Selecione o servico > Attach policies > Nomeie a role > Create role.
Lab · Passo 5

Politica customizada em JSON

Criar a politica DataLakeReadOnly e anexar ao grupo DataAnalysts.

{ "Version": "2012-10-17", "Statement": [ { "Effect": "Allow", "Action": ["s3:GetObject", "s3:ListBucket"], "Resource": [ "arn:aws:s3:::data-lake-refined-*", "arn:aws:s3:::data-lake-refined-*/*" ] }, { "Effect": "Allow", "Action": "athena:StartQueryExecution", "Resource": "*" } ] }
Caminho: IAM > Policies > Create policy > JSON tab > Cole o JSON > Nome: DataLakeReadOnly > Create. Depois: User groups > DataAnalysts > Attach policy.
Lab · Passo 6

Validacao com Policy Simulator e criacao de buckets S3

Policy Simulator
  1. Acesse IAM > Policy Simulator.
  2. Selecione o usuario eng-dados-01.
  3. Teste: S3 > PutObject → deve ser Allowed.
  4. Teste: IAM > CreateUser → deve ser Denied.
Buckets S3 por camada
  1. S3 > Create bucket.
  2. data-lake-raw-{account-id}
  3. data-lake-trusted-{account-id}
  4. data-lake-refined-{account-id}
  5. Regiao: us-east-1. Manter demais opcoes padrao.
Lab · Passo 7 + Desafio

Verificacao final e desafio extra

Checklist de verificacao
  • Grupo DataEngineers criado com 3 politicas
  • Grupo DataAnalysts criado com 2 politicas + custom
  • Usuario eng-dados-01 no grupo DataEngineers
  • Role GlueJobRole com trusted entity Glue
  • Role LambdaExecutionRole com trusted entity Lambda
  • Politica DataLakeReadOnly anexada a DataAnalysts

  • Policy Simulator validando permissoes
Desafio (opcional)
  • Habilite MFA para o usuario eng-dados-01 (se suportado no AWS Student).
  • Crie uma permission boundary que impeca qualquer usuario de criar outros usuarios IAM, e aplique-a ao grupo DataEngineers.
  • Documente: qual e a diferenca entre uma boundary e uma policy normal?

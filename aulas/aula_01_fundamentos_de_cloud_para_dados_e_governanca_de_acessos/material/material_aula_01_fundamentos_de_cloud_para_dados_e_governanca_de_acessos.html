Introdução — contexto antes de mergulhar

Antes de entrar em Cloud Computing, IAM e arquiteturas de dados, vale dedicar algum tempo a entender o terreno. Engenharia de dados em 2026 é um papel que cresceu muito nos últimos anos, e a cloud é o substrato por baixo desse crescimento. Quem é o engenheiro de dados em 2026

Em 2015 o engenheiro de dados era, em boa parte, uma evolução do DBA somada a quem escrevia jobs de ETL. Em 2026 a função cobre pelo menos quatro frentes simultâneas, e entender essa expansão ajuda a calibrar a carga de conhecimento que o curso pretende construir.

  • Pipeline. Ingerir, transformar e entregar dados com qualidade. Escolher entre batch e streaming conforme o caso de uso. Definir contratos entre produtor e consumidor e garantir que eles sejam respeitados.
  • Governança. Catálogo, linhagem, classificação, controle de acesso por domínio e conformidade com LGPD. Dados certos, nas mãos certas, com rastreabilidade.
  • Reliability. Definir SLOs de frescor e qualidade, instrumentar observabilidade, manter runbooks e fazer postmortems. A cultura de SRE aplicada ao mundo de dados é tema da Aula 07.
  • Custo. Saber quanto custa cada pipeline, escolher storage class, formato e compressão, e manter a conta AWS previsível. FinOps é o que separa um pipeline saudável de uma surpresa no fim do mês.

Por que a cloud mudou engenharia de dados

Há quatro ganhos concretos que explicam por que, hoje, praticamente toda plataforma de dados nova nasce na cloud. Não é modismo: são propriedades difíceis (ou impossíveis) de replicar em data center próprio.

  • Elasticidade real. Escala para cima e para baixo sob demanda, sem comprar hardware. Na Black Friday o cluster cresce, depois encolhe, sem conversa com compras.
  • OpEx em vez de CapEx. Paga-se pelo consumo, não por capacidade ociosa. O Athena cobra por TB escaneado, sem nenhum servidor fixo rodando no fundo.
  • Serviços gerenciados. Menos tempo em backup, patch e tuning, mais tempo em lógica de negócio. O RDS cuida do Postgres; você cuida do dado.
  • Time-to-market. Um MVP de pipeline sai em dias, não em meses. S3 mais Athena geram BI básico no mesmo dia em que você cria a conta.

O que NÃO é engenharia de dados

Ajuda a calibrar expectativa listar os papéis vizinhos. O engenheiro de dados conversa com todos eles e toma emprestado várias práticas, mas não substitui nenhum.

  • Não é DBA clássico. O DBA cuida de tuning, backup, replicação e alta disponibilidade de um banco específico. O engenheiro de dados move dado entre sistemas e prepara para análise, com menos profundidade em um banco único.
  • Não é cientista de dados. O cientista constrói modelos, explora hipóteses e usa estatística. O engenheiro de dados entrega o dataset confiável que o cientista consome, e frequentemente também o coloca em produção.
  • Não é analista de dados / BI. O analista responde perguntas de negócio com dashboards e relatórios. O engenheiro de dados garante que o dado chegue limpo, fresco e governado à camada de BI.
  • Não é DevOps puro. O DevOps entrega aplicações e infraestrutura genérica. O engenheiro de dados aplica as mesmas práticas (IaC, CI/CD, observabilidade) ao mundo específico de dados e pipelines.

Módulo 1 — Cloud Computing para Engenharia de Dados

O que é Cloud Computing

Definição NIST

O National Institute of Standards and Technology (NIST), na publicação SP 800-145, define Cloud Computing como um modelo que permite acesso de rede ubíquo, conveniente e sob demanda a um pool compartilhado de recursos computacionais configuráveis (por exemplo: redes, servidores, armazenamento, aplicações e serviços) que podem ser rapidamente provisionados e liberados com mínimo esforço de gerenciamento ou interação com o provedor de serviço.

Essa definição estabelece cinco características essenciais que diferenciam a cloud de qualquer outra forma de hospedagem ou virtualização:

  1. Self-service sob demanda (On-demand self-service): o consumidor pode provisionar recursos computacionais — como servidores virtuais ou armazenamento — automaticamente, sem necessidade de interação humana com o provedor. Na prática, isso significa que um engenheiro de dados pode criar um cluster de processamento Spark via console ou API em minutos, sem abrir chamado de infraestrutura.
  2. Acesso amplo à rede (Broad network access): os recursos estão disponíveis pela rede (tipicamente a internet) e são acessados por mecanismos padronizados que promovem o uso por uma variedade de plataformas (laptops, smartphones, tablets, estações de trabalho). Qualquer membro da equipe de dados pode acessar o console AWS ou fazer chamadas de API de qualquer localização.
  3. Pool de recursos compartilhados (Resource pooling): os recursos do provedor são agrupados para atender múltiplos consumidores usando um modelo multi-tenant, com diferentes recursos físicos e virtuais atribuídos e reatribuídos dinamicamente conforme a demanda. O consumidor não tem controle ou conhecimento sobre a localização exata dos recursos, embora possa especificar a região geográfica. Isso permite economia de escala que seria impossível em infraestrutura privada.
  4. Elasticidade rápida (Rapid elasticity): os recursos podem ser provisionados e liberados elasticamente, em alguns casos automaticamente, para escalar rapidamente tanto para cima quanto para baixo, proporcionalmente à demanda. Para o consumidor, os recursos frequentemente parecem ilimitados. Um pipeline de dados que processa 100 GB por dia pode, sem alteração de arquitetura, processar 10 TB em um dia de pico.
  5. Serviço medido (Measured service): os sistemas de cloud controlam e otimizam automaticamente o uso de recursos por meio de medição em um nível de abstração apropriado ao tipo de serviço (armazenamento, processamento, largura de banda, contas de usuário ativas). O uso é monitorado, controlado e reportado, proporcionando transparência tanto para o provedor quanto para o consumidor. O modelo "pay-per-use" é a consequência direta dessa característica.

Contexto histórico: de mainframes à nuvem

Para entender por que a cloud mudou fundamentalmente a engenharia de dados, é útil revisitar a evolução da computação corporativa:

  • Era dos mainframes (1960-1980): computação centralizada, com terminais "burros" acessando uma máquina única e poderosa. O processamento de dados era batch, com filas de jobs agendados. O custo era proibitivo e o acesso limitado a grandes corporações.
  • Era cliente-servidor (1980-2000): descentralização parcial. Servidores departamentais, bancos de dados relacionais (Oracle, SQL Server), data warehouses on-premises. A engenharia de dados nasceu aqui, com ETL, modelagem dimensional (Kimball, Inmon) e ferramentas de BI.
  • Era da virtualização (2000-2010): VMware e tecnologias similares permitiram rodar múltiplos servidores virtuais em hardware físico compartilhado. Isso melhorou a utilização de recursos, mas o modelo econômico ainda era CapEx: compre o hardware, instale, mantenha.
  • Era da cloud (2006-presente): em 2006, a Amazon lançou o EC2 e o S3, inaugurando a cloud pública. O Google Cloud Platform (GCP) e o Microsoft Azure seguiram. O modelo econômico mudou de CapEx para OpEx, e a elasticidade tornou viável processar volumes de dados antes inimagináveis para a maioria das empresas.
  • Era da cloud nativa para dados (2015-presente): serviços gerenciados como Redshift, BigQuery, Snowflake, Athena e Databricks eliminaram a necessidade de gerenciar infraestrutura para análise de dados. O conceito de "serverless" — onde o consumidor não gerencia nenhum servidor — tornou-se o padrão aspiracional para pipelines de dados.

Por que a cloud mudou o jogo para engenharia de dados

Antes da cloud, construir uma plataforma de dados exigia investimento inicial significativo: servidores físicos, storage arrays, licenças de software de banco de dados, switches de rede, espaço em data center, equipe de operações 24x7. O ciclo de aquisição era de semanas a meses. Se a demanda crescesse além da capacidade instalada, era necessário um novo ciclo de aquisição. Se a demanda caísse, o hardware ficava ocioso, mas os custos continuavam.

A cloud resolveu esses problemas de forma estrutural:

  • Eliminação do planejamento de capacidade estático: em vez de dimensionar para o pico esperado (com margem de segurança), você provisiona para a demanda atual e escala automaticamente.
  • Experimentação de baixo custo: testar uma nova arquitetura de pipeline custa centavos na cloud. On-premises, exigiria um projeto de infraestrutura.
  • Acesso a tecnologias de ponta: serviços como GPUs para treinamento de ML, bancos de dados especializados (graph, time-series, document) e streaming em tempo real estão disponíveis com alguns cliques, sem necessidade de expertise em hardware.
  • Globalização de dados: replicar dados entre regiões geográficas para compliance (LGPD, GDPR) ou para performance é nativo na cloud.

Modelo econômico: CapEx vs OpEx

A diferença entre CapEx (Capital Expenditure) e OpEx (Operational Expenditure) é central para entender a economia da cloud:

Aspecto CapEx (On-premises) OpEx (Cloud)
Investimento inicial Alto (compra de hardware, licenças) Zero ou muito baixo
Modelo de custo Fixo, independente do uso Variável, proporcional ao uso
Escalabilidade Limitada pela capacidade instalada Virtualmente ilimitada
Depreciação 3-5 anos de amortização contábil Sem depreciação (despesa operacional)
Risco de sub-utilização Alto (hardware ocioso) Baixo (paga apenas pelo consumo)
Tempo para provisionar Semanas a meses Minutos a horas

Para workloads de dados, o modelo OpEx é especialmente vantajoso porque a demanda de processamento é frequentemente variável: pipelines batch rodam à noite, cargas de treinamento de ML são esporádicas, campanhas de marketing geram picos temporários de ingestão. Na cloud, você paga pelo pico apenas enquanto ele dura; on-premises, você paga o tempo todo por capacidade que pode ficar ociosa 80% do dia.

Modelos de Serviço

IaaS — Infrastructure as a Service

No modelo IaaS, o provedor de cloud entrega a camada de infraestrutura virtualizada: computação (servidores virtuais), armazenamento (discos) e rede (VPCs, load balancers). O consumidor é responsável por tudo que roda acima dessa camada: sistema operacional, patches de segurança, middleware, runtime, aplicações e dados.

Serviços AWS representativos:

  • Amazon EC2 (Elastic Compute Cloud): servidores virtuais com centenas de tipos de instância diferentes, desde t3.micro (1 vCPU, 1 GB RAM) até instâncias com 448 vCPUs e 24 TB de RAM. Para dados, as famílias de instância mais relevantes são: r6i/r7i (otimizadas para memória, ideais para Spark e bancos in-memory), c6i/c7i (otimizadas para computação, ideais para transformações CPU-intensive) e p4d/p5 (com GPUs NVIDIA, para treinamento de modelos de ML).
  • Amazon EBS (Elastic Block Store): discos de alta performance anexados a instâncias EC2. O tipo gp3 oferece até 16.000 IOPS a custo acessível; o tipo io2 oferece até 64.000 IOPS para workloads que exigem latência sub-milissegundo (como bancos de dados autogerenciados).
  • Amazon VPC (Virtual Private Cloud): rede virtual isolada onde seus recursos rodam. Permite controle total sobre sub-redes, tabelas de roteamento, gateways de internet e conexões VPN com redes corporativas. Essencial para compliance e isolamento de ambientes.

Quando usar IaaS para dados: quando você precisa de controle total sobre a pilha de software — por exemplo, para rodar uma versão específica do Apache Spark que não é suportada pelo EMR, para instalar drivers de GPU customizados para treinamento de modelos, ou para rodar bancos de dados open-source (ClickHouse, TimescaleDB) que não têm equivalente gerenciado na AWS.

PaaS — Platform as a Service

No modelo PaaS, o provedor gerencia a infraestrutura e o runtime. O consumidor entrega a aplicação (ou a configuração) e os dados, sem se preocupar com sistema operacional, patches ou escalabilidade da infraestrutura subjacente.

Serviços AWS representativos:

  • Amazon RDS (Relational Database Service): bancos de dados relacionais gerenciados. Suporta PostgreSQL, MySQL, MariaDB, Oracle, SQL Server e Amazon Aurora. O RDS cuida de backups automáticos, patching, failover, réplicas de leitura e monitoramento. O engenheiro de dados interage apenas com o banco via SQL, sem gerenciar o servidor que o hospeda.
  • Amazon EMR (Elastic MapReduce): clusters Spark, Hive, Presto e Flink gerenciados. Você define o tamanho do cluster, submete seus jobs e o EMR cuida do provisionamento, configuração e desligamento das instâncias. Combina a flexibilidade do Spark com a conveniência de não gerenciar o cluster manualmente.
  • AWS Elastic Beanstalk: plataforma para deploy de aplicações web. Menos usado diretamente por engenheiros de dados, mas relevante quando o pipeline inclui APIs de ingestão que precisam escalar automaticamente.

Quando usar PaaS para dados: quando você quer a funcionalidade de tecnologias como PostgreSQL ou Spark sem o overhead de instalar, configurar, atualizar e monitorar a infraestrutura. É a escolha mais comum para a maioria das plataformas de dados empresariais.

SaaS — Software as a Service

No modelo SaaS, o provedor entrega a funcionalidade completa. O consumidor interage com uma interface ou API para obter resultados, sem gerenciar nada da infraestrutura ou da aplicação subjacente.

Serviços AWS representativos:

  • Amazon Athena: serviço de consultas SQL serverless sobre dados no S3. Não há infraestrutura para provisionar: você escreve a query, o Athena a executa sobre os dados no S3 e cobra por volume de dados escaneado (US$ 5 por TB). Ideal para consultas ad-hoc e exploração de dados no Data Lake.
  • Amazon QuickSight: ferramenta de BI e visualização de dados serverless. Conecta-se a fontes como Athena, Redshift, RDS e S3 para gerar dashboards interativos. Possui engine in-memory (SPICE) para performance e suporta ML integrado para detecção de anomalias e previsões.
  • Amazon Rekognition: serviço de visão computacional para análise de imagens e vídeos. Relevante em pipelines de dados que incluem conteúdo multimídia — por exemplo, classificação automática de imagens de produtos em e-commerce.

Quando usar SaaS para dados: quando você quer resultado imediato, sem escrever código de infraestrutura. O Athena é o exemplo mais emblemático: em vez de manter um cluster de banco de dados para consultas, você simplesmente executa SQL diretamente sobre arquivos no S3.

Arquiteturas de Dados na Nuvem

Data Lake

Um Data Lake é um repositório centralizado que armazena dados em seu formato original — estruturados, semiestruturados e não estruturados — sem transformação prévia no momento da gravação. O conceito central é schema-on-read: o esquema dos dados é aplicado no momento da leitura (quando alguém executa uma query ou um job de transformação), não no momento da gravação. Isso confere máxima flexibilidade para exploração, experimentação e reprocessamento.

Na AWS, o componente central de um Data Lake é o Amazon S3, que oferece durabilidade de 99,999999999% (11 noves), disponibilidade de 99,99% e custo de armazenamento extremamente baixo (a partir de US$ 0,023/GB/mês no tier Standard, e US$ 0,004/GB/mês no tier Glacier Instant Retrieval). Sobre o S3, serviços como AWS Glue (catálogo de metadados e ETL serverless) e Amazon Athena (consultas SQL serverless) formam a base analítica do Lake no mercado AWS. Como o Learner Lab da disciplina não libera o Glue, nós substituímos o Glue por PySpark em EC2 (Aula 05) e por tabelas externas criadas diretamente no Athena (a partir da Aula 02). O Athena permanece disponível e continua sendo o motor de consulta serverless da disciplina.

A organização canônica de um Data Lake utiliza três camadas, cada uma com propósito, formato de dados e ciclo de vida específicos:

Camada Raw / Bronze

  • Propósito: armazenar os dados exatamente como chegaram da fonte, sem nenhuma transformação, limpeza ou deduplicação. Esta camada serve como a fonte da verdade (source of truth) e permite reprocessamento completo em caso de erro em estágios posteriores.
  • Formatos típicos: JSON bruto de APIs, CSV exportado de bancos transacionais, Avro de sistemas de streaming (Kafka, Kinesis), logs de texto plano, imagens, vídeos, PDFs.
  • Organização: particionamento por data de ingestão e fonte. Exemplo: s3://data-lake-raw/source=crm/year=2026/month=04/day=16/
  • Ciclo de vida: dados brutos são mantidos por períodos longos (frequentemente anos) para compliance e possibilidade de reprocessamento. Políticas de lifecycle do S3 podem migrar dados antigos para tiers de menor custo (Infrequent Access, Glacier).
  • Acesso: restrito a engenheiros de dados e pipelines automatizados. Analistas e cientistas de dados não devem acessar esta camada diretamente.

Camada Trusted / Silver

  • Propósito: dados limpos, validados, deduplicados e padronizados. Erros de tipo, valores nulos inválidos e registros duplicados foram tratados. Os dados estão em formato otimizado para consulta e são consumíveis por times técnicos (engenheiros de dados, cientistas de dados).
  • Formatos típicos: Apache Parquet (colunar, comprimido, com schema embutido) ou Apache ORC. A conversão de JSON/CSV para Parquet é uma das primeiras transformações do pipeline.
  • Organização: particionamento por domínio de negócio e dimensões analíticas relevantes. Exemplo: s3://data-lake-trusted/domain=vendas/year=2026/month=04/
  • Ciclo de vida: dados são mantidos enquanto tiverem valor analítico. Geralmente reprocessados periodicamente conforme as regras de limpeza evoluem.
  • Acesso: engenheiros de dados (leitura e escrita), cientistas de dados (leitura), pipelines automatizados (leitura e escrita).

Camada Refined / Gold

  • Propósito: dados agregados, modelados e prontos para consumo direto por ferramentas de BI, dashboards executivos e modelos de machine learning. Frequentemente organizados por domínio de negócio ou caso de uso específico (ex.: "métricas de vendas diárias", "features para modelo de churn").
  • Formatos típicos: Parquet com schema rígido e bem documentado. Em alguns casos, os dados são também materializados em um Data Warehouse (Redshift no mercado AWS; ClickHouse em EC2 no contexto da disciplina, que não libera Redshift no Learner Lab) para performance máxima em consultas recorrentes.
  • Organização: particionamento otimizado para os padrões de consulta mais frequentes. Exemplo: s3://data-lake-refined/kpi=vendas_diarias/year=2026/month=04/
  • Ciclo de vida: dados são recalculados periodicamente. Versões anteriores podem ser mantidas para auditoria.
  • Acesso: analistas de BI (leitura), cientistas de dados (leitura), ferramentas de BI (QuickSight, Tableau), modelos de ML em produção. Engenheiros de dados controlam a escrita.

Data Warehouse

Um Data Warehouse é um sistema otimizado para consultas analíticas de alta performance sobre dados estruturados e transformados. O modelo de dados é definido na gravação (schema-on-write): antes de inserir dados, é necessário criar tabelas com schemas rígidos. Isso permite ao banco de dados otimizar o layout físico dos dados para velocidade de leitura.

Na AWS, o Amazon Redshift é o serviço de Data Warehouse gerenciado. Como o Learner Lab da disciplina não libera Redshift, a Aula 06 usará um ClickHouse em EC2 como substituto didático — um DW colunar MPP open-source que reproduz as mesmas características arquiteturais descritas abaixo. Conceitualmente, o que você aprende sobre Redshift vale diretamente para ClickHouse. Suas características arquiteturais incluem:

  • Processamento Massivamente Paralelo (MPP): as queries são divididas entre múltiplos nós de computação que trabalham em paralelo. Cada nó processa uma fatia (slice) dos dados, e o nó líder agrega os resultados.
  • Armazenamento colunar: em vez de armazenar dados linha por linha (como um banco OLTP), o Redshift armazena cada coluna em blocos contíguos. Isso é ideal para queries analíticas que tipicamente leem poucas colunas de tabelas com muitas colunas (ex.: SELECT SUM(valor) FROM vendas WHERE ano = 2026 lê apenas as colunas valor e ano, ignorando todas as outras).
  • Compressão automática: o Redshift analisa os dados e aplica o algoritmo de compressão mais eficiente para cada coluna, reduzindo o armazenamento e melhorando a performance de I/O.
  • Redshift Spectrum: permite executar queries que combinam dados de tabelas internas do Redshift com dados armazenados no S3, sem necessidade de carregá-los. Isso integra Data Warehouse e Data Lake em uma única interface SQL.
  • Redshift Serverless: versão sem cluster para provisionar. O Redshift escala automaticamente a capacidade de computação conforme a demanda das queries.

O Data Warehouse é a escolha certa quando você precisa de latência consistentemente baixa em consultas complexas (joins entre tabelas grandes, agregações com filtros), relatórios regulares com SLA de performance definido e dashboards de BI que exigem resposta em segundos.

Data Lakehouse

O Data Lakehouse é uma arquitetura que combina as melhores características do Data Lake e do Data Warehouse em uma única plataforma. A ideia central é: manter os dados no Data Lake (S3), mas adicionar uma camada de gerenciamento que fornece funcionalidades tradicionalmente exclusivas de Data Warehouses — transações ACID, schema enforcement, indexação, time travel e atualizações incrementais.

Essa camada de gerenciamento é implementada por formatos de tabela abertos (open table formats):

  • Apache Iceberg: desenvolvido pela Netflix e depois doado à Apache Foundation. Suporta transações ACID, time travel, schema evolution e partition evolution. É o formato adotado pelo AWS como padrão em serviços como Athena v3, EMR e Glue. A AWS oferece suporte nativo ao Iceberg em múltiplos serviços, tornando-o a escolha mais natural no ecossistema AWS.
  • Delta Lake: desenvolvido pela Databricks. Suporta transações ACID, time travel, schema enforcement e operações MERGE (upsert). Amplamente usado por clientes Databricks, mas também compatível com Spark open-source e com o EMR.
  • Apache Hudi: desenvolvido pela Uber. Foco em ingestão incremental eficiente (upserts em grande escala) e compactação de dados. Ideal para cenários de CDC (Change Data Capture) onde registros são frequentemente atualizados.

Implementação na AWS (mercado): dados armazenados no S3 em formato Parquet, com metadados de tabela gerenciados pelo Apache Iceberg. O AWS Glue Data Catalog serve como metastore. O Athena e o Redshift Spectrum consultam esses dados com suporte a transações e time travel. O resultado é uma plataforma que tem a flexibilidade e o custo do Data Lake com a confiabilidade e a performance do Data Warehouse.

Implementação na disciplina (Learner Lab): como Glue e Redshift não estão disponíveis, usamos S3 + Athena + PySpark em EC2. O Athena oferece suporte nativo a Iceberg (engine v3) e permite consultar dados no S3 com semântica de tabela. O metastore fica self-hosted em EC2 (Hive Metastore ou o próprio Iceberg catalog file). Conceitualmente é equivalente à implementação de mercado.

Data Mesh

Data Mesh é uma abordagem arquitetural e organizacional proposta por Zhamak Dehghani (2019) que trata dados como produto e distribui a responsabilidade pelos dados para os domínios de negócio que os originam. É fundamentalmente diferente das arquiteturas anteriores porque não é apenas uma escolha técnica — é uma mudança organizacional.

O Data Mesh se baseia em quatro princípios:

  1. Propriedade de domínio (Domain Ownership): cada domínio de negócio (vendas, logística, financeiro, marketing) é responsável por seus próprios dados, incluindo qualidade, pipeline e disponibilização. Não existe um time central de dados que é o guardião de todos os dados da organização.
  2. Dados como produto (Data as a Product): cada domínio trata seus dados como um produto com consumidores reais. Isso implica documentação, SLAs de qualidade, contratos de interface (schemas), monitoramento de uso e evolução planejada.
  3. Plataforma self-service (Self-Serve Data Platform): a organização fornece uma plataforma de infraestrutura que permite aos domínios publicar e consumir produtos de dados sem depender de um time centralizado para cada operação. A plataforma oferece templates, automação, governança e observabilidade como serviço.
  4. Governança federada computacional (Federated Computational Governance): a governança é definida centralmente (padrões de qualidade, segurança, interoperabilidade), mas executada de forma automática e descentralizada. Políticas são implementadas como código que é aplicado automaticamente pela plataforma.

Quando o Data Mesh faz sentido: em organizações grandes (tipicamente com mais de 50-100 engenheiros de dados) onde um time centralizado se torna gargalo. Organizações menores geralmente não têm a complexidade organizacional que justifica a overhead de um Data Mesh.

Implementação na AWS (referência de mercado): AWS Lake Formation para controle de acesso fine-grained por domínio, Amazon EventBridge para integração assíncrona entre domínios, AWS Glue Data Catalog como catálogo federado onde cada domínio registra seus produtos de dados, e AWS RAM (Resource Access Manager) para compartilhamento de recursos entre contas AWS de diferentes domínios. Como o Learner Lab da disciplina não libera Glue nem Lake Formation, não iremos montar um Data Mesh real nas práticas — mas o conceito é fundamental para o panorama de arquitetura.

AWS como Plataforma de Dados

Infraestrutura global

A AWS opera a maior infraestrutura de cloud pública do mundo:

  • Regiões (32+): conjuntos geográficos independentes, cada um com governança de dados isolada. Dados criados em uma região não são automaticamente replicados para outras. Exemplos: us-east-1 (N. Virginia), sa-east-1 (São Paulo), eu-west-1 (Irlanda).
  • Zonas de Disponibilidade / AZs (100+): dentro de cada região, existem múltiplas AZs (tipicamente 3 a 6). Cada AZ é composta por um ou mais data centers com energia, resfriamento e conectividade de rede independentes. As AZs são fisicamente separadas por distância suficiente para resistir a desastres localizados (enchentes, incêndios), mas conectadas por fibra óptica de baixa latência (tipicamente menos de 2ms entre AZs na mesma região).
  • Edge Locations (400+): pontos de presença para Amazon CloudFront (CDN) e AWS Global Accelerator. Usados para entregar conteúdo com baixa latência a usuários finais.
  • Local Zones: extensões de regiões AWS colocadas próximas a centros populacionais para workloads que exigem latência de um dígito de milissegundos (ex.: aplicações de streaming de vídeo, gaming).

Por que usaremos us-east-1 nesta disciplina

A região us-east-1 (N. Virginia) foi escolhida por três razões práticas:

  1. Cobertura máxima de serviços: us-east-1 é tipicamente a primeira região a receber novos serviços e features da AWS. Todos os serviços que usaremos na disciplina estão disponíveis nela.
  2. Custo ligeiramente menor: os preços de muitos serviços são mais baixos em us-east-1 comparados a outras regiões, incluindo sa-east-1 (São Paulo).
  3. Compatibilidade com AWS Student: o ambiente de laboratório da AWS Academy opera primariamente em us-east-1.
Atenção: sempre confirme que a região no console está definida como us-east-1 antes de criar qualquer recurso. Recursos criados em regiões diferentes não são visíveis entre si por padrão, e criar recursos na região errada é uma das causas mais comuns de confusão em laboratórios.

Modelo de Responsabilidade Compartilhada

O Shared Responsibility Model da AWS é o conceito de segurança mais importante para qualquer profissional que trabalha com cloud. Ele define claramente o que é responsabilidade da AWS e o que é responsabilidade do cliente:

A AWS é responsável pela segurança DA nuvem (security OF the cloud):

  • Hardware físico dos data centers (servidores, discos, switches)
  • Segurança física (acesso biométrico, vigilância, destruição de hardware)
  • Infraestrutura de rede global (backbone, Edge Locations)
  • Hypervisors e camada de virtualização
  • Disponibilidade das regiões e zonas de disponibilidade

O cliente é responsável pela segurança NA nuvem (security IN the cloud):

  • Configuração de IAM (quem pode acessar o quê)
  • Configuração de rede (Security Groups, NACLs, VPC)
  • Criptografia de dados em repouso e em trânsito
  • Configuração de buckets S3 (público vs privado)
  • Gerenciamento de credenciais e rotação de chaves
  • Patching de sistema operacional (em instâncias EC2)
  • Configuração de firewalls de aplicação (WAF)
Exemplo concreto para engenharia de dados: se um bucket S3 contendo dados sensíveis de clientes for configurado com acesso público e os dados forem expostos, a responsabilidade é inteiramente do cliente, não da AWS. A AWS fornece as ferramentas para bloquear acesso público (S3 Block Public Access, bucket policies, ACLs), mas a configuração correta é responsabilidade de quem administra a conta. Da mesma forma, se um job PySpark rodando em EC2 (ou qualquer outro serviço gerenciado como Lambda) usar uma role com permissões excessivas (por exemplo, AdministratorAccess) e essa role for comprometida, a responsabilidade pela escalação de privilégio é do cliente.

Módulo 2 — Governança de Acessos com AWS IAM

Por que Governança de Acessos é o Alicerce

Incidentes reais que demonstram a importância

A governança de acessos não é um tema abstrato ou meramente teórico. Falhas de configuração IAM e de controle de acesso estão entre as causas mais frequentes e mais caras de incidentes de segurança em cloud. Alguns exemplos ilustrativos:

  • Incidente Capital One (2019): uma ex-funcionária de um provedor de cloud explorou uma configuração incorreta de WAF (Web Application Firewall) combinada com uma role IAM com privilégios excessivos para acessar dados de mais de 100 milhões de clientes armazenados no S3. O custo para a empresa ultrapassou US$ 150 milhões entre multas, processos e remediação. A causa raiz não foi uma vulnerabilidade zero-day — foi uma configuração de acesso inadequada.
  • Buckets S3 expostos publicamente: ao longo dos anos, diversas organizações tiveram dados sensíveis expostos por buckets S3 configurados com acesso público. Dados de eleitores, registros médicos, informações financeiras e credenciais de acesso foram encontrados por pesquisadores de segurança em buckets abertos. Em todos os casos, a AWS fornecia as ferramentas para bloquear o acesso — a responsabilidade pela configuração era do cliente.
  • Mineração de criptomoedas com credenciais roubadas: access keys de IAM vazadas em repositórios Git públicos são rapidamente descobertas por bots automatizados que provisionam instâncias EC2 de alto custo para mineração de criptomoedas. Empresas já reportaram faturas surpresa de dezenas de milhares de dólares por uso não autorizado de instâncias EC2 com GPUs.

Contexto regulatório

Além dos riscos operacionais e financeiros, falhas de governança de acesso geram consequências regulatórias:

  • LGPD (Lei Geral de Proteção de Dados — Brasil): exige que dados pessoais sejam protegidos por medidas técnicas e administrativas. A falta de controle de acesso adequado é uma violação direta da lei, com multas de até 2% do faturamento (limitadas a R$ 50 milhões por infração).
  • GDPR (General Data Protection Regulation — UE): exige proteção de dados pessoais de cidadãos europeus. Multas podem chegar a 4% do faturamento global ou 20 milhões de euros, o que for maior.
  • SOC 2 (Service Organization Control): auditoria comum em empresas de tecnologia que demonstra controles de segurança adequados. A ausência de controle de acesso granular é um achado crítico que pode impedir a certificação.

Custo total da governança inadequada

O custo de uma governança de acessos inadequada vai muito além de multas:

  • Custos financeiros diretos: faturas inesperadas por uso não autorizado de recursos.
  • Custos de remediação: investigação forense, rotação de credenciais, reconfiguração de ambientes.
  • Perda de dados: dados deletados ou corrompidos por usuários com permissões excessivas.
  • Danos reputacionais: perda de confiança de clientes e parceiros após um vazamento.
  • Custos legais: processos judiciais, negociações regulatórias, auditorias extraordinárias.

AWS IAM em Profundidade

O AWS Identity and Access Management (IAM) é o serviço central de segurança da AWS. Cada chamada de API feita na AWS — seja pelo console, pela CLI, por um SDK ou por um serviço AWS — passa pelo IAM, que avalia as políticas associadas à identidade que faz a chamada e decide se concede ou nega o acesso. Compreender o IAM em profundidade é pré-requisito para trabalhar com qualquer serviço AWS de forma segura.

IAM Users (Usuários)

Um IAM User representa uma pessoa ou sistema com credenciais de longo prazo na conta AWS. Cada usuário pode ter dois tipos de credenciais:

  • Senha do console (Console password): usada para acessar o AWS Management Console (interface web). Pode ter política de complexidade, expiração e obrigatoriedade de troca no primeiro login.
  • Chaves de acesso (Access keys): par de Access Key ID + Secret Access Key, usado para acessar a AWS via CLI (aws cli), SDKs (boto3, AWS SDK for Java) ou chamadas diretas de API REST. As access keys são credenciais de longo prazo e devem ser tratadas com o mesmo cuidado que senhas.

Boas práticas para IAM Users:

  • Nunca use a conta root para operações do dia a dia. A conta root tem acesso irrestrito a todos os recursos e serviços da conta AWS, e suas permissões não podem ser limitadas por políticas IAM. O root deve ser usado apenas para operações que exigem exclusivamente credenciais root (como fechar a conta ou alterar o plano de suporte).
  • Habilite MFA (Multi-Factor Authentication) para todos os usuários. O MFA adiciona um segundo fator de autenticação (tipicamente um código de 6 dígitos gerado por um aplicativo como Google Authenticator ou Authy) ao login no console.
  • Rotacione access keys periodicamente. A AWS recomenda rotação a cada 90 dias. O IAM Credential Report permite identificar chaves que não foram rotacionadas.
  • Nunca compartilhe credenciais entre pessoas. Cada pessoa deve ter seu próprio usuário IAM. Compartilhar credenciais impede auditoria individual e viola o princípio de accountability.
  • Nunca coloque access keys no código-fonte. Use variáveis de ambiente, AWS Secrets Manager, ou (preferencialmente) roles IAM que eliminam a necessidade de credenciais estáticas.

IAM Groups (Grupos)

Um IAM Group é uma coleção de IAM Users. Políticas são atribuídas ao grupo, e todos os usuários membros herdam essas permissões automaticamente. Grupos simplificam drasticamente o gerenciamento de permissões em equipes:

  • Quando um novo membro entra na equipe, basta adicioná-lo ao grupo correto.
  • Quando alguém muda de função, basta movê-lo para outro grupo.
  • Quando uma política precisa ser atualizada, a mudança é feita uma vez no grupo e se aplica a todos os membros.

Padrões de design de grupos para equipes de dados:

Grupo Perfil Políticas típicas
DataEngineers Engenheiros que constroem e mantêm pipelines S3 Full, EC2 (Spark workers), Athena Full, Kinesis Full, CloudWatch Logs, Step Functions
DataScientists Cientistas que exploram dados e treinam modelos S3 Read (Silver/Gold), Athena Full, SageMaker
DataAnalysts Analistas que consultam dados e criam relatórios Athena Full, S3 Read (Gold), ClickHouse Read, QuickSight
DataAdmins Administradores da plataforma de dados IAM Full, Lake Formation, CloudTrail, Billing Read

Convenções de nomenclatura: use PascalCase para grupos (DataEngineers, não data-engineers), pois é o padrão adotado na documentação AWS e facilita a leitura em políticas e logs.

IAM Roles (Papéis)

Uma IAM Role é uma identidade que pode ser assumida temporariamente por um usuário IAM, um serviço AWS, ou uma identidade de outra conta AWS. A diferença fundamental entre um Role e um User é que o Role não possui credenciais de longo prazo. Quando uma entidade assume um Role, o AWS Security Token Service (STS) gera credenciais temporárias que expiram automaticamente (por padrão, em 1 hora, configurável até 12 horas).

Uma Role é composta por dois elementos:

  • Trust Policy (Política de confiança): define quem pode assumir a role. Por exemplo: "o serviço AWS Glue pode assumir esta role" ou "usuários da conta 123456789012 podem assumir esta role".
  • Permission Policy (Política de permissão): define o que a role pode fazer depois de assumida. Por exemplo: "pode ler objetos no bucket S3 X e gravar logs no CloudWatch".

Por que Roles são fundamentais para serviços AWS:

Quando uma função Lambda (ou uma instância EC2 rodando um job PySpark) precisa ler dados de um bucket S3 e gravar resultados em outro bucket, ela precisa de permissões para essas operações. A forma correta de conceder essas permissões é criar uma Role com uma Trust Policy que permite ao serviço (lambda.amazonaws.com ou ec2.amazonaws.com) assumi-la e Permission Policies que concedem acesso S3. O serviço assume a role automaticamente no início da execução e as credenciais expiram quando o job termina.

A forma incorreta (e perigosa) seria criar um IAM User com access keys e configurar essas chaves dentro do código ou nas variáveis de ambiente do job. Isso é inseguro porque: as chaves são de longo prazo e não expiram automaticamente; se vazarem, o atacante tem acesso permanente; não há como revogar as chaves sem impactar todos os jobs que as usam; e o CloudTrail não diferencia ações feitas por jobs diferentes usando a mesma chave.

Tipos de Roles mais usados em dados:

  • Service Role: assumida por um serviço AWS (Lambda, EC2, Kinesis Firehose, Step Functions).
  • Cross-Account Role: assumida por identidades de outra conta AWS. Usada em Data Mesh para compartilhar dados entre domínios.
  • Service-Linked Role: role pré-configurada pela AWS para um serviço específico. Criada automaticamente quando você habilita o serviço.

IAM Policies (Políticas)

Uma IAM Policy é um documento JSON que define formalmente o que é permitido ou negado. Toda decisão de acesso na AWS é baseada na avaliação de uma ou mais políticas. Compreender a anatomia de uma política é essencial: 

{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "AllowReadFromDataLakeGold",
      "Effect": "Allow",
      "Action": [
        "s3:GetObject",
        "s3:ListBucket"
      ],
      "Resource": [
        "arn:aws:s3:::olist-refined-*",
        "arn:aws:s3:::olist-refined-*/*"
      ],
      "Condition": {
        "StringEquals": {
          "aws:RequestedRegion": "us-east-1"
        }
      }
    }
  ]
}

Anatomia de cada campo:

  • Version: versão da linguagem de políticas IAM. Sempre use "2012-10-17" (a versão mais recente). A versão anterior ("2008-10-17") não suporta variáveis de política.
  • Statement: array de declarações. Cada declaração é uma regra de acesso independente. Uma política pode ter múltiplos statements.
  • Sid (Statement ID): identificador opcional e legível para humanos. Não afeta a avaliação, mas facilita a auditoria e a documentação.
  • Effect: "Allow" ou "Deny". Define se a declaração concede ou nega acesso.
  • Principal: quem é afetado pela política. Usado em resource-based policies (como bucket policies do S3). Em identity-based policies (atachadas a Users, Groups ou Roles), o Principal é implicitamente a identidade à qual a política está atachada.
  • Action: lista de operações de API que a declaração cobre. Formato: serviço:OperaçãoDeAPI. Exemplos: s3:GetObject, ec2:RunInstances, athena:StartQueryExecution. O wildcard * significa "todas as ações".
  • Resource: ARN (Amazon Resource Name) do recurso ao qual a ação se aplica. O formato ARN é arn:aws:serviço:região:conta:recurso. O wildcard * significa "todos os recursos".
  • Condition: condições opcionais que devem ser verdadeiras para a declaração se aplicar. Exemplos: permitir acesso apenas de um determinado IP, apenas se o MFA estiver ativo, apenas em uma determinada região, apenas em horário comercial.

Tipos de políticas:

  • Identity-based policies (políticas baseadas em identidade): atachadas a Users, Groups ou Roles. Definem o que a identidade pode fazer.
  • Resource-based policies (políticas baseadas em recurso): atachadas diretamente a recursos AWS (como bucket policies do S3 ou key policies do KMS). Definem quem pode acessar o recurso.
  • Permissions Boundaries (limites de permissão): definem o limite máximo de permissões que uma identity-based policy pode conceder. Funcionam como um "teto": mesmo que a policy conceda AdministratorAccess, se o permissions boundary permitir apenas S3 e Athena, apenas essas permissões serão efetivas.
  • SCPs (Service Control Policies): usadas no AWS Organizations para definir o limite máximo de permissões para toda a organização ou para OUs (Organizational Units) específicas.
  • Session policies: políticas passadas no momento de assumir uma role, limitando ainda mais as permissões da sessão.

Lógica de avaliação de políticas

A avaliação de políticas IAM segue uma lógica precisa que todo profissional de dados deve dominar:

  1. Deny explícito? Se qualquer política aplicável contém um "Effect": "Deny" que se aplica à ação e ao recurso, o acesso é negado. Um Deny explícito sempre prevalece sobre qualquer Allow. Não há como "sobrescrever" um Deny.
  2. Allow explícito? Se alguma política aplicável contém um "Effect": "Allow" que se aplica à ação e ao recurso (e todas as condições são satisfeitas), o acesso é concedido.
  3. Nem Deny nem Allow? Se nenhuma política se pronuncia sobre a ação e o recurso, o acesso é negado por padrão (implicit deny).

Resumo: Deny explícito > Allow explícito > Deny implícito (padrão).

Dica prática: na dúvida, lembre-se de que na AWS tudo é negado por padrão. Você precisa explicitamente conceder acesso. E se alguém explicitamente negar, essa negação prevalece sobre tudo.

Princípio do Menor Privilégio

O Princípio do Menor Privilégio (Principle of Least Privilege) é a pedra angular da segurança em cloud: conceda a cada identidade apenas as permissões estritamente necessárias para que ela realize sua função, e nada além disso. Se uma função Lambda precisa ler de um bucket S3 específico e gravar logs no CloudWatch, a role dessa função deve ter permissão apenas para essas duas operações, nesses recursos específicos.

Estratégias de implementação

  • Comece com zero permissões: crie a identidade (user ou role) sem nenhuma política. Adicione permissões incrementalmente conforme a necessidade for documentada e validada.
  • Use AWS managed policies como ponto de partida, depois refine: políticas gerenciadas pela AWS (como AmazonS3ReadOnlyAccess) são um bom começo para prototipação e aprendizado, mas em produção devem ser substituídas por políticas customizadas com escopo mais restrito (por exemplo, leitura apenas em buckets específicos, não em todos).
  • Especifique recursos explicitamente: em vez de "Resource": "*" (todos os recursos), use ARNs específicos: "Resource": "arn:aws:s3:::meu-data-lake-gold/*".
  • Especifique ações explicitamente: em vez de "Action": "s3:*" (todas as ações S3), liste apenas as necessárias: "Action": ["s3:GetObject", "s3:ListBucket"].
  • Use condições: restrinja acesso por região (aws:RequestedRegion), IP (aws:SourceIp), MFA (aws:MultiFactorAuthPresent), ou horário.

IAM Access Analyzer

O AWS IAM Access Analyzer é uma ferramenta que ajuda a identificar permissões excessivas. Ele analisa políticas atachadas a identidades e identifica ações e recursos que a identidade tem permissão para acessar mas que nunca foram efetivamente utilizados (com base nos logs do CloudTrail). Isso permite refinar políticas removendo permissões que foram concedidas mas nunca usadas.

O Access Analyzer também identifica recursos compartilhados externamente (como buckets S3 acessíveis por outras contas) e gera políticas de menor privilégio automaticamente com base no uso observado.

Permission Boundaries como guardrails

Em ambientes onde múltiplas equipes criam suas próprias roles e políticas (como em um Data Mesh), Permission Boundaries são essenciais como guardrails. Eles definem o limite máximo de permissões que qualquer role criada por uma equipe pode ter, mesmo que a equipe atache uma política com AdministratorAccess.

Exemplo: um permission boundary que permite apenas acesso a S3, Athena, EC2 e CloudWatch garante que nenhuma role criada pela equipe de dados possa, acidentalmente ou deliberadamente, acessar serviços fora do escopo do time (como IAM completo ou Lambda).

Anti-padrões comuns (o que NÃO fazer)

  • Wildcards em tudo: "Action": "*", "Resource": "*" é equivalente a AdministratorAccess. Nunca use em roles de serviço ou em políticas de equipe.
  • Admin para todos: atribuir AdministratorAccess a todos os membros da equipe "para facilitar" é o caminho mais curto para um incidente de segurança.
  • Credenciais compartilhadas: usar a mesma access key para múltiplas pessoas ou serviços impede auditoria individual e aumenta o raio de impacto se a chave vazar.
  • Credenciais no código: access keys hardcoded em scripts Python, notebooks Jupyter ou repositórios Git são um dos vetores de ataque mais explorados.
  • Não revisar permissões: permissões concedidas temporariamente ("só para testar") que nunca são revogadas acumulam-se ao longo do tempo, criando um ambiente cada vez mais permissivo.

Mapeando Perfis de Acesso para o Data Lake

Antes de criar qualquer recurso de IAM, é essencial mapear quais perfis existem na equipe de dados e quais permissões cada um precisa. Esse exercício previne retrabalho, evita concessão excessiva de permissões e garante que o modelo de segurança seja coerente com o modelo organizacional.

Data Engineer (Engenheiro de Dados)

O engenheiro de dados constrói, opera e mantém os pipelines de ingestão, transformação e disponibilização de dados. Suas permissões devem refletir a necessidade de acesso completo às ferramentas de pipeline, mas sem acesso à administração da plataforma (IAM, billing, redes).

  • S3: leitura e escrita em todas as camadas (Raw, Trusted, Refined). Precisa ingerir dados no Raw, transformar e gravar no Trusted, e materializar no Refined.
  • EC2: criar e gerenciar instâncias que rodam o PySpark para ETL e o ClickHouse para DW (substitutos de Glue e Redshift no Learner Lab).
  • Athena: criar tabelas externas, workgroups e executar queries SQL sobre dados no S3.
  • Kinesis: criar e gerenciar streams e firehoses para ingestão em tempo real.
  • Step Functions: criar e executar state machines para orquestração de pipelines.
  • CloudWatch: visualizar logs de execução de jobs, criar métricas customizadas e alarmes.
  • Lambda: criar e executar funções serverless usadas como componentes de pipelines.
  • IAM: sem permissão. A administração de acessos é responsabilidade do DataAdmin.

Data Scientist (Cientista de Dados)

O cientista de dados explora dados, treina modelos de machine learning e gera insights. Precisa de acesso de leitura a dados limpos e prontos, e de ferramentas de experimentação, mas não deve modificar os pipelines de produção.

  • S3: somente leitura nas camadas Trusted (Silver) e Refined (Gold). Não deve acessar a camada Raw (dados brutos não confiáveis) nem gravar em nenhuma camada.
  • Athena: executar queries SQL para exploração de dados.
  • Catálogo de dados: leitura do catálogo (Athena/Hive Metastore self-hosted neste curso) para descobrir datasets disponíveis.
  • SageMaker: criar e executar notebooks, training jobs e endpoints de inferência.
  • ECR (Elastic Container Registry): leitura para acessar imagens Docker de modelos.

BI Analyst (Analista de BI)

O analista de BI consome dados prontos para gerar relatórios, dashboards e análises de negócio. Suas permissões devem ser as mais restritivas: apenas leitura em dados refinados.

  • Athena: executar queries SQL (somente leitura, sem DDL).
  • ClickHouse em EC2 (DW da disciplina): conexão somente leitura em schemas de reporting.
  • QuickSight: acesso a dashboards publicados e criação de análises.
  • S3: somente leitura na camada Refined (Gold). Sem acesso a Raw ou Trusted.

Data Admin (Administrador de Dados)

O administrador gerencia a plataforma de dados: controle de acessos, governance, monitoramento e custos.

  • IAM: gerenciamento completo de users, groups, roles e policies.
  • Lake Formation: configuração de acesso fine-grained ao Data Lake.
  • CloudTrail: auditoria de todas as chamadas de API na conta.
  • CloudWatch: visualização de logs, métricas e alarmes de todos os serviços.
  • AWS Config: monitoramento de conformidade de recursos.
  • Billing: leitura de custos e uso para FinOps.

Módulo 3 — O LabRole do AWS Academy Learner Lab

O ambiente que usamos nas práticas é o AWS Academy Learner Lab. Ele é uma conta AWS real, com limites de tempo, créditos de consumo e um conjunto restrito de serviços habilitados. Para evitar que cada aluno gaste tempo criando roles de serviço nas primeiras aulas, a AWS Academy pré-configura uma role chamada LabRole, que os serviços disponíveis no lab assumem automaticamente (EC2, Lambda, Athena, Kinesis Firehose, SageMaker, Step Functions).

O que a LabRole permite

  • Acessos amplos a S3 (leitura, escrita, gerenciamento de buckets).
  • Criação, gerenciamento e término de instâncias EC2 (essenciais para PySpark e ClickHouse na disciplina).
  • Execução e gerenciamento de funções Lambda e streams Kinesis.
  • Queries Athena e gerenciamento de workgroups.
  • Logs em CloudWatch Logs e métricas.
  • Leitura de informações do IAM (listar users, groups, roles, policies) — mas não escrita.
  • Trust policy configurada para múltiplos serviços (ec2.amazonaws.com, lambda.amazonaws.com, firehose.amazonaws.com etc.).

O que a LabRole não permite (e por quê)

  • Acesso a AWS Glue e Amazon Redshift: esses dois serviços não estão habilitados no Learner Lab. Por esse motivo, a disciplina substitui Glue por PySpark em EC2 (Aula 05) e Redshift por ClickHouse em EC2 (Aula 06). Tentar chamar APIs desses serviços resulta em erro de autorização.
  • Criar usuários IAM persistentes: o ambiente é reciclado a cada sessão do lab, então não faz sentido criar usuários permanentes.
  • Alterar billing e organização: você não controla a conta raiz nem a conta do AWS Organizations da universidade.
  • Provisionar recursos de alto custo: instâncias com GPUs, clusters muito grandes e storage de tiers premium estão bloqueados.
  • Modificar a própria LabRole: a trust policy e as políticas atachadas vêm prontas; você não consegue adicionar nem remover permissões.
Implicação didática importante: em uma conta AWS real corporativa, não usaríamos uma role única com permissões amplas. Criaríamos uma role dedicada por serviço, com políticas de menor privilégio. Aqui, porém, o objetivo é aprender os conceitos de dados; o LabRole elimina o atrito inicial de IAM para que você foque no que importa nesta primeira hora de prática. Em aulas posteriores, revisitamos o tema.

Como a LabRole aparece no dia a dia do lab

Quando você subir uma instância EC2 na Aula 05 (para rodar PySpark) ou na Aula 06 (para rodar ClickHouse), por exemplo, o console perguntará qual IAM role associar à instância. A resposta padrão será LabRole. O mesmo vale para funções Lambda e entregas do Kinesis Firehose. Saber disso evita confusão quando o console pedir uma "IAM role" e a lista não estiver vazia.

Passo 3 — Criar a estrutura de buckets S3 do Data Lake (25 min) · em aula

Agora criamos a estrutura física do Data Lake com três buckets S3, um por camada. Os nomes usam o prefixo olist- porque o dataset Olist ocupará essa estrutura a partir da Aula 02.

  1. Na barra de busca do console, digite S3 e clique no serviço Amazon S3.
  2. Clique em Create bucket.
  3. Configure o primeiro bucket:
    • Bucket name: olist-raw-<iniciais> (substitua <iniciais> pelas suas iniciais em minúsculo, por exemplo olist-raw-jps). Nomes de bucket são globalmente únicos — se estiver em uso, acrescente números.
    • AWS Region: confirme que está US East (N. Virginia) us-east-1.
    • Object Ownership: mantenha "ACLs disabled" (recomendado).
    • Block Public Access: mantenha todas as 4 opções marcadas (Block all public access). Confirme marcando "I acknowledge...".
    • Bucket Versioning: selecione Enable.
    • Mantenha as demais opções com valores padrão.
  4. Clique em Create bucket.
  5. Repita o processo para criar mais dois buckets com exatamente as mesmas configurações:
    • olist-trusted-<iniciais>
    • olist-refined-<iniciais>

Por que estas configurações?

  • Block Public Access: garante que nenhum objeto nestes buckets pode ser acessado publicamente, mesmo que alguém configure uma ACL ou bucket policy incorretamente. É a principal defesa contra vazamentos.
  • Versioning: mantém um histórico de todas as versões de cada objeto. Se um arquivo for acidentalmente sobrescrito ou deletado, a versão anterior pode ser recuperada. Essencial para a camada Raw, que é a fonte da verdade.
  • Prefixo olist-: torna imediato, em qualquer operação, qual dataset estamos manipulando. Em uma conta compartilhada, isso é boa prática de organização.

Passo 5 — Criar grupos IAM (20 min) · aprofundamento

Vamos projetar dois grupos que representam perfis diferentes de acesso em uma equipe de dados.

Grupo 1: DataEngineers

  1. No menu lateral do IAM, clique em User groups.
  2. Clique em Create group.
  3. No campo "User group name", insira: DataEngineers
  4. Na seção "Attach permissions policies", use o campo de busca para encontrar e selecionar as seguintes políticas (marque o checkbox de cada uma):
    • AmazonS3FullAccess — acesso completo ao S3 (leitura, escrita, gerenciamento de buckets)
    • AmazonEC2FullAccess — acesso completo ao EC2 (instâncias para PySpark e ClickHouse)
    • AmazonAthenaFullAccess — acesso completo ao Athena (queries e workgroups)
    • CloudWatchLogsFullAccess — acesso completo a logs do CloudWatch (ler, escrever, criar log groups)
  5. Revise as seleções e clique em Create group.

Por que estas políticas? O engenheiro de dados precisa de acesso completo ao S3 para gerenciar as camadas do Data Lake, ao EC2 para subir instâncias que rodam o PySpark (ETL) e o ClickHouse (DW), ao Athena para consultar dados e ao CloudWatch Logs para monitorar a execução dos jobs. Em um ambiente de produção, refinaríamos estas políticas para escopo mais restrito (buckets específicos, instâncias específicas, log groups específicos), mas para o lab, as managed policies da AWS são adequadas para aprendizado.

Grupo 2: DataAnalysts

  1. Ainda em User groups, clique novamente em Create group.
  2. No campo "User group name", insira: DataAnalysts
  3. Busque e selecione as seguintes políticas:
    • AmazonAthenaFullAccess — acesso completo ao Athena (executar queries, gerenciar workgroups)
    • AmazonS3ReadOnlyAccess — somente leitura no S3 (listar buckets, ler objetos, sem escrita)
  4. Clique em Create group.

Por que estas políticas? O analista de BI precisa consultar dados via Athena e ler objetos no S3, mas nunca deve poder modificar dados no Data Lake. A política AmazonS3ReadOnlyAccess garante que o analista pode ler de qualquer bucket, mas não pode escrever nem deletar.

Validação do Passo 5: na tela de User groups, você deve ver os dois grupos (DataEngineers com 3 políticas e DataAnalysts com 2 políticas). Clique em cada grupo para confirmar que as políticas estão listadas na aba "Permissions". No Learner Lab essa operação pode falhar — nesse caso, apenas registre o design do grupo no seu caderno.

Validação do Passo 6: na lista de Users, ambos os usuários devem aparecer com status "Enabled". Clique em engenheiro-dados-lab e verifique: na aba "Groups", o grupo DataEngineers deve estar listado. Clique em analista-bi-lab e verifique que pertence ao grupo DataAnalysts. Caso o Learner Lab bloqueie a criação de usuários, apenas registre o design.

Passo 7 — Criar roles para serviços (20 min) · aprofundamento

Role 1: SparkEC2Role (substituto da GlueJobRole, já que o lab não libera Glue)

  1. No menu lateral do IAM, clique em Roles e depois em Create role.
  2. Em "Trusted entity type", selecione "AWS service".
  3. Em "Use case", na lista de serviços, selecione "EC2".
  4. Clique em Next.
  5. Na tela "Add permissions", busque e selecione as seguintes políticas:
    • AmazonS3FullAccess
    • CloudWatchLogsFullAccess
    • AmazonAthenaFullAccess
  6. Clique em Next.
  7. No campo "Role name", insira: SparkEC2Role
  8. Em "Description", insira: Role para instancias EC2 que rodam PySpark (ETL) - acesso a S3, Athena e CloudWatch Logs
  9. Revise o resumo (Trusted entities deve mostrar "ec2.amazonaws.com") e clique em Create role.

O que acabamos de fazer? Criamos uma role que uma instância EC2 (rodando PySpark) pode assumir. A Trust Policy (criada automaticamente ao selecionar "EC2" como trusted entity) contém: 

{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Principal": {
        "Service": "ec2.amazonaws.com"
      },
      "Action": "sts:AssumeRole"
    }
  ]
}

Isso significa: "o serviço EC2 tem permissão para chamar sts:AssumeRole nesta role". Quando uma instância EC2 é lançada com esta role anexada (como instance profile), o EC2 chama o STS, recebe credenciais temporárias com as permissões da role (S3, Athena, CloudWatch) e disponibiliza essas credenciais ao código Python/PySpark rodando dentro da instância via metadata endpoint.

Role 2: LambdaExecutionRole

  1. Clique novamente em Create role.
  2. Trusted entity type: "AWS service".
  3. Use case: selecione "Lambda".
  4. Clique em Next.
  5. Busque e selecione:
    • AmazonS3ReadOnlyAccess
    • AmazonDynamoDBReadOnlyAccess
    • CloudWatchLogsFullAccess
  6. Clique em Next.
  7. Role name: LambdaExecutionRole
  8. Description: Role para funções Lambda - leitura em S3 e DynamoDB, escrita em CloudWatch Logs
  9. Clique em Create role.

Diferença de escopo: note que a SparkEC2Role tem AmazonS3FullAccess (leitura e escrita, porque o PySpark precisa gravar resultados de transformação no Trusted/Refined), enquanto a LambdaExecutionRole tem apenas AmazonS3ReadOnlyAccess (porque, neste cenário, a Lambda apenas lê dados para processamento leve, sem gravar no S3).

Validação do Passo 7: na lista de Roles, ambas as roles devem ser visíveis. Clique em SparkEC2Role: na aba "Trust relationships", confirme que o serviço confiável é ec2.amazonaws.com; na aba "Permissions", confirme 3 políticas. Repita para LambdaExecutionRole: serviço confiável lambda.amazonaws.com, 3 políticas. No Learner Lab a criação de roles customizadas pode estar bloqueada; nesse caso, compare mentalmente com a LabRole já existente.

Passo 8 — Escrever política IAM customizada (20 min) · aprofundamento

Até agora, usamos apenas políticas gerenciadas pela AWS (managed policies). Agora vamos criar uma política customizada que implementa o princípio do menor privilégio para o cenário de um analista que deve ler dados refinados do Data Lake mas nunca pode modificá-los.

  1. No menu lateral do IAM, clique em Policies e depois em Create policy.
  2. Alterne para a aba "JSON" (não use o editor visual neste exercício — é fundamental saber escrever políticas JSON diretamente).
  3. Apague o conteúdo padrão e cole a seguinte política: 
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "AllowReadGoldLayer",
      "Effect": "Allow",
      "Action": [
        "s3:GetObject"
      ],
      "Resource": "arn:aws:s3:::olist-refined-*/*"
    },
    {
      "Sid": "AllowListGoldBuckets",
      "Effect": "Allow",
      "Action": [
        "s3:ListBucket"
      ],
      "Resource": "arn:aws:s3:::olist-refined-*"
    },
    {
      "Sid": "AllowAthenaQueries",
      "Effect": "Allow",
      "Action": [
        "athena:StartQueryExecution",
        "athena:GetQueryResults",
        "athena:GetQueryExecution"
      ],
      "Resource": "*"
    },
    {
      "Sid": "DenyWriteAndDelete",
      "Effect": "Deny",
      "Action": [
        "s3:PutObject",
        "s3:DeleteObject"
      ],
      "Resource": "*"
    }
  ]
}
  1. Clique em Next.
  2. No campo "Policy name", insira: DataLakeGoldReadOnly
  3. No campo "Description", insira: Permite leitura na camada Gold do Data Lake e queries no Athena. Nega explicitamente escrita e exclusao no S3.
  4. Clique em Create policy.

Análise da política criada:

  • Statement 1 (AllowReadGoldLayer): permite s3:GetObject apenas em objetos dentro de buckets cujo nome começa com olist-refined-. O wildcard * no final do ARN significa "qualquer objeto dentro desses buckets".
  • Statement 2 (AllowListGoldBuckets): permite s3:ListBucket para listar o conteúdo dos buckets Gold. Sem esta permissão, o analista poderia ler objetos individuais (se soubesse o caminho), mas não poderia listar quais objetos existem.
  • Statement 3 (AllowAthenaQueries): permite executar queries no Athena, verificar o status de execução e recuperar os resultados. O "Resource": "*" aqui é aceitável porque o acesso aos dados subjacentes é controlado pelas permissões S3.
  • Statement 4 (DenyWriteAndDelete): nega explicitamente s3:PutObject e s3:DeleteObject em qualquer recurso. Este Deny explícito garante que, mesmo se outra política conceder permissão de escrita (por exemplo, uma managed policy atachada ao grupo), a escrita será negada. Lembre-se: Deny explícito sempre prevalece.

Atachar a política ao grupo DataAnalysts:

  1. Vá para User groups e clique no grupo DataAnalysts.
  2. Clique na aba Permissions.
  3. Clique em Add permissions > Attach policies.
  4. Busque por DataLakeGoldReadOnly e selecione-a.
  5. Clique em Attach policies.
Validação do Passo 8: na lista de Policies, a política DataLakeGoldReadOnly deve ser visível (tipo "Customer managed"). No grupo DataAnalysts, a aba Permissions deve agora listar 3 políticas: AmazonAthenaFullAccess, AmazonS3ReadOnlyAccess e DataLakeGoldReadOnly.

Passo 9 — Validar com Policy Simulator (15 min) · aprofundamento

O IAM Policy Simulator é uma ferramenta que permite testar as permissões efetivas de um usuário, grupo ou role sem executar ações reais na conta. Vamos usá-lo para validar que nosso modelo de permissões está correto.

  1. Abra o Policy Simulator. Você pode acessá-lo de duas formas:
    • No menu lateral do IAM, procure o link "Policy Simulator" no final da página.
    • Ou acesse diretamente: https://policysim.aws.amazon.com/

Teste 1: engenheiro-dados-lab

  1. No painel esquerdo "Users, Groups, and Roles", expanda Users e selecione engenheiro-dados-lab.
  2. No painel direito, em "Policy Simulator", selecione o serviço Amazon S3.
  3. Em "Select actions", selecione GetObject.
  4. Clique em Run Simulation. Resultado esperado: allowed.
  5. Limpe a seleção e selecione PutObject.
  6. Run Simulation. Resultado esperado: allowed (engenheiro tem S3FullAccess).
  7. Agora mude o serviço para IAM e selecione CreateUser.
  8. Run Simulation. Resultado esperado: denied (engenheiro não tem permissão IAM).

Teste 2: analista-bi-lab

  1. No painel esquerdo, selecione analista-bi-lab.
  2. Serviço: Amazon S3. Ação: GetObject.
  3. Run Simulation. Resultado esperado: allowed (analista tem S3ReadOnlyAccess).
  4. Ação: PutObject.
  5. Run Simulation. Resultado esperado: denied (a política DataLakeGoldReadOnly tem Deny explícito para PutObject, e a S3ReadOnlyAccess não concede escrita).
  6. Serviço: Amazon Athena. Ação: StartQueryExecution.
  7. Run Simulation. Resultado esperado: allowed.
  8. Serviço: IAM. Ação: CreateUser.
  9. Run Simulation. Resultado esperado: denied.

O que estes resultados confirmam:

  • O engenheiro pode ler e escrever no S3 (necessário para pipelines), mas não pode criar usuários IAM (separação de responsabilidades).
  • O analista pode ler dados e executar queries no Athena (necessário para análise), mas não pode modificar dados no S3 (proteção da camada Gold).
  • Nenhum dos dois pode administrar IAM (responsabilidade exclusiva do DataAdmin).
  • O princípio do menor privilégio está sendo respeitado: cada perfil tem exatamente as permissões necessárias para sua função.
Validação do Passo 9: todos os 8 testes devem retornar os resultados esperados (allowed ou denied conforme listado acima). Anote os resultados para referência futura. Se o Policy Simulator não estiver disponível no Learner Lab, faça o exercício mental de prever o resultado de cada combinação usuário × ação — o objetivo é treinar o raciocínio.

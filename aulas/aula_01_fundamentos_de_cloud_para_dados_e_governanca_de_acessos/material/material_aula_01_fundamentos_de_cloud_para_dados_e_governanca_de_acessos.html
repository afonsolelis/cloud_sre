Fundamentos de Cloud para Dados e Governança de Acessos

Este material aprofunda os conceitos apresentados nos slides da Aula 01. Cobre Cloud Computing aplicada à engenharia de dados, os modelos arquiteturais de Data Lake, Data Warehouse e Data Mesh, os fundamentos do AWS IAM e o passo a passo completo do laboratório de configuração de conta e permissões.

Módulo 1 — Cloud Computing para Dados

O que é Cloud Computing?

Cloud Computing é o modelo de entrega de recursos de TI — servidores, armazenamento, bancos de dados, redes, software e inteligência — pela internet, com cobrança baseada no consumo real. Em vez de comprar e manter infraestrutura própria, você aluga capacidade de um provedor e paga apenas pelo que usa.

Para a engenharia de dados, esse modelo resolve dois problemas históricos: o custo fixo de manter servidores ociosos em períodos de baixa demanda e a impossibilidade de crescer rapidamente quando o volume de dados aumenta de forma inesperada.

Modelos de serviço

IaaS (Infrastructure as a Service): o provedor entrega computação, armazenamento e rede virtualizados. Você é responsável pelo sistema operacional, middleware e aplicação. Exemplos AWS: EC2 (servidores virtuais), EBS (discos), VPC (rede privada). Usado quando você precisa de controle granular sobre a pilha.

PaaS (Platform as a Service): o provedor gerencia o hardware e o runtime. Você entrega a aplicação e os dados. Exemplos AWS: RDS (banco relacional gerenciado), Elastic Beanstalk. Usado quando você quer focar no código sem operar sistema operacional.

SaaS (Software as a Service): o provedor entrega a funcionalidade completa. Você consome via interface ou API. Exemplos AWS: QuickSight (BI), Athena (query serverless), Rekognition (visão computacional). Usado quando você quer resultado imediato sem escrever ou manter código de infraestrutura.

Por que Cloud é fundamental para dados em escala

Pipelines de dados modernos têm cargas imprevisíveis: uma campanha de marketing pode multiplicar o volume de eventos por 10x em minutos. Na infraestrutura on-premises, você dimensiona para o pico e mantém capacidade ociosa o restante do tempo. Na nuvem, você escala para o pico e retorna ao estado base automaticamente, pagando apenas pelo tempo utilizado.

Além da elasticidade, a cloud elimina o overhead operacional de instalar, atualizar e monitorar serviços complexos como bancos de dados distribuídos, clusters Spark e sistemas de mensageria. Isso libera a equipe de engenharia para focar no que gera valor: transformar dados em produto.

Modelos Arquiteturais: Data Lake, Data Warehouse e Data Mesh

Data Lake

Um Data Lake é um repositório centralizado que armazena dados em seu formato original — estruturados, semiestruturados e não estruturados — sem transformação prévia. O conceito central é schema on read: o esquema é aplicado no momento da leitura, não no momento da gravação, o que confere máxima flexibilidade para exploração e experimentação.

Na AWS, o componente principal de um Data Lake é o Amazon S3, que oferece durabilidade de 99,999999999% (11 noves) e custo de armazenamento muito baixo. Sobre o S3, serviços como AWS Glue (catálogo e ETL) e Amazon Athena (consultas SQL serverless) formam a base analítica do Lake.

A organização canônica do Data Lake usa três camadas:

  • Raw / Bronze: dados exatamente como chegaram da fonte, sem nenhuma transformação. Servem como fonte da verdade e permitem reprocessamento completo em caso de erro em estágios posteriores.
  • Trusted / Silver: dados limpos, validados e padronizados. Erros de tipo, valores nulos e duplicatas foram tratados. Prontos para consumo por times técnicos.
  • Refined / Gold: dados agregados e modelados para consumo direto por BI, dashboards e modelos de machine learning. Geralmente organizados por domínio de negócio.

Data Warehouse

Um Data Warehouse é otimizado para consultas analíticas de alta performance sobre dados estruturados e transformados. O modelo de dados é definido na gravação (schema on write), o que permite ao banco de dados otimizar o layout físico dos dados para velocidade de leitura.

Na AWS, o Amazon Redshift é o serviço de Data Warehouse gerenciado. Usa processamento massivamente paralelo (MPP), armazenamento colunar e compressão para entregar consultas rápidas mesmo em tabelas com bilhões de linhas. O Redshift Spectrum permite estender queries para dados diretamente no S3, integrando Data Warehouse e Data Lake.

O Data Warehouse é a escolha certa quando você precisa de latência baixa em consultas complexas, relatórios regulares e dashboards de BI com SLA de performance definido.

Data Mesh

Data Mesh é uma abordagem arquitetural e organizacional que trata dados como produto e distribui a responsabilidade pelos dados para os domínios de negócio que os originam. Em vez de um time central de dados ser o guardião de todos os dados da organização, cada domínio (ex.: time de vendas, time de logística) publica seus próprios produtos de dados com contratos de interface bem definidos.

O Data Mesh resolve o gargalo de centralização que aparece em organizações grandes, onde um único time de plataforma de dados não consegue atender todos os domínios de negócio com a velocidade necessária. A governança é federada: cada domínio tem autonomia, mas segue padrões centrais de qualidade, segurança e interoperabilidade.

Na AWS, o Data Mesh é tipicamente implementado com Lake Formation (controle de acesso fine-grained), Event Bridge (integração entre domínios) e Glue Data Catalog (catálogo federado).

AWS como Plataforma de Dados

Infraestrutura global

A AWS opera em Regiões distribuídas pelo mundo, cada uma composta por múltiplas Zonas de Disponibilidade (AZs). Cada AZ é um ou mais data centers com energia, rede e resfriamento independentes, fisicamente separados por distância suficiente para resistir a falhas localizadas (enchentes, incêndios), mas próximos o suficiente para comunicação de baixa latência.

Para a disciplina, usaremos a região us-east-1 (N. Virginia), que tem cobertura mais ampla de serviços e geralmente custo menor. É importante fixar a região antes de criar qualquer recurso: recursos criados em regiões diferentes não são visíveis entre si por padrão.

Modelo de responsabilidade compartilhada

A AWS é responsável pela segurança da nuvem: hardware físico, rede global, hypervisors, data centers. Você é responsável pela segurança na nuvem: quem acessa seus dados, como eles são criptografados, quais portas estão abertas, quais permissões IAM estão configuradas.

Compreender esse modelo é essencial: uma falha de segurança causada por configuração incorreta de IAM ou por um bucket S3 público é sua responsabilidade, não da AWS.

Mapa de serviços da disciplina

Ao longo das 8 aulas, trabalharemos com o seguinte conjunto de serviços:

  • Armazenamento: S3 (objeto), EBS (bloco), EFS (arquivo).
  • Banco de dados: RDS (relacional), DynamoDB (NoSQL), Redshift (DW).
  • Processamento: AWS Glue (ETL serverless), EMR (Spark gerenciado).
  • Streaming: Kinesis Data Streams, Kinesis Data Firehose.
  • Orquestração: Step Functions, MWAA (Airflow gerenciado).
  • Governança: IAM, Lake Formation, CloudTrail.
  • Observabilidade: CloudWatch Logs, CloudWatch Metrics, CloudWatch Alarms.
  • Consulta: Amazon Athena.

Módulo 2 — Governança de Acessos com AWS IAM

Por que o controle de acesso é crítico

Dados são ativo estratégico. Sem controle de acesso adequado, dados sensíveis de clientes ficam expostos, pipelines podem ser modificados por usuários não autorizados e logs de auditoria não existem para rastrear o que aconteceu.

Os erros de configuração IAM são, consistentemente, uma das principais causas de incidentes de segurança em cloud. Os mais comuns são: uso da conta root para operações do dia a dia, chaves de acesso (access keys) armazenadas em código-fonte ou variáveis de ambiente não protegidas, políticas com wildcard (*) em recursos e ações, e buckets S3 configurados como públicos por engano.

AWS IAM: conceitos fundamentais

O AWS Identity and Access Management (IAM) é o serviço que controla quem pode chamar quais APIs da AWS e em quais condições. Cada chamada de API na AWS passa pelo IAM, que avalia as políticas associadas à identidade que faz a chamada e decide se concede ou nega o acesso.

Usuários (IAM Users)

Representam uma pessoa ou sistema com credenciais de longo prazo (senha para console ou access key/secret key para API). Devem ser usados para acesso humano ao console AWS. A conta root nunca deve ser usada para operações do dia a dia — ela tem acesso irrestrito a tudo e suas ações não podem ser limitadas por políticas IAM.

Grupos (IAM Groups)

São coleções de usuários. Em vez de atribuir políticas individualmente a cada usuário, você atribui políticas ao grupo e adiciona usuários ao grupo. Quando o perfil de um usuário muda, basta movê-lo para outro grupo — as permissões mudam automaticamente.

Exemplo de estrutura de grupos para uma equipe de dados:

  • DataEngineers: acesso de leitura/escrita em S3, Glue, Kinesis, CloudWatch.
  • DataScientists: leitura em S3 Silver/Gold, Athena, SageMaker.
  • DataAnalysts: somente leitura em Athena e Redshift.
  • DataAdmins: acesso total ao IAM e ao gerenciamento da plataforma.

Roles (IAM Roles)

Uma Role é uma identidade que pode ser assumida temporariamente por um usuário, um serviço AWS, ou uma conta externa. Ao contrário de usuários, roles não têm credenciais de longo prazo — as credenciais são geradas temporariamente no momento em que a role é assumida e expiram automaticamente.

Roles são o mecanismo correto para conceder permissões a serviços AWS. Por exemplo: um job do AWS Glue precisa ler de um bucket S3 e gravar logs no CloudWatch. Em vez de criar um usuário e colocar as credenciais no código do job, você cria uma Glue Job Role com as permissões necessárias e configura o Glue para assumir essa role automaticamente.

Políticas (IAM Policies)

Políticas são documentos JSON que definem o que é permitido ou negado. Uma política é composta por um ou mais statements, cada um com:

  • Effect: Allow ou Deny.
  • Action: lista de operações de API (ex.: s3:GetObject, glue:StartJobRun).
  • Resource: ARN do recurso ao qual a ação se aplica.
  • Condition: condições opcionais (ex.: somente se MFA estiver ativo, somente de determinado IP).

Regras de avaliação:

  • Deny explícito sempre prevalece sobre qualquer Allow.
  • Sem Allow = negado por padrão (implicit deny).
  • Todas as políticas associadas à identidade são avaliadas em conjunto.

Princípio do menor privilégio

O princípio do menor privilégio (Least Privilege) é a pedra angular da segurança em cloud: conceda apenas as permissões estritamente necessárias para que uma identidade realize sua função, e nada além disso.

Na prática, isso significa:

  • Nunca usar "Action": "*" ou "Resource": "*" em políticas de produção sem justificativa explícita.
  • Começar com permissões mínimas e adicionar conforme necessário, em vez de começar com permissão total e tentar restringir depois.
  • Revisar periodicamente permissões ativas e revogar as que não são mais usadas. O AWS IAM Access Analyzer ajuda a identificar permissões excessivas.
  • Separar ambientes (dev, staging, prod) em contas AWS diferentes, nunca em namespaces dentro da mesma conta.

Mapeando Permissões para o Data Lake

Antes de criar qualquer recurso de IAM, é importante mapear quais perfis existem na equipe de dados e quais permissões cada um precisa. Esse exercício previne retrabalho e garante que o modelo de segurança seja coerente com o modelo de negócio.

Permissões por perfil

Engenheiro de Dados

  • S3: leitura e escrita em todas as camadas (Raw, Trusted, Refined).
  • Glue: criar crawlers, jobs, catálogos.
  • Kinesis: produzir e consumir streams e firehose.
  • Step Functions: criar e executar state machines.
  • CloudWatch: visualizar logs e métricas.
  • IAM: sem permissão (administração de acesso é separada).

Cientista de Dados

  • S3: somente leitura nas camadas Trusted e Refined.
  • Athena: executar queries e criar workgroups.
  • Glue Data Catalog: leitura do catálogo.
  • SageMaker: criar e executar notebooks e treinamentos.

Analista de BI

  • Athena: executar queries (somente leitura).
  • Redshift: conexão somente leitura em schemas de reporting.
  • QuickSight: acesso a dashboards publicados.
  • S3: somente leitura em Refined.

Roles para serviços AWS

  • GlueJobRole: S3 (leitura/escrita), Glue (execução), CloudWatch (logs).
  • LambdaRole: S3 (leitura), DynamoDB (leitura/escrita), CloudWatch (logs).
  • StepFunctionsRole: invocar Lambda, iniciar Glue Jobs, publicar no CloudWatch.
  • KinesisFirehoseRole: S3 (escrita), CloudWatch (logs).

Módulo 3 — Laboratório: Configuração Segura da Conta AWS

A prática ocorre entre 21h00 e 22h00 utilizando o ambiente AWS Student. Siga os passos abaixo na ordem indicada. Cada passo tem uma validação — não avance sem confirmar que o passo anterior funcionou.

Passo 1 — Acessar o ambiente AWS Student

  1. Acesse o portal da sua instituição e obtenha o link de acesso ao AWS Academy ou AWS Student Lab.
  2. Clique em "Start Lab" e aguarde o status ficar verde (pode levar 1-2 minutos).
  3. Clique em "AWS" para abrir o console com as credenciais temporárias do lab.
  4. No canto superior direito do console, confirme que a região selecionada é us-east-1 (N. Virginia). Se não estiver, altere agora.
  5. Validação: o console deve aparecer com uma barra de serviços visível e a região correta no header.

Passo 2 — Explorar o IAM Dashboard

  1. No campo de busca do console, digite IAM e clique no serviço.
  2. O IAM Dashboard exibe o resumo de segurança da conta. Observe:
    • Status do MFA na conta root (no lab gerenciado pela AWS Academy, isso já está configurado).
    • Número de usuários, grupos, roles e políticas existentes.
  3. No menu lateral, clique em Users. Anote os usuários pré-existentes, se houver.
  4. Validação: você consegue navegar pelo IAM sem erros de permissão.

Passo 3 — Criar grupo DataEngineers

  1. No menu lateral do IAM, clique em User groups.
  2. Clique em Create group.
  3. No campo "User group name", insira: DataEngineers.
  4. Na seção "Attach permissions policies", busque e selecione:
    • AmazonS3FullAccess
    • AWSGlueConsoleFullAccess
    • CloudWatchLogsFullAccess
  5. Clique em Create group.
  6. Validação: o grupo DataEngineers aparece na lista com 3 políticas associadas.

Passo 4 — Criar usuário IAM

  1. No menu lateral, clique em Users e depois em Create user.
  2. Nome do usuário: engenheiro-dados-lab.
  3. Marque a opção "Provide user access to the AWS Management Console".
  4. Selecione "I want to create an IAM user".
  5. Escolha uma senha e marque a opção de troca obrigatória no primeiro login.
  6. Clique em Next.
  7. Em "Set permissions", selecione "Add user to group" e marque o grupo DataEngineers.
  8. Clique em Next e depois em Create user.
  9. Na tela de confirmação, anote o link de login e a senha temporária.
  10. Validação: o usuário aparece na lista com status "Active".

Passo 5 — Criar a Role GlueJobRole

  1. No menu lateral do IAM, clique em Roles e depois em Create role.
  2. Em "Trusted entity type", selecione "AWS service".
  3. Em "Use case", escolha "Glue" na lista de serviços.
  4. Clique em Next.
  5. Busque e selecione as seguintes políticas:
    • AmazonS3FullAccess
    • AWSGlueServiceRole
    • CloudWatchLogsFullAccess
  6. Clique em Next.
  7. No campo "Role name", insira: GlueJobRole.
  8. Revise o resumo e clique em Create role.
  9. Validação: a role GlueJobRole aparece na lista. Clique nela e confirme que as 3 políticas estão listadas na aba "Permissions".

Passo 6 — Verificar as permissões com o Policy Simulator

  1. No menu lateral do IAM, role até o final e clique em "Policy Simulator".
  2. Em "Users, Groups, and Roles", selecione o usuário engenheiro-dados-lab.
  3. Em "Select service", escolha S3.
  4. Em "Select actions", escolha GetObject e PutObject.
  5. Clique em Run Simulation.
  6. Validação: ambas as ações devem retornar "Allowed".
  7. Repita o teste escolhendo a ação CreateUser do serviço IAM. O resultado esperado é "Denied" — confirmando que o princípio do menor privilégio está sendo respeitado.

Exercícios de fixação

  1. Comparativo de arquiteturas: descreva um cenário de negócio real (ex.: e-commerce, banco, plataforma de streaming) e justifique se a melhor escolha inicial seria um Data Lake, um Data Warehouse ou uma combinação dos dois. Explique o raciocínio considerando volume, variedade e velocidade dos dados.
  2. Design de política IAM: escreva (em português estruturado, não necessariamente em JSON) uma política IAM para um cientista de dados que precisa: ler arquivos Parquet de um bucket S3 específico chamado meu-data-lake-gold, executar queries no Athena, mas não tem permissão para escrever no S3 nem para acessar DynamoDB. Liste Effect, Action e Resource para cada permissão necessária.
  3. Diagnóstico de segurança: uma empresa tem a seguinte configuração: todos os engenheiros de dados usam a mesma access key compartilhada, o bucket S3 com dados de produção está configurado como público para facilitar o acesso, e não há nenhuma role configurada para os jobs do Glue. Liste pelo menos cinco riscos concretos dessa configuração e proponha uma solução para cada um.
  4. Diferença entre usuário e role: explique com suas palavras por que é uma má prática criar um usuário IAM e colocar suas credenciais dentro de um job do AWS Glue, e por que usar uma role resolve esse problema do ponto de vista de segurança e operação.
  5. Modelo de responsabilidade: um bucket S3 de uma empresa foi acessado indevidamente e dados de clientes foram expostos. A empresa afirma que a AWS deveria ter impedido o acesso. Com base no modelo de responsabilidade compartilhada, quem é responsável por esse incidente e por quê?

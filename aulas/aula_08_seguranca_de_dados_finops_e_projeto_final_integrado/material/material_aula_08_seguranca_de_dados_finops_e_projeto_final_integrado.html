Segurança de Dados, FinOps e Projeto Final Integrado

Este material aprofunda os conceitos apresentados nos slides da Aula 08 e fecha a disciplina com uma visão integrada. Cobre o AWS Lake Formation para governança de dados com controle de acesso fino, as práticas de FinOps para redução de custo em workloads analíticos, as melhores práticas de segurança para o pipeline completo, e a revisão do fluxo integrado end-to-end construído ao longo das 8 aulas.

Módulo 1 — AWS Lake Formation: Governança de Dados em Escala

Por que o IAM sozinho não é suficiente para governança de dados

O AWS IAM controla o acesso a APIs da AWS: quem pode chamar o s3:GetObject, quem pode criar um Glue Crawler, quem pode executar uma query no Athena. Mas o IAM não entende dados — ele não sabe que dentro de um bucket S3 existem dados sensíveis de clientes ao lado de dados públicos de catálogo de produtos.

O resultado prático é que, com apenas IAM, você precisaria criar um bucket separado para cada nível de sensibilidade de dado, ou dar acesso total ao bucket inteiro a qualquer usuário que precise de qualquer dado dentro dele. Nenhuma das duas opções escala bem em um Data Lake com dezenas de fontes, centenas de tabelas e múltiplos perfis de acesso.

O AWS Lake Formation resolve isso adicionando uma camada de controle de acesso no nível de tabela, coluna e linha sobre o Glue Data Catalog. Em vez de gerenciar permissões de bucket S3 no IAM, você gerencia permissões de database/tabela/coluna no Lake Formation, e ele traduz isso para permissões no nível de objeto S3 automaticamente.

Conceitos do Lake Formation

Data Administrators: usuários com permissão de administrar o Lake Formation. Eles concedem permissões a outros usuários e definem quais buckets S3 estão registrados como parte do Lake.

Registered locations: prefixos S3 que estão sob governança do Lake Formation. Após o registro, o acesso a esses prefixos é controlado pelo Lake Formation em vez de pelo IAM diretamente.

Data permissions: permissões no nível de database, tabela, coluna ou linha. Exemplos:

  • Analista de BI: SELECT nas colunas não-sensíveis da tabela trusted_clientes, sem acesso às colunas cpf e data_nascimento.
  • Cientista de dados: SELECT em todas as tabelas do database trusted, mas sem acesso ao database raw.
  • Engenheiro de dados: CREATE TABLE, ALTER TABLE, INSERT, SELECT em todos os databases do lake.

Column-level security (CLS): o Lake Formation pode mascarar ou excluir colunas inteiras para determinados usuários. Um analista que consulta a tabela de clientes via Athena simplesmente não verá as colunas para as quais não tem permissão — elas não aparecem no resultado da query.

Row-level security (RLS): via Lake Formation filters, é possível restringir quais linhas um usuário pode ver. Por exemplo, um analista regional só pode ver registros onde regiao = 'Sul'.

Data Catalog e descoberta de dados sensíveis

O AWS Glue Sensitive Data Detection (baseado em Macie) pode escanear tabelas no Catalog para identificar automaticamente dados pessoais (PII): números de CPF, cartões de crédito, emails, telefones. Os resultados são registrados e podem ser usados para aplicar automaticamente as políticas de CLS/RLS corretas.

Módulo 2 — Melhores Práticas de Segurança para o Pipeline Completo

Criptografia em repouso e em trânsito

Em repouso (at rest): todo dado no S3 deve ser criptografado. Use SSE-S3 (chave gerenciada pela AWS, sem custo extra) para dados internos de baixa sensibilidade, ou SSE-KMS (AWS Key Management Service) para dados sensíveis onde você precisa de controle sobre a chave de criptografia, auditoria de uso da chave via CloudTrail, e possibilidade de revogar acesso invalidando a chave.

Em trânsito (in transit): toda comunicação com serviços AWS usa HTTPS/TLS por padrão. Nunca use endpoints HTTP não criptografados. Para conexões JDBC de Glue para RDS, certifique-se de que a conexão está configurada com SSL.

VPC e isolamento de rede

Para pipelines de produção, execute Glue Jobs e outras cargas de trabalho dentro de uma VPC (Virtual Private Cloud) em vez de na rede pública da AWS. Isso permite:

  • Comunicação com RDS e outros bancos de dados sem expô-los à internet.
  • Controle de saída via Security Groups e NACLs (Network Access Control Lists).
  • Uso de VPC Endpoints para S3 e outros serviços AWS, eliminando tráfego pela internet mesmo para serviços públicos.

CloudTrail: auditoria de todas as chamadas de API

O AWS CloudTrail registra todas as chamadas de API feitas na conta AWS: quem fez, de onde (IP), quando, qual ação, em qual recurso, e qual foi o resultado. Para governança de dados, o CloudTrail permite auditar:

  • Quem acessou quais objetos no S3 (via S3 Data Events).
  • Quais queries foram executadas no Athena.
  • Quem criou ou modificou tabelas no Glue Catalog.
  • Quais mudanças foram feitas em políticas IAM e no Lake Formation.

Configure o CloudTrail para enviar logs para um bucket S3 separado e de longa retenção, e ative a validação de integridade dos logs para garantir que não foram modificados.

Gestão de segredos com AWS Secrets Manager

Senhas de banco de dados, chaves de API e outros segredos nunca devem ser hardcoded em código ou armazenados em variáveis de ambiente não criptografadas. O AWS Secrets Manager armazena, rotaciona automaticamente e controla o acesso a segredos. Em Glue Jobs, use o SDK do Secrets Manager para buscar a senha do banco em tempo de execução: 

import boto3
import json

def get_secret(secret_name):
    client = boto3.client('secretsmanager', region_name='us-east-1')
    response = client.get_secret_value(SecretId=secret_name)
    return json.loads(response['SecretString'])

rds_creds = get_secret('prod/rds/datalake')

Módulo 3 — FinOps para Workloads Analíticos

O que é FinOps

FinOps (Financial Operations) é a prática de gestão financeira em cloud que une equipes de engenharia, finanças e negócio para otimizar custos sem sacrificar performance ou agilidade. O princípio central é: quem usa os recursos é responsável por gerenciar seus custos — não apenas o time de infra ou financeiro.

Principais alavancas de custo em workloads analíticos na AWS

Amazon S3

  • Use S3 Lifecycle policies para migrar dados para classes mais baratas (Standard-IA, Glacier) após períodos de inatividade. Dados Raw com mais de 90 dias raramente são relidos.
  • Ative S3 Intelligent-Tiering para dados com padrões de acesso imprevisíveis — o S3 move automaticamente os objetos entre tiers sem custo de monitoramento adicional acima de 128 KB.
  • Monitore e delete versões antigas de objetos em buckets com versionamento habilitado. Versões antigas acumulam custo sem valor se não houver processo de limpeza.

Amazon Athena

  • Converta dados de CSV/JSON para Parquet com compressão Snappy. Uma tabela de 1 TB em CSV pode ser reduzida para 100-200 GB em Parquet, reduzindo o custo por query em 5-10x.
  • Adicione particionamento às tabelas mais consultadas. Filtros por partição eliminam o scan de dados irrelevantes.
  • Configure Athena Workgroups com limites de bytes por query para evitar queries acidentalmente custosas de analistas.
  • Use o Athena Query Results Cache para queries repetitivas — resultados de queries idênticas são reutilizados por até 7 dias sem custo adicional.

AWS Glue

  • Dimensione o número de DPUs corretamente. Muitos jobs são configurados com 10 DPUs quando 2-3 seriam suficientes. Use o Glue Job Metrics para identificar jobs com baixa utilização de workers.
  • Prefira Glue Flex execution tier (40% mais barato que Standard) para jobs não urgentes que podem tolerar latência variável.
  • Configure o número máximo de retries adequado — um job que falha 3 vezes cobra 3x o custo de execução.

Amazon Redshift

  • Use o Redshift Serverless para workloads intermitentes — você paga por RPU (Redshift Processing Unit) segundo enquanto consultas estão rodando, sem custo em períodos ociosos.
  • Configure o Redshift Auto Pause para clusters provisionados — pausa automática após X minutos de inatividade e retoma na próxima query.
  • Use Redshift Spectrum para dados históricos que são consultados com baixa frequência, mantendo apenas dados quentes no cluster.
  • Execute VACUUM regularmente para reclamar espaço de linhas deletadas e manter a performance de scan.

Tags de custo: visibilidade por projeto e equipe

Todo recurso AWS deve ser tagueado com metadados de negócio: projeto, equipe, ambiente (dev/prod), e centro de custo. As tags são propagadas automaticamente para o AWS Cost Explorer e AWS Budgets, permitindo visualizar e alertar sobre custos por dimensão de negócio, não apenas por serviço.

Configure AWS Budgets com alertas quando o custo mensal de uma tag específica atingir 80% e 100% do orçamento definido. Isso permite intervenção proativa antes que o orçamento seja estourado.

Módulo 4 — Revisão: O Pipeline Integrado da Disciplina

Fluxo completo end-to-end

Ao longo das 8 aulas, construímos os blocos de um pipeline analítico completo na AWS. O fluxo integrado funciona da seguinte forma:

  1. Aula 01 — Fundação de segurança: criação das roles IAM (GlueJobRole, etc.), grupos de usuários (DataEngineers, DataScientists), e políticas de menor privilégio. Todo o pipeline roda com as identidades corretas definidas nesta aula.
  2. Aula 02 — Data Lake no S3: bucket criado com estrutura de prefixos Raw/Trusted/Refined, versionamento e criptografia habilitados. Primeira exploração com Athena.
  3. Aula 03 — Fontes de dados: dados de RDS extraídos via CDC com DMS para S3 Raw. Tabela DynamoDB exportada via PITR para S3 Raw. Ambas as fontes alimentam a camada Raw do lake.
  4. Aula 04 — Streaming: eventos em tempo real chegando via Kinesis Firehose, entregues ao S3 Raw com particionamento dinâmico por data e tipo.
  5. Aula 05 — ETL e catálogo: Glue Crawlers descobrem o schema dos dados Raw. Glue Jobs transformam Raw → Trusted (limpeza, tipagem, Parquet). Glue Catalog mantém o metastore central.
  6. Aula 06 — Data Warehouse: dados da camada Refined são carregados no Redshift via COPY command. Star schema criado para análise de vendas. Redshift Spectrum acessa dados históricos no S3 sem mover.
  7. Aula 07 — Confiabilidade: DAG no MWAA orquestra a sequência Crawler → ETL → Crawler → Carga Redshift com retries configurados. CloudWatch Alarms alertam sobre falhas. SLOs definidos para freshness e completeness.
  8. Aula 08 — Governança e custo: Lake Formation controla o acesso fino por tabela e coluna. Lifecycle policies no S3 otimizam custo. CloudTrail audita todos os acessos. Tags aplicadas em todos os recursos.

Arquiteturas Lambda e Kappa aplicadas ao pipeline da disciplina

O pipeline construído ao longo da disciplina implementa uma arquitetura Lambda: existe um caminho batch (CDC do RDS via DMS, processado em Glue Jobs que rodam periodicamente) e um caminho streaming (Kinesis Firehose entregando eventos em near real-time ao S3). Os consumidores finais (Redshift, Athena) combinam as visões das duas fontes.

Uma evolução para Kappa seria migrar também a extração do RDS para um stream contínuo (Kinesis + DMS em modo CDC streaming), eliminando o caminho batch e unificando toda a ingestão em um único paradigma de eventos. O trade-off é maior complexidade de reprocessamento histórico e necessidade de maior retenção no stream.

Módulo 5 — Laboratório: Governança com Lake Formation e Revisão Final

A prática utiliza o ambiente AWS Student. O foco é explorar as configurações do Lake Formation e revisar o pipeline construído.

Passo 1 — Configurar o Lake Formation

  1. No console AWS, acesse AWS Lake Formation.
  2. Se for o primeiro acesso, clique em Get started.
  3. Em Administrators, verifique se seu usuário está listado. Se não estiver, clique em Add myself.
  4. Em Data lake settings, desmarque a opção "Use only IAM access control for new databases and tables". Isso ativa o controle pelo Lake Formation.
  5. Clique em Save.
  6. Validação: a configuração do Lake Formation está ativa.

Passo 2 — Registrar a localização S3

  1. No menu lateral do Lake Formation, clique em Data lake locations.
  2. Clique em Register location.
  3. Em Amazon S3 location, insira o bucket do Data Lake: s3://seu-bucket-datalake/
  4. Em IAM role, selecione a LakeFormationServiceRole (ou crie uma nova com as permissões necessárias).
  5. Clique em Register location.
  6. Validação: o bucket aparece na lista de localizações registradas.

Passo 3 — Conceder permissões granulares de tabela

  1. No Lake Formation, clique em Permissions e depois em Data lake permissions.
  2. Clique em Grant.
  3. Em IAM users and roles, selecione o usuário engenheiro-dados-lab criado na Aula 01.
  4. Em LF-Tags or catalog resources, selecione Named data catalog resources.
  5. Database: datalake_lab. Table: All tables.
  6. Em Table permissions, marque: SELECT, DESCRIBE.
  7. Clique em Grant.
  8. Validação: a permissão aparece na lista com o usuário, database e tipo de acesso corretos.

Passo 4 — Cheklist de revisão do pipeline completo

  1. Verifique que o bucket S3 está com criptografia SSE-S3 habilitada (Aula 02).
  2. Confirme que a GlueJobRole tem as permissões mínimas necessárias e não usa "Resource": "*" sem necessidade (Aula 01).
  3. Verifique que a camada Raw tem prefixos com o formato Hive (chave=valor) para particionamento automático (Aula 02).
  4. Confirme que os dados da camada Trusted estão em formato Parquet com compressão Snappy (Aula 05).
  5. Verifique que o CloudWatch Alarm de falha do Glue Job está configurado e com estado OK (Aula 07).
  6. Confirme que os recursos estão tagueados com projeto=datalake-mba e ambiente=lab.
  7. Ao final do lab, delete os recursos criados para evitar cobranças: Glue Jobs, Crawlers, tabelas do Catalog, DynamoDB table, Firehose stream. O bucket S3 pode ser mantido se o lab for encerrado com o ambiente.
  8. Validação final: você consegue descrever cada componente do pipeline e sua relação com os outros sem consultar o material.

Exercícios de fixação

  1. Governança com Lake Formation: uma empresa de saúde armazena dados de pacientes no Data Lake. A tabela trusted_pacientes tem as colunas: paciente_id, nome, cpf, data_nascimento, diagnostico, medico_responsavel. Defina uma matriz de permissões para três perfis — médico, pesquisador e auditor —, especificando para cada perfil quais colunas podem ser vistas, com ou sem máscara, e justificando com base em requisitos de LGPD.
  2. Otimização FinOps: a conta AWS do time de dados está custando R$ 15.000/mês. A análise do Cost Explorer mostra: 40% em Athena (queries sem particionamento em CSV), 30% em Glue (jobs com 10 DPUs para datasets de 500 MB), 20% em S3 (todos os dados em Standard incluindo 5 anos de histórico), 10% em outros. Estime o potencial de redução de cada categoria e descreva as ações concretas para implementar cada otimização.
  3. Pipeline end-to-end: descreva o fluxo completo de dados de uma compra realizada em um e-commerce — desde o INSERT no banco de dados de produção até estar disponível em um dashboard de vendas em tempo real e em um relatório histórico mensal. Para cada etapa, identifique o serviço AWS utilizado nesta disciplina, o formato dos dados, a latência esperada, e o mecanismo de recuperação de falhas.
  4. Arquitetura Lambda versus Kappa — decisão final: com base em tudo que foi visto na disciplina, você está projetando a arquitetura de dados para uma empresa de logística com 10 milhões de entregas/mês. Os dados vêm de sistemas OLTP, GPS de veículos (streaming), e APIs de terceiros (batch diário). O time de BI precisa de dashboards diários e o time de operações precisa de alertas em 2 minutos. Escolha entre Lambda e Kappa (ou uma combinação), justifique sua escolha, e esboce a arquitetura com os serviços AWS específicos para cada componente.
  5. Checklist de segurança: antes de fazer o deploy de um pipeline de dados para produção, você precisa garantir que atende a todos os requisitos de segurança. Crie um checklist de pelo menos 10 itens verificáveis (não genéricos — cada item deve especificar o serviço AWS, a configuração exata e como verificar que está correto), cobrindo: IAM, criptografia, rede, auditoria, governança de dados e FinOps.

1. Tese da aula

Ao longo de cinco aulas (Aulas 04 a 07) construímos, em conjunto, uma plataforma de dados que sai do laptop e vai para a nuvem. Hoje cada aluno mostra, por conta própria, que sabe replicar e estender esse caminho. O semestre, que começou como sequência de exercícios assistidos, agora se materializa como repositório versionado. O critério é simples e severo: se um colega abre seu repositório e não consegue subir a stack em 15 minutos a partir do README, a entrega não está pronta.

Tese

O semestre virou repositório. Hoje é dia de provar que a stack vive sem o professor por perto — com arquitetura clara, infra reprodutível, testes verdes e evidências reais commitadas.

A aula é deliberadamente sem conteúdo teórico novo. Não há slide com conceito a memorizar, nem laboratório guiado a copiar. Há um desafio individual, três horas de trabalho assistido em sala e um repositório no GitHub com a sua entrega. Esse é o exame prático da disciplina.

3. Convenção do repositório

Todos os projetos seguem o mesmo padrão de naming usado desde a Aula 04. O nome do repositório é parte da entrega: não use nome criativo, não inverta a ordem, não use acento ou caractere especial. O matching do nome com o padrão simplifica avaliação, busca e comparação entre projetos.

20261sre-<primeironome>-<ultimonome>

3.1 Como criar o repositório

  1. Acesse o repositório base: https://github.com/afonsolelis/20261sre-primeironome-ultimonome.
  2. Clique em Fork para criar uma cópia na sua conta pessoal do GitHub.
  3. Renomeie o fork em Settings → Repository name seguindo o padrão acima, em minúsculas e sem acento. Exemplo: 20261sre-maria-souza.
  4. Garanta que o repositório está público — repositórios privados não são aceitos para avaliação.
  5. Confirme que a branch padrão é main e que branch protection está ativa (mesmo padrão da Aula 05).

3.2 Fluxo de trabalho até a entrega

  • Trabalhe em branches de feature (feat/..., fix/...); não commite direto em main.
  • Abra pull request contra main para cada bloco relevante de mudança.
  • Mantenha CI verde antes de fazer merge: workflow wb.yml e bb.yml herdados das aulas anteriores.
  • Antes da apresentação, garanta que o último commit em main contém a entrega completa.
  • (Opcional, recomendado) Aplique a tag aula-08-entrega no commit final para marcar o ponto de avaliação.
Critério de pronto do naming

Repositório público, nome no padrão 20261sre-<primeironome>-<ultimonome>, branch main contendo a entrega final, README na raiz, último CI verde.

4. Critérios de avaliação (esqueleto)

Os critérios abaixo formam o esqueleto fixo da avaliação. O peso de cada eixo e os cut-offs exatos serão fechados na semana da entrega, em conjunto com o desafio específico. O conjunto, porém, não muda: sem repositório no padrão, sem README, sem evidência, não há nota.

# Critério O que avaliamos no repositório
01 Arquitetura documentada README.md com seção de arquitetura, diagrama (Mermaid ou imagem), descrição dos serviços e justificativa das decisões.
02 Infra reprodutível docker-compose.yml e/ou templates AWS (CloudFormation) com comando único de subida (make up, docker compose up, aws cloudformation deploy).
03 Testes automatizados Suite de testes rodando em CI (GitHub Actions), com badge ou print do último run verde anexado ao README.
04 Evidências de execução Logs, prints, vídeo curto (até 3 min) ou pasta documents/ com saída real do pipeline, métricas observadas e respostas a falhas induzidas.
05 Commits limpos Histórico legível, mensagens descritivas, sem WIP, fix fix ou force push destrutivo em main.

Cada eixo é binário em sua versão mínima (existe / não existe) e graduado em sua versão completa. O peso final de cada eixo será publicado junto com o enunciado do desafio.

5. Como organizar o README de um projeto SRE/Cloud

O README.md é o cartão de visita do seu repositório. Em projetos SRE/Cloud, ele deveria conseguir responder cinco perguntas para um leitor que nunca abriu seu código antes. Estruture seu README respondendo, nesta ordem, às perguntas abaixo.

5.1 O que é este projeto?

Em um parágrafo: qual problema ele resolve, em que domínio, com qual dataset. Evite buzzwords; escreva como quem explica para um analista de negócio.

5.2 Qual é a arquitetura?

Inclua um diagrama (Mermaid no próprio README ou imagem em docs/) mostrando os componentes: fontes de dados, serviços de ingestão, processamento, armazenamento, observabilidade. Explique cada caixa em uma frase. Cite as escolhas de Bass/ATAM que justificam o desenho.

5.3 Como subo isso na minha máquina?

Seção Quick start com pré-requisitos (Docker, AWS CLI, conta AWS Student), comando único de subida (make up ou equivalente) e tempo estimado até up & running. Um colega deve conseguir subir tudo em até 15 minutos sem perguntar nada.

5.4 Como sei que está funcionando?

Seção de validação: como rodar os testes (make test), onde ver métricas em tempo real (dashboard, CloudWatch, Grafana), exemplos de queries de validação, badges de CI. Inclua um screenshot ou snippet de log de uma execução real.

5.5 O que aprendi construindo isso?

Seção curta de trade-offs e decisões: o que você considerou e descartou, onde a arquitetura tem dívida técnica conhecida, o que faria diferente em produção. Essa seção transforma o repositório de exercício em portfólio.

Estruture a raiz do repositório com pastas reconhecíveis: infra/ (IaC), services/ ou src/ (código), tests/ (testes), documents/ (Bass, ATAM, relatórios), .github/workflows/ (CI). Quem abre o repositório enxerga a estrutura antes de ler qualquer linha de prosa.

6. Orientação para a apresentação em sala

No bloco 3 da aula (apresentações curtas), cada aluno tem um intervalo de alguns minutos para mostrar o repositório. Tempo exato por apresentação será calibrado em sala conforme o tamanho da turma. Prepare-se para apresentar nesta ordem:

  1. Abra o GitHub — mostre o repositório, o naming, a árvore de pastas e o último CI verde.
  2. Mostre o diagrama de arquitetura — explique a stack em 60 segundos, citando Bass/ATAM.
  3. Mostre uma evidência de execução — abra um log, um dashboard, um print de run de teste sob carga.
  4. Apresente uma decisão de trade-off — o que você escolheu, o que descartou, por quê.
  5. Encerre com a próxima coisa — o que faria se tivesse mais uma semana.

Não use slide de PowerPoint — o repositório é o slide. Apresente direto do GitHub, do terminal e do console de observabilidade. Apresentações que terminam dentro do tempo, com evidência real na tela, valem mais que apresentações longas com promessas.

9. Atividade prática em sala — entrega final no GitHub

Esta é a aula prática de fechamento da disciplina. O exercício prático em sala é a própria entrega do projeto final. Use as três horas para:

  1. Antes da aula: garanta que seu fork do repositório base está renomeado para 20261sre-<primeironome>-<ultimonome>, público, com a stack das Aulas 04 a 07 commitada em main.
  2. Bloco 1 (briefing): tome nota do desafio específico, das regras e dos critérios. Faça perguntas — esse é o único momento em que o professor responde dúvidas sobre o enunciado.
  3. Bloco 2 (trabalho assistido): abra Codespace ou ambiente local, crie uma branch de feature, implemente a entrega, faça push, abra PR contra main, merge.
  4. Bloco 3 (apresentação): apresente seu repositório seguindo a ordem da seção 6 deste material.
  5. Após a apresentação: aplique a tag aula-08-entrega no commit final; deixe o repositório público para avaliação posterior.

Não há lista de exercícios escritos nesta aula. A entrega é o repositório, e o repositório é o exame. Boa entrega — e obrigado por percorrerem o arco completo do semestre.

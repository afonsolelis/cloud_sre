Projeto final — desafio entregue no GitHub

Este material acompanha a Aula 08 da disciplina Cloud Computing e SRE — Visão Prática com AWS, ofertada no âmbito do MBA em Engenharia de Dados. A presente sessão não contempla a introdução de conteúdo teórico novo: a aula é integralmente dedicada à entrega do projeto final. O artefato de saída da disciplina consolida-se em um repositório público no GitHub, criado do zero, com nome fixo de turma (20261sre-projeto-final), contendo modelagens, implementação aplicando táticas arquiteturais e evidências de execução sobre o dataset Northwind. A entrega é submetida via formulário oficial da disciplina ao final da aula.

Pré-requisitos Familiaridade com a stack construída ao longo das Aulas 04 a 07 (docker-compose Olist da Aula 04, Bass local da Aula 05, migração para AWS com ATAM da Aula 06 e Bass distribuído com estresse sob carga da Aula 07). Conta no GitHub ativa e Codespace ou ambiente Docker local pronto para uso.

1. Tese da aula

Ao longo das Aulas 04 a 07 foi construída, de forma colaborativa, uma plataforma de dados que parte do ambiente local e progride em direção à nuvem. Nesta sessão, cada aluno demonstra, de forma autônoma, sua capacidade de replicar e estender esse percurso. O semestre, iniciado como uma sequência de exercícios assistidos, materializa-se em um repositório versionado. O critério adotado é objetivo e exigente: caso um colega não consiga colocar a stack em execução, a partir do README, em até quinze minutos, a entrega não está concluída.

Tese

Esta sessão verifica a autonomia dos alunos na operação e evolução da arquitetura proposta, demandando arquitetura claramente documentada, infraestrutura reprodutível, testes automatizados aprovados e evidências reais devidamente versionadas.

A aula é, por concepção, isenta de conteúdo teórico novo. Não são previstos slides com conceitos a memorizar nem laboratórios guiados a reproduzir. Propõem-se um desafio individual, três horas de trabalho assistido em sala e um repositório no GitHub contendo a entrega correspondente. Este é o exame prático da disciplina.

3. Convenção do repositório

Para esta entrega final, o nome do repositório é fixo e idêntico para toda a turma. Não há mais variação por aluno, tampouco fork de um repositório base do curso: cada aluno cria um repositório novo, do zero, na própria conta do GitHub, e adota a nomenclatura padrão da turma 2026/1. Essa padronização tem propósito operacional: facilita a correção da disciplina, permite a busca sistemática dos repositórios e elimina ambiguidades de identificação.

20261sre-projeto-final

O nome adota o prefixo 20261sre (turma 2026/1 de SRE) seguido do sufixo projeto-final. Toda a turma utiliza exatamente esse nome — em minúsculas, sem acentos e sem caracteres especiais. A identificação do aluno é feita pelo proprietário do repositório (owner), e não pelo nome do projeto.

3.1 Procedimento de criação do repositório

  1. Acessar o GitHub autenticado na conta pessoal e clicar em New repository, ou utilizar o GitHub CLI: gh repo create 20261sre-projeto-final --public --add-readme.
  2. Definir o nome exatamente como 20261sre-projeto-final, sem prefixos, sufixos ou abreviações adicionais.
  3. Configurar a visibilidade como pública; repositórios privados não são aceitos para fins de avaliação.
  4. Inicializar com um README.md mínimo e confirmar que a branch padrão é main.
  5. Ativar a branch protection sobre main, em conformidade com o padrão estabelecido nas aulas anteriores.

3.2 Fluxo de trabalho até a entrega

  • Trabalhar em branches de funcionalidade (feat/..., fix/...), evitando commits diretos em main.
  • Abrir um pull request em direção à main para cada bloco relevante de modificação.
  • Manter a esteira de CI aprovada antes de efetuar o merge: workflows herdados das aulas anteriores.
  • Antes do envio no Forms, assegurar que o último commit em main contemple a entrega completa, com push efetivado.
  • (Opcional, porém recomendado) Aplicar a tag aula-08-entrega no commit final, demarcando o ponto de avaliação.
Critérios de conformidade do repositório

Repositório público, nomeado exatamente como 20261sre-projeto-final, branch main contendo a entrega final, README.md na raiz, modelagens e diagramas presentes e link enviado via Forms da disciplina.

4. Roadmap da entrega — cinco etapas

A entrega da Aula 08 segue cinco etapas sequenciais. As etapas substituem a antiga sequência de fork + renomeação, e refletem o caráter do projeto final como repositório novo de consolidação, e não como derivação de um repositório base. Cada etapa deve estar concluída antes da submissão do link no formulário de entrega.

  1. Criar um novo repositório

    Criar um repositório totalmente novo no GitHub, pela interface web (New repository) ou pelo GitHub CLI (gh repo create). O repositório deve nascer público, com a branch padrão main e um README.md inicial. Não se trata de fork: não há repositório base a partir do qual derivar a entrega.

  2. Nomear como 20261sre-projeto-final

    O nome do repositório é fixo e idêntico para toda a turma. A padronização do nome simplifica a correção pelo professor, permite buscar todos os repositórios da turma com um critério único e elimina a necessidade de manter um mapa de aluno → repositório. A identificação individual é feita pelo owner da conta GitHub.

  3. Criar as modelagens

    Antes de codar, modelar o problema. Espera-se: (i) modelo conceitual, lógico e físico das tabelas do dataset Northwind (entidades, atributos, relacionamentos, tipos e chaves); (ii) diagrama de arquitetura da stack adotada (Mermaid no próprio README ou imagem em docs/), mostrando ingestão, armazenamento (MinIO/S3), processamento e camada analítica (ClickHouse, Streamlit); (iii) registro, no README.md, das decisões e trade-offs considerados — quais alternativas foram descartadas e por quê.

  4. Codar e aplicar as técnicas de táticas arquiteturais

    Implementar a stack aplicando táticas arquiteturais de Bass e considerações ATAM trabalhadas nas Aulas 05 a 07. Em termos práticos: implementar a ingestão dos CSVs do Northwind, as transformações que alimentam o ClickHouse e a camada analítica do Streamlit, identificando, no código e na documentação, quais táticas (disponibilidade, desempenho, modificabilidade, segurança) foram aplicadas em cada componente. Cenários e trade-offs ATAM devem ser referenciados no README.

  5. Finalizar com commit e push, e enviar no Forms

    Concluída a implementação, efetuar o commit final em uma branch de funcionalidade, abrir o pull request contra main, realizar o merge e o push para o repositório remoto. Em seguida, abrir o formulário de entrega da disciplina e submeter o link público do repositório. A entrega só é considerada efetuada após o envio no Forms.

5. Critérios de avaliação (estrutura)

Os critérios apresentados a seguir constituem a estrutura fixa da avaliação. O peso atribuído a cada eixo e os cut-offs precisos serão definidos na semana da entrega, em conjunto com o desafio específico. O conjunto de critérios, no entanto, permanece inalterado: a ausência de repositório em conformidade com o padrão, de README.md ou de evidências de execução impossibilita a atribuição de nota.

# Critério O que é avaliado no repositório
01 Arquitetura documentada README.md contendo seção de arquitetura, diagrama (Mermaid ou imagem), descrição dos serviços e justificativa das decisões adotadas.
02 Infraestrutura reprodutível docker-compose.yml e/ou templates AWS (CloudFormation) acompanhados de comando único de provisionamento (make up, docker compose up, aws cloudformation deploy).
03 Testes automatizados Suíte de testes em execução na esteira de CI (GitHub Actions), com badge ou captura de tela da última execução aprovada anexada ao README.
04 Evidências de execução Logs, capturas de tela, vídeo de curta duração (até 3 minutos) ou diretório documents/ contendo saída real do pipeline, métricas observadas e respostas às falhas induzidas.
05 Histórico de commits consistente Histórico legível, mensagens descritivas, sem ocorrências de WIP, fix fix ou force push destrutivo em main.

Cada eixo apresenta caráter binário em sua versão mínima (presente ou ausente) e admite graduação em sua versão completa. O peso final atribuído a cada eixo será publicado em conjunto com o enunciado do desafio.

6. Organização do README de um projeto SRE/Cloud

O README.md constitui o documento de apresentação do repositório. Em projetos de SRE e Cloud, espera-se que ele seja capaz de responder a cinco perguntas fundamentais para um leitor que nunca tenha acessado o código anteriormente. Recomenda-se estruturar o README de modo a responder, nesta ordem, às perguntas elencadas a seguir.

6.1 Qual é o objeto deste projeto?

Em um parágrafo, identificar o problema abordado, o domínio de aplicação e o dataset utilizado. Recomenda-se evitar buzzwords e adotar um registro acessível, equivalente ao utilizado em comunicações destinadas a um analista de negócio.

6.2 Qual é a arquitetura adotada?

Incluir um diagrama (em Mermaid, no próprio README, ou em imagem armazenada em docs/) que represente os componentes principais: fontes de dados, serviços de ingestão, processamento, armazenamento e observabilidade. Cada elemento deve ser descrito em uma frase, com referência explícita às decisões de Bass e ATAM que sustentam o desenho proposto.

6.3 Como executar o projeto em ambiente local?

Seção de Quick Start contendo pré-requisitos (Docker, AWS CLI, conta AWS Student), comando único de provisionamento (make up ou equivalente) e estimativa de tempo até up & running. Espera-se que um colega seja capaz de executar todo o ambiente em até quinze minutos, sem necessidade de esclarecimentos adicionais.

6.4 Como verificar o funcionamento adequado?

Seção de validação: instruções para execução dos testes (make test), localização das métricas em tempo real (dashboard, CloudWatch, Grafana), exemplos de queries de validação e badges de CI. Recomenda-se incluir uma captura de tela ou trecho de log proveniente de execução real.

6.5 Quais foram os aprendizados obtidos?

Seção sintética de trade-offs e decisões: alternativas consideradas e descartadas, dívida técnica conhecida na arquitetura e ajustes que seriam recomendados em ambiente produtivo. Esta seção converte o repositório de exercício didático em peça de portfólio profissional.

Recomenda-se estruturar a raiz do repositório com diretórios reconhecíveis: infra/ (IaC), services/ ou src/ (código-fonte), tests/ (testes), documents/ (Bass, ATAM, relatórios) e .github/workflows/ (CI). A organização permite que o leitor depreenda a estrutura do projeto antes mesmo de examinar qualquer trecho de prosa.

7. Orientações para a apresentação em sala

Embora a entrega oficial seja feita por meio do formulário, reserva-se uma janela em sala para breves apresentações entre pares. O objetivo é compartilhar decisões de modelagem e táticas aplicadas, e receber comentários do professor e dos colegas. O tempo individual será calibrado em sala, em função do tamanho da turma. Recomenda-se organizar a apresentação na seguinte ordem:

  1. Acessar o GitHub — apresentar o repositório 20261sre-projeto-final, a árvore de diretórios e a última execução aprovada na esteira de CI.
  2. Apresentar as modelagens — mostrar o modelo conceitual/lógico/físico do Northwind e o diagrama de arquitetura da stack.
  3. Demonstrar uma evidência de execução — exibir um log, um dashboard ou uma captura de tela de execução de teste sob carga.
  4. Discutir uma decisão de tática arquitetural — identificar qual tática Bass foi aplicada e qual cenário ATAM motivou a decisão.
  5. Encerrar com perspectivas futuras — indicar o que seria realizado caso houvesse mais uma semana de trabalho.

Não se recomenda o uso de slides em PowerPoint: o próprio repositório cumpre essa função. A apresentação deve ser conduzida diretamente a partir do GitHub, do terminal e do console de observabilidade. Apresentações que se encerram dentro do tempo previsto, com evidências reais exibidas em tela, são consideradas superiores a exposições longas pautadas em promessas.

8. O desafio específico — Northwind

O desafio da Aula 08 substitui o dataset Olist, utilizado nas Aulas 04 a 07, pelo Northwind clássico: um conjunto relacional de pedidos e itens de pedido, de domínio de distribuição de alimentos e bebidas. A complexidade é equivalente ao Olist, mas o domínio é distinto — o aluno é forçado a repensar a modelagem e os KPIs sem reaproveitar diretamente o que foi feito ao longo do semestre.

A stack MinIO + ClickHouse + ingestor + Streamlit permanece como referência: o núcleo arquitetural é o mesmo, e o desafio incide sobre as extremidades — fontes de dados de entrada, schemas intermediários, modelos analíticos e camada de apresentação.

8.1 Fonte de dados — Google Drive oficial

Os dois arquivos do Northwind estão disponíveis no Google Drive da disciplina. Faça o download diretamente da pasta abaixo:

Pasta oficial no Google Drive — Northwind CSVs

Arquivo Tamanho Papel
northwind_orders.csv ~96 KB Cabeçalho de pedidos (um registro por pedido).
northwind_order_details.csv ~40 KB Linhas de itens de pedido (várias por pedido).

8.2 O que deve ser entregue no repositório

  • Modelagens — modelo conceitual, lógico e físico das duas tabelas, com chaves, tipos e cardinalidades documentados no README ou em docs/.
  • Diagrama de arquitetura — representação da stack (ingestão, MinIO, ClickHouse, Streamlit) em Mermaid ou imagem.
  • Implementação — código de ingestão dos CSVs, transformações para a camada analítica e dashboard Streamlit consumindo o ClickHouse.
  • Táticas aplicadas — identificação explícita das táticas Bass utilizadas e dos cenários ATAM endereçados, em uma seção dedicada do README.
  • Evidências — capturas de tela, logs ou trechos de saída comprovando execução real sobre os dados do Northwind.
Padrão de camadas

Mantenha o mesmo padrão raw / refined adotado no projeto Olist. Documente, no README.md, as decisões de adaptação, os KPIs reformulados e os trade-offs considerados.

Distribuição preliminar do tempo. A divisão respeita o intervalo regulamentar entre 19h00 e 22h00 e será calibrada no início da aula, em função do número de alunos presentes e do número de apresentações previstas.

10. Atividade prática em sala — entrega final no GitHub e envio no Forms

Esta é a aula prática de encerramento da disciplina. A atividade prática conduzida em sala corresponde à própria entrega do projeto final. Recomenda-se aproveitar o período de três horas conforme as orientações a seguir:

  1. Antes da aula: revisar as Aulas 04 a 07 (stack Docker Olist, Bass local, migração AWS com ATAM, Bass distribuído sob estresse), de modo a chegar à aula com o referencial arquitetural fresco. Não é necessário ter um repositório pré-criado — o passo 1 do roadmap é justamente criar o repositório novo.
  2. Bloco 1 (briefing): acompanhar a apresentação do desafio Northwind, do nome fixo do repositório (20261sre-projeto-final) e dos critérios de avaliação. Apresentar dúvidas, uma vez que este é o único momento em que o professor responde a questionamentos sobre o enunciado.
  3. Bloco 2 (trabalho assistido): percorrer as cinco etapas do roadmap descritas na Seção 4 deste material — criar o repositório, nomeá-lo, criar as modelagens, codar aplicando táticas arquiteturais, e preparar o commit final. Pode-se utilizar o Codespace, o ambiente local em Docker, ferramentas de IA para acelerar a modelagem e o AWS Student Lab caso opte-se por demonstrar a versão em nuvem.
  4. Bloco 3 (apresentação e envio): apresentar o repositório de acordo com a ordem estabelecida na Seção 7 deste material e, ao final, abrir o formulário oficial da disciplina e submeter o link público do repositório. A entrega só é considerada efetuada após o envio no Forms.
  5. Após a apresentação: aplicar a tag aula-08-entrega ao commit final e manter o repositório público para fins de avaliação posterior.

Esta aula não contempla uma lista de exercícios escritos. A entrega corresponde ao próprio repositório e o repositório constitui o instrumento de avaliação. Agradecemos aos alunos pela dedicação ao longo do arco completo do semestre e desejamos um bom trabalho na entrega final.

Encerramento da disciplina

Esta é a última aula do curso. O repositório de cada aluno permanece como portfólio público: recomenda-se mantê-lo em evolução após o término da disciplina, utilizá-lo em processos seletivos e demonstrar sua progressão ao longo do tempo. Os fundamentos de Cloud e SRE consolidam-se por meio da prática, e o conteúdo disponível no GitHub constitui a evidência dessa prática.

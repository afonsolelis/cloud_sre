2. O briefing — cliente fictício, escopo apertado

A diretoria de uma empresa imaginária pede um mini-observatório de série temporal sobre o tema escolhido pelo aluno. O pedido vem em uma frase: "Quero entender o passado, decompor o sinal e ver para onde isso vai nos próximos 30 períodos — com um link público que funciona enquanto eu apresento." A frase carrega três entregas distintas e uma restrição operacional.

2.1 As três abas do front

O front, em Streamlit ou Plotly Dash, precisa ter exatamente três abas. Não duas, não quatro. Cada aba responde uma pergunta diferente sobre a série, e a ausência de qualquer uma derruba o critério 4 da avaliação. As três abas, em ordem, são:

  • Histórico — gráfico de linha da série completa, com seletor de janela (7d, 30d, 1y, "all"). No rodapé, estatísticas básicas: mínimo, máximo, média, desvio padrão. Esta aba estabelece a baseline e ancora a leitura das outras duas.
  • Decomposição STL — três sub-gráficos verticais mostrando tendência, sazonalidade e resíduo. A função recomendada é statsmodels.tsa.seasonal.STL, com período inferido do dataset (12 para mensal, 7 para diário, 24 para horário, etc.). STL é mais robusto que clássicos como X-13 porque tolera tendências não-lineares e outliers.
  • Forecast 30 períodos — previsão dos próximos 30 pontos, com intervalo de confiança 95%. Modelos aceitos: ARIMA (via statsmodels), ExponentialSmoothing/Holt-Winters (também statsmodels), ou Prophet. No rodapé, MAPE e RMSE calculados sobre split temporal (últimos 20% como validação).

2.2 O que fica fora do escopo

Tão importante quanto o que entra é o que não entra. O aluno é tentado a sofisticar — a aula penaliza esse instinto. Ficam fora: análise multi-séries, modelos avançados (LSTM, Transformer, N-BEATS), pipeline ETL ao vivo, autenticação de usuário, CDN, domínio próprio, deploy em múltiplas regiões. O CSV vai estático no S3, o front é HTTP simples na porta 8501, o domínio é o IP público do EC2.

Princípio operacional Em três horas, o melhor projeto não é o mais ambicioso — é o que terminou e está demonstrável. Sofisticação custa tempo; tempo é a restrição mais dura da aula.

3. Padrão de naming — tsa-{iniciais}-{slug-tema}

Toda aula cria nome. Um padrão de naming previsível elimina ambiguidade no momento de avaliar e cria um índice navegável da turma. A convenção desta aula é estrita:

  • Prefixo — tsa-. TSA é Time Series Analysis. Não é Transportation Security Administration.
  • Iniciais — três letras minúsculas, sem ponto, sem espaço. Ex.: jps para João Pedro Silva.
  • Slug do tema — minúsculas, hífens entre palavras, sem acento, sem caractere especial.

Nada nessa figura é negociável. Substituir EC2 por Lambda complica o cold start do Streamlit e mata a aba Forecast por timeout. Substituir S3 por upload direto no EC2 mata a portabilidade do dataset. Tirar o EventBridge mata o closed loop. A arquitetura é o teto da entrega — o aluno otimiza dentro dela.

6. SLOs mínimos — o contrato com a demo

O aluno declara três SLOs antes de fazer deploy. Os três viram metric filters em CloudWatch e widgets no dashboard. Sem essa declaração explícita, a demo de 3 minutos não tem critério para ser julgada saudável.

IDSLOSLI medindoMetaOnde vive
SLO-01Disponibilidade durante a demoHTTP 200 a cada 30s no /≥ 95% das checagensCW alarm + EventBridge
SLO-02Latência de carregamentotempo até primeiro byte< 5 s p95CW metric filter
SLO-03Latência do forecasttempo de render da aba 3< 10 s p95CW metric filter

SLO-01 é especial: ele é o gatilho do closed loop. Se o alarm associado dispara, o EventBridge chama o SSM Run Command restart_streamlit e o serviço sobe sem intervenção humana. Esse é o ato de mais alto valor da demo — provar, ao vivo, que o sistema se cura sozinho dentro do error budget definido.

7. Expectativas SRE mínimas — cinco regras duras

O reviewer recusa pull request que falha em qualquer um dos cinco pontos abaixo. As regras existem para impedir entrega que parece pronta mas não é defensável.

  1. IaC obrigatório. Toda criação de recurso passa por aws cloudformation deploy. Console AWS é proibido em sala — nem para criar SG, nem para anexar role, nem para subir log group. CloudTrail mostra quem clicou; o reviewer audita.
  2. Observabilidade desde o deploy. Log group, dashboard e alarm aparecem na primeira versão do template, não em uma atualização posterior. Sem essas três peças, o app não é deployável.
  3. SLOs declarados. Os três SLOs do projeto vivem no README.md e no release_checklist.md, com meta numérica e janela de medição. Aspirações em prosa não contam.
  4. Closed loop ativo. Alarm de health check → EventBridge rule → SSM Run Command que reinicia o serviço. O aluno deve conseguir simular falha (kill do processo) e mostrar o sistema voltando, na frente da turma.
  5. README com runbook ≤ 10 min. Reviewer cronometra. O texto deve permitir, a partir de zero, recriar a stack inteira em menos de dez minutos. Se o reviewer precisar perguntar algo durante a recriação, o critério falha.
Restrição zero Sem credencial permanente em código. Apenas session token do Learner Lab, vivo durante a aula. ~/.aws/credentials fora do git, git secrets rodando no pre-commit hook, e nenhuma ARN vazada no repo público.

13. Sprint 3 · app + observabilidade (21h00 – 21h30)

O terceiro sprint é o de maior densidade. DataDev produz o app. DevOps adiciona log group, metric filters, dashboard, alarm e EventBridge rule. SSM document do runbook é commitado. O template é re-deployado.

13.1 DataDev — app.py com 3 abas

A skill gen_streamlit_tsa_app recebe três parâmetros: as abas desejadas (sempre historico,stl,forecast), o modelo de forecast (sugestão: expsmoothing por estabilidade) e o bucket S3 de origem. Saída: um app.py de cerca de 200 linhas, mais um requirements.txt com pandas, numpy, statsmodels, plotly, streamlit, boto3 e httpx. 

> /gen_streamlit_tsa_app --abas historico,stl,forecast \
                         --modelo expsmoothing \
                         --bucket tsa-{iniciais}-data

O app emite um log JSON estruturado a cada render. Esse log é o input para os metric filters do CloudWatch. Exemplo de log de render da aba forecast: 

{"event":"render","aba":"forecast","duration_ms":4820,"points":120,
 "mape":0.073,"rmse":12.4,"ts":"2026-06-11T20:18:32Z"}

13.2 DevOps — observabilidade no template

A skill gen_observability aceita os SLOs e o nome do runbook. Ela aumenta o template com:

  • Log group /tsa/{iniciais}/app com retention 7 dias.
  • 3 metric filters: http_2xx_rate, load_p95_ms, forecast_p95_ms.
  • Dashboard tsa-{iniciais} com 3 widgets (um por SLI).
  • Alarm SLO-01-availability em http_2xx_rate < 0.95 por 1 datapoint de 60s.
  • EventBridge rule disparada pelo alarm, target = SSM Run Command document restart-streamlit-{iniciais}.
  • SSM Document local em runbooks/restart_streamlit.yaml, registrado no template.

13.3 Deploy do app no EC2 (sem SSH)

O aluno não usa SSH. O deploy do código no EC2 passa por SSM Run Command, que executa um shell script que faz pull do repositório, instala dependências e sobe o Streamlit: 

aws ssm send-command \
  --document-name "AWS-RunShellScript" \
  --targets "Key=tag:Project,Values=tsa-{iniciais}" \
  --parameters 'commands=[
    "cd /opt/app && git pull origin main",
    "pip install -r requirements.txt -q",
    "pkill -f streamlit || true",
    "nohup streamlit run app.py --server.port 8501 > /var/log/streamlit.log 2>&1 &"
  ]'

# Re-deploy do stack com observabilidade
aws cloudformation deploy --stack-name tsa-{iniciais} \
  --template-file iac/cloudformation/app.yaml \
  --parameter-overrides Iniciais={iniciais} DatasetSlug={slug} \
  --capabilities CAPABILITY_IAM
Checkpoint Sprint 3 URL pública abre as 3 abas. Dashboard CW mostra os 3 SLIs preenchidos. Alarm em estado OK. EventBridge rule ligada ao SSM document. SSM document executável manualmente como teste.

15. Erros frequentes e atalhos defensivos

A lista abaixo concentra os defeitos vistos em turmas anteriores ao executar projeto sob restrição de tempo. Antecipar cada um economiza minutos preciosos no Sprint 3 ou 4.

SintomaCausa raiz comumAtalho defensivo
UnauthorizedOperation no meio do Sprint 3 Token expirado. Renovar token antes de cada cloudformation deploy.
URL não abre, EC2 está rodando SG sem inbound 8501. Conferir SecurityGroupIngress no template antes do deploy.
SSM Run Command não responde EC2 sem LabRole anexada. IamInstanceProfile: !Ref LabRoleProfile obrigatório no recurso EC2.
STL falha em runtime statsmodels ausente do requirements.txt. Skill gen_streamlit_tsa_app já injeta — não remover.
Dashboard CW vazio Metric filter criado mas widget aponta para namespace errado. Conferir o MetricNamespace dos filters e dos widgets antes do re-deploy.
Alarm não dispara após kill Janela de avaliação alta (5 datapoints) ou TreatMissingData errado. 1 datapoint, 60s, TreatMissingData: breaching.

16. Restrições duras — linhas vermelhas

Cinco regras inegociáveis
  1. Sem console AWS. Só CLI + CFN. Reviewer audita histórico do CloudTrail.
  2. Sem credencial permanente em código. Só session token. git secrets roda no pre-commit.
  3. Repositório no padrão tsa-{iniciais}-{slug-tema}. Sem variação.
  4. Sem trabalho prévio à aula. Apenas escolher dataset e abrir o repo. Aluno que codar antes não cumpre o exercício.
  5. Demo de 3 minutos ao fim. Quem não entrega demo não fecha o projeto.

As cinco regras existem por razão didática: o objetivo da aula não é avaliar quem chegou com mais código pronto, e sim provar que o aluno é capaz de orquestrar IA, IaC e observabilidade dentro de um cronograma curto. A entrega ideal é produzida em sala, com cavalaria, sob pressão.

21. Atividade prática em sala — o que executar

O fechamento deste material aponta para a atividade prática que ocupa as duas horas finais. A trilha é a sequência dos Sprints 1 a 4, executada com Codespace aberto, OpenCode (ou Gemini CLI) carregado com as skills do AGENTS.md, e AWS CLI autenticado pelo session token do Learner Lab.

  1. Sprint 1 · 20h00–20h30. Renovar token. Abrir Codespace do tsa-{iniciais}-{slug}. Rodar /bootstrap_tsa_project. Escrever spec/00_problem.md com 5 RFs e 5 RNFs.
  2. Sprint 2 · 20h30–21h00. Rodar /gen_app_template em modo system_design e em modo cfn. Validar template. aws cloudformation deploy. Subir CSV para o S3.
  3. Sprint 3 · 21h00–21h30. Rodar /gen_streamlit_tsa_app. Deploy do app via SSM. Rodar /gen_observability. Re-deploy do template com log group, dashboard, alarm e EventBridge.
  4. Sprint 4 · 21h30–22h00. Rodar pytest tests/smoke/. Marcar release_checklist.md. Simular falha via SSM. Mostrar closed loop. git tag final-v1 && git push --tags. Demo de 3 minutos.
Critério de pronto da aula Tag final-v1 empurrada para o GitHub público, URL pública aberta na frente da turma, três SLIs verdes no dashboard, simulação de falha demonstrada com closed loop respondendo. Repositório navegável pelo reviewer com README que recria a stack em ≤ 10 minutos.

Teoria · Abertura

O briefing do cliente

Você foi contratado pela diretoria para construir um mini-observatório de série temporal sobre o tema escolhido. 3 abas, deploy AWS, demo viva ao fim do dia.

Pedido em uma frase

"Quero entender o passado, decompor o sinal e ver para onde isso vai nos próximos 30 períodos — com um link público que funciona enquanto eu apresento."

O que entra no escopo

  • 1 dataset, 1 série numérica principal.
  • 3 abas: Histórico, Decomposição STL, Forecast 30 períodos.
  • Front leve em Streamlit ou Plotly Dash.

O que fica fora

  • Multi-séries cruzadas, modelos avançados (LSTM, Transformer).
  • Pipeline ETL ao vivo — o CSV vai estático no S3.
  • Autenticação, CDN, domínio próprio.

Teoria · produto

As 3 abas que o cliente vai abrir

Cada aba responde uma pergunta diferente sobre a série. Sem essas três, o front não está completo.

Aba 1 · Histórico

  • Gráfico de linha da série.
  • Seletor de janela (7d, 30d, 1y, all).
  • Estatísticas básicas: min, max, média, desvio.

Aba 2 · Decomposição STL

  • Tendência, sazonalidade e resíduo separados.
  • statsmodels.tsa.seasonal.STL.
  • Period inferido do dataset (mensal=12, diário=7, etc).

Aba 3 · Forecast 30

  • Modelo simples: ARIMA, Prophet ou ExponentialSmoothing.
  • Intervalo de confiança 95%.
  • MAPE/RMSE no rodapé do gráfico.
Teoria · expectativas SRE

SLOs mínimos — não-negociáveis

Três SLOs declarados antes do deploy. Sem isso, não há critério para julgar se a entrega está saudável durante a demo.

IDSLOSLI medindoMeta
SLO-01Disponibilidade durante a demoHTTP 200 a cada 30s no /≥ 95% das checagens
SLO-02Latência de carregamentotempo até primeiro byte da home< 5 s p95
SLO-03Latência do forecasttempo de render da aba 3< 10 s p95

SLOs viram metric filters em CloudWatch e alimentam o dashboard. SLO-01 vira alarm + closed loop.

Teoria · entrega mínima

SRE mínimo — o que não pode faltar

Cinco regras duras que separam um projeto entregue de um projeto demonstrado. Reviewer rejeita PR sem todas as cinco.

1. IaC obrigatório

  • Tudo via CloudFormation.
  • aws cloudformation deploy ... e nada mais.
  • Zero clique no console — nem para criar SG.

2. Observabilidade desde o deploy

  • 1 log group + 1 dashboard + 1 alarm por aplicação.
  • Metric filters para os 3 SLIs.

3. SLOs declarados

  • 3 SLOs no README e no documento de release.
  • Mensuráveis e visíveis no dashboard.

4. Closed loop

  • Alarm de health check → EventBridge → SSM Run Command.
  • Reinício automático do Streamlit. Demonstrável.

5. README com runbook

  • Recriar a stack do zero em ≤ 10 min.
  • Comando único copiável para deploy.
  • Comando único copiável para destruir.

0. Restrição dura

  • Sem credencial permanente em código.
  • Apenas session token do Learner Lab.
  • ~/.aws/credentials fora do git.

Restrições duras — ler com atenção

Cinco linhas vermelhas. Cruzar qualquer uma invalida a entrega, mesmo que o app esteja perfeito.

  3. Repositório no padrão tsa-{iniciais}-{slug-tema}. Sem variação.

Prática · Bloco 1 · Bootstrap & arquitetura

System design + CFN base

Architect produz Mermaid de 1 página. DevOps gera app.yaml. Deploy roda. EC2 ativa, S3 recebe o CSV via aws s3 cp.

# Architect (skill gen_app_template -> diagrama Mermaid)
> /gen_app_template --kind system_design

# DevOps (skill gen_app_template -> CFN)
> /gen_app_template --kind cfn --output iac/cloudformation/app.yaml

# Validacao local
aws cloudformation validate-template \
  --template-body file://iac/cloudformation/app.yaml

# Deploy
aws cloudformation deploy \
  --stack-name tsa-{iniciais} \
  --template-file iac/cloudformation/app.yaml \
  --parameter-overrides Iniciais={iniciais} DatasetSlug={slug} \
  --capabilities CAPABILITY_IAM

# Upload do dataset
aws s3 cp data/dataset.csv s3://tsa-{iniciais}-data/dataset.csv
Checkpoint do Bloco 1 · fechamento

EC2 visível em aws ec2 describe-instances, IP público anotado, S3 com CSV listado, SG aberto na porta 8501.

Prática · Bloco 2 · App + observabilidade

App 3 abas + observabilidade ligada

DataDev gera app.py. DevOps adiciona log group, metric filters, dashboard, alarm e EventBridge rule. Re-deploy.

# DataDev (skill gen_streamlit_tsa_app)
> /gen_streamlit_tsa_app --abas historico,stl,forecast \
                         --modelo expsmoothing \
                         --bucket tsa-{iniciais}-data

# Deploy do app no EC2 via SSM (sem SSH)
aws ssm send-command \
  --document-name "AWS-RunShellScript" \
  --targets "Key=tag:Project,Values=tsa-{iniciais}" \
  --parameters 'commands=[
    "cd /opt/app && git pull origin main",
    "pip install -r requirements.txt -q",
    "pkill -f streamlit || true",
    "nohup streamlit run app.py --server.port 8501 &"
  ]'

# DevOps (skill gen_observability)
> /gen_observability --slos SLO-01,SLO-02,SLO-03 \
                     --runbook restart_streamlit

# Re-deploy do stack com observabilidade
aws cloudformation deploy --stack-name tsa-{iniciais} ...

Prática · Bloco 3 · Validação & demo

Smoke + checklist + demo

Tester roda smoke contra a URL. Reviewer audita o checklist. Aluno simula falha (kill do processo via SSM) e mostra o closed loop subindo o serviço de novo.

# Tester
pytest tests/smoke/test_url.py -v
# TC-SMOKE-01: GET / -> 200 em < 5s
# TC-SMOKE-02: GET /aba=stl -> 200
# TC-SMOKE-03: GET /aba=forecast -> 200 em < 10s

# Reviewer
> /gen_release_checklist
# Marca os 8 criterios em documents/release_checklist.md

# Simulacao de falha (demonstra closed loop)
aws ssm send-command --document-name "AWS-RunShellScript" \
  --targets "Key=tag:Project,Values=tsa-{iniciais}" \
  --parameters 'commands=["pkill -9 -f streamlit"]'
# Em < 60s o alarm dispara, EventBridge chama o runbook
# e o servico volta. Ato 1 da demo.

# Tag de release + push
git tag final-v1
git push origin main --tags

Prática · Bloco 3 · armadilhas

O que costuma quebrar — e como evitar

Lista curta dos erros mais comuns no Learner Lab. Antecipá-los economiza muitas idas e vindas no bloco final.

Erros frequentes
  • Token expirado no meio do Bloco 2 → renovar antes de cada deploy.
  • SG sem inbound 8501 → conferir no template.
  • EC2 sem LabRole anexado → IamInstanceProfile obrigatório.
  • requirements.txt sem statsmodels → STL falha em runtime.
  • Dashboard sem widget → metric filters criados mas não plotados.
Atalhos defensivos
  • aws cloudformation describe-stack-events a cada erro.
  • aws logs tail em modo follow durante o restart.
  • Smoke test rodando local em loop enquanto valida UI.
  • Plano B se forecast estourar — cair para média móvel.

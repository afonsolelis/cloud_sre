Testcontainers · Definição

Containers Docker descartáveis controlados pelos testes

A suíte de teste sobe o container, espera o readiness, conecta, roda o teste, desliga o container. O código de produção nunca sabe que estava conversando com um container efêmero.

O ciclo do TC
  1. start() — pull da imagem, run, wait strategy.
  2. get_connection_url() — URI dinâmica com porta efêmera.
  3. Teste roda contra a infra real.
  4. stop() — container some, volume some.
Pacotes Python · testcontainers-python
  • PostgresContainer — nosso DW dimensional do gold.
  • KafkaContainer — streaming da Aula 04.
  • LocalStackContainer — S3, SSM, IAM off-cloud.
  • GenericContainer — Toxiproxy, Redis, qualquer imagem.
Trade-off

Mock vs Testcontainer — quem ganha o quê

Não é guerra de "melhor". Cada um cobre um nicho. Mock é rápido e pobre; TC é lento e rico. A pirâmide convive.

CritérioMockTestcontainer
Tempo por teste1–5 ms200 ms–2 s (1ª subida do container)
Paridade com produçãoBaixa — só o que o mock prevêAlta — SQL real, driver real, lock real
Bug de driver / dialetoNão pegaPega na primeira execução
Dependência de DockerNenhumaObrigatória no runner
Custo de manutençãoAlto — mock vira ficçãoMédio — subiu a imagem certa, acabou
Onde aplicarUnit, lógica puraIntegration, contract com infra
Testcontainers · Pytest

Fixture session vs function — o trade-off

Container compartilhado entre testes é rápido mas exige reset; container por teste é seguro mas explode o tempo total. A escolha depende da granularidade do dado.

# tests/integration/conftest.py
import pytest
from testcontainers.postgres import PostgresContainer

@pytest.fixture(scope="session")
def pg_container():
    with PostgresContainer("postgres:16-alpine") as pg:
        yield pg

@pytest.fixture
def pg_conn(pg_container, db_session_reset):
    # db_session_reset trunca tabelas entre testes -> isolation barata
    return pg_container.get_connection_url()

Regra prática: escopo session + reset transacional ou TRUNCATE entre testes — 10× mais rápido que recriar o container.

LocalStack · AWS local

Quando vale subir LocalStack no teste

LocalStack emula APIs AWS dentro de um container. Para S3, SSM e SQS é fiel. Para serviços profundos (IAM policy boundary, KMS HSM) não — teste em conta real.

Vale — rodar local
  • S3 — put_object, list_objects_v2.
  • SSM Parameter Store — get_parameter, hierarquias.
  • SQS — ordenação FIFO e visibility timeout.
  • EventBridge — bus + rule + target básico.
  • Lambda em modo docker-reuse.
Não vale — testar em conta real
  • IAM — SCP, permissions boundary, condições de tag.
  • KMS — chaves HSM e grants.
  • Redes complexas — VPC peering, transit gateway.
  • Quotas — throttling do serviço real.
  • Custo — LocalStack não fatura.
Pré-requisito do runner

Docker tem que existir — e ser confiável

Testcontainer sem Docker é fixture vazia. No nosso self-hosted runner (EC2 da Aula 04), o daemon precisa estar instalado, ativo no boot, e com o usuário do runner no grupo docker.

# cloud-init / userdata da EC2 runner
yum install -y docker
systemctl enable --now docker
usermod -aG docker ec2-user

# Sanity-check (rodado pelo workflow antes dos testes)
docker info | grep -E "Server Version|Storage Driver"
docker run --rm hello-world
Anti-padrão

Rodar Testcontainers em GitHub-hosted runner gratuito sem cache de imagens — a primeira pull de cada job paga 5–15 s. Em runner self-hosted com cache, paga uma vez por boot.

Chaos Engineering · Origem

Quebrar de propósito antes que quebre por acidente

Netflix, 2010. Chaos Monkey desliga máquinas aleatoriamente em produção para forçar a arquitetura a tolerar falhas. Em 2014 vira disciplina formal: principlesofchaos.org.

Definição operacional

Chaos engineering é a disciplina de experimentar em um sistema para construir confiança na sua capacidade de tolerar condições turbulentas em produção. É experimento, não teste. Tem hipótese, observação e conclusão.

Princípios · principlesofchaos.org

Os cinco princípios canônicos

Saem dos slides; entram em todo experimento que vamos rodar a partir de hoje.

1. Estado estacionário

  • Hipótese sobre o output observável.
  • Métrica única, mensurável.
  • Antes do experimento, mede-se a baseline.

2. Variar eventos reais

  • Latência, perda de pacote, kill de processo.
  • Falha que ocorre no mundo real.
  • Não inventar falha exótica.

3. Rodar em prod

  • Com salvaguardas e blast radius mínimo.
  • Por enquanto, no nosso lab.
  • Aula 08 discute prod real.

4. Automatizar

  • Experimento como código (chaos/experiments/*.yaml).
  • Roda em game day, vira regressão.

5. Minimizar raio

  • Kill switch obrigatório.
  • Janela curta, área isolada.
  • Quem aborta? Como?

Bonus · cultura

  • Postmortem sem culpa.
  • Lições viram ADR.
  • Falha vira regressão.

Restrição do Learner Lab

Sem AWS FIS — usamos SSM + Toxiproxy

FIS (Fault Injection Service) é o serviço gerenciado de chaos da AWS. No Learner Lab ele não está habilitado. A boa notícia: tudo que a FIS faz pode ser reproduzido com SSM Run Command e ferramentas instaladas na EC2.

O que a FIS daria de graça
  • Templates de experimento versionáveis.
  • Stop conditions vinculadas a alarme do CW.
  • Targets selecionados por tag.
  • Histórico de runs no console.
O que substitui no nosso lab
  • chaos/experiments/*.yaml versionado no repo.
  • CW alarm + SSM cancel-command como kill switch.
  • aws ssm send-command --targets "tag:Role,Value=etl".
  • documents/15_chaos_runs/<ts>.md + ADR.

Experimento · CHAOS-02

Hipótese, ação, observação, kill switch

Forma canônica do experimento. Nada fora dela é aceito como chaos engineering.

id: CHAOS-02
titulo: Latencia ETL -> Postgres por 60s
hipotese: |
  Se a latencia entre ETL e Postgres subir para 2000ms por 60s,
  o circuit breaker abre em <= 10s e o ETL faz freeze do
  proximo run sem perder mensagens em flight.
estado_estacionario:
  sli: pg_query_p95_ms
  baseline: "< 80 ms por 5 min"
acoes:
  - toxiproxy-cli toxic add -t latency -a latency=2000 etl-pg
  - sleep 60
  - toxiproxy-cli toxic remove --toxicName etl-pg-latency etl-pg
salvaguardas:
  - kill_switch: "alarm SLO_BREACH ON => ssm cancel-command"
  - blast_radius: "apenas instancia tag Role=etl-lab"
  - duracao_max: 90s
aceitacao:
  - circuit_breaker_aberto_em: "<= 10 s"
  - mensagens_perdidas: "= 0"
  - retorno_ao_baseline_em: "<= 30 s pos-fim"
Cultura · Game day

Game day — o ritual

Game day é a cerimônia em que o time roda chaos junto, na mesma sala (virtual ou física), com papéis claros. Sem ritual, vira blame.

Papéis

  • Comandante — decide go/no-go.
  • Operador — injeta a falha.
  • Observador — lê CW Logs.
  • Escriba — transcreve.

Roteiro

  • Ler hipótese em voz alta.
  • Confirmar baseline.
  • Anunciar T-0.
  • Injetar.
  • Observar 5 min.
  • Postmortem ao final.

Saída

  • 15_chaos_runs/<ts>.md
  • 1 ADR por descoberta.
  • Issue para cada bug.
  • Próximo experimento agendado.
Qualidade · Definição

Código de baixa qualidade vira toil

SRE chama de toil trabalho repetitivo, sem valor duradouro, que escala linearmente com o serviço. Função com complexidade alta vira toil: cada bug fix dura horas, cada feature pequena trava em revisão.

Tese

Medimos a qualidade do código antes de ela virar incidente. Se a função está acima do limiar de complexidade, ela é candidata a refactor patrocinado pelo error budget — não pela boa vontade.

QM-01 · Complexidade ciclomática

McCabe (1976) — caminhos linearmente independentes

Conta os pontos de decisão. Cada if, elif, and, or, for, while, except, case incrementa. Acima de 10, a função é candidata a quebrar.

Fórmula

M = E − N + 2P

  • E — arestas do grafo de controle.
  • N — nós.
  • P — componentes conexos.
  • Equivale a "nº de decisões + 1".
Limites práticos
  • 1–5 — função simples, baixa.
  • 6–10 — média, aceitável.
  • 11–20 — alta, refatorar.
  • > 20 — muito alta, dívida.
  • Gate do curso: McCabe ≤ 10.
QM-02 · Cognitive Complexity

SonarSource — o que dói para o leitor humano

McCabe não distingue if aninhado de if sequencial — mas o cérebro distingue. Cognitive complexity penaliza aninhamento, recursão e quebra de fluxo (break/continue) progressivamente.

Regras de incremento
  • +1 a cada quebra de fluxo (if, else, switch, for, catch).
  • +1 extra por nível de aninhamento.
  • +1 por sequência de operadores lógicos heterogêneos.
  • Recursão e goto — sempre +1.
Por que importa
  • Mais fiel ao tempo de leitura humano.
  • Captura "código que parece simples mas é confuso".
  • Alvo: ≤ 15 por função.
  • Em Python, flake8-cognitive-complexity ou radon mi + revisão manual.
QM-03 · Halstead

Halstead (1977) — vocabulário, volume, esforço

Olha a função como texto e conta operadores e operandos. Revela duplicação semântica e densidade de informação que McCabe não vê.

MétricaFórmulaO que diz
Vocabulárion = n1 + n2Tipos distintos de tokens
ComprimentoN = N1 + N2Total de tokens
VolumeV = N · log₂(n)Tamanho da informação
DificuldadeD = (n1/2) · (N2/n2)Quão denso é o código
EsforçoE = D · VCusto de entender
Tempo (s)T = E / 18Estimado para 1 leitor
QM-04 · Maintainability Index

MI — combina Halstead, McCabe e LOC

Métrica única, normalizada de 0 a 100. Acima de 65 é manutenível. Abaixo, é dívida. Cada projeto define seu limiar — o nosso é 65.

Fórmula clássica (normalizada 0–100)

MI = max(0, (171 − 5.2 · ln(V) − 0.23 · G − 16.2 · ln(LOC)) · 100 / 171)

  • V — volume de Halstead.
  • G — complexidade ciclomática (McCabe).
  • LOC — linhas de código.
  • Em Python: radon mi src/.

Política de qualidade

Limites duros — sem ADR, sem merge

Limiares ligados à RTM. Função crítica que cruza um limiar precisa de ADR justificando, ou ela não entra na main.

Limiares do projeto
  • McCabe ≤ 10 por função; ≤ 8 média por arquivo.
  • Cognitive ≤ 15 por função.
  • MI ≥ 65 em todo módulo crítico.
  • Duplicação ≤ 5% (regra SIM do ruff).
  • Cobertura unit ≥ 80%.
Como ligar à RTM
  • Cada função crítica tem RF/RNF correspondente.
  • Ultrapassar limiar = ADR QUALITY-NN obrigatório.
  • ADR explica trade-off e plano de refactor.
  • Refactor é orçado pelo error budget restante.
Anti-padrões

O que não fazer hoje

Cada um desses já travou um time real. Estão na aula porque são tentadores.

Testcontainers

  • Container persistente entre testes sem reset → testes não-determinísticos.
  • Pull em todo job → runner lento — cache de imagem é obrigatório.
  • Mock + TC misturados sem critério.

Chaos

  • Rodar em prod sem kill switch.
  • Toxiproxy esquecido ligado pós-experimento.
  • Experimento sem hipótese — vandalismo.
  • Game day sem postmortem.

Qualidade

  • Métrica vista, nunca aplicada.
  • Limiar negociado por sprint.
  • Refactor sem orçamento de error budget.
  • Ignorar deps cíclicas (madge).

Prática · Passo 6b

Injetar latência via SSM Run Command

SSM Run Command roda toxiproxy-cli dentro da EC2 alvo. O ID do comando volta para o terminal — é ele que usamos no kill switch.

# Injetar latencia 2000ms por 60s no proxy etl-pg
CMD_ID=$(aws ssm send-command \
  --document-name "AWS-RunShellScript" \
  --targets "Key=tag:Role,Values=etl-lab" \
  --parameters 'commands=[
    "docker exec toxiproxy /toxiproxy-cli toxic add -t latency -a latency=2000 etl-pg",
    "sleep 60",
    "docker exec toxiproxy /toxiproxy-cli toxic remove --toxicName etl-pg-latency etl-pg"
  ]' \
  --query "Command.CommandId" --output text)

echo "$CMD_ID" > /tmp/last_chaos_cmd

# Kill switch (executar se algo sair do controle)
aws ssm cancel-command --command-id "$CMD_ID"

# Observar
aws logs tail /aws/ec2/etl-lab --follow --since 2m

# .github/workflows/wb.yml (adendo)
quality:
  runs-on: [self-hosted, lab]
  needs: [unit, integration]
  steps:
    - uses: actions/checkout@v4
    - run: pip install radon xenon ruff
    - name: McCabe gate
      run: xenon --max-absolute B --max-modules A --max-average A src/
    - name: MI gate
      run: |
        python - <<'EOF'
        import subprocess, sys, json, re
        out = subprocess.check_output(["radon","mi","-j","src/"]).decode()
        data = json.loads(out)
        bad = [f for f, m in data.items() if m["mi"] < 65]
        if bad:
            print("MI < 65 em:", bad); sys.exit(1)
        EOF
    - name: Ruff gate
      run: ruff check src/

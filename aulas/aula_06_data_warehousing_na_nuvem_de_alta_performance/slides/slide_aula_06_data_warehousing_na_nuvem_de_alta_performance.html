Tese da aula

A migração como execução da arquitetura planejada

A análise arquitetural foi conduzida previamente, por meio da ATAM elaborada na Aula 05. A presente aula concentra-se na execução: realiza-se primeiramente o lift-and-shift, com o objetivo de assegurar a continuidade operacional, e em seguida a decomposição, que permite obter os benefícios próprios de uma arquitetura distribuída.

Três princípios orientadores desta aula
  1. Lift-and-shift antes da redesenhar a arquitetura — o compose integral é executado dentro de uma única EC2 antes de ser decomposto em serviços individuais.
  2. Restrição ao uso do console — todas as operações são realizadas via AWS CLI, sendo os comandos gerados pelo Gemini CLI e submetidos à revisão humana antes da execução.
  3. Observância dos sensitivity points da Aula 05 — cada decisão de provisionamento é avaliada à luz dos pontos de sensibilidade identificados na análise ATAM.
Arquitetura alvo · Mapeamento

Equivalência entre componentes Docker e serviços AWS

Realiza-se uma tradução direta dos componentes do compose para os serviços AWS correspondentes, respeitando-se os limites estabelecidos pelo Student Lab. O conjunto utilizado restringe-se a EC2, S3, Lambda, IAM, Security Groups e Secrets Manager.

Ambiente · AWS Academy

AWS Student Lab — Learner Lab

Trata-se do sandbox gratuito disponibilizado pela AWS Academy, caracterizado por sessões de curta duração, credenciais rotativas, uma única região disponível e uma role pré-configurada. O ambiente é suficiente para a realização integral da aula, desde que sejam observados os limites estabelecidos.

Procedimento para obtenção das credenciais
  • Realizar o login em awsacademy.instructure.com.
  • Acionar o botão Start Lab, que inicia uma sessão com duração aproximada de quatro horas.
  • Em AWS Details > AWS CLI, recuperar as variáveis AWS_ACCESS_KEY_ID, AWS_SECRET_ACCESS_KEY e AWS_SESSION_TOKEN.
  • Em SSH key, efetuar o download do arquivo labsuser.pem e aplicar chmod 400.
Limites estabelecidos
  • Região única disponível: us-east-1.
  • Role pré-configurada: LabRole. Não é permitida a criação de IAM personalizada.
  • Instance profile fixo: LabInstanceProfile.
  • As credenciais expiram em aproximadamente quatro horas e devem ser renovadas por meio de novo acionamento do Start Lab.
Ferramenta · Gemini CLI

Gemini CLI como assistente de provisionamento

O Gemini CLI é o agente de linha de comando do Google, que dispõe de um free tier com cota generosa, configurando-se como alternativa ao Claude Code e ao AIOX para os alunos que não possuem assinatura paga. O fluxo de uso compreende as seguintes etapas: formulação do prompt, geração do comando AWS CLI pelo agente, revisão humana, execução e validação do resultado.

instalar e autenticar 
# 1. instalar (precisa Node 18+)
npm install -g @google/gemini-cli

# 2. autenticar (abre navegador para login Google)
gemini auth

# 3. confirmar que esta funcionando
gemini "diga ola em uma linha"

Fluxo esperado: o aluno descreve a intenção em linguagem natural; o Gemini gera o comando aws ec2 run-instances ...; o aluno revisa o comando integralmente; executa o procedimento; e, em seguida, o Gemini auxilia na validação por meio de um comando describe-*. Recomenda-se que nenhum comando destrutivo (a exemplo de terminate, delete ou force) seja executado sem revisão prévia.

Ambiente · Limites

Restrições do Learner Lab a serem observadas

O ambiente, embora amplo em recursos, apresenta limitações específicas. O conhecimento prévio dessas restrições evita interrupções durante a etapa prática.

Região e IAM

  • Apenas us-east-1 está disponível.
  • Não é possível criar roles personalizadas.
  • Utiliza-se sempre a LabRole.
  • As EC2 são instanciadas com LabInstanceProfile.

Serviços

  • EC2, S3, Lambda e Secrets Manager estão habilitados.
  • A VPC default já está provisionada; não se deve criar uma nova.
  • Redshift, Glue e Athena não são utilizados nesta aula.
  • A Lambda pode encontrar-se limitada; recomenda-se manter o fallback baseado em cron.

Operação

  • A sessão tem duração aproximada de quatro horas, com credenciais rotativas.
  • As instâncias EC2 devem ser desligadas ao final para evitar o consumo desnecessário de créditos.
  • O acesso SSH utiliza o key pair gerado no próprio Lab.
  • Não há suporte a domínios personalizados nem ao ACM.
Etapa 1 · Lift-and-shift

Priorização da continuidade na primeira etapa

O compose integral será executado dentro de uma única EC2, preservando-se a estrutura original, agora hospedada em uma instância AWS. Trata-se de uma configuração provisória, cuja função é estabelecer uma baseline migrada antes do início da decomposição arquitetural.

Razões para a precedência do lift-and-shift
  1. Permite validar credenciais, região e Security Groups sem a complexidade adicional de serviços distintos.
  2. Mantém o sistema em operação com o mesmo comportamento, viabilizando uma decomposição incremental e controlada.
  3. Estabelece o ponto de comparação a ser utilizado ao término da Etapa 2.
  4. Em cenários reais de produção, este é o caminho que minimiza o tempo de indisponibilidade durante a migração.

Etapa 2 · Decomposição

Decomposição do monolito em serviços AWS apropriados a cada responsabilidade

Nesta etapa, cada serviço da stack é alocado em um recurso AWS dedicado. As interfaces são preservadas (S3 API, HTTP e SQL); apenas os provedores subjacentes mudam. Cada passo é independente e pode ser conduzido de forma incremental.

Os oito passos da decomposição
  1. Substituição do MinIO pelo S3.
  2. Provisionamento de EC2 dedicada para o ClickHouse.
  3. Configuração do user-data do ClickHouse com integração ao Secrets Manager.
  4. Provisionamento de EC2 dedicada para o Streamlit, executado via systemd.
  5. Conversão do ingestor em função Lambda event-driven.
  6. Implementação do handler Python da Lambda.
  7. Configuração de IAM e dos Security Groups encadeados.
  8. Validação end-to-end.
Etapa 2 · Passo 2.1

Substituição do MinIO pelo S3

A API S3 é preservada, alterando-se apenas o provedor que responde às requisições. Não há necessidade de um endpoint_url customizado no boto3; as credenciais são herdadas da IAM role atribuída à EC2 ou à Lambda.

criar bucket e subir os CSVs 
BUCKET="olist-${USER}-raw"
aws s3 mb "s3://${BUCKET}" --region us-east-1

aws s3 cp data/ "s3://${BUCKET}/olist/" --recursive \
  --exclude "*" --include "*.csv"

aws s3 ls "s3://${BUCKET}/olist/"

No código do ingestor, recomenda-se a remoção do parâmetro endpoint_url e das variáveis AWS_ACCESS_KEY_ID e SECRET, deixando-se que o boto3 herde as credenciais do instance profile.

Etapa 2 · Passo 2.2

Provisionamento de EC2 dedicada ao ClickHouse

Provisiona-se uma nova instância EC2 t3.medium. O Security Group sg-clickhouse autoriza tráfego na porta 8123 unicamente a partir dos Security Groups associados ao Streamlit e à Lambda. A senha do ClickHouse é armazenada no Secrets Manager e deve ser criada previamente à subida da EC2.

criar SG e secret antes da instância 
VPC_ID=$(aws ec2 describe-vpcs --filters Name=is-default,Values=true \
  --query 'Vpcs[0].VpcId' --output text)

SG_CH=$(aws ec2 create-security-group \
  --group-name sg-clickhouse \
  --description "ClickHouse - 8123 interno" \
  --vpc-id "$VPC_ID" --query 'GroupId' --output text)

# senha forte vai para o Secrets Manager
CH_PASSWORD=$(openssl rand -base64 24)
aws secretsmanager create-secret \
  --name olist/clickhouse \
  --secret-string "{\"password\":\"${CH_PASSWORD}\"}"

Provisionamento de EC2 dedicada ao Streamlit, executado via systemd

Provisiona-se uma instância EC2 t3.small. O Security Group sg-streamlit autoriza tráfego na porta 8501 a partir da internet e na porta 22 unicamente a partir do IP do aluno. A conexão com o ClickHouse utiliza o IP privado, e a senha é recuperada no Secrets Manager. O serviço é executado por meio de unidade systemd.

  https://github.com/<aluno>/20261sre-primeironome-ultimonome.git app

Etapa 2 · Passo 2.5

Conversão do ingestor em função Lambda event-driven

Uma operação PUT no S3 aciona a Lambda, que efetua a leitura do CSV, sua conversão para NDJSON e a inserção no ClickHouse via HTTP. Caso o Lab apresente limitação ao serviço Lambda, o fallback consiste em um cron systemd, hospedado em uma das instâncias EC2, que invoca o mesmo script.

criar a Lambda + trigger S3 
# empacotar
mkdir -p build
pip install -r lambda/requirements.txt -t build/
cp lambda/lambda_handler.py build/
cd build && zip -r ../olist-ingestor.zip . && cd ..

# criar funcao
LAB_ROLE=$(aws iam get-role --role-name LabRole --query Role.Arn --output text)
aws lambda create-function \
  --function-name olist-ingestor \
  --runtime python3.11 \
  --role "$LAB_ROLE" \
  --handler lambda_handler.handler \
  --timeout 300 --memory-size 1024 \
  --zip-file fileb://olist-ingestor.zip \
  --environment "Variables={CLICKHOUSE_HOST=<ip-ch>,CLICKHOUSE_PASSWORD=$CH_PASSWORD}"

# permitir s3 chamar
aws lambda add-permission --function-name olist-ingestor \
  --action lambda:InvokeFunction --principal s3.amazonaws.com \
  --statement-id s3-trigger --source-arn "arn:aws:s3:::${BUCKET}"

# trigger S3:PutObject
aws s3api put-bucket-notification-configuration --bucket "$BUCKET" \
  --notification-configuration '{
    "LambdaFunctionConfigurations": [{
      "LambdaFunctionArn": "ARN_DA_LAMBDA",
      "Events": ["s3:ObjectCreated:*"],
      "Filter":{"Key":{"FilterRules":[{"Name":"suffix","Value":".csv"}]}}
    }]
  }'
Etapa 2 · Passo 2.6

Lambda handler — lambda_handler.py

Trata-se de uma adaptação direta do ingest_minio.py apresentado na Aula 04: o handler recebe um event S3, realiza a leitura do objeto e procede à inserção no ClickHouse em lotes de cinquenta mil registros, por meio do clickhouse_connect.

lambda/lambda_handler.py 
import os, json, time, urllib.parse, boto3
import pandas as pd, clickhouse_connect

s3 = boto3.client("s3")
ch = clickhouse_connect.get_client(
    host=os.environ["CLICKHOUSE_HOST"],
    username="dashboard",
    password=os.environ["CLICKHOUSE_PASSWORD"],
)

def handler(event, context):
    unixtime = int(time.time())
    total = 0
    for rec in event["Records"]:
        bucket = rec["s3"]["bucket"]["name"]
        key = urllib.parse.unquote_plus(rec["s3"]["object"]["key"])
        obj = s3.get_object(Bucket=bucket, Key=key)
        df = pd.read_csv(obj["Body"], dtype=str).where(pd.notnull, None)
        rows = [(unixtime, json.dumps(r.to_dict()), key) for _, r in df.iterrows()]
        for i in range(0, len(rows), 50_000):
            ch.insert("olist_raw.ingestion", rows[i:i+50_000],
                      column_names=["unixtime","data","tag"])
        total += len(rows)
    return {"ok": True, "rows": total}
Etapa 2 · Passo 2.7

IAM e Security Groups encadeados

No Lab, a role utilizada é a LabRole. Os Security Groups operam como elementos de conexão entre os três componentes principais. O princípio orientador é o de que cada SG admite apenas o tráfego estritamente necessário, na porta correspondente, proveniente da origem autorizada.

SG · ClickHouse

  • Inbound 8123 ← sg-streamlit.
  • Inbound 8123 ← sg-lambda.
  • Inbound 22 ← IP do aluno.
  • Outbound livre.

SG · Streamlit

  • Inbound 8501 ← 0.0.0.0/0.
  • Inbound 22 ← IP do aluno.
  • Outbound livre.
  • Permissão de acesso ao sg-clickhouse.

IAM

  • Todas as operações são realizadas via LabRole.
  • As EC2 são instanciadas com LabInstanceProfile.
  • A Lambda também utiliza a LabRole.
  • Secrets Manager: a permissão secretsmanager:GetSecretValue já se encontra atribuída à role.
Etapa 2 · Passo 2.8

Validação end-to-end

Procede-se ao upload de um arquivo CSV; a Lambda é automaticamente acionada; o volume de dados do ClickHouse é incrementado; e o dashboard reflete a atualização. Todo o pipeline é reativo de ponta a ponta, sem intervenção manual no console.

teste de fluxo reativo

aws s3 cp data/olist_orders_dataset.csv \
  "s3://${BUCKET}/olist/olist_orders_dataset.csv"

# 2. Lambda dispara automaticamente (logs no CloudWatch)
aws logs tail /aws/lambda/olist-ingestor --since 2m

# 3. Conferir no ClickHouse via Streamlit EC2
ssh -i ~/labsuser.pem ec2-user@$ST_EC2_IP \
  "curl -s http://<ch-private>:8123/?query=SELECT+count()+FROM+olist_raw.ingestion"

# 4. Abrir o dashboard
echo "http://${ST_EC2_IP}:8501"

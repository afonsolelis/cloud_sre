Teoria · 19h00 – 20h30

O que é um Data Warehouse?

Repositório central, orientado a assuntos, integrado, não volátil e histórico, otimizado para consultas analíticas — não para transações.

Quatro características clássicas (Bill Inmon)
  1. Subject-oriented — organizado por domínio de negócio (vendas, clientes, produtos).
  2. Integrado — dados padronizados a partir de múltiplas fontes.
  3. Non-volatile — dados não são atualizados no lugar; só lidos e acrescentados.
  4. Time-variant — guarda histórico ao longo do tempo, não só o estado atual.

Teoria · Evolução

OLAP cubes vs DW moderno em nuvem

OLAP tradicional (anos 1990-2000)
  • Cubos MOLAP pré-agregados (Essbase, SSAS).
  • Dimensões e medidas fixas.
  • Processamento batch noturno.
  • Pouca flexibilidade para ad–hoc.
DW moderno em nuvem
  • SQL direto sobre grandes volumes.
  • MPP + armazenamento colunar.
  • Separação de compute e storage.
  • Elasticidade sob demanda (Serverless).
Teoria · Arquitetura

MPP: Massively Parallel Processing

Arquitetura distribuída com um nó coordenador e múltiplos nós de cálculo trabalhando em paralelo sobre partições diferentes dos dados.

Fluxo de uma query MPP
  1. Leader recebe o SQL, valida e gera o plano de execução.
  2. Leader distribui fragmentos para os compute nodes.
  3. Cada compute node processa sua fatia local (slices).
  4. Resultados intermediários são trocados quando necessário (redistribution/broadcast).
  5. Leader consolida o resultado final e devolve ao cliente.
Teoria · Armazenamento

Armazenamento colunar e compressão por coluna

DW analíticos armazenam dados por coluna, não por linha. Consultas analíticas leem poucas colunas de muitas linhas — o oposto de OLTP.

Vantagens colunares

  • Lê só as colunas usadas (menos IO).
  • Compressão específica por tipo de dado.
  • Vetorização de operações agregadas.

Codificações típicas

  • AZ64 — padrão do Redshift para numéricos e datas.
  • ZSTD — forte em strings.
  • RUNLENGTH — colunas com poucos valores repetidos.
  • DELTA — sequências ordenadas.
Teoria · Otimização

Zone maps / block min-max

Cada bloco de 1 MB de dados guarda, em metadados, o valor mínimo e máximo de cada coluna. O planejador pula blocos inteiros que não atendem ao filtro.

Exemplo prático

Uma tabela fato_vendas ordenada por data_pedido e filtrada por WHERE data_pedido BETWEEN '2018-01-01' AND '2018-01-31' pode descartar 95% dos blocos antes mesmo de lê-los. Sem ordenação coerente com o filtro, zone maps perdem efetividade.

Teoria · Modelagem dimensional

Star schema vs snowflake

Star schema
  • Fato no centro, dimensões ao redor.
  • Dimensões desnormalizadas (1 tabela por dimensão).
  • Menos joins, queries mais rápidas.
  • Padrão em DWs analíticos.
Snowflake
  • Dimensões normalizadas (hierárquicas).
  • Economiza armazenamento.
  • Mais joins, mais complexidade.
  • Justificável quando dimensão é enorme.
Teoria · Modelagem dimensional

Tabela fato: grain e tipos

A primeira decisão da modelagem é declarar o grain: o nível de detalhe de uma linha da fato. Sem grain claro, o modelo colapsa.

Tipo de fatoDescriçãoExemplo Olist

Periodic snapshotEstado em um intervalo fixo de tempo.Estoque por SKU no fim do dia.
Accumulating snapshotCiclo de vida de um processo, atualizado conforme avança.Pedido: criado → pago → enviado → entregue.
Teoria · Dimensões

Slow Changing Dimensions (SCD)

O mundo muda: clientes trocam endereço, produtos mudam de categoria. O DW precisa decidir como guardar essa história.

TipoComportamentoQuando usar
SCD 0Nunca muda.Atributos imutáveis (data de nascimento).
SCD 1Sobrescreve o valor antigo.Correções, sem necessidade histórica.
SCD 2Cria nova linha versionada (valid_from/valid_to).Histórico completo (padrão).
SCD 3Mantém colunas current_value e previous_value.Só uma mudança recente importa.
Teoria · Chaves

Surrogate keys vs natural keys

Natural key
  • Identificador do sistema de origem (CPF, email, SKU).
  • Muda quando a fonte muda.
  • Difícil de unificar entre sistemas.
Surrogate key
  • Inteiro sequencial gerado pelo DW.
  • Estável, imune a mudanças de origem.
  • Essencial para SCD 2 (uma linha por versão).
  • Chave de join leve e rápida.

Teoria · Redshift interno

Distribution styles

Como as linhas de uma tabela são espalhadas entre os nós de compute. Escolher errado custa shuffle pesado em todo join.

EstiloComportamentoQuando usar
KEYHash de uma coluna distribui a linha.Fato grande com join frequente em mesma chave.
ALLCópia completa em todos os nós.Dimensão pequena usada em quase toda query.
EVENRound-robin uniforme.Tabela sem chave de join dominante.
AUTORedshift decide e evolui.Default recomendado para início.
Teoria · Redshift interno

Sort keys: compound vs interleaved

Sort key define a ordem física dos blocos. Ela ativa zone maps e pode transformar um full scan em leitura pontual.

Compound
  • Ordena pela sequência declarada (col1, col2, col3).
  • Excelente quando o filtro usa as primeiras colunas.
  • Padrão para filtros temporais.
Interleaved
  • Peso equivalente para todas as colunas.
  • Bom quando filtros variam entre colunas diferentes.
  • VACUUM mais caro; cair em desuso na maioria dos casos.
Teoria · Carga

COPY: ingestar do S3 para Redshift

COPY é a forma canonical de carregar dados. Usa os nós de compute em paralelo, lê direto do S3 e aplica compressão automática.

COPY fato_vendas
FROM 's3://aluno-refined/olist/gold/fato_vendas/'
IAM_ROLE 'arn:aws:iam::<account>:role/LabRole'
FORMAT AS PARQUET;
Dicas operacionais
  • Use arquivos de 100 MB a 1 GB e múltiplos arquivos para paralelismo.
  • Prefira Parquet ao CSV: tipos, compressão e menor custo de parse.
  • Use MANIFEST para controlar exatamente quais arquivos entram.
Teoria · Lakehouse leve

Redshift Spectrum

Consulta SQL direto em arquivos no S3 sem carregar. Usa o Glue Data Catalog como metastore. Ideal para dados frios ou para juntar gold do DL com fatos no DW.

CREATE EXTERNAL SCHEMA olist_ext
FROM DATA CATALOG
DATABASE 'olist_gold'
IAM_ROLE 'arn:aws:iam::<account>:role/LabRole'
REGION 'us-east-1';

SELECT categoria, SUM(receita_total)
FROM olist_ext.fato_vendas
GROUP BY categoria;
Teoria · Alternativa lakehouse

Athena + Apache Iceberg

Iceberg é um formato de tabela aberto sobre Parquet no S3. Traz ACID, snapshots, time travel e evolução de schema — sem precisar de um DW tradicional.

Por que Iceberg

  • Transações ACID sobre o data lake.
  • UPSERT e DELETE via MERGE INTO.
  • Time travel por snapshot id.
  • Schema evolution sem reescrever arquivos.

Como roda na AWS

  • Athena engine v3 com suporte nativo.
  • Metadata manifesto em s3://.../iceberg/....
  • Glue Catalog como catálogo.
  • Sem cluster dedicado.

Prática · Plano A · DDL

Criar schema e dimensões

CREATE SCHEMA IF NOT EXISTS olist;

CREATE TABLE olist.dim_cliente (
  sk_cliente      BIGINT IDENTITY(1,1),
  customer_id     VARCHAR(64) NOT NULL,
  cidade          VARCHAR(120),
  uf              CHAR(2),
  valid_from      TIMESTAMP,
  valid_to        TIMESTAMP,
  is_current      BOOLEAN
) DISTSTYLE ALL;

CREATE TABLE olist.dim_produto (
  sk_produto      BIGINT IDENTITY(1,1),
  product_id      VARCHAR(64) NOT NULL,
  categoria       VARCHAR(120)
) DISTSTYLE ALL;

CREATE TABLE olist.dim_tempo (
  sk_tempo        INTEGER NOT NULL,
  data            DATE,
  ano             SMALLINT,
  mes             SMALLINT,
  dia             SMALLINT
) DISTSTYLE ALL;
Prática · Plano A · DDL

Criar tabela fato

CREATE TABLE olist.fato_vendas (
  sk_cliente      BIGINT,
  sk_produto      BIGINT,
  sk_tempo        INTEGER,
  order_id        VARCHAR(64),
  price           DECIMAL(12,2),
  freight_value   DECIMAL(12,2),
  receita_total   DECIMAL(12,2),
  data_pedido     TIMESTAMP
)
DISTSTYLE KEY
DISTKEY (sk_cliente)
COMPOUND SORTKEY (data_pedido, sk_produto);
Por que essas escolhas
  • DISTKEY em sk_cliente: joins frequentes com dim_cliente.
  • Dimensões pequenas com DISTSTYLE ALL: sem shuffle nos joins.
  • SORTKEY compound em data_pedido: acelera filtros temporais via zone maps.
Prática · Plano A · Carga

COPY do gold e primeira consulta

COPY olist.fato_vendas
FROM 's3://<aluno>-refined/olist/gold/fato_vendas/'
IAM_ROLE 'arn:aws:iam::<account>:role/LabRole'
FORMAT AS PARQUET;

SELECT p.categoria,
       DATE_TRUNC('month', f.data_pedido) AS mes,
       SUM(f.receita_total)               AS receita
FROM   olist.fato_vendas f
JOIN   olist.dim_produto p USING (sk_produto)
GROUP  BY 1,2
ORDER  BY mes, receita DESC;

Criar tabela Iceberg

CREATE TABLE olist_iceberg.fato_vendas (
  order_id        string,
  customer_id     string,
  product_id      string,
  price           decimal(12,2),
  freight_value   decimal(12,2),
  receita_total   decimal(12,2),
  data_pedido     timestamp
)
PARTITIONED BY (month(data_pedido))
LOCATION 's3://<aluno>-refined/olist/iceberg/fato_vendas/'
TBLPROPERTIES (
  'table_type'='ICEBERG',
  'format'='parquet'
);
Prática · Plano B · DML

INSERT e MERGE INTO

INSERT INTO olist_iceberg.fato_vendas
SELECT order_id, customer_id, product_id,
       price, freight_value,
       price + freight_value AS receita_total,
       data_pedido
FROM   olist_gold.fato_vendas;

MERGE INTO olist_iceberg.fato_vendas t
USING atualizacoes_vendas u
ON    t.order_id = u.order_id
WHEN MATCHED THEN UPDATE SET
      price = u.price,
      freight_value = u.freight_value,
      receita_total = u.price + u.freight_value
WHEN NOT MATCHED THEN INSERT VALUES (
      u.order_id, u.customer_id, u.product_id,
      u.price, u.freight_value,
      u.price + u.freight_value, u.data_pedido
);
Prática · Plano B · Features

Time travel e schema evolution

-- Descobrir os snapshots
SELECT snapshot_id, committed_at
FROM   "olist_iceberg"."fato_vendas$snapshots"
ORDER  BY committed_at DESC;

-- Consultar estado anterior
SELECT COUNT(*) FROM olist_iceberg.fato_vendas
FOR VERSION AS OF 1234567890123456789;

-- Evoluir o schema sem reescrever arquivos
ALTER TABLE olist_iceberg.fato_vendas
  ADD COLUMN canal_origem string;
O que demonstrar em sala
  • Antes e depois de um MERGE: mesma tabela, dois snapshots.
  • Contagem diferente com FOR VERSION AS OF.
  • Nova coluna aparece sem quebrar queries existentes.

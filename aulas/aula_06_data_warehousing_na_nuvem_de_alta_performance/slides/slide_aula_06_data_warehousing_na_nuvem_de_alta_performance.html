Tese da aula

Migrar é executar a arquitetura planejada

A análise arquitetural já foi feita (ATAM da Aula 05). Hoje é execução: primeiro lift-and-shift para garantir continuidade, depois decomposição para colher os benefícios do distribuído.

Três regras desta aula
  1. Lift-and-shift antes de redesenhar — o compose inteiro roda dentro de uma EC2 antes de virar serviços separados.
  2. Console é proibido — tudo via AWS CLI, e o AWS CLI é digitado pelo Gemini CLI. Revisão humana antes de executar.
  3. Honrar os sensitivity points da Aula 05 — cada decisão de provisionamento confere com o que a ATAM disse que pode quebrar.

Ambiente · AWS Academy

AWS Student Lab — Learner Lab

Sandbox gratuito da AWS Academy. Sessão curta, credenciais rotativas, region única, role pré-pronta. Suficiente para a aula inteira — desde que respeite os limites.

Como obter credenciais
  • Login em awsacademy.instructure.com.
  • Botão Start Lab → sessão de ~4h.
  • AWS Details > AWS CLI entrega AWS_ACCESS_KEY_ID, AWS_SECRET_ACCESS_KEY, AWS_SESSION_TOKEN.
  • Em SSH key, baixar labsuser.pem; chmod 400.
Limites fixos
  • Region única: us-east-1.
  • Role pré-pronta: LabRole — nada de IAM custom.
  • Instance profile fixo: LabInstanceProfile.
  • Credenciais expiram em ~4h — renovar voltando no Start Lab.
Ferramenta · Gemini CLI

Gemini CLI como copiloto

Agente de linha de comando do Google, com free tier generoso. Substitui Claude Code/AIOX para alunos sem assinatura paga. Loop: prompt → comando AWS CLI → revisão humana → execução → validação.

instalar e autenticar 
# 1. instalar (precisa Node 18+)
npm install -g @google/gemini-cli

# 2. autenticar (abre navegador para login Google)
gemini auth

# 3. confirmar que esta funcionando
gemini "diga ola em uma linha"

Loop esperado: aluno descreve a intenção → Gemini gera aws ec2 run-instances ... → aluno revisa palavra a palavra → executa → Gemini ajuda a validar com um describe-*. Nunca auto-executar comando destrutivo (terminate, delete, force) sem revisão.

Ambiente · Limites

Restrições do Learner Lab — o que respeitar

O Lab é generoso mas opinativo. Saber o que não tem evita travas no meio da prática.

Region & IAM

  • Só us-east-1.
  • Sem criação de roles custom.
  • Usar sempre LabRole.
  • EC2 sobe com LabInstanceProfile.

Serviços

  • EC2, S3, Lambda, Secrets Manager — OK.
  • VPC default já existe — não criar nova.
  • Sem Redshift, Glue, Athena hoje.
  • Lambda pode estar limitada: ter fallback cron.

Operação

  • Sessão ~4h — credenciais rotativas.
  • Desligar EC2 ao final ou queima crédito.
  • SSH via key pair gerado no Lab.
  • Sem domínios custom, sem ACM.
Etapa 1 · Lift-and-shift

Mover primeiro. Melhorar depois.

O compose inteiro vai rodar dentro de uma única EC2. Tudo no docker-compose, mas agora numa instância da AWS. É feio? É. Funciona? Funciona. E nos dá uma baseline migrada antes de começar a decompor.

Por que lift-and-shift primeiro
  1. Valida que credenciais, region e SG estão certos — sem entrar no labirinto de serviços distintos.
  2. Mantém o sistema rodando "igualzinho" para depois decompor por partes.
  3. Cria o ponto de comparação que será usado quando a Etapa 2 terminar.
  4. Em produção real, este é o caminho que minimiza downtime de migração.

Renovar credenciais do Student Lab

Etapa 1 · Passo 1.2

Provisionar EC2 via Gemini CLI

Descreva a intenção em linguagem natural. Gemini gera o comando AWS CLI. Aluno revisa. Aluno executa.

prompt para o Gemini CLI 
gemini "gere um comando aws ec2 run-instances que:
- usa AMI Amazon Linux 2023 mais recente em us-east-1
- t3.medium
- instance profile LabInstanceProfile
- security group abrindo 22 e 8501
- user-data que instale docker, docker-compose-plugin, git
- key pair vockey
- tag Name=olist-monolito"
comando gerado (revisar antes de rodar) 
aws ec2 run-instances \
  --image-id $(aws ec2 describe-images \
    --owners amazon \
    --filters "Name=name,Values=al2023-ami-2023.*-x86_64" \
    --query 'sort_by(Images, &CreationDate)[-1].ImageId' --output text) \
  --instance-type t3.medium \
  --key-name vockey \
  --iam-instance-profile Name=LabInstanceProfile \
  --security-group-ids $SG_ID \
  --user-data file://userdata-monolito.sh \
  --tag-specifications 'ResourceType=instance,Tags=[{Key=Name,Value=olist-monolito}]'
Etapa 1 · Passo 1.3 · user-data

userdata-monolito.sh — o que a EC2 faz no boot

Bootstrap autônomo: instala Docker, habilita o serviço, adiciona o usuário ec2-user ao grupo e clona o repo. Quando o SSH liga, a stack já está pronta para compose up.

userdata-monolito.sh 
#!/bin/bash
set -euxo pipefail

dnf update -y
dnf install -y docker git
systemctl enable --now docker
usermod -aG docker ec2-user

# Plugin compose v2 oficial
mkdir -p /usr/local/lib/docker/cli-plugins
curl -SL "https://github.com/docker/compose/releases/latest/download/docker-compose-linux-x86_64" \
  -o /usr/local/lib/docker/cli-plugins/docker-compose
chmod +x /usr/local/lib/docker/cli-plugins/docker-compose

# Repo da aula
cd /home/ec2-user
sudo -u ec2-user git clone https://github.com/<aluno>/20261sre-primeironome-ultimonome.git
chown -R ec2-user:ec2-user /home/ec2-user/20261sre-primeironome-ultimonome

SSH na EC2, subir o compose

Quebrar o monolito — cada serviço no seu lugar canônico AWS

Oito passos da decomposição

Etapa 2 · Passo 2.1

S3 substitui MinIO

Mesma API S3 — só muda quem responde. Sem endpoint_url custom no boto3, credenciais via IAM role da EC2 / Lambda.

criar bucket e subir os CSVs 
BUCKET="olist-${USER}-raw"
aws s3 mb "s3://${BUCKET}" --region us-east-1

aws s3 cp data/ "s3://${BUCKET}/olist/" --recursive \
  --exclude "*" --include "*.csv"

Etapa 2 · Passo 2.2

EC2 dedicada para ClickHouse

Nova EC2 t3.medium. SG sg-clickhouse aceita 8123 só da SG do Streamlit e da Lambda. Senha do CH em Secrets Manager (criar antes da EC2 subir).

criar SG e secret antes da instância 
VPC_ID=$(aws ec2 describe-vpcs --filters Name=is-default,Values=true \
  --query 'Vpcs[0].VpcId' --output text)

SG_CH=$(aws ec2 create-security-group \
  --group-name sg-clickhouse \
  --description "ClickHouse - 8123 interno" \
  --vpc-id "$VPC_ID" --query 'GroupId' --output text)

# senha forte vai para o Secrets Manager
CH_PASSWORD=$(openssl rand -base64 24)
aws secretsmanager create-secret \
  --name olist/clickhouse \
  --secret-string "{\"password\":\"${CH_PASSWORD}\"}"
Etapa 2 · Passo 2.3 · user-data

userdata-clickhouse.sh — bootstrap autônomo

No boot, a EC2 pega a senha do Secrets Manager (via LabRole), instala o ClickHouse, cria o usuário dashboard e sobe o serviço.

userdata-clickhouse.sh 
#!/bin/bash
set -euxo pipefail
dnf install -y curl jq

curl https://clickhouse.com/ | sh
./clickhouse install --noninteractive

CH_PASSWORD=$(aws secretsmanager get-secret-value \
  --secret-id olist/clickhouse \
  --query SecretString --output text | jq -r .password)

cat > /etc/clickhouse-server/users.d/dashboard.xml <<EOF
<clickhouse>
  <users>
    <dashboard>
      <password>${CH_PASSWORD}</password>
      <networks><ip>10.0.0.0/8</ip></networks>
      <profile>default</profile>
    </dashboard>
  </users>
</clickhouse>
EOF

systemctl enable --now clickhouse-server
clickhouse-client -q "CREATE DATABASE IF NOT EXISTS olist_raw"
Etapa 2 · Passo 2.4

EC2 dedicada para Streamlit (systemd)

EC2 t3.small. SG sg-streamlit aceita 8501 da internet, 22 do IP do aluno. Conecta no CH usando IP privado e senha vinda do Secrets Manager. Roda como service systemd.

userdata-streamlit.sh 
#!/bin/bash
set -euxo pipefail
dnf install -y python3-pip git jq

cd /home/ec2-user
sudo -u ec2-user git clone \
  https://github.com/<aluno>/20261sre-primeironome-ultimonome.git app
cd app
pip3 install -r requirements.txt

CH_PASSWORD=$(aws secretsmanager get-secret-value \
  --secret-id olist/clickhouse \
  --query SecretString --output text | jq -r .password)

cat > .env <<EOF
CLICKHOUSE_HOST=<ip-privado-do-ch>
CLICKHOUSE_USER=dashboard
CLICKHOUSE_PASSWORD=${CH_PASSWORD}
EOF

cat > /etc/systemd/system/streamlit.service <<EOF
[Unit] Description=Streamlit dashboard
[Service]
WorkingDirectory=/home/ec2-user/app
EnvironmentFile=/home/ec2-user/app/.env
ExecStart=/usr/bin/streamlit run streamlit_app.py --server.address=0.0.0.0
Restart=always
User=ec2-user
[Install] WantedBy=multi-user.target
EOF
systemctl enable --now streamlit
Etapa 2 · Passo 2.5

Ingestor vira Lambda event-driven

PUT no S3 dispara a Lambda; ela lê o CSV, transforma em NDJSON e insere no ClickHouse via HTTP. Se o Lab limitar Lambda, fallback: cron systemd dentro de uma das EC2 chamando o mesmo script.

criar a Lambda + trigger S3 
# empacotar
mkdir -p build
pip install -r lambda/requirements.txt -t build/
cp lambda/lambda_handler.py build/
cd build && zip -r ../olist-ingestor.zip . && cd ..

# criar funcao
LAB_ROLE=$(aws iam get-role --role-name LabRole --query Role.Arn --output text)
aws lambda create-function \
  --function-name olist-ingestor \
  --runtime python3.11 \
  --role "$LAB_ROLE" \
  --handler lambda_handler.handler \
  --timeout 300 --memory-size 1024 \
  --zip-file fileb://olist-ingestor.zip \
  --environment "Variables={CLICKHOUSE_HOST=<ip-ch>,CLICKHOUSE_PASSWORD=$CH_PASSWORD}"

# permitir s3 chamar
aws lambda add-permission --function-name olist-ingestor \
  --action lambda:InvokeFunction --principal s3.amazonaws.com \
  --statement-id s3-trigger --source-arn "arn:aws:s3:::${BUCKET}"

# trigger S3:PutObject
aws s3api put-bucket-notification-configuration --bucket "$BUCKET" \
  --notification-configuration '{
    "LambdaFunctionConfigurations": [{
      "LambdaFunctionArn": "ARN_DA_LAMBDA",
      "Events": ["s3:ObjectCreated:*"],
      "Filter":{"Key":{"FilterRules":[{"Name":"suffix","Value":".csv"}]}}
    }]
  }'
Etapa 2 · Passo 2.6

Lambda handler — lambda_handler.py

Adaptação direta do ingest_minio.py da Aula 04: recebe event S3, lê o objeto, batches de 50k para o ClickHouse via clickhouse_connect.

lambda/lambda_handler.py 
import os, json, time, urllib.parse, boto3
import pandas as pd, clickhouse_connect

s3 = boto3.client("s3")
ch = clickhouse_connect.get_client(
    host=os.environ["CLICKHOUSE_HOST"],
    username="dashboard",
    password=os.environ["CLICKHOUSE_PASSWORD"],
)

def handler(event, context):
    unixtime = int(time.time())
    total = 0
    for rec in event["Records"]:
        bucket = rec["s3"]["bucket"]["name"]
        key = urllib.parse.unquote_plus(rec["s3"]["object"]["key"])
        obj = s3.get_object(Bucket=bucket, Key=key)
        df = pd.read_csv(obj["Body"], dtype=str).where(pd.notnull, None)
        rows = [(unixtime, json.dumps(r.to_dict()), key) for _, r in df.iterrows()]
        for i in range(0, len(rows), 50_000):
            ch.insert("olist_raw.ingestion", rows[i:i+50_000],
                      column_names=["unixtime","data","tag"])
        total += len(rows)
    return {"ok": True, "rows": total}
Etapa 2 · Passo 2.7

IAM e Security Groups encadeados

No Lab a role é LabRole. Os SGs viram a rede de cola entre as 3 pontas. Princípio: cada SG só aceita tráfego de quem precisa, na porta que precisa.

SG · ClickHouse

  • Inbound 8123 ← sg-streamlit.
  • Inbound 8123 ← sg-lambda.
  • Inbound 22 ← IP do aluno.
  • Outbound livre.

SG · Streamlit

  • Inbound 8501 ← 0.0.0.0/0.
  • Inbound 22 ← IP do aluno.
  • Outbound livre.
  • Permissão de chegar no sg-clickhouse.

IAM

  • Tudo via LabRole.
  • EC2 sobe com LabInstanceProfile.
  • Lambda usa LabRole também.
  • Secrets Manager: secretsmanager:GetSecretValue já vem na role.

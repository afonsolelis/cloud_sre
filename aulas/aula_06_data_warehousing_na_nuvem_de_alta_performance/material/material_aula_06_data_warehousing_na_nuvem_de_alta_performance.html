Migração para AWS Student Lab via Gemini CLI

Este material acompanha a Aula 06 da disciplina Cloud Computing e SRE — Visão Prática com AWS do MBA em Engenharia de Dados. A aula faz a primeira migração de verdade: a stack Olist — subida em Docker Compose na Aula 04 e analisada via ATAM + endurecida com táticas locais de Bass na Aula 05 — deixa o laptop e vai para a AWS. O caminho usa o AWS Academy Learner Lab como ambiente e o Gemini CLI como copiloto de provisionamento. A análise arquitetural já foi feita na Aula 05; hoje é execução: lift-and-shift primeiro, depois decomposição.

Pré-requisitos Conta AWS Academy ativa · Codespace ou laptop com AWS CLI v2 instalado · Node 18+ para instalar o Gemini CLI · repositório da Aula 04/05 clonado (fork pessoal) · chave SSH labsuser.pem baixada do Lab.

Restrições da aula Tudo via AWS CLI · AWS CLI é digitado pelo Gemini CLI (não pelo aluno direto) · revisão humana obrigatória antes de executar · nada de console clicado · só serviços liberados no Lab: EC2, S3, Lambda, IAM, Secrets Manager, SG, VPC default.

Repositório Mesmo repositório das Aulas 04/05. Hoje ele ganha os arquivos da infra: infra/userdata-*.sh, infra/sg.json, lambda/lambda_handler.py.

1. Recap das Aulas 04 e 05

A Aula 04 desceu do plano abstrato para o teclado: subimos uma stack inteira de ingestão near real-time no laptop, com MinIO no papel de S3, ClickHouse como warehouse colunar, um ingestor Python em Docker e um dashboard Streamlit consumindo as métricas. O dataset foi o Olist Brazilian E-commerce Public Dataset. O padrão de modelagem foi schema-on-read: tudo cai numa tabela landing olist_raw.ingestion(unixtime Int64, data JSON, tag LowCardinality(String)) e a modelagem vive em views. A idempotência vem de filtrar max(unixtime) por tag; reingestões nunca corrompem o estado.

A Aula 05 olhou para essa stack pela lente de arquitetura. Construiu uma análise ATAM (Architecture Tradeoff Analysis Method) do sistema completo — mapeando quality attributes, scenarios, decisões, sensitivity points, tradeoffs, risks e non-risks — e em seguida aplicou as táticas de Len Bass ainda no laptop: healthchecks robustos, depends_on: condition: service_healthy, lote ótimo do ingestor, cache no Streamlit, testes por componente, smoke tests pós-compose up e stress por ponto fraco. Ao final da Aula 05 a stack estava madura para o laptop e existia uma tabela ATAM concreta documentando o que sustentava o sistema e o que poderia quebrar.

A Aula 06 fecha esse ciclo movendo a mesma stack para a AWS. Não estamos refazendo o sistema nem construindo nova análise arquitetural: estamos portando o sistema e honrando a leitura arquitetural feita na Aula 05. As interfaces (S3 API, HTTP do ClickHouse, Streamlit em 8501) continuam as mesmas; muda quem responde a elas.

ATAM como insumo, não como conteúdo da Aula 06 A tabela ATAM construída na Aula 05 é o checklist arquitetural que vamos honrar nesta migração. Cada sensitivity point que apareceu lá — healthcheck do ClickHouse, single point of failure do ingestor, retry no boto3, latência da view sob carga, credenciais em texto plano — informa as decisões de provisionamento desta aula. A Aula 07 vai voltar à mesma tabela e estressar cada sensitivity point agora no contexto distribuído da AWS.
Continuidade narrativa Aula 04 = construir · Aula 05 = analisar (ATAM) e endurecer local (Bass) · Aula 06 = migrar (lift-and-shift + decomposição) · Aula 07 = estressar distribuído (Bass para sistemas distribuídos, atacando os sensitivity points da ATAM) · Aula 08 = projeto final integrado.

2. Tese da aula — migrar é executar a arquitetura planejada

A tentação ao olhar uma nuvem nova é abrir o console e começar a clicar. O custo dessa rota é alto: cada clique não deixa rastro reproduzível, cada erro custa hora-recurso de uma sessão de Lab que dura quatro horas, e o time inteiro fica preso na criatividade de quem domina o console primeiro. Vamos inverter a ordem: tudo via AWS CLI, e o AWS CLI é digitado pelo Gemini CLI — o aluno descreve a intenção em linguagem natural, revisa o comando que o agente gera, executa, valida.

Como a análise arquitetural já foi feita na Aula 05, hoje a aula é execução pura. A sequência é deliberada: primeiro um lift-and-shift que move o docker-compose inteiro para uma EC2 (continuidade primeiro), depois uma decomposição que separa cada serviço em recurso AWS canônico (benefícios do distribuído depois).

As três regras desta aula:

  1. Lift-and-shift antes de redesenhar. O docker-compose.yaml inteiro vai rodar dentro de uma única EC2 antes de virar serviços separados. Só depois decompomos.
  2. Console é proibido. Tudo via AWS CLI, e o AWS CLI é digitado pelo Gemini CLI. O aluno descreve a intenção em linguagem natural, revisa o comando que o agente gera, executa, valida.
  3. Honrar os sensitivity points da Aula 05. Cada decisão de provisionamento confere com o que a ATAM disse que poderia quebrar — credenciais saem do .env e vão para Secrets Manager, ingestor ganha capacidade reativa via Lambda, EC2 do ClickHouse fica numa SG fechada.

5. AWS Student Lab

5.1 O Learner Lab

O AWS Academy Learner Lab entrega contas AWS descartáveis para uso educacional. Cada aluno tem um sandbox com créditos limitados, uma role pré-configurada (LabRole) e um instance profile pré-configurado (LabInstanceProfile). A operação é simples e deliberadamente restrita:

  1. Acessar awsacademy.instructure.com e fazer login.
  2. Entrar no curso, abrir o módulo "Learner Lab".
  3. Clicar em Start Lab. Um indicador vermelho fica verde quando a sessão sobe (~1 minuto).
  4. Clicar em AWS Details → AWS CLI → Show. O painel mostra AWS_ACCESS_KEY_ID, AWS_SECRET_ACCESS_KEY e AWS_SESSION_TOKEN. Copie os três.
  5. Em AWS Details → SSH key, baixe labsuser.pem. Salve em ~/labsuser.pem e chmod 400.
  6. Region única: us-east-1. Não tente outra — vai negar permissão.

A sessão dura cerca de quatro horas. Quando expira, as credenciais param de funcionar com erro ExpiredToken. Renovar é só clicar em Start Lab novamente e re-exportar as novas três variáveis.

5.2 Restrições do Learner Lab

O Lab é generoso, mas opinativo. Cinco restrições afetam diretamente as escolhas desta aula:

  • Apenas us-east-1. Outras regiões retornam access denied. Configure AWS_DEFAULT_REGION=us-east-1.
  • Sem criação de IAM custom. Você não pode criar roles novas, mas tem a LabRole pronta com permissão em S3, Lambda, Secrets Manager, CloudWatch Logs e EC2. EC2 sobe com LabInstanceProfile — o instance profile que envelopa essa role.
  • VPC default. Não crie nova VPC. Use a default da region. Security Groups são livres para criar.
  • Lambda pode estar limitada. Em alguns períodos do semestre, a criação de Lambda no Lab fica limitada por quota. Por isso temos um fallback: ingestor vira cron systemd rodando dentro da EC2 do Streamlit, chamando aws s3 sync e o script Python a cada N minutos.
  • Serviços de DW e analytics gerenciados (Redshift, Glue, Athena) não estão liberados nesta aula. Atenha-se a EC2 + S3 + Lambda + IAM + Secrets Manager + SG.

6. Gemini CLI — o copiloto da aula

O Gemini CLI é um agente de linha de comando do Google, com free tier generoso (no momento, milhões de tokens/mês para contas pessoais) e contexto longo. Para a aula, ele substitui Claude Code / AIOX para os alunos que não têm assinatura paga. A instalação é via npm e a autenticação usa conta Google. 

# 1. Instalar
npm install -g @google/gemini-cli

# 2. Autenticar (abre navegador para login Google)
gemini auth

# 3. Confirmar que está funcionando
gemini "diga ola em uma linha"

O loop esperado da aula é simples:

  1. Prompt em linguagem natural: aluno descreve a intenção. "Gere um aws ec2 run-instances com t3.medium, AMI Amazon Linux 2023, SG abrindo 22 e 8501, instance profile LabInstanceProfile, key vockey, tag Name=olist-monolito, e user-data que instale docker."
  2. Geração: Gemini devolve o comando AWS CLI completo.
  3. Revisão: aluno lê o comando palavra por palavra. Confere flags destrutivas, region, recursos referenciados. Se algo soa estranho, pergunta o porquê.
  4. Execução: aluno cola no terminal e roda.
  5. Validação: Gemini pode ajudar a escrever a verificação ("escreva um aws ec2 describe-instances que liste só instâncias com tag Name=olist-monolito").
Regra de ouro Nunca rodar comando destrutivo sem revisão (terminate, delete, force). O agente é um copiloto, não o piloto.

7. Etapa 1 — Lift-and-shift

A primeira metade da prática (20h30 — ~21h15) move o compose inteiro para dentro de uma única EC2. Esta etapa não é uma "etapa descartável de aquecimento": ela é o caminho que minimiza downtime em uma migração real. Primeiro mover, depois melhorar. Sem ela, todo bug que aparecer na decomposição (Etapa 2) será confundido com bug do lift-and-shift.

7.1 Passo 1.1 — renovar credenciais do Student Lab

No painel do Lab, em AWS Details > AWS CLI > Show, copie as três variáveis e exporte:

export AWS_SECRET_ACCESS_KEY="..."
export AWS_SESSION_TOKEN="..."

7.2 Passo 1.2 — criar SG e provisionar a EC2 monolito via Gemini CLI

Primeiro o Security Group, depois a instância. Prompt para o Gemini:

1. crie um security group chamado sg-olist-monolito na VPC default, com regras de inbound:
   - 22 TCP de 0.0.0.0/0

2. gere o comando aws ec2 run-instances que:
   - usa AMI Amazon Linux 2023 mais recente em us-east-1

Comando esperado para criar o SG:

  --query 'Vpcs[0].VpcId' --output text)

SG_ID=$(aws ec2 create-security-group \
  --group-name sg-olist-monolito \
  --description "Olist monolito - 22, 8501" \
  --vpc-id "$VPC_ID" \
  --query 'GroupId' --output text)

aws ec2 authorize-security-group-ingress --group-id "$SG_ID" \
  --protocol tcp --port 22   --cidr 0.0.0.0/0
aws ec2 authorize-security-group-ingress --group-id "$SG_ID" \
  --protocol tcp --port 8501 --cidr 0.0.0.0/0

Comando esperado para subir a EC2:

aws ec2 run-instances \
  --image-id $(aws ec2 describe-images \
    --owners amazon \
    --filters "Name=name,Values=al2023-ami-2023.*-x86_64" \
    --query 'sort_by(Images, &CreationDate)[-1].ImageId' --output text) \
  --instance-type t3.medium \
  --key-name vockey \
  --iam-instance-profile Name=LabInstanceProfile \
  --security-group-ids "$SG_ID" \
  --user-data file://userdata-monolito.sh \
  --tag-specifications 'ResourceType=instance,Tags=[{Key=Name,Value=olist-monolito}]'

8. Etapa 2 — Decomposição

A segunda metade da prática (~21h15 — 22h00) tira a stack do monolito e distribui cada serviço em recurso AWS dedicado. As interfaces continuam as mesmas (S3 API, HTTP/SQL do ClickHouse, HTTP do Streamlit) — muda quem responde a elas. A ordem dos passos é importante: ClickHouse precisa existir antes do Streamlit; o S3 precisa existir antes da Lambda.

8.1 Passo 2.1 — S3 substitui MinIO

BUCKET="olist-${USER}-raw"
aws s3 mb "s3://${BUCKET}" --region us-east-1

# Sobe os CSVs da pasta data/

8.5 Passo 2.5 — ingestor vira Lambda event-driven

O lambda_handler.py é uma adaptação direta do ingest_minio.py da Aula 04. As diferenças: (1) recebe events S3 em vez de listar bucket inteiro; (2) credenciais herdam da role da Lambda; (3) host do ClickHouse e senha vêm de variáveis de ambiente da função.

Fallback se Lambda estiver bloqueada no Lab: rodar o mesmo handler como cron systemd dentro da EC2 do Streamlit, fazendo aws s3 sync e processando arquivos novos a cada 60s. O código do handler é exatamente o mesmo — só muda como ele é invocado.

8.6 Passo 2.6 — Lambda handler Python

import os, json, time, urllib.parse, boto3
import pandas as pd, clickhouse_connect

s3 = boto3.client("s3")
ch = clickhouse_connect.get_client(
    host=os.environ["CLICKHOUSE_HOST"],
    username="dashboard",
    password=os.environ["CLICKHOUSE_PASSWORD"],
)

def handler(event, context):
    unixtime = int(time.time())
    total = 0
    for rec in event["Records"]:
        bucket = rec["s3"]["bucket"]["name"]
        key = urllib.parse.unquote_plus(rec["s3"]["object"]["key"])
        obj = s3.get_object(Bucket=bucket, Key=key)
        df = pd.read_csv(obj["Body"], dtype=str).where(pd.notnull, None)
        rows = [(unixtime, json.dumps(r.to_dict()), key) for _, r in df.iterrows()]
        for i in range(0, len(rows), 50_000):
            ch.insert("olist_raw.ingestion", rows[i:i+50_000],
                      column_names=["unixtime", "data", "tag"])
        total += len(rows)
    return {"ok": True, "rows": total}

Empacotamento e deploy: 

# 1. Empacotar com dependencias
mkdir -p build
pip install -r lambda/requirements.txt -t build/
cp lambda/lambda_handler.py build/
cd build && zip -r ../olist-ingestor.zip . && cd ..

# 2. Criar a funcao (LabRole é a role pré-aprovada do Lab)
LAB_ROLE=$(aws iam get-role --role-name LabRole --query Role.Arn --output text)

aws lambda create-function \
  --function-name olist-ingestor \
  --runtime python3.11 \
  --role "$LAB_ROLE" \
  --handler lambda_handler.handler \
  --timeout 300 \
  --memory-size 1024 \
  --zip-file fileb://olist-ingestor.zip \
  --environment "Variables={CLICKHOUSE_HOST=<ip-privado-ch>,CLICKHOUSE_PASSWORD=$CH_PASSWORD}"

# 3. Conectar trigger S3 -> Lambda
aws lambda add-permission \
  --function-name olist-ingestor \
  --action lambda:InvokeFunction \
  --principal s3.amazonaws.com \
  --statement-id s3-trigger \
  --source-arn "arn:aws:s3:::${BUCKET}"

aws s3api put-bucket-notification-configuration \
  --bucket "$BUCKET" \
  --notification-configuration '{
    "LambdaFunctionConfigurations": [{
      "LambdaFunctionArn": "'"$(aws lambda get-function --function-name olist-ingestor --query Configuration.FunctionArn --output text)"'",
      "Events": ["s3:ObjectCreated:*"],
      "Filter": {"Key": {"FilterRules": [{"Name": "suffix", "Value": ".csv"}]}}
    }]
  }'

8.7 Passo 2.7 — IAM e Security Groups encadeados

A regra é simples: cada SG aceita só o que precisa, na porta que precisa, vindo de quem precisa. No Lab, IAM é toda LabRole.

RecursoSG / RoleRegra
EC2 ClickHousesg-clickhouse + LabInstanceProfileInbound 8123 ← sg-streamlit e sg-lambda; 22 ← IP do aluno; outbound livre.
EC2 Streamlitsg-streamlit + LabInstanceProfileInbound 8501 ← 0.0.0.0/0; 22 ← IP do aluno; outbound livre.
Lambda ingestorLabRoleAcesso S3 e Secrets Manager via role; rede default da Lambda (sem VPC) ou attach numa subnet privada que enxergue o CH.
S3 bucketBucket policy defaultSem acesso público; somente LabRole acessa.

8.8 Passo 2.8 — validação end-to-end

Suba um CSV novo no S3. A Lambda dispara automaticamente; logs vão para CloudWatch. ClickHouse ganha linhas. Dashboard atualiza. 

# 1. Subir um CSV novo no S3
aws s3 cp data/olist_orders_dataset.csv \
  "s3://${BUCKET}/olist/olist_orders_dataset.csv"

# 2. Lambda dispara automaticamente (logs no CloudWatch)
aws logs tail /aws/lambda/olist-ingestor --since 2m

# 3. Conferir no ClickHouse (via Streamlit EC2 que ja sabe falar com o CH)
ssh -i ~/labsuser.pem ec2-user@$ST_EC2_IP \
  "curl -s http://<ch-private>:8123/?query=SELECT+count()+FROM+olist_raw.ingestion"

# 4. Abrir o dashboard
echo "http://${ST_EC2_IP}:8501"

Critério de pronto da Etapa 2: upload de CSV no S3 popula o dashboard via Streamlit sem nenhuma intervenção manual.

Migração para AWS Student Lab via Gemini CLI

Este material acompanha a Aula 06 da disciplina Cloud Computing e SRE — Visão Prática com AWS, integrante do MBA em Engenharia de Dados. A aula conduz a primeira migração efetiva do curso: a stack Olist, originalmente provisionada em Docker Compose na Aula 04 e analisada por meio do método ATAM com posterior aplicação das táticas locais de Bass na Aula 05, é portada do ambiente local para a AWS. Adota-se o AWS Academy Learner Lab como ambiente de execução e o Gemini CLI como assistente para o provisionamento via AWS CLI. Considerando que a análise arquitetural foi previamente conduzida na Aula 05, esta aula concentra-se na execução, contemplando a etapa de lift-and-shift, seguida pela decomposição.

Pré-requisitos Conta AWS Academy ativa; Codespace ou laptop com AWS CLI v2 instalado; Node 18 ou superior para a instalação do Gemini CLI; repositório das Aulas 04 e 05 clonado em fork pessoal; chave SSH labsuser.pem obtida no Lab.

Restrições da aula Todas as operações são realizadas via AWS CLI, sendo os comandos gerados pelo Gemini CLI, e não digitados diretamente pelo aluno. Exige-se revisão humana antes da execução. Não se utiliza o console gráfico. Apenas os serviços liberados no Lab são empregados: EC2, S3, Lambda, IAM, Secrets Manager, Security Groups e VPC default.

Repositório Utiliza-se o mesmo repositório das Aulas 04 e 05, agora acrescido dos arquivos da infraestrutura: infra/userdata-*.sh, infra/sg.json e lambda/lambda_handler.py.

1. Recap das Aulas 04 e 05

A Aula 04 conduziu a passagem do plano conceitual à implementação prática: provisionou-se uma stack completa de ingestão near real-time no ambiente local, na qual o MinIO desempenhou o papel de S3, o ClickHouse operou como warehouse colunar, um ingestor Python foi executado em Docker e um dashboard Streamlit realizou o consumo das métricas. O conjunto de dados utilizado foi o Olist Brazilian E-commerce Public Dataset. O padrão de modelagem adotado foi o schema-on-read: todos os dados são depositados em uma tabela de landing, olist_raw.ingestion(unixtime Int64, data JSON, tag LowCardinality(String)), e a modelagem é mantida em views. A garantia de idempotência decorre da filtragem por max(unixtime) agrupada por tag, o que impede que reingestões corrompam o estado da base.

A Aula 05 examinou essa stack sob a perspectiva da arquitetura. Foi elaborada uma análise pelo método ATAM (Architecture Tradeoff Analysis Method) do sistema integral, mapeando-se os quality attributes, scenarios, decisões arquiteturais, sensitivity points, tradeoffs, risks e non-risks. Em seguida, foram aplicadas as táticas de Len Bass ainda em ambiente local: healthchecks robustos, depends_on: condition: service_healthy, dimensionamento adequado dos lotes do ingestor, cache no Streamlit, testes por componente, smoke tests após compose up e ensaios de stress sobre os pontos identificados como críticos. Ao final da Aula 05, a stack encontrava-se preparada para o ambiente local e dispunha de uma tabela ATAM documentando, de forma concreta, os elementos de sustentação e os pontos de fragilidade do sistema.

A Aula 06 conclui esse ciclo ao portar a referida stack para a AWS. Não se trata de uma reconstrução do sistema nem da elaboração de uma nova análise arquitetural; o objetivo é portar o sistema, em conformidade com a leitura arquitetural conduzida na Aula 05. As interfaces (S3 API, HTTP do ClickHouse e Streamlit na porta 8501) são mantidas; altera-se apenas o provedor que responde por elas.

ATAM como insumo, não como conteúdo da Aula 06 A tabela ATAM elaborada na Aula 05 constitui o roteiro arquitetural a ser observado nesta migração. Cada sensitivity point identificado naquela ocasião — healthcheck do ClickHouse, single point of failure do ingestor, retry no boto3, latência da view sob carga e credenciais armazenadas em texto plano — informa as decisões de provisionamento desta aula. A Aula 07 retomará a mesma tabela e submeterá cada sensitivity point a ensaios no contexto distribuído da AWS.
Continuidade narrativa Aula 04: construção da stack. Aula 05: análise arquitetural (ATAM) e endurecimento local (Bass). Aula 06: migração (lift-and-shift e decomposição). Aula 07: ensaios de stress no ambiente distribuído (aplicação das táticas de Bass para sistemas distribuídos, com foco nos sensitivity points da ATAM). Aula 08: projeto final integrado.

2. Tese da aula — a migração como execução da arquitetura planejada

Diante de um ambiente em nuvem desconhecido, é comum recorrer ao console gráfico como ponto de partida. Tal abordagem apresenta custos elevados: cada interação manual não produz rastro reproduzível, cada erro consome tempo de uma sessão de Lab limitada a quatro horas, e o ritmo de trabalho da turma fica condicionado à fluência individual de quem opera o console em primeiro lugar. Esta aula adota o caminho inverso: todas as operações são realizadas via AWS CLI, sendo os comandos AWS CLI gerados pelo Gemini CLI. O aluno descreve a intenção em linguagem natural, revisa o comando produzido pelo agente, executa o procedimento e valida o resultado.

Como a análise arquitetural foi conduzida na Aula 05, esta aula concentra-se na execução. A sequência adotada é deliberada: realiza-se primeiramente um lift-and-shift, que transfere o docker-compose integral para uma instância EC2, priorizando-se a continuidade operacional. Em seguida, conduz-se a decomposição, que aloca cada serviço em um recurso AWS apropriado a sua responsabilidade, com vistas à obtenção dos benefícios próprios das arquiteturas distribuídas.

Os três princípios orientadores desta aula são apresentados a seguir:

  1. Lift-and-shift antes do redesenho. O docker-compose.yaml integral é executado dentro de uma única EC2 antes de ser decomposto em serviços individuais. A decomposição ocorre apenas em momento subsequente.
  2. Restrição ao uso do console. Todas as operações são conduzidas via AWS CLI, sendo os comandos gerados pelo Gemini CLI. O aluno descreve a intenção em linguagem natural, revisa o comando proposto pelo agente, executa o procedimento e valida o resultado.
  3. Observância dos sensitivity points da Aula 05. Cada decisão de provisionamento é avaliada à luz dos pontos de sensibilidade identificados na ATAM: as credenciais são retiradas do .env e armazenadas no Secrets Manager; o ingestor ganha capacidade reativa por meio da Lambda; e a EC2 do ClickHouse é mantida em um Security Group fechado.

5. AWS Student Lab

5.1 O Learner Lab

O AWS Academy Learner Lab disponibiliza contas AWS descartáveis para uso educacional. Cada aluno dispõe de um sandbox com créditos limitados, uma role pré-configurada (LabRole) e um instance profile pré-configurado (LabInstanceProfile). A operação é simples e deliberadamente restrita, conforme detalhado a seguir:

  1. Acessar awsacademy.instructure.com e efetuar o login.
  2. Acessar o curso e abrir o módulo "Learner Lab".
  3. Acionar o botão Start Lab. Um indicador vermelho passa à coloração verde quando a sessão é estabelecida, em aproximadamente um minuto.
  4. Acessar AWS Details → AWS CLI → Show. O painel exibe as variáveis AWS_ACCESS_KEY_ID, AWS_SECRET_ACCESS_KEY e AWS_SESSION_TOKEN, que devem ser copiadas.
  5. Em AWS Details → SSH key, efetuar o download do arquivo labsuser.pem, salvá-lo em ~/labsuser.pem e aplicar chmod 400.
  6. A única região disponível é us-east-1. Tentativas em outras regiões resultam em negação de permissão.

A sessão tem duração aproximada de quatro horas. Quando expira, as credenciais deixam de operar e retornam o erro ExpiredToken. A renovação consiste em acionar novamente o Start Lab e reexportar as três variáveis atualizadas.

5.2 Restrições do Learner Lab

O ambiente, embora amplo em recursos, apresenta limitações específicas. Cinco restrições impactam diretamente as escolhas desta aula:

  • Apenas us-east-1. Outras regiões retornam access denied. Recomenda-se configurar AWS_DEFAULT_REGION=us-east-1.
  • Não há criação de IAM personalizada. Não é permitido criar novas roles, porém a LabRole está pré-configurada com permissões para S3, Lambda, Secrets Manager, CloudWatch Logs e EC2. As EC2 são instanciadas com LabInstanceProfile, o instance profile associado a essa role.
  • Utilização da VPC default. Não se deve criar uma nova VPC. Emprega-se a VPC default da região. A criação de Security Groups é livre.
  • Lambda pode encontrar-se limitada. Em determinados períodos do semestre, a criação de Lambda no Lab é restringida por quota. Por esse motivo, prevê-se um fallback: o ingestor passa a operar como cron systemd hospedado na EC2 do Streamlit, invocando aws s3 sync e o script Python em intervalos regulares.
  • Os serviços gerenciados de data warehouse e analytics (Redshift, Glue, Athena) não estão habilitados nesta aula. Restringem-se as operações ao conjunto EC2, S3, Lambda, IAM, Secrets Manager e Security Groups.

6. Gemini CLI — assistente para o provisionamento

O Gemini CLI é um agente de linha de comando do Google, que dispõe de free tier com cota generosa (no momento, da ordem de milhões de tokens mensais para contas pessoais) e janela de contexto extensa. Para os propósitos desta aula, configura-se como alternativa ao Claude Code e ao AIOX para os alunos que não possuem assinatura paga. A instalação é realizada via npm e a autenticação utiliza conta Google. 

# 1. Instalar
npm install -g @google/gemini-cli

# 2. Autenticar (abre navegador para login Google)
gemini auth

# 3. Confirmar que está funcionando
gemini "diga ola em uma linha"

O fluxo previsto para a aula compreende as etapas descritas a seguir:

  1. Formulação do prompt em linguagem natural: o aluno descreve a intenção. Por exemplo: "Gere um aws ec2 run-instances com t3.medium, AMI Amazon Linux 2023, Security Group abrindo as portas 22 e 8501, instance profile LabInstanceProfile, key vockey, tag Name=olist-monolito e user-data para instalação do docker."
  2. Geração: o Gemini produz o comando AWS CLI integral.
  3. Revisão: o aluno lê o comando integralmente, verifica eventuais flags destrutivas, a região e os recursos referenciados. Diante de qualquer inconsistência aparente, recomenda-se solicitar esclarecimento.
  4. Execução: o aluno transcreve o comando no terminal e procede à execução.
  5. Validação: o Gemini pode auxiliar na elaboração da verificação (por exemplo, "escreva um aws ec2 describe-instances que liste apenas instâncias com tag Name=olist-monolito").
Princípio fundamental Nenhum comando destrutivo (terminate, delete, force) deve ser executado sem revisão prévia. O agente atua como assistente; a responsabilidade pela execução permanece com o operador humano.

7. Etapa 1 — Lift-and-shift

A primeira parte da etapa prática (compreendida entre 20h30 e, aproximadamente, 21h15) transfere o compose integral para o interior de uma única EC2. Não se trata de uma etapa preparatória descartável: configura-se como o caminho que minimiza o tempo de indisponibilidade em uma migração real. Prioriza-se a transferência operacional e, em momento subsequente, conduzem-se as otimizações. Caso esta etapa fosse omitida, qualquer falha emergente na decomposição (Etapa 2) poderia ser confundida com defeito oriundo do lift-and-shift.

7.1 Passo 1.1 — renovação das credenciais do Student Lab

No painel do Lab, em AWS Details > AWS CLI > Show, copiam-se as três variáveis e procede-se à exportação no terminal: 

export AWS_ACCESS_KEY_ID="ASIA..."
export AWS_SECRET_ACCESS_KEY="..."
export AWS_SESSION_TOKEN="..."
export AWS_DEFAULT_REGION="us-east-1"

aws sts get-caller-identity
# Esperado:
# {
#   "UserId": "AROA.../user...",
#   "Account": "012345678901",
#   "Arn": "arn:aws:sts::012345678901:assumed-role/voclabs/..."
# }

Caso ocorra retorno com erro ExpiredToken durante a aula, recomenda-se retornar a esta etapa e renovar as credenciais. Não se deve recorrer ao aws configure: o Lab utiliza session token e o comando aws configure não armazena AWS_SESSION_TOKEN de maneira adequada.

7.2 Passo 1.2 — criação do Security Group e provisionamento da EC2 monolito via Gemini CLI

Procede-se primeiramente à criação do Security Group e, em seguida, ao provisionamento da instância. Prompt sugerido para o Gemini: 

gemini "gere um par de comandos aws cli:
1. crie um security group chamado sg-olist-monolito na VPC default, com regras de inbound:
   - 22 TCP de 0.0.0.0/0
   - 8501 TCP de 0.0.0.0/0
2. gere o comando aws ec2 run-instances que:
   - usa AMI Amazon Linux 2023 mais recente em us-east-1
   - t3.medium
   - instance profile LabInstanceProfile
   - security group: sg-olist-monolito
   - user-data: file://userdata-monolito.sh
   - key pair: vockey
   - tag Name=olist-monolito"

Comando esperado para a criação do Security Group: 

VPC_ID=$(aws ec2 describe-vpcs --filters Name=is-default,Values=true \
  --query 'Vpcs[0].VpcId' --output text)

SG_ID=$(aws ec2 create-security-group \
  --group-name sg-olist-monolito \
  --description "Olist monolito - 22, 8501" \
  --vpc-id "$VPC_ID" \
  --query 'GroupId' --output text)

aws ec2 authorize-security-group-ingress --group-id "$SG_ID" \
  --protocol tcp --port 22   --cidr 0.0.0.0/0
aws ec2 authorize-security-group-ingress --group-id "$SG_ID" \
  --protocol tcp --port 8501 --cidr 0.0.0.0/0

Comando esperado para o provisionamento da EC2:

aws ec2 run-instances \
  --image-id $(aws ec2 describe-images \
    --owners amazon \
    --filters "Name=name,Values=al2023-ami-2023.*-x86_64" \
    --query 'sort_by(Images, &CreationDate)[-1].ImageId' --output text) \
  --instance-type t3.medium \
  --key-name vockey \
  --iam-instance-profile Name=LabInstanceProfile \
  --security-group-ids "$SG_ID" \
  --user-data file://userdata-monolito.sh \
  --tag-specifications 'ResourceType=instance,Tags=[{Key=Name,Value=olist-monolito}]'

7.3 Passo 1.3 — user-data da EC2 monolito

O arquivo infra/userdata-monolito.sh realiza o bootstrap de forma autônoma: instala o Docker, habilita o serviço, adiciona o usuário ec2-user ao grupo correspondente, instala o plugin Compose v2 e efetua o clone do repositório. 

#!/bin/bash
set -euxo pipefail

dnf update -y
dnf install -y docker git
systemctl enable --now docker
usermod -aG docker ec2-user

# Plugin compose v2 oficial
mkdir -p /usr/local/lib/docker/cli-plugins
curl -SL "https://github.com/docker/compose/releases/latest/download/docker-compose-linux-x86_64" \
  -o /usr/local/lib/docker/cli-plugins/docker-compose
chmod +x /usr/local/lib/docker/cli-plugins/docker-compose

# Repo da aula
cd /home/ec2-user
sudo -u ec2-user git clone https://github.com/<aluno>/20261sre-primeironome-ultimonome.git
chown -R ec2-user:ec2-user /home/ec2-user/20261sre-primeironome-ultimonome

Observa-se que esse user-data é integralmente autônomo: assim que o acesso SSH for disponibilizado, a stack já se encontrará pronta para a execução de compose up. Recomenda-se substituir <aluno> pelo respectivo usuário GitHub antes do provisionamento da instância.

7.5 Passo 1.5 — população do MinIO, execução do ingestor e validação do dashboard

Acessa-se o console MinIO em http://<ec2-public-ip>:9003 com o login minioadmin / minioadmin. Cria-se o bucket olist e realiza-se o upload dos arquivos CSV armazenados na pasta data/. Em seguida, no interior da EC2, executam-se os comandos a seguir: 

docker exec -i olist_analytics clickhouse-client < sql/raw/01_schema.sql
docker exec olist_ingestor python /app/ingest_minio.py

Critério de conclusão da Etapa 1: ao acessar http://<ec2-public-ip>:8501 no navegador, o dashboard de top produtos deve estar renderizado adequadamente. Realiza-se a captura de tela e seu armazenamento no fork do repositório. Somente após a confirmação procede-se ao avanço para a Etapa 2.

8. Etapa 2 — Decomposição

A segunda parte da etapa prática (compreendida entre, aproximadamente, 21h15 e 22h00) desfaz a configuração monolítica e distribui cada serviço em um recurso AWS dedicado. As interfaces são preservadas (S3 API, HTTP e SQL do ClickHouse e HTTP do Streamlit); altera-se apenas o provedor que responde por elas. A ordem de execução dos passos é relevante: o ClickHouse deve existir antes do Streamlit, e o S3 deve existir antes da Lambda.

8.1 Passo 2.1 — substituição do MinIO pelo S3

BUCKET="olist-${USER}-raw"
aws s3 mb "s3://${BUCKET}" --region us-east-1

# Sobe os CSVs da pasta data/
aws s3 cp data/ "s3://${BUCKET}/olist/" --recursive \
  --exclude "*" --include "*.csv"

aws s3 ls "s3://${BUCKET}/olist/"

No código do ingestor, recomenda-se a remoção do parâmetro endpoint_url e das variáveis aws_access_key_id e aws_secret_access_key. O boto3 herda as credenciais do instance profile da EC2 ou da execution role da Lambda. O nome do bucket passa a ser a única variável a ser fornecida via env. Esta substituição também endereça o sensitivity point relativo a "credenciais em texto plano" identificado na ATAM da Aula 05.

8.2 Passo 2.2 — EC2 dedicada ao ClickHouse

Previamente ao provisionamento da EC2, cria-se o secret com a senha do ClickHouse: 

# Gerar senha forte e armazenar
CH_PASSWORD=$(openssl rand -base64 24)
aws secretsmanager create-secret \
  --name olist/clickhouse \
  --secret-string "{\"password\":\"${CH_PASSWORD}\"}"

Security Group do ClickHouse: autoriza tráfego na porta 8123 unicamente a partir dos Security Groups internos correspondentes ao Streamlit e à Lambda. O acesso SSH é restrito ao IP do aluno. 

SG_CH=$(aws ec2 create-security-group \
  --group-name sg-clickhouse \
  --description "ClickHouse - 8123 interno" \
  --vpc-id "$VPC_ID" --query 'GroupId' --output text)

# SSH apenas do meu IP
MY_IP=$(curl -s https://checkip.amazonaws.com)
aws ec2 authorize-security-group-ingress --group-id "$SG_CH" \
  --protocol tcp --port 22 --cidr "${MY_IP}/32"

8.3 Passo 2.3 — user-data da EC2 do ClickHouse

O arquivo infra/userdata-clickhouse.sh contém o script de bootstrap. A EC2 obtém a senha junto ao Secrets Manager por meio da LabRole, instala o ClickHouse, cria o usuário dashboard com escopo restrito à VPC e aplica o schema inicial. 

#!/bin/bash
set -euxo pipefail
dnf install -y curl jq

curl https://clickhouse.com/ | sh
./clickhouse install --noninteractive

CH_PASSWORD=$(aws secretsmanager get-secret-value \
  --secret-id olist/clickhouse \
  --query SecretString --output text | jq -r .password)

cat > /etc/clickhouse-server/users.d/dashboard.xml <<EOF
<clickhouse>
  <users>
    <dashboard>
      <password>${CH_PASSWORD}</password>
      <networks><ip>10.0.0.0/8</ip></networks>
      <profile>default</profile>
    </dashboard>
  </users>
</clickhouse>
EOF

systemctl enable --now clickhouse-server

# Bootstrap do schema e da view (mesmo SQL da Aula 04)
clickhouse-client -q "CREATE DATABASE IF NOT EXISTS olist_raw"
clickhouse-client --multiquery < /home/ec2-user/app/sql/raw/01_schema.sql
clickhouse-client --multiquery < /home/ec2-user/app/sql/views/01_top_products.sql

8.4 Passo 2.4 — EC2 dedicada ao Streamlit

Security Group do Streamlit: a porta 8501 é aberta para a internet, a porta 22 é restrita ao IP do aluno e o tráfego de saída permanece livre. Autoriza-se o acesso de sg-streamlit a sg-clickhouse: 

aws ec2 authorize-security-group-ingress --group-id "$SG_CH" \
  --protocol tcp --port 8123 --source-group "$SG_ST"

User-data do Streamlit, definido em infra/userdata-streamlit.sh. O serviço é executado como unidade systemd, com persistência após reboots. 

#!/bin/bash
set -euxo pipefail
dnf install -y python3-pip git jq

cd /home/ec2-user
sudo -u ec2-user git clone \
  https://github.com/<aluno>/20261sre-primeironome-ultimonome.git app
cd app
pip3 install -r requirements.txt

CH_PASSWORD=$(aws secretsmanager get-secret-value \
  --secret-id olist/clickhouse \
  --query SecretString --output text | jq -r .password)

CLICKHOUSE_HOST=<ip-privado-do-ch>
CLICKHOUSE_USER=dashboard

# rodar como service systemd para sobreviver a reboot
cat > /etc/systemd/system/streamlit.service <<EOF
[Unit]

After=network-online.target

[Service]

EnvironmentFile=/home/ec2-user/app/.env
ExecStart=/usr/bin/streamlit run streamlit_app.py --server.address=0.0.0.0
Restart=always
User=ec2-user

[Install]
WantedBy=multi-user.target
EOF

systemctl daemon-reload
systemctl enable --now streamlit

8.5 Passo 2.5 — conversão do ingestor em Lambda event-driven

O lambda_handler.py constitui uma adaptação direta do ingest_minio.py apresentado na Aula 04. As diferenças são: (1) o handler recebe events S3, em substituição à listagem integral do bucket; (2) as credenciais são herdadas da role da Lambda; e (3) o host do ClickHouse e a respectiva senha são fornecidos por variáveis de ambiente da função.

Procedimento de fallback caso a Lambda esteja bloqueada no Lab: executar o mesmo handler como cron systemd hospedado na EC2 do Streamlit, realizando aws s3 sync e processando arquivos novos em intervalos de 60 segundos. O código do handler é mantido inalterado; modifica-se apenas o mecanismo de invocação.

8.6 Passo 2.6 — Lambda handler Python

import os, json, time, urllib.parse, boto3
import pandas as pd, clickhouse_connect

s3 = boto3.client("s3")
ch = clickhouse_connect.get_client(
    host=os.environ["CLICKHOUSE_HOST"],
    username="dashboard",
    password=os.environ["CLICKHOUSE_PASSWORD"],
)

def handler(event, context):
    unixtime = int(time.time())
    total = 0
    for rec in event["Records"]:
        bucket = rec["s3"]["bucket"]["name"]
        key = urllib.parse.unquote_plus(rec["s3"]["object"]["key"])
        obj = s3.get_object(Bucket=bucket, Key=key)
        df = pd.read_csv(obj["Body"], dtype=str).where(pd.notnull, None)
        rows = [(unixtime, json.dumps(r.to_dict()), key) for _, r in df.iterrows()]
        for i in range(0, len(rows), 50_000):
            ch.insert("olist_raw.ingestion", rows[i:i+50_000],
                      column_names=["unixtime", "data", "tag"])
        total += len(rows)
    return {"ok": True, "rows": total}

Empacotamento e deploy da função: 

# 1. Empacotar com dependencias
mkdir -p build
pip install -r lambda/requirements.txt -t build/
cp lambda/lambda_handler.py build/
cd build && zip -r ../olist-ingestor.zip . && cd ..

# 2. Criar a funcao (LabRole é a role pré-aprovada do Lab)
LAB_ROLE=$(aws iam get-role --role-name LabRole --query Role.Arn --output text)

aws lambda create-function \
  --function-name olist-ingestor \
  --runtime python3.11 \
  --role "$LAB_ROLE" \
  --handler lambda_handler.handler \
  --timeout 300 \
  --memory-size 1024 \
  --zip-file fileb://olist-ingestor.zip \
  --environment "Variables={CLICKHOUSE_HOST=<ip-privado-ch>,CLICKHOUSE_PASSWORD=$CH_PASSWORD}"

# 3. Conectar trigger S3 -> Lambda
aws lambda add-permission \
  --function-name olist-ingestor \
  --action lambda:InvokeFunction \
  --principal s3.amazonaws.com \
  --statement-id s3-trigger \
  --source-arn "arn:aws:s3:::${BUCKET}"

aws s3api put-bucket-notification-configuration \
  --bucket "$BUCKET" \
  --notification-configuration '{
    "LambdaFunctionConfigurations": [{
      "LambdaFunctionArn": "'"$(aws lambda get-function --function-name olist-ingestor --query Configuration.FunctionArn --output text)"'",
      "Events": ["s3:ObjectCreated:*"],
      "Filter": {"Key": {"FilterRules": [{"Name": "suffix", "Value": ".csv"}]}}
    }]
  }'

8.7 Passo 2.7 — IAM e Security Groups encadeados

O princípio orientador é simples: cada Security Group autoriza apenas o tráfego estritamente necessário, na porta correspondente, proveniente da origem autorizada. No Lab, a totalidade das operações IAM utiliza a LabRole.

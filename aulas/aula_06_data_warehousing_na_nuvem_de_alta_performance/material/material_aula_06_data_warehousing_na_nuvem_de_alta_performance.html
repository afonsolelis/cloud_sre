Em SRE, dois conceitos sustentam a aula. O primeiro é paridade de ambiente: o teste só vale se o que ele exercita se parece com o que vai rodar em produção. O segundo é toil: trabalho manual repetitivo, sem valor duradouro, que escala linearmente com a operação. Código de baixa qualidade vira toil — cada bug fix custa horas, cada PR pequeno trava em revisão. Medir qualidade é o primeiro passo para tirar toil da conta.

2. Testcontainers — infra real, descartável e controlada pelo teste

Testcontainers é uma família de bibliotecas (Java original em 2015, depois Python, Go, .NET, Node) que coloca a suíte de teste no comando do Docker. Em Python usamos testcontainers-python: o teste invoca PostgresContainer("postgres:16-alpine").start(), recebe uma URI dinâmica com porta efêmera, executa o caso, e ao final da fixture o container é destruído. Volume some, dado some, ambiente volta ao zero. Isso ataca duas dores clássicas do mock: discrepância semântica (o mock não conhece todos os edge cases do dialeto) e drift (o mock envelhece, a produção não).

2.1 Quando vale e quando não vale

Mock continua certo na base da pirâmide — testes unitários puros, lógica sem efeito colateral. Testcontainer entra na faixa de integração: SQL real, transações reais, drivers reais. A regra de bolso é: se o erro só aparece com infra real, o teste tem que ter infra real.

CritérioMockTestcontainer
Tempo por teste1–5 ms200 ms–2 s (1ª subida)
Paridade com produçãoBaixaAlta
Pega bug de driver/dialetoNãoSim
Dependência de DockerNenhumaObrigatória no runner
Custo de manutençãoAlto (mock vira ficção)Médio
Faixa adequadaUnit, lógica puraIntegration, contract com infra

2.2 Padrões de fixture — session vs function

O escopo da fixture é o trade-off central. Container por teste é seguro mas explode o tempo total — cada fixture function-scoped paga 200–500 ms só para subir. Container compartilhado em session é 10× mais rápido, mas exige reset entre testes para evitar acoplamento por estado. A receita do curso: escopo session + TRUNCATE ou ROLLBACK automatizado em fixture function-scoped. 

# tests/integration/conftest.py
import pytest
from testcontainers.postgres import PostgresContainer
from testcontainers.localstack import LocalStackContainer

@pytest.fixture(scope="session")
def pg():
    with PostgresContainer("postgres:16-alpine") as c:
        yield c

@pytest.fixture(scope="session")
def aws():
    with LocalStackContainer(image="localstack/localstack:3").with_services(
        "s3", "ssm"
    ) as c:
        yield c

@pytest.fixture
def db(pg, _truncate_all):
    return pg.get_connection_url()

2.3 LocalStack como variante

LocalStack emula APIs AWS dentro de um container. Para S3, SSM Parameter Store, SQS e EventBridge a fidelidade é alta — o teste de integração pode rodar inteiramente local. Para serviços profundos — IAM com permissions boundary, KMS HSM, redes complexas — LocalStack não substitui conta real. A regra: use LocalStack para o que cabe e teste em conta real o que LocalStack não simula com fidelidade.

2.4 Docker no runner

Sem Docker, fixture vazia. No nosso self-hosted runner (EC2 da Aula 04), o cloud-init precisa instalar o Docker, habilitá-lo no boot e dar permissão ao usuário do runner. Se o pull da imagem postgres:16-alpine for feito a cada job, paga 5–15 s só de download. Em runner self-hosted com cache local, paga uma vez por boot.

3. Chaos engineering — quebrar de propósito antes que quebre por acidente

Chaos engineering nasceu na Netflix em 2010, quando a empresa migrava do datacenter para a AWS. O Chaos Monkey desligava instâncias EC2 aleatoriamente para forçar a arquitetura a tolerar falhas que aconteciam de qualquer jeito. Em 2014 a prática vira disciplina formal — principlesofchaos.org. A definição operacional é simples: chaos engineering é a disciplina de experimentar em um sistema para construir confiança na sua capacidade de tolerar condições turbulentas em produção. É experimento, não teste. Tem hipótese, observação e conclusão.

3.1 Os cinco princípios canônicos

  1. Hipótese sobre estado estacionário — antes do experimento, o sistema é caracterizado por uma métrica observável (SLI). A hipótese diz que o estado estacionário se mantém apesar da falha.
  2. Variar eventos do mundo real — latência, perda de pacote, kill de processo, disco cheio, queda de zona. Não inventar falha exótica.
  3. Rodar em produção — com salvaguardas. Por enquanto, no nosso laboratório. A Aula 08 discute produção real.
  4. Automatizar — experimento como código (chaos/experiments/*.yaml). Game day pontual, regressão recorrente.
  5. Minimizar o raio de explosão — janela curta, área isolada, kill switch obrigatório.

3.2 Métodos — o que cabe no Learner Lab

AWS FIS (Fault Injection Service) é a oferta gerenciada para chaos. No Learner Lab a FIS não está habilitada. Substituímos com SSM Run Command (que distribui comandos para EC2 alvo) e ferramentas Linux instaladas na EC2: Toxiproxy para latência e perda em TCP, tc netem para qdisc do kernel, stress-ng para CPU/memória, fallocate para disco. O efeito é equivalente; muda o empacotamento.

3.4 Game day — a cerimônia

Game day é o ritual em que o time roda chaos junto, na mesma sala (virtual ou física), com papéis claros: Comandante (decide go/no-go), Operador (injeta a falha), Observador (lê CW Logs em tempo real) e Escriba (transcreve o que vê). Sem ritual, chaos vira blame. Saída obrigatória: arquivo 15_chaos_runs/<timestamp>.md com hipótese, execução, resultado e ADR.

5. Qualidade de código mensurada

Engenharia de software produz duas coisas: comportamento e custo de manutenção. Comportamento é exercitado por testes (Aula 05) e por chaos (esta aula). Custo de manutenção é exercitado por métricas que olham o código como artefato — quantos caminhos ele tem, quão denso é, quanto esforço cognitivo um leitor humano paga para entender. Em SRE, código de baixa qualidade não é estética — é toil que retorna toda vez que algo precisa mudar.

5.1 QM-01 — Complexidade ciclomática (McCabe, 1976)

McCabe propôs em 1976 a métrica que conta caminhos linearmente independentes pelo grafo de controle de uma função. Em prática, conta os pontos de decisão: cada if, elif, and, or, for, while, except, case incrementa. A fórmula formal é M = E − N + 2P (arestas, nós, componentes conexos), mas equivale a "número de decisões + 1". A leitura prática:

  • 1–5 — baixa, função simples.
  • 6–10 — média, aceitável.
  • 11–20 — alta, candidata a refactor.
  • > 20 — muito alta, dívida explícita.

O gate do curso é McCabe ≤ 10 por função e média ≤ 8 por arquivo.

5.2 QM-02 — Cognitive Complexity (SonarSource, 2017)

McCabe não distingue if aninhado de if sequencial — mas o cérebro humano distingue. Cognitive complexity ajusta McCabe penalizando aninhamento, recursão e quebra de fluxo progressivamente. Cada quebra de fluxo conta +1; cada nível de aninhamento conta +1 extra; sequências de operadores lógicos heterogêneos contam +1; recursão e goto contam +1. O resultado é mais fiel ao tempo que um humano leva para reler a função. Limiar do curso: ≤ 15.

5.3 QM-03 — Halstead (1977)

Halstead olha a função como texto e conta operadores e operandos — tipos distintos (n1, n2) e ocorrências totais (N1, N2). A partir disso deriva volume, dificuldade, esforço e tempo estimado. As métricas revelam densidade que McCabe não vê: função curta com vocabulário pesado, função longa com vocabulário pobre.

MétricaFórmulaO que diz
Vocabulárion = n1 + n2Tipos distintos de tokens
ComprimentoN = N1 + N2Total de tokens
VolumeV = N · log₂(n)Tamanho da informação
DificuldadeD = (n1/2) · (N2/n2)Densidade do código
EsforçoE = D · VCusto cognitivo
Tempo (s)T = E / 18Estimado para 1 leitor

5.4 QM-04 — Maintainability Index

Oman e Hagemeister (1992) compuseram McCabe, Halstead e LOC em uma métrica única, o Maintainability Index. Microsoft popularizou a versão normalizada de 0 a 100 que usamos hoje:

Fórmula clássica (normalizada 0–100)

MI = max(0, (171 − 5.2 · ln(V) − 0.23 · G − 16.2 · ln(LOC)) · 100 / 171)

Onde V é o volume de Halstead, G é a complexidade ciclomática (McCabe) e LOC é o número de linhas de código. Acima de 65 é considerado manutenível; abaixo, dívida. Em Python: radon mi src/.

5.6 Política de qualidade do projeto

  • McCabe ≤ 10 por função; média ≤ 8 por arquivo.
  • Cognitive complexity ≤ 15 por função.
  • MI ≥ 65 em todo módulo crítico.
  • Duplicação ≤ 5% (regra SIM do ruff).
  • Cobertura unit ≥ 80% (já vinha da Aula 05).

Cada função crítica é vinculada via RTM a pelo menos um RF ou RNF. Quando ela cruza um limiar, o PR não merge sem ADR QUALITY-NN que justifique e proponha plano de refactor. O refactor é orçado pelo error budget restante — não pela boa vontade.

6. Anti-padrões que vamos evitar

Cada item desta lista travou um time real. Eles voltam a aparecer porque são tentadores.

  • Container persistente entre testes sem reset. Faz teste 5 passar porque o teste 3 deixou dado lá. Quando rodam isolados, falham. Sintoma: flaky no CI.
  • Pull de imagem em todo job. Runner gasta 15 s antes de cada execução. Cache local resolve.
  • Mock e Testcontainer misturados sem critério. O mesmo módulo usa mock em um teste e TC em outro — o teste com mock vira referência falsa.
  • Chaos sem kill switch. Algo dá errado, ninguém sabe parar. Por menor que seja a janela, kill switch é obrigatório.
  • Toxiproxy esquecido ligado. Experimento acaba, ninguém remove o toxic. Daqui a duas horas, todo mundo culpa rede.
  • Experimento sem hipótese. Quebrar coisa por curiosidade não é chaos engineering — é vandalismo. A hipótese diz o que esperar.
  • Métrica vista nunca aplicada. Relatório bonito que ninguém usa para decidir merge. O conteúdo só vira realidade quando vira gate.
  • Limiar negociado por sprint. Hoje McCabe é 12, semana que vem é 15. Métrica vira piada.
  • Refactor sem orçamento. "Quando der tempo" nunca dá. Refactor patrocinado pelo error budget é ritual: a equipe vê quanto sobrou e decide o que pagar.

7.6 Passo 6 — injetar a falha via SSM Run Command

Com o YAML pronto, transformamos a sequência de ações em um aws ssm send-command que distribui para a EC2 alvo (selecionada por tag). O ID do comando volta como variável de ambiente; é ele que usamos no kill switch. 

# Injetar latencia 2000ms por 60s no proxy etl-pg
CMD_ID=$(aws ssm send-command \
  --document-name "AWS-RunShellScript" \
  --targets "Key=tag:Role,Values=etl-lab" \
  --parameters 'commands=[
    "docker exec toxiproxy /toxiproxy-cli toxic add -t latency -a latency=2000 etl-pg",
    "sleep 60",
    "docker exec toxiproxy /toxiproxy-cli toxic remove --toxicName etl-pg-latency etl-pg"
  ]' \
  --query "Command.CommandId" --output text)

echo "$CMD_ID" > /tmp/last_chaos_cmd

# Kill switch (executar se algo sair do controle)
aws ssm cancel-command --command-id "$CMD_ID"

# Observar
aws logs tail /aws/ec2/etl-lab --follow --since 2m

        if bad:
            print("MI < 65 em:", bad); sys.exit(1)
        EOF
    - name: Ruff gate
      run: ruff check src/

8. Restrições duras desta aula

  • Sem console AWS. Tudo via CLI / SSM.
  • Chaos só no stack lab. Nunca em produção sem ADR específico e janela negociada.
  • Kill switch obrigatório em todo experimento — sem ele, a skill recusa.
  • Toxiproxy nunca em prod sem ADR e sem revisão de blast radius.
  • FIS está bloqueado no Learner Lab — usar SSM Run Command como substituto.
  • Métrica de qualidade não negociável por sprint — só por ADR QUALITY-NN.

9. Continuidade — ponte para a Aula 07

As métricas medidas hoje (McCabe médio, MI, taxa de chaos passados, tempo médio de retorno ao baseline) viram SLIs internos na Aula 07. Entram nos dashboards CloudWatch consolidados e passam a ter SLO próprio. Quando o SLI cair abaixo do SLO, o controlador SRE da Aula 03 dispara — freeze de feature, prioridade para refactor, escalonamento. O ciclo da Aula 07 é: medir, comparar com meta, decidir trabalho.

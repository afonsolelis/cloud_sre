Data Warehousing na Nuvem de Alta Performance

Este material aprofunda os conceitos apresentados nos slides da Aula 06. Cobre a arquitetura MPP do Amazon Redshift, as decisões físicas de distribution style e sort keys, a carga de dados com o COPY command, a modelagem analítica com star schema, e a integração entre Data Warehouse e Data Lake via Redshift Spectrum.

Módulo 1 — Arquitetura MPP do Amazon Redshift

O que é MPP e por que importa

MPP (Massively Parallel Processing) é um modelo de processamento onde uma query é decomposta em tarefas menores executadas simultaneamente em múltiplos nós. No Redshift, o nó leader recebe a query do cliente, cria um plano de execução e distribui as tarefas para os nós compute. Cada nó compute processa apenas os dados armazenados nele, em paralelo com todos os outros. O leader coleta os resultados parciais, os combina e retorna para o cliente.

O resultado prático é que uma query que escaneia 1 bilhão de linhas pode ser processada em segundos quando distribuída por dezenas de nós compute, cada um processando uma fatia dos dados em paralelo. Isso é fundamentalmente diferente de um banco relacional tradicional (como PostgreSQL), que processa queries em um único servidor.

Armazenamento colunar no Redshift

O Redshift usa armazenamento colunar (como Parquet): os dados de cada coluna são armazenados sequencialmente no disco, separados dos dados das outras colunas. Quando uma query precisa de apenas algumas colunas de uma tabela com dezenas, o Redshift lê apenas os blocos das colunas necessárias. A combinação de armazenamento colunar com compressão específica por tipo de dado (Run Length Encoding para colunas com muita repetição, Delta para séries temporais, etc.) pode reduzir o tamanho físico dos dados em 3-5x em relação ao armazenamento row-based não comprimido.

Distribution styles: como os dados são distribuídos entre nós

A escolha do distribution style define como as linhas de uma tabela são distribuídas entre os nós compute. Essa decisão impacta diretamente a performance de queries que fazem JOIN entre tabelas.

  • DISTSTYLE EVEN: o Redshift distribui linhas em round-robin entre os nós. Cada nó tem aproximadamente a mesma quantidade de dados. Bom para tabelas sem padrão claro de JOIN. O problema é que JOINs entre duas tabelas EVEN exigem movimento de dados (redistribuição) entre nós — operação custosa chamada de data movement.
  • DISTSTYLE KEY (DISTKEY): linhas com o mesmo valor de DISTKEY são armazenadas no mesmo nó. Quando duas tabelas grandes têm o mesmo DISTKEY e são unidas por essa coluna, o JOIN acontece localmente em cada nó sem movimentação de dados. Escolha o DISTKEY pela coluna mais frequente em JOINs, com alta cardinalidade e distribuição uniforme.
  • DISTSTYLE ALL: uma cópia completa da tabela é armazenada em cada nó. JOINs com essa tabela nunca precisam de redistribuição. Use apenas para tabelas pequenas e raramente modificadas (dimensões pequenas em star schemas), pois o custo de armazenamento e de WRITE cresce linearmente com o número de nós.
  • DISTSTYLE AUTO: o Redshift escolhe automaticamente o melhor estilo com base no tamanho da tabela. Recomendado para começar.

Sort keys: otimizando leituras com zone maps

O sort key define a ordem física em que os dados são armazenados nos blocos de disco. O Redshift mantém zone maps — metadados que registram o valor mínimo e máximo de cada bloco para as colunas de sort key. Quando uma query filtra por uma coluna de sort key (ex.: WHERE data_venda BETWEEN '2026-01-01' AND '2026-03-31'), o Redshift usa os zone maps para pular blocos inteiros que não contêm dados no intervalo filtrado, sem lê-los.

Tipos de sort key:

  • COMPOUND SORTKEY: os dados são ordenados pela primeira coluna, depois pela segunda dentro de grupos da primeira, etc. Mais eficiente para filtros que usam as colunas na mesma ordem do sort key.
  • INTERLEAVED SORTKEY: trata cada coluna de sort key com igual importância. Melhor quando os filtros variam amplamente entre diferentes colunas.

Módulo 2 — Modelagem Analítica: Star Schema

Modelo estrela: fatos e dimensões

O star schema (modelagem estrela) é o padrão de modelagem dimensional usado em Data Warehouses. Uma tabela de fatos central contém as métricas numéricas mensuráveis (vendas, receita, quantidade), e tabelas de dimensão ao redor descrevem o contexto (quem, quando, onde, o quê).

Exemplo de star schema para análise de vendas:

  • Fato: fato_vendas — colunas: pedido_id (SK), cliente_sk (FK), produto_sk (FK), data_sk (FK), loja_sk (FK), quantidade, valor_unit, valor_total, desconto. Cada linha é uma venda individual.
  • Dim: dim_cliente — cliente_sk, nome, segmento, cidade, estado, pais.
  • Dim: dim_produto — produto_sk, nome, categoria, subcategoria, marca, preco_lista.
  • Dim: dim_data — data_sk, data, dia_semana, mes, trimestre, ano, is_fim_semana, is_feriado.
  • Dim: dim_loja — loja_sk, nome, regiao, tipo (fisica/online), gerente.

As tabelas de dimensão são desnormalizadas — atributos que em um modelo OLTP seriam separados em múltiplas tabelas (ex.: cidade, estado, país em tabelas separadas) são "achatados" em uma única dimensão. Isso elimina JOINs em profundidade e simplifica as queries dos analistas.

COPY command: a forma eficiente de carregar dados

O COPY é o comando otimizado para cargas em massa no Redshift. Em vez de múltiplos INSERTs individuais (que são lentos porque o Redshift precisa manter os índices e estatísticas a cada inserção), o COPY carrega dados diretamente do S3 em paralelo para os nós compute.

Exemplo de COPY de arquivos Parquet do S3: 

COPY schema_vendas.fato_vendas
FROM 's3://meu-datalake/refined/produto=vendas/ano=2026/mes=04/'
IAM_ROLE 'arn:aws:iam::123456789:role/RedshiftCopyRole'
FORMAT AS PARQUET;

Após uma carga com COPY, execute sempre ANALYZE para atualizar as estatísticas do query planner e VACUUM para reorganizar os dados e reclamar espaço de linhas deletadas.

Módulo 3 — Redshift Spectrum: Data Warehouse + Data Lake

O que é o Redshift Spectrum

O Redshift Spectrum permite executar queries SQL sobre dados armazenados no S3 diretamente do Redshift, sem precisar carregar os dados no cluster. Ele usa uma frota separada de nós (independente do cluster Redshift) para escanear e processar os dados no S3, e retorna os resultados para o nó leader do Redshift para combinação com dados internos.

Isso cria um padrão híbrido poderoso: dados quentes e frequentemente consultados ficam carregados no Redshift (acesso rápido, custo de armazenamento maior), enquanto dados históricos ou raramente consultados ficam no S3 (armazenamento barato) e são acessados via Spectrum quando necessário.

Configurando o Spectrum

Para usar o Spectrum, você precisa:

  1. Criar um external schema no Redshift apontando para o Glue Data Catalog.
  2. As tabelas externas no Catalog ficam automaticamente visíveis no Redshift via esse schema.
CREATE EXTERNAL SCHEMA spectrum_raw
FROM DATA CATALOG
DATABASE 'datalake_lab'
IAM_ROLE 'arn:aws:iam::123456789:role/RedshiftSpectrumRole'
CREATE EXTERNAL DATABASE IF NOT EXISTS;

Após isso, você pode fazer JOINs entre tabelas internas do Redshift e tabelas externas no S3 na mesma query: 

SELECT c.nome, SUM(f.valor_total) AS receita_total
FROM schema_vendas.fato_vendas f
JOIN spectrum_raw.trusted_pedidos s3_p ON f.pedido_id = s3_p.pedido_id
JOIN schema_vendas.dim_cliente c ON f.cliente_sk = c.cliente_sk
WHERE f.data_sk BETWEEN 20260101 AND 20260430
GROUP BY c.nome
ORDER BY receita_total DESC
LIMIT 10;

Módulo 4 — Laboratório: Explorando o Redshift com o Query Editor

Devido a restrições de custo no ambiente AWS Student, o laboratório usa o Redshift Serverless (se disponível no lab) ou o Redshift Query Editor v2 para explorar conceitos. Siga os passos adaptando para o que estiver disponível no seu ambiente.

Passo 1 — Acessar ou provisionar o Redshift

  1. No console AWS, acesse Amazon Redshift.
  2. Se o lab já tiver um cluster ou workgroup serverless, clique nele para acessar.
  3. Se não houver, clique em Try Amazon Redshift Serverless e siga o assistente de configuração com as configurações padrão.
  4. Aguarde o status ficar Available.
  5. Validação: você consegue acessar o Query Editor v2.

Passo 2 — Criar o schema e as tabelas dimensão e fato

  1. No Query Editor v2, crie um novo script e execute:
CREATE SCHEMA IF NOT EXISTS vendas;

CREATE TABLE vendas.dim_cliente (
    cliente_sk INT IDENTITY(1,1),
    cliente_id VARCHAR(20) NOT NULL,
    nome       VARCHAR(100),
    segmento   VARCHAR(50),
    cidade     VARCHAR(50),
    estado     CHAR(2)
)
DISTSTYLE ALL;

CREATE TABLE vendas.dim_data (
    data_sk    INT,
    data       DATE,
    ano        SMALLINT,
    mes        SMALLINT,
    dia        SMALLINT,
    trimestre  SMALLINT
)
DISTSTYLE ALL
SORTKEY (data);

CREATE TABLE vendas.fato_vendas (
    pedido_sk   BIGINT IDENTITY(1,1),
    pedido_id   VARCHAR(20),
    cliente_sk  INT,
    data_sk     INT,
    valor       DECIMAL(10,2),
    status      VARCHAR(20)
)
DISTKEY (cliente_sk)
SORTKEY (data_sk);
  1. Insira dados de exemplo nas dimensões:
INSERT INTO vendas.dim_cliente (cliente_id, nome, segmento, cidade, estado)
VALUES ('C001','Ana Silva','premium','São Paulo','SP'),
       ('C002','João Lima','standard','Rio de Janeiro','RJ');

INSERT INTO vendas.dim_data VALUES
(20260401, '2026-04-01', 2026, 4, 1, 2),
(20260402, '2026-04-02', 2026, 4, 2, 2);

INSERT INTO vendas.fato_vendas (pedido_id, cliente_sk, data_sk, valor, status)
VALUES ('P001', 1, 20260401, 150.00, 'entregue'),
       ('P002', 2, 20260401, 89.90, 'processando'),
       ('P003', 1, 20260402, 320.50, 'entregue');
  1. Execute uma query analítica com JOIN:
SELECT c.nome, d.data, SUM(f.valor) AS total
FROM vendas.fato_vendas f
JOIN vendas.dim_cliente c ON f.cliente_sk = c.cliente_sk
JOIN vendas.dim_data d ON f.data_sk = d.data_sk
WHERE f.status = 'entregue'
GROUP BY c.nome, d.data
ORDER BY total DESC;
  1. Validação: a query retorna os totais por cliente e data apenas para pedidos entregues.

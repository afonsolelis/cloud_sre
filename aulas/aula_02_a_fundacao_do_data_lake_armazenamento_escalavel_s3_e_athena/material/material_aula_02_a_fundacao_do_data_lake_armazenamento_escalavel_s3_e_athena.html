A Fundação do Data Lake: Armazenamento Escalável com S3 e Athena

Este material aprofunda os conceitos apresentados nos slides da Aula 02. Cobre a arquitetura de um Data Lake moderno na AWS, a estruturação de buckets e camadas no Amazon S3, estratégias de particionamento e escolha de formatos de arquivo, e o uso do Amazon Athena para consultas SQL serverless diretamente sobre os dados armazenados no lake.

Módulo 1 — Amazon S3: A Fundação do Data Lake

Por que o S3 se torna a fundação do Data Lake?

O Amazon Simple Storage Service (S3) é um serviço de armazenamento de objetos projetado para escala praticamente ilimitada. Diferente de sistemas de arquivos tradicionais, o S3 não tem diretórios reais — os "prefixos" (caminhos que parecem pastas) são apenas convenções de nomenclatura que os serviços interpretam como estrutura hierárquica. Essa distinção importa porque afeta como você lê, lista e cobra pelo acesso aos dados.

A durabilidade do S3 é de 99,999999999% (11 noves), conseguida por replicação automática dos objetos em múltiplas instalações físicas dentro de uma região. Isso significa que a perda de um arquivo por falha de hardware da AWS é essencialmente impossível na prática. Para dados de missão crítica, o S3 Cross-Region Replication adiciona ainda mais camadas de proteção geográfica.

A separação entre armazenamento e computação é o argumento definitivo para usar o S3 como base: o S3 armazena os dados, enquanto serviços como Athena, Glue, EMR, Redshift Spectrum e SageMaker os processam sem precisar mover terabytes para dentro de um cluster. Isso elimina o acoplamento entre custo de armazenamento e custo de processamento, permitindo escalá-los independentemente.

Buckets: criação e configuração essencial

Um bucket é o container raiz no S3. Todo objeto (arquivo) precisa pertencer a um bucket. Cada bucket tem um nome globalmente único em toda a AWS — dois clientes diferentes não podem ter buckets com o mesmo nome no mesmo momento. Isso significa que nomes simples como dados ou analytics certamente já estão ocupados; a convenção é usar prefixos de domínio ou identificadores de organização, como minha-empresa-datalake-prod.

Ao criar um bucket, as configurações mais importantes são:

  • Região: escolha a mesma região dos serviços que vão consumir os dados (ex.: us-east-1). Transferência de dados entre regiões gera custo adicional e latência.
  • Bloqueio de acesso público: mantenha todas as opções de bloqueio ativadas por padrão. Buckets públicos são uma das principais causas de vazamento de dados em cloud. Acesso público só deve ser liberado deliberadamente para casos como sites estáticos.
  • Versionamento: habilite especialmente nas camadas Raw e Trusted. O versionamento mantém versões anteriores dos objetos, permitindo recuperação de arquivos sobrescritos ou deletados acidentalmente.
  • Criptografia padrão: ative SSE-S3 (criptografia gerenciada pela AWS) ou SSE-KMS (criptografia com chave gerenciada pelo AWS KMS) para garantir que todos os objetos sejam criptografados em repouso automaticamente.

Classes de armazenamento e custo

O S3 oferece múltiplas classes de armazenamento com diferentes perfis de custo e disponibilidade. A escolha correta pode reduzir significativamente o custo do lake:

  • S3 Standard: alta disponibilidade (99,99%), acesso frequente. Custo mais alto de armazenamento por GB. Use para camadas Trusted e Refined com acesso analítico frequente.
  • S3 Standard-IA (Infrequent Access): custo de armazenamento menor, mas cobra por GB lido. Use para dados Raw acessados apenas para reprocessamento.
  • S3 Glacier Instant Retrieval: custo muito baixo de armazenamento, acesso em milissegundos. Ideal para dados históricos que raramente são consultados mas precisam de acesso imediato quando necessário.
  • S3 Glacier Flexible Retrieval: custo mínimo de armazenamento, recuperação em minutos a horas. Use para arquivamento de longo prazo.

As políticas de Lifecycle do S3 automatizam a migração entre classes. Por exemplo: dados na camada Raw mais antigos que 90 dias podem ser movidos automaticamente para Standard-IA, e após 1 ano para Glacier Instant Retrieval, sem nenhuma ação manual.

Módulo 2 — Camadas do Data Lake: Raw, Trusted e Refined

Por que organizar em camadas?

A organização em camadas é a espinha dorsal de um Data Lake bem governado. Sem ela, dados brutos se misturam com dados processados, é impossível rastrear a origem de um valor incorreto, e o reprocessamento após uma correção de lógica se torna inviável. Cada camada tem uma responsabilidade clara e uma audiência definida.

Camada Raw / Bronze

A camada Raw recebe dados exatamente como chegaram da fonte, sem nenhuma transformação. Isso inclui erros de digitação, valores nulos inesperados, formatos inconsistentes e duplicatas — tudo preservado fielmente. O objetivo é ter um ponto de verdade auditável: se algo der errado em qualquer etapa posterior, você pode sempre reler o dado bruto e reprocessar a partir do zero.

Boas práticas para a camada Raw:

  • Nunca sobrescreva um arquivo Raw já gravado. Use particionamento por data de ingestão para garantir imutabilidade.
  • Não aplique transformações de schema. Se a fonte envia JSON com campos extras, preserve todos os campos.
  • Aplique compressão leve (gzip, snappy) para economizar espaço sem comprometer a fidelidade dos dados.
  • Configure o S3 Lifecycle para migrar para Standard-IA após 60-90 dias, pois o Raw raramente é relido depois que as camadas superiores estão populadas.

Convenção de prefixo recomendada para Raw: raw/fonte=orders/ano=2026/mes=04/dia=23/arquivo.json.gz

Camada Trusted / Silver

A camada Trusted contém dados que passaram por um processo de qualidade: tipos de dados validados e padronizados, valores nulos tratados (eliminados ou imputados com justificativa), duplicatas removidas, encoding normalizado para UTF-8, e campos renomeados para seguir convenções de nomenclatura da organização.

Os dados na camada Trusted são o ponto de consumo para times técnicos — engenheiros de ML, analistas avançados e jobs de Glue que alimentam a camada Refined. O formato preferencial é Parquet com compressão Snappy, que oferece boa razão de compressão sem overhead excessivo de CPU na leitura.

Convenção de prefixo recomendada para Trusted: trusted/dominio=vendas/entidade=pedidos/ano=2026/mes=04/parte-0001.parquet

Camada Refined / Gold

A camada Refined contém dados modelados para consumo direto por ferramentas de BI, dashboards operacionais e modelos de machine learning em produção. Aqui os dados já estão agregados, enriquecidos com atributos de outras fontes e organizados por domínio ou produto analítico.

Exemplos de datasets na camada Refined: tabela de fatos de vendas diárias por região, indicadores de NPS calculados por coorte de clientes, dataset de features para modelo de churn. Cada um tem um proprietário de negócio e um SLA de atualização definido.

Convenção de prefixo recomendada para Refined: refined/produto=dashboard-vendas/versao=v2/snapshot=2026-04-23/dados.parquet

Módulo 3 — Formatos de Arquivo e Particionamento

CSV: simples, mas custoso em escala

O CSV é o formato mais universalmente suportado e ideal para a ingestão inicial na camada Raw, especialmente quando a fonte é um sistema legado ou um arquivo exportado manualmente. A leitura analítica de CSV, porém, tem custo elevado no Athena porque o mecanismo precisa ler a linha inteira mesmo quando a consulta só precisa de duas colunas de uma tabela com 50.

JSON: flexível, mas volumoso

O JSON é preferido quando os dados têm schema variável — por exemplo, eventos de plataforma onde cada tipo de ação tem atributos diferentes. O JSON Lines (JSONL, onde cada linha é um objeto JSON independente) facilita o processamento paralelo. Assim como o CSV, o JSON é row-oriented e custoso para leituras analíticas que acessam subconjuntos de colunas.

Parquet: o formato analítico padrão

O Apache Parquet é um formato colunar: em vez de armazenar linha por linha, armazena coluna por coluna. Quando uma query do Athena precisa apenas das colunas data_pedido e valor_total de uma tabela com 30 colunas, o Athena lê apenas os blocos físicos dessas duas colunas, ignorando completamente as outras 28. O resultado é uma redução drástica no volume de dados lido — e o Athena cobra por volume lido.

Além da leitura seletiva, o Parquet inclui estatísticas por bloco (valor mínimo, máximo e contagem de nulos por coluna). O Athena usa essas estatísticas para pular blocos inteiros que não satisfazem os filtros da query, sem nem precisar ler os dados. Isso é chamado de predicate pushdown.

A compressão padrão para Parquet na AWS é Snappy, que oferece boa velocidade de compressão/descompressão em troca de uma razão de compressão ligeiramente menor que Gzip. Para dados históricos raramente acessados, Gzip pode reduzir ainda mais o tamanho, ao custo de mais CPU na leitura.

Particionamento estratégico

Particionar significa organizar os dados em subpastas baseadas em valores de uma ou mais colunas. O Athena usa a estrutura de partições para pular prefixos inteiros do S3 que não satisfazem os filtros de uma query, sem precisar listar ou ler nenhum arquivo dentro deles.

Critérios para escolher colunas de particionamento:

  • A coluna deve aparecer frequentemente nos filtros WHERE das consultas. Se quase toda query filtra por data, particionar por data é a escolha óbvia.
  • A cardinalidade deve ser moderada. Particionar por user_id em uma tabela com 10 milhões de usuários criaria 10 milhões de partições, causando problemas de listagem e overhead de metadados no Glue.
  • Evite partições com poucos arquivos pequenos (menos de ~100 MB cada). O overhead de abertura de arquivo no S3 se torna dominante quando há muitos arquivos diminutos.

Exemplo de hierarquia de particionamento para uma tabela de eventos: trusted/dominio=eventos/tipo=click/ano=2026/mes=04/dia=23/parte-0001.parquet

O formato chave=valor nos prefixos é a convenção do Hive Metastore, que o AWS Glue Catalog reconhece automaticamente ao executar um Crawler, eliminando a necessidade de cadastrar partições manualmente.

O problema dos arquivos pequenos

Cada arquivo no S3 tem um custo mínimo de operação: o Athena faz uma requisição HTTP separada para abrir cada arquivo. Quando um pipeline gera milhares de arquivos de 1-2 MB (o "small files problem"), o tempo de abertura de arquivos domina o tempo total da query. A solução é compactar esses arquivos em um job periódico de compaction — um job Glue que lê todos os arquivos pequenos de uma partição e os reescreve como um ou dois arquivos grandes.

Módulo 4 — Amazon Athena: Consultas Serverless sobre o Data Lake

O que é o Athena e como funciona

O Amazon Athena é um serviço de query interativo baseado no motor open-source Presto (agora Trino). Ele executa SQL ANSI sobre dados armazenados no S3 sem precisar provisionar ou gerenciar nenhum servidor. Você define as tabelas no AWS Glue Data Catalog (que atua como metastore compatível com Hive), submete a query, e o Athena lê os arquivos diretamente do S3, processa em paralelo em uma frota de workers gerenciados pela AWS, e retorna os resultados.

O modelo de precificação é baseado em dados lidos: US$ 5,00 por terabyte de dados varridos. Isso cria um incentivo direto para usar Parquet (reduz volume lido por projeção de colunas), compressão (reduz o tamanho físico), e particionamento (reduz o número de arquivos abertos). Uma query que escaneia 10 GB em CSV pode escanear apenas 500 MB em Parquet particionado para o mesmo conjunto de resultados, reduzindo o custo em 20x.

Configuração inicial do Athena

Antes de executar a primeira query, o Athena precisa de um bucket S3 configurado para armazenar os resultados. Essa configuração é feita em Settings no console do Athena. Os arquivos de resultado ficam nesse bucket e podem ser consultados ou baixados. Crie um bucket separado exclusivamente para resultados do Athena, como minha-empresa-athena-results, para manter o lake organizado.

Criando tabelas externas no Athena

Uma tabela externa no Athena é apenas um mapeamento de metadados: ela descreve como interpretar os arquivos no S3 (formato, schema, partições), mas não move nem copia os dados. Deletar a tabela não apaga os arquivos.

Exemplo de DDL para criar uma tabela CSV na camada Raw: 

CREATE EXTERNAL TABLE raw_pedidos (
  pedido_id   STRING,
  cliente_id  STRING,
  data_pedido STRING,
  valor       DOUBLE,
  status      STRING
)
ROW FORMAT DELIMITED
FIELDS TERMINATED BY ','
LINES TERMINATED BY '\n'
TBLPROPERTIES ('skip.header.line.count'='1')
LOCATION 's3://meu-datalake/raw/fonte=pedidos/';

Exemplo de DDL para criar uma tabela Parquet particionada na camada Trusted: 

CREATE EXTERNAL TABLE trusted_pedidos (
  pedido_id   STRING,
  cliente_id  STRING,
  data_pedido DATE,
  valor       DOUBLE,
  status      STRING
)
PARTITIONED BY (ano STRING, mes STRING, dia STRING)
STORED AS PARQUET
LOCATION 's3://meu-datalake/trusted/dominio=vendas/entidade=pedidos/';

Após criar a tabela particionada, é necessário registrar as partições para que o Athena as reconheça. O comando MSCK REPAIR TABLE trusted_pedidos; escaneia os prefixos no S3 e registra automaticamente todas as partições que seguem o formato Hive.

Otimizando consultas no Athena

As quatro regras mais impactantes para reduzir custo e tempo de consulta:

  • Selecione apenas as colunas necessárias: use SELECT coluna1, coluna2 em vez de SELECT *. Com Parquet, isso reduz o volume lido na proporção das colunas não selecionadas.
  • Filtre por colunas de partição: sempre inclua WHERE ano='2026' AND mes='04' quando possível. O Athena pula prefixos inteiros que não correspondem ao filtro.
  • Use Parquet com compressão: converta dados de Raw (CSV/JSON) para Parquet na camada Trusted com um job Glue. A diferença de custo por consulta justifica o custo do job de conversão em poucas execuções.
  • Evite SELECT COUNT(*) em tabelas grandes sem filtro: esse tipo de query escaneia a tabela inteira. Use estimativas ou tabelas de summary pré-calculadas para métricas operacionais frequentes.

Workgroups no Athena

Os Workgroups permitem isolar consultas por equipe, controlar limites de custo e configurar output locations diferentes. Por exemplo, o workgroup DataScientists pode ter um limite de 50 GB por query e apontar resultados para um bucket separado, enquanto o workgroup DataEngineers tem limite de 500 GB e acesso a tabelas de todas as camadas.

AWS Glue Data Catalog como metastore

O Glue Data Catalog é o repositório central de metadados do ecossistema analítico da AWS. Ele armazena databases e tables, que descrevem onde os dados estão no S3 e como interpretá-los. O Athena, o Redshift Spectrum, o EMR e o Glue Jobs compartilham o mesmo Glue Catalog, garantindo que todos os serviços vejam o mesmo schema de tabelas sem duplicação de definições.

Os Glue Crawlers descobrem automaticamente o schema de dados no S3, criam ou atualizam tabelas no Catalog e detectam novas partições. Um Crawler configurado para rodar diariamente mantém o Catalog sempre atualizado sem intervenção manual.

Módulo 5 — Laboratório: Estruturando o Data Lake no S3 e Consultando com Athena

A prática ocorre entre 21h00 e 22h00 utilizando o ambiente AWS Student. Siga cada passo na ordem indicada e valide o resultado antes de avançar.

Passo 1 — Acessar o ambiente AWS Student e verificar a região

  1. Acesse o portal da sua instituição e inicie o lab clicando em Start Lab.
  2. Aguarde o ícone de status ficar verde e clique em AWS para abrir o console.
  3. No canto superior direito, confirme que a região é us-east-1 (N. Virginia). Se não estiver, altere agora — recursos criados em outras regiões não serão visíveis.
  4. No campo de busca do console, acesse o S3 para confirmar que você tem acesso.
  5. Validação: você consegue visualizar a lista de buckets (vazia ou com buckets existentes) sem erros.

Passo 2 — Criar o bucket do Data Lake

  1. No console do S3, clique em Create bucket.
  2. Em Bucket name, insira um nome único globalmente, por exemplo: datalake-mba-seusobrenome-2026.
  3. Em AWS Region, confirme US East (N. Virginia) us-east-1.
  4. Em Block Public Access settings, mantenha todos os bloqueios ativados (configuração padrão e correta para um Data Lake privado).
  5. Em Bucket Versioning, selecione Enable.
  6. Em Default encryption, selecione SSE-S3.
  7. Clique em Create bucket.
  8. Validação: o bucket aparece na lista com o status correto. Clique nele para confirmar que está vazio e as configurações foram aplicadas.

Passo 3 — Criar a estrutura de prefixos (camadas)

  1. Dentro do bucket, clique em Create folder.
  2. No campo Folder name, insira raw e clique em Create folder.
  3. Repita o processo criando as pastas trusted e refined.
  4. Entre na pasta raw e crie a subpasta fonte=pedidos.
  5. Dentro de fonte=pedidos, crie a subpasta ano=2026.
  6. Dentro de ano=2026, crie mes=04.
  7. Validação: a estrutura raw/fonte=pedidos/ano=2026/mes=04/ existe e está acessível no bucket.

Passo 4 — Criar e fazer upload de um arquivo de dados CSV de exemplo

  1. No seu computador, crie um arquivo chamado pedidos_2026_04.csv com o seguinte conteúdo:
pedido_id,cliente_id,data_pedido,valor,status
P001,C001,2026-04-01,150.00,entregue
P002,C002,2026-04-01,89.90,processando
P003,C001,2026-04-02,320.50,entregue
P004,C003,2026-04-03,45.00,cancelado
P005,C004,2026-04-03,210.75,processando
  1. No console do S3, navegue até raw/fonte=pedidos/ano=2026/mes=04/.
  2. Clique em Upload, depois em Add files e selecione o arquivo CSV.
  3. Clique em Upload para concluir.
  4. Validação: o arquivo pedidos_2026_04.csv aparece no prefixo correto com o tamanho esperado.

Passo 5 — Configurar o Athena e criar a tabela

  1. No console AWS, acesse o Amazon Athena pelo campo de busca.
  2. Se for o primeiro acesso, o console exibirá um aviso para configurar o output location. Clique em Edit settings.
  3. Em Query result location, insira o caminho de um prefixo no seu bucket: s3://datalake-mba-seusobrenome-2026/athena-results/
  4. Clique em Save.
  5. No editor de queries, clique em + para abrir uma nova query.
  6. Execute o seguinte DDL para criar o database:
CREATE DATABASE IF NOT EXISTS datalake_lab;
  1. No seletor de database à esquerda, selecione datalake_lab.
  2. Execute o seguinte DDL para criar a tabela externa, substituindo o nome do bucket:
CREATE EXTERNAL TABLE datalake_lab.raw_pedidos (
  pedido_id   STRING,
  cliente_id  STRING,
  data_pedido STRING,
  valor       DOUBLE,
  status      STRING
)
ROW FORMAT DELIMITED
FIELDS TERMINATED BY ','
LINES TERMINATED BY '\n'
TBLPROPERTIES ('skip.header.line.count'='1')
LOCATION 's3://datalake-mba-seusobrenome-2026/raw/fonte=pedidos/';
  1. Validação: a tabela raw_pedidos aparece no painel de tabelas à esquerda, sob o database datalake_lab.

Passo 6 — Executar a primeira consulta

  1. No editor do Athena, execute a query de validação:
SELECT * FROM datalake_lab.raw_pedidos LIMIT 10;
  1. Observe o painel de resultados — os 5 registros do CSV devem aparecer com os tipos corretos.
  2. Observe também o indicador de dados varridos no canto superior direito do resultado, que deve mostrar o tamanho aproximado do arquivo CSV lido.
  3. Agora execute uma query com filtro para ver como o Athena reporta os dados varridos:
SELECT pedido_id, valor
FROM datalake_lab.raw_pedidos
WHERE status = 'entregue';
  1. Validação: a query retorna apenas os registros com status "entregue". Note que os dados varridos são iguais ao da query anterior — em CSV, a tabela inteira é lida mesmo com filtro, porque o formato não suporta predicate pushdown.

Passo 7 — Explorar o AWS Glue Crawler (opcional, se houver tempo)

  1. No console AWS, acesse o AWS Glue.
  2. No menu lateral, clique em Crawlers e depois em Create crawler.
  3. Nome: crawler-raw-pedidos.
  4. Em Data source, selecione S3 e insira o caminho: s3://datalake-mba-seusobrenome-2026/raw/
  5. Em IAM Role, selecione ou crie uma role com permissões de Glue e S3.
  6. Em Target database, selecione datalake_lab.
  7. Em Schedule, selecione On demand.
  8. Revise e clique em Create crawler.
  9. Na lista de Crawlers, selecione o crawler criado e clique em Run.
  10. Aguarde o status mudar para Succeeded.
  11. Verifique no Athena que uma nova tabela foi criada automaticamente pelo Crawler no database datalake_lab.
  12. Validação: o Crawler criou a tabela com o schema correto inferido automaticamente a partir dos arquivos CSV no S3.

Exercícios de fixação

  1. Design de particionamento: uma plataforma de e-commerce captura eventos de comportamento de usuários (cliques, visualizações de produto, adições ao carrinho). O volume é de aproximadamente 50 milhões de eventos por dia, com consultas predominantemente filtrando por data (últimos 7 dias) e tipo de evento. Defina uma estratégia de particionamento para a camada Trusted com justificativa para cada nível da hierarquia de prefixos. Explique também quais partições você evitaria e por quê.
  2. Comparativo de formatos: você tem uma tabela de transações financeiras com 200 colunas armazenada em CSV, ocupando 2 TB no S3. As queries analíticas típicas selecionam 8-10 colunas e filtram por data. Estime a redução de custo no Athena após converter para Parquet com Snappy, explicando o impacto de cada técnica (projeção de colunas, predicate pushdown, compressão) separadamente.
  3. Política de Lifecycle: descreva uma política de ciclo de vida para um bucket de Data Lake com as camadas Raw, Trusted e Refined, especificando para cada camada: a classe de armazenamento inicial, as transições de classe ao longo do tempo com os limiares de idade, e a justificativa de custo e disponibilidade para cada decisão.
  4. Diagnóstico de query cara: uma analista reclama que sua query no Athena está custando US$ 8,00 por execução e levando 4 minutos. A query é: SELECT COUNT(*) FROM eventos WHERE tipo = 'compra' sobre uma tabela com 5 TB em formato CSV sem particionamento. Liste todas as melhorias que você implementaria (formato, particionamento, query rewrite) e estime o novo custo e tempo após cada melhoria.
  5. Imutabilidade e reprocessamento: um bug no job de transformação da camada Trusted causou cálculo incorreto do campo valor_com_desconto por 30 dias. Explique como a arquitetura de camadas com Raw imutável permite corrigir esse problema, descrevendo o fluxo de reprocessamento passo a passo, sem perda de dados e sem impacto nos consumidores da camada Refined durante a correção.

Módulo 1 — Data Lake: definição e princípios

O que é um Data Lake

Um Data Lake é um repositório centralizado projetado para armazenar volumes arbitrariamente grandes de dados em seu formato original, sem modelagem prévia. A expressão foi popularizada por James Dixon (CTO da Pentaho) em 2010 para contrastar com o modelo rígido dos Data Warehouses. A ideia central é simples: em vez de projetar um esquema antes de ingerir os dados (abordagem típica de DW), o Data Lake aceita qualquer coisa — CSV, JSON, imagens, logs, eventos, Parquet — e deixa a decisão sobre como interpretar para o momento da leitura.

Essa inversão resolve três dores históricas da engenharia de dados: (1) o tempo entre coletar e analisar um novo conjunto de dados deixa de ser semanas ou meses de modelagem formal; (2) dados raros ou exploratórios que não justificariam um datamart podem ser preservados com custo marginal; (3) múltiplos consumidores (BI, ciência de dados, ML) podem usar os mesmos bytes sem disputa pelo schema "oficial".

Schema-on-read vs Schema-on-write

O contraste com o Data Warehouse é direto. Em um DW, aplica-se schema-on-write: toda linha precisa caber no esquema da tabela de destino no momento da ingestão. Isso garante qualidade e consistência, mas torna a estrutura cara de evoluir.

Em um Data Lake, aplica-se schema-on-read: os arquivos são guardados como chegaram, e cada consumidor aplica seu próprio esquema ao ler. Um cientista de dados pode interpretar um JSON aninhado com uma visão; um analista pode achatar o mesmo JSON com outra. Ambos partem do mesmo objeto imutável em S3. Isso traz flexibilidade, mas exige disciplina: sem governança, o Data Lake vira "Data Swamp" — dados presentes, porém inusáveis.

Princípios fundamentais

  1. Desacoplamento de armazenamento e computação: os arquivos ficam em S3 (barato, durável, elástico); engines diferentes (Athena, EMR, Redshift Spectrum, SageMaker, Spark, Flink) leem os mesmos dados. Se a demanda analítica dobra, não é preciso redimensionar o "banco".
  2. Formatos abertos: Parquet, ORC, Avro, JSON e CSV são padrões públicos. Um pipeline construído corretamente pode ser lido por qualquer ferramenta compatível. Isso elimina lock-in — trocar de engine não exige migração de dados.
  3. Escala praticamente ilimitada: o S3 absorve de kilobytes a exabytes com o mesmo modelo de preço por GB armazenado. Não há planejamento de capacidade.
  4. Multi-workload: o mesmo objeto S3 alimenta um dashboard Power BI, um notebook Jupyter, um job Spark e um modelo SageMaker. Os metadados descritivos ficam em um catálogo compartilhado (Glue Data Catalog).
  5. Custo linear e previsível: armazenamento por GB/mês, classes Glacier para dados frios e cobrança por TB escaneado nas engines. O custo acompanha o uso.

Módulo 2 — Data Lake vs Data Warehouse

Muita discussão "Lake vs Warehouse" é falsa dicotomia: são ferramentas diferentes que convivem bem. O DW otimiza analytics tabulares de alta performance com schema e integridade bem definidos; o Lake otimiza escala, variedade e custo. Organizações grandes quase sempre têm os dois, conectados por pipelines que promovem dados refinados do Lake para o DW.

Data Lakehouse

Entre as duas abordagens surgiu o Data Lakehouse, que combina custo e flexibilidade do Lake com transações ACID e performance do DW, via formatos de tabela como Apache Iceberg, Delta Lake e Apache Hudi. Nesses formatos, os dados permanecem como Parquet em S3, mas ganham metadados transacionais que permitem upserts, time travel, schema evolution controlada e otimizações sem perder interoperabilidade. A AWS suporta Iceberg nativamente via Athena, Glue, EMR e Redshift Spectrum. Aula 06 retomará Lakehouse com mais profundidade.

Módulo 3 — Medallion Architecture

Medallion é um padrão arquitetural popularizado pela Databricks que organiza o Data Lake em camadas de qualidade e valor crescentes: Bronze → Silver → Gold. No curso, usamos os nomes equivalentes raw → trusted → refined para alinhar com convenções comuns em engenharia de dados brasileira.

Raw / Bronze

Camada de ingestão fiel. Os dados chegam no formato original da fonte e são gravados no S3 sem modificação. É intencionalmente "feia": CSVs com encoding problemático, JSONs aninhados, logs semi-estruturados, tudo convive. A regra de ouro é imutabilidade: nada se sobrescreve, cada ingestão produz novos arquivos, normalmente com carimbo temporal no prefixo. Essa camada é a garantia de reprocessamento: se um bug for descoberto semanas depois na camada trusted, sempre é possível regenerá-la a partir da raw.

Trusted / Silver

Dados limpos, validados e deduplicados. Os tipos são corretamente convertidos (datas viram timestamp, números viram double), registros duplicados são removidos, chaves nulas tratadas, e o formato passa a ser Parquet comprimido e particionado. A camada trusted tem schema estável — mudanças exigem versionamento explícito. Esta é a "fonte de verdade operacional" do Data Lake: a maioria das análises e modelos ML consome daqui.

Refined / Gold

Dados modelados para casos de uso específicos. Aqui vivem tabelas agregadas, modelos dimensionais (fatos e dimensões), marts por área de negócio (vendas, marketing, logística), features de ML prontas para inferência. A camada gold é opinativa: existe para servir consumidores bem definidos, com SLAs claros de atualização e qualidade. Ferramentas de BI conectam majoritariamente aqui.

Aplicação no curso

Nosso pipeline Olist percorrerá as três camadas ao longo das aulas: hoje populamos raw e criamos a primeira tabela trusted (orders em Parquet particionado); nas aulas 04 e 05 adicionamos streaming e ETL serverless que alimentam trusted; na aula 06 promovemos para refined em Redshift/Iceberg.

Módulo 4 — Amazon S3 em profundidade

Visão geral

O Amazon Simple Storage Service (S3) é o serviço mais antigo da AWS, lançado em 2006. É um sistema de armazenamento de objetos distribuído globalmente, acessível via API HTTP, com modelo de preço pay-per-use. Praticamente todo Data Lake moderno na AWS é construído sobre S3, e serviços como Glue, Athena, EMR, Redshift Spectrum e SageMaker leem e escrevem nele nativamente.

Conceitos centrais

  • Bucket: contêiner raiz de objetos. Nomes são globalmente únicos no mundo todo. O bucket vive em uma região específica (us-east-1 no nosso caso) e herda dela características de latência, preço e compliance.
  • Object: a unidade de armazenamento. Cada objeto tem um corpo (bytes), uma key (caminho lógico) e metadados (Content-Type, tamanho, checksum, tags opcionais). O tamanho varia de 0 bytes a 5 TB por objeto.
  • Key: o identificador único dentro do bucket. Apesar de parecer um caminho hierárquico (olist/orders/2026/04/23/file.csv), na verdade é uma string plana — não existem "diretórios" de fato. O S3 usa prefixos (a parte inicial da key) para agrupar objetos logicamente.
  • Prefix: convenção que transforma o S3 em um pseudo-filesystem. Ferramentas como o console AWS exibem prefixos como pastas. Para Data Lakes, os prefixos são essenciais: cada camada, dataset e partição é um prefixo.

Durabilidade e disponibilidade

O S3 Standard oferece 99,999999999% de durabilidade (os famosos "eleven nines"). Na prática isso significa que, se você armazenar 10 milhões de objetos, a probabilidade estatística de perder um é de aproximadamente um a cada 10.000 anos. A AWS atinge esse número replicando cada objeto automaticamente em pelo menos três Availability Zones dentro da região, com checksums verificados em cada operação e auto-recuperação de corrupção detectada.

A disponibilidade (capacidade de responder a requisições) é de 99,99% para S3 Standard, com SLAs menores para classes de arquivamento.

Consistência forte (strong read-after-write)

Desde dezembro de 2020 o S3 oferece consistência forte para todas as operações: após um PUT bem-sucedido, qualquer GET ou LIST subsequente enxerga a versão final do objeto. Antes, objetos recém-criados tinham read-after-write consistente, mas sobrescritas e deletes eram eventualmente consistentes — o que exigia workarounds em pipelines. Hoje isso não é mais necessário, o que simplifica ETL, ingestão em streaming e orquestração.

Classes de armazenamento

Cada objeto tem uma storage class associada, que determina custo por GB, custo por requisição, latência de recuperação e disponibilidade. A tabela abaixo resume as classes mais usadas para Data Lakes:

Classe Uso típico Latência de acesso Custo relativo

Dica prática: configure Lifecycle Rules no bucket raw para mover automaticamente objetos com mais de 90 dias para Glacier Instant, e com mais de 365 dias para Glacier Deep Archive. Isso reduz custo em 70% ou mais para dados históricos sem trabalho manual.

Recursos complementares

  • Versioning: cada overwrite cria uma nova versão; objetos "deletados" ganham delete marker. Permite desfazer erros.
  • Replication: cópia automática para outro bucket (mesma região ou cross-region) para DR e latência geográfica.
  • Object Lock: bloqueio WORM (write once, read many) para compliance regulatória.
  • Server-Side Encryption: SSE-S3 (chaves gerenciadas pelo S3), SSE-KMS (integração com AWS KMS) e SSE-C (chaves do cliente).

Módulo 5 — Formatos de arquivo no Data Lake

Por que o formato importa

A escolha do formato determina custo, performance de consulta, interoperabilidade e facilidade de evolução. Formatos baseados em texto (CSV, JSON) são humanamente legíveis e fáceis de produzir, mas são lentos para analytics em larga escala e ocupam muito espaço. Formatos binários colunares (Parquet, ORC) são o padrão de facto em camadas trusted e refined.

Panorama dos formatos

Formato Tipo Quando usar Quando evitar
CSVTexto, linhasIngestão inicial, intercâmbio com áreas não técnicas, dumps manuaisAnalytics em escala, schemas complexos
JSON / JSONLTexto, semi-estruturadoAPIs REST, eventos, schemas aninhados e flexíveisVolumes grandes (alto custo de parse e tamanho)
AvroBinário, linhaStreaming (Kafka, Kinesis), evolução de schema automáticaConsultas analíticas colunares
ParquetBinário, colunarAnalytics OLAP, camadas trusted e refined, formato padrão do cursoEscrita frequente de registros individuais (OLTP)
ORCBinário, colunarAmbientes Hive/Hadoop on-premises legadosAmbientes AWS modernos (prefira Parquet)

Row-based vs Column-based

Em formatos row-based (CSV, JSON, Avro), registros são armazenados completos em sequência: linha1[col1,col2,col3] linha2[col1,col2,col3].... Para ler uma coluna específica, o engine precisa ler o arquivo inteiro. Isso é ótimo para OLTP, onde cada transação lê ou grava uma linha inteira.

Em formatos column-based (Parquet, ORC), os dados são reorganizados por coluna dentro de blocos chamados row groups: rowgroup[col1(linha1,2,3...) col2(linha1,2,3...) col3(linha1,2,3...)]. Isso habilita duas otimizações cruciais:

  • Projection pushdown: o engine lê apenas as colunas mencionadas no SELECT. Se você seleciona 2 colunas de uma tabela com 20, lê 10% dos bytes.
  • Predicate pushdown: Parquet guarda estatísticas (min/max) por row group. Um WHERE price > 1000 permite pular row groups cujo max <= 1000 sem abrir o conteúdo.

Além disso, valores similares ficam juntos dentro de uma coluna, o que melhora drasticamente a compressão. Uma coluna order_status com 5 valores distintos em 100 mil linhas comprime para poucos bytes em Parquet.

Parquet em detalhes

Apache Parquet é um formato colunar open source criado em 2013 pelo Twitter e Cloudera, hoje mantido pela Apache Foundation. Especificação e codecs são abertos; leitores existem em praticamente toda linguagem moderna (Java, Python/pandas/PyArrow, R, Rust, Go).

A estrutura interna de um arquivo Parquet é:

  1. Header com magic bytes (PAR1).
  2. Um ou mais row groups, cada um contendo column chunks (um por coluna).
  3. Cada column chunk é dividido em pages comprimidas, com estatísticas.
  4. Footer com metadados globais (schema, localização de cada row group, estatísticas).

Compressão

Parquet suporta múltiplos codecs de compressão por coluna. A escolha é um trade-off entre taxa de compressão, CPU de leitura/escrita e se o arquivo é splittable (capaz de ser processado em paralelo por múltiplos workers).

  • Snappy: compressão média, CPU mínima, splittable em Parquet. É o padrão recomendado para analytics. Parquet+Snappy é a combinação usada no curso.
  • Gzip: taxa de compressão alta, CPU alto, não splittable em CSV mas splittable em Parquet (Parquet comprime por página, não o arquivo inteiro). Bom para dados frios, arquivamento.
  • Zstd: codec moderno da Facebook. Taxa comparável ao Gzip com muito menos CPU. Substituto natural do Gzip em pipelines novos.
  • Sem compressão: usado apenas para depuração.

Módulo 6 — Particionamento

Particionar um dataset é dividir os arquivos em subdiretórios (prefixos S3) de acordo com o valor de uma ou mais colunas. Quando um engine SQL sabe como as partições estão organizadas, ele pode eliminar partições inteiras do plano de execução com base no WHERE da consulta — técnica conhecida como partition pruning.

Convenção estilo Hive

A convenção mais usada é a herdada do Apache Hive, no formato coluna=valor em cada nível de prefixo: 

s3://aluno-trusted/olist/orders/
  year=2026/month=04/day=23/part-00000.parquet
  year=2026/month=04/day=24/part-00000.parquet
  year=2026/month=05/day=01/part-00000.parquet

Uma consulta como SELECT ... FROM orders WHERE year=2026 AND month=04 fará o Athena abrir apenas os arquivos do prefixo year=2026/month=04/. Sem partição, o Athena teria de ler o dataset inteiro.

Boas práticas

  • Particione por colunas frequentemente usadas em WHERE, tipicamente datas (year/month/day, ingestion_date).
  • Evite cardinalidade muito alta: particionar por user_id gera milhões de prefixos com arquivos minúsculos, degrada performance e aumenta custos de metadados.
  • Prefira arquivos de 128–512 MB: row groups menores têm overhead alto de metadados, e arquivos muito grandes reduzem paralelismo.
  • Partição composta: year=.../month=.../day=.../ é melhor que apenas day=.../ porque permite queries de intervalo em cada nível.
  • Partition projection: recurso do Athena que dispensa registrar partições explicitamente — útil quando a estrutura é determinística (data diária, por exemplo).
Atenção — small file problem: particionar com granularidade muito fina (por exemplo, uma partição por minuto) gera milhares de arquivos pequenos. Cada arquivo exige uma requisição S3 e contribui ao overhead de planning do Athena. Consolide com processos de compaction (união periódica de arquivos pequenos) ou reduza a granularidade da partição.

Módulo 7 — AWS Glue Data Catalog

O AWS Glue Data Catalog é um metastore gerenciado pela AWS, compatível com a API do Apache Hive Metastore. Ele guarda a descrição dos dados — schema das tabelas, localização dos arquivos no S3, formatos, partições registradas — mas não guarda os dados em si, que permanecem em S3.

Objetos do catálogo

  • Database: namespace lógico que agrupa tabelas relacionadas (ex.: olist_raw, olist_trusted).
  • Table: definição de schema (colunas e tipos), formato (CSV, Parquet), SerDe (Serializer/Deserializer), location no S3 e colunas de partição.
  • Partition: combinação de valores das colunas de partição (ex.: year=2026, month=04) com um location específico.
  • Crawler: programa gerenciado que varre um prefixo S3, infere schema automaticamente (inclusive partições) e cria/atualiza tabelas.
  • Classifier: regra usada pelo Crawler para reconhecer formatos customizados.

Quem consome

O Glue Data Catalog é a camada de metadados compartilhada da plataforma de dados AWS:

  • Athena: executa SQL diretamente sobre tabelas do catálogo.
  • Redshift Spectrum: estende clusters Redshift para consultar S3 via Catalog.
  • Amazon EMR: clusters Spark/Hive leem o catálogo como Hive Metastore.
  • AWS Glue Jobs (Spark): jobs ETL gerenciados.
  • AWS Lake Formation: governa permissões em nível fino no catálogo.
  • Amazon SageMaker: leitura de datasets para treino de modelos.

Preço

O Data Catalog tem free tier generoso: o primeiro milhão de objetos armazenados e o primeiro milhão de requisições por mês são gratuitos. A cobrança começa em cerca de US$ 1 por 100 mil objetos adicionais. Para o escopo do curso, o uso é gratuito.

Módulo 8 — Amazon Athena

O que é

Amazon Athena é um serviço de consulta SQL serverless baseado no engine de código aberto Trino (antigo PrestoSQL). Não há cluster para provisionar, dimensionar ou manter — você envia uma query, o Athena a executa usando capacidade interna da AWS e devolve o resultado.

Três propriedades essenciais

  1. Serverless: zero administração. Escala automaticamente para queries simultâneas de múltiplos usuários.
  2. SQL ANSI + extensões Trino: suporte completo a SELECT, JOIN, GROUP BY, window functions, CTEs, subqueries correlacionadas, funções analíticas, CAST, JSON functions, geoespaciais (via extensões), expressões regex.
  3. Pay-per-query: cobrado pelos terabytes escaneados (aproximadamente US$ 5,00 por TB, na região us-east-1). Queries que leem muito pagam muito, queries eficientes pagam pouco. Isso muda o jogo: otimização não é só performance, é economia direta.

Como o Athena reduz scan

Três alavancas combinadas, cada uma multiplicando o efeito das outras:

  • Parquet (ou ORC) + compressão: formato colunar lê apenas as colunas mencionadas no SELECT. Snappy/Zstd reduzem tamanho em 2–5x.
  • Particionamento: filtros no WHERE que combinam com colunas de partição eliminam prefixos inteiros antes de abrir qualquer arquivo.
  • Colunas explícitas no SELECT: nunca use SELECT * em produção. Liste apenas as colunas que você realmente precisa.

Exemplo real: a mesma consulta "total de pedidos entregues em 2017" sobre CSV bruto pode ler 50 MB; sobre Parquet particionado por ano-mês, pode ler 1 MB. Em escala de terabytes, isso é a diferença entre US$ 5 e US$ 0,10 por execução.

Workgroups e guardrails

Workgroups são agrupamentos lógicos de queries e usuários. Cada workgroup isola métricas, histórico, bucket de resultado e — crucial em ambiente corporativo — permite configurar limite máximo de TB escaneados por query. Uma consulta que tente ler mais é abortada antes de executar, prevenindo "query bomb" acidental.

O Athena também integra com CloudWatch, enviando métricas de bytes scanned, tempo de execução, queries bem-sucedidas e falhas. Alarmes e budgets são construídos sobre essas métricas.

Query result location

Cada query grava seu resultado em um bucket S3 configurado no workgroup. Antes da primeira query, é obrigatório configurar essa localização, por exemplo s3://<aluno>-athena-results/. Esse bucket também acumula metadados de execução. Configure uma lifecycle rule para apagar objetos com mais de 30 dias e evitar crescimento indefinido.

Passo 6 — Primeiras consultas SQL

Rode cada query e, após a execução, clique em Run details para ver o Data scanned. Anote esse número — usaremos no Passo 8. Ajuste os nomes de tabela se o Crawler gerou variações (ex.: olist_orders_dataset). 

-- 1. Amostra rapida dos pedidos
SELECT *
FROM olist_orders_dataset
LIMIT 10;

-- 2. Pedidos por status
SELECT order_status, COUNT(*) AS total
FROM olist_orders_dataset
GROUP BY order_status
ORDER BY total DESC;

-- 3. Top 10 estados por receita de pedidos entregues
SELECT c.customer_state,
       COUNT(DISTINCT o.order_id) AS pedidos,
       SUM(oi.price)              AS receita
FROM olist_orders_dataset o
JOIN olist_order_items_dataset oi USING (order_id)
JOIN olist_customers_dataset   c  USING (customer_id)
WHERE o.order_status = 'delivered'
GROUP BY c.customer_state
ORDER BY receita DESC
LIMIT 10;

-- 4. Categoria com maior ticket medio
SELECT p.product_category_name,
       AVG(oi.price) AS ticket_medio,
       COUNT(*)      AS itens_vendidos
FROM olist_order_items_dataset oi
JOIN olist_products_dataset p USING (product_id)
WHERE p.product_category_name IS NOT NULL
GROUP BY p.product_category_name
ORDER BY ticket_medio DESC
LIMIT 10;

Passo 7 — CTAS: materializar a camada trusted em Parquet

O comando CREATE TABLE AS SELECT do Athena cria uma nova tabela no catálogo e, simultaneamente, grava os dados no S3 no formato desejado. Vamos converter olist_orders_dataset para Parquet particionado por ano-mês da compra. 

CREATE TABLE olist_raw.orders_parquet
WITH (
  format = 'PARQUET',
  parquet_compression = 'SNAPPY',
  external_location = 's3://<aluno>-trusted/olist/orders/',
  partitioned_by = ARRAY['order_purchase_year_month']
) AS
SELECT
  order_id,
  customer_id,
  order_status,
  CAST(order_purchase_timestamp AS timestamp)            AS order_purchase_ts,
  CAST(order_approved_at        AS timestamp)            AS order_approved_ts,
  CAST(order_delivered_carrier_date AS timestamp)        AS order_delivered_carrier_ts,
  CAST(order_delivered_customer_date AS timestamp)       AS order_delivered_customer_ts,
  CAST(order_estimated_delivery_date AS timestamp)       AS order_estimated_delivery_ts,
  date_format(CAST(order_purchase_timestamp AS timestamp), '%Y-%m')
    AS order_purchase_year_month
FROM olist_raw.olist_orders_dataset;

Depois do CTAS, abra o S3 em s3://<aluno>-trusted/olist/orders/ e veja os prefixos criados, um para cada ano-mês presente nos dados (order_purchase_year_month=2017-01/, =2017-02/, etc.), cada um com um arquivo Parquet.

Passo 8 — Comparativo: CSV bruto vs Parquet particionado

Rode as duas queries abaixo e anote o Data scanned de cada uma. As duas respondem à mesma pergunta lógica. 

-- Versao CSV bruto (tabela criada pelo Crawler)
SELECT COUNT(*) AS pedidos_entregues
FROM olist_raw.olist_orders_dataset
WHERE order_status = 'delivered';

-- Versao Parquet particionado (tabela criada pelo CTAS)
SELECT COUNT(*) AS pedidos_entregues
FROM olist_raw.orders_parquet
WHERE order_status = 'delivered'
  AND order_purchase_year_month BETWEEN '2017-01' AND '2017-12';
Resultado esperado: a query CSV lê o arquivo inteiro (dezenas de MB). A query Parquet lê apenas as partições de 2017 e, dentro delas, somente a coluna order_status. A redução típica é de 80 a 95% no volume escaneado. Em cargas de produção de TB, isso se traduz em reduções diretas de dezenas de dólares por dia na fatura do Athena.

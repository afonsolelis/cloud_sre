Tese da aula

Por que modelar antes de codar

A ferramenta já foi vista. Agora o foco é o problema. Confiabilidade não se programa depois — ela se modela antes.

A armadilha do código prematuro
  1. Abrir o editor antes de entender os stakeholders leva a retrabalho sistemático.
  2. Sem problema bem definido, nenhuma solução pode ser correta — só "funcional".
  3. IA codifica muito rápido o problema errado. A spec é o freio de qualidade.
  4. Confiabilidade é propriedade emergente: aparece no todo, não em componentes isolados.
  5. Modelar é mais barato que reescrever; especificar é mais barato que depurar.
Teoria · SWEBOK V4

Software Engineering Body of Knowledge

Guia consolidado pelo IEEE Computer Society. Define os blocos de conhecimento que um engenheiro de software competente domina. Versão 4 publicada em 2024.

O que é o SWEBOK
  • Knowledge Areas (KAs) organizadas por ciclo de vida.
  • Referência para certificações e currículos.
  • Linguagem comum entre praticantes e academia.
  • Não é um método — é um inventário de conhecimentos.
15 áreas de conhecimento (V4)
  • Requirements, Architecture, Design, Construction, Testing
  • Operations, Maintenance, Configuration Management
  • Engineering Management, Process, Models & Methods
  • Quality, Security, Professional Practice, Economics
Teoria · SWEBOK KA 1

Software Quality

Qualidade não é sinônimo de "código bonito". É o grau em que o software atende requisitos declarados e expectativas implícitas de seus stakeholders.

Pilares da KA Quality no SWEBOK
  1. Fundamentals — cultura, ética, custo da não-qualidade.
  2. Quality Models — modelos como ISO/IEC 25010 que decompõem qualidade em atributos (reliability, performance efficiency, security, maintainability).
  3. Quality Management Processes — V&V, auditoria, revisão, inspeção estática.
  4. Software Quality Tools — análise estática, testes automatizados, code review, observabilidade.

Liga direta com a Aula 03: qualidade se concretiza em RNFs mensuráveis (SLIs/SLOs).

Teoria · SWEBOK KA 2

Software Engineering Models and Methods

Métodos dão disciplina a decisões que seriam, de outro modo, arbitrárias. Modelos reduzem a complexidade do sistema ao necessário para decidir.

Tipos de modelo
  • Informacionais (entidades, relações, fluxos)
  • Comportamentais (estados, sequências)
  • Estruturais (componentes, conectores)
  • Domínio (regras de negócio)
Métodos aplicáveis ao curso
  • Modelos formais leves: EARS para requisitos
  • RM-ODP para arquitetura (Aula 04)
  • FMEA para falhas (Aula 07)
  • ATAM para validação arquitetural (Aula 08)
Teoria · SWEBOK KA 3

Software Engineering Professional Practice

Conhecimentos, habilidades e atitudes de um engenheiro responsável. O que o separa de alguém que apenas escreve código.

Componentes
  1. Professionalism — códigos de conduta, responsabilidade sobre impactos sistêmicos.
  2. Group dynamics & psychology — colaboração, decisões por consenso, revisão de pares.
  3. Communication skills — escrever specs compreensíveis, documentar decisões, apresentar trade-offs.

Nesta aula: a pesquisa em grupo sobre guias de spec é exercício direto de Professional Practice.

Teoria · SEBoK

Systems Engineering Body of Knowledge

Mantido pelo BKCASE (INCOSE + IEEE-CS + Stevens Institute). Enquanto o SWEBOK olha o software, o SEBoK olha o sistema — o todo do qual o software é parte.

Por que engenheiros de dados precisam disso
  • Pipeline de dados é um sistema composto (ETL + DB + obs + rede).
  • Cada parte pode estar correta e o todo falhar mesmo assim.
  • Reliability, availability e performance são propriedades do sistema, não de módulos.
Conceitos centrais que vamos usar
  • Systems thinking
  • Propriedades emergentes
  • System of Systems (SoS)
  • Interfaces e fronteiras
Teoria · SEBoK

Systems thinking — partes vs todo

Pensamento sistêmico: olhar para as relações e feedbacks entre componentes, não apenas para os componentes em si.

Cinco movimentos do raciocínio sistêmico
  1. Identificar o sistema de interesse — onde começam e terminam suas fronteiras?
  2. Mapear os componentes e suas interfaces — o que passa entre eles?
  3. Rastrear fluxos — de dados, de controle, de valor, de falha.
  4. Detectar loops de feedback — onde o sistema se auto-regula ou se auto-amplifica?
  5. Observar o comportamento emergente — o que o todo faz que nenhuma parte faz sozinha?
Teoria · SEBoK

Propriedades emergentes

Propriedade emergente é aquela que existe no sistema e não em nenhum de seus componentes isolados. É o coração da Engenharia de Confiabilidade.

Propriedade Do componente isolado? Do sistema? Como surge
ReliabilityNão tem sentidoSimDa composição correta, redundância, tratamento de falhas
AvailabilityLimitadaSimDa topologia, do failover, do MTTR
PerformanceParcialSimDa coordenação, do caminho crítico, dos gargalos
SegurançaParcialSimDo controle nas fronteiras e entre componentes
EscalabilidadeRaramenteSimDas decisões de concorrência, sharding, estado
Teoria · SEBoK

System of Systems aplicado ao Olist ETL

Um pipeline moderno não é um sistema — é um sistema de sistemas. Cada subsistema tem seu próprio ciclo de vida, governança e falhas independentes.

Subsistemas do Olist Pipeline
  • Fonte de dados (Kaggle / CSVs)
  • Codespace (dev + IA)
  • GitHub (repo, CI)
  • AWS (EC2 app, EC2 data, rede)
  • Observabilidade (CloudWatch, Grafana)
Características de um SoS
  • Independência operacional de cada parte
  • Independência gerencial (times/fornecedores distintos)
  • Distribuição geográfica
  • Evolução assíncrona
  • Comportamento emergente no todo
Aplicação

Do genérico ao concreto — problema SRE do Olist

A aula agora aterrissa no projeto real. Vamos modelar o problema antes de escrever uma linha de código.

Contexto do problema

O negócio Olist precisa processar ~100 mil pedidos do marketplace, carregar em um banco analítico e gerar dashboards diários. O ETL deve ser confiável: idempotente, observável e resiliente a falhas parciais. Não podemos perder dados, nem duplicá-los, nem sofrer silenciosamente.

Pergunta orientadora: o que o sistema precisa garantir para que o negócio confie nos números do dashboard?

Teoria · SDD

Spec-Driven Development — por que especificar é o novo "codar"

IAs generativas invertem o custo relativo das atividades: escrever código ficou barato; escrever o problema certo virou o gargalo.

Antes das IAs
  • Codificar era lento e caro
  • Spec era documento burocrático
  • Revisão focava no código
  • Dívida nascia no teclado
Com IAs produtivas
  • Código é commodity barata
  • Spec vira contrato e input primário
  • Revisão foca na intenção e no fit
  • Dívida nasce em specs ruins

Case do curso

O que vamos construir juntos até a Aula 08

A diretoria da Olist precisa tomar decisões de growth e operação baseadas em dados confiáveis. Ao longo das aulas 02 – 08, a turma entrega um dashboard Grafana alimentado por um pipeline Python. SRE é o que torna isso digno de produção.

1. CSV Olist

  • Dataset Brazilian E-commerce
  • ~100k pedidos históricos
  • Fonte fora do nosso controle

2. ETL Python

  • Extração, transformação, carga
  • Idempotente e observável
  • Orquestrado em EC2

3. Postgres

  • Banco analítico
  • Modelo dimensional mínimo
  • Contrato de dados estável

4. Grafana → Diretoria

  • Entregável visível ao negócio
  • KPIs de growth e operação
  • SLOs de frescor e disponibilidade
O ângulo SRE — por que isso é mais do que BI
  • Frescor: dados prontos todo dia às 06h (SLO de frescor).
  • Disponibilidade: dashboard no ar durante o horário comercial.
  • Completude/Unicidade: zero duplicatas e zero gaps entre execuções.
  • MTTR: pipeline se recupera sozinho de falhas transitórias da fonte.
Case do curso

O que a diretoria quer ver

O dashboard Grafana precisa responder a estas perguntas de negócio. Elas são o input do problema e, portanto, o input da modelagem com IA.

Growth e receita
  • Receita mensal e tendência — total, por estado, por categoria, por vendedor.
  • Top produtos e categorias por receita e volume.
  • Métodos de pagamento e taxa de parcelamento.
  • Taxa de recompra e comportamento de clientes.
Operação e qualidade
  • Volume de pedidos por status (delivered, shipped, canceled, unavailable).
  • Tempo médio de entrega vs SLA prometido — onde estamos atrasando?
  • Score de avaliação por vendedor e categoria — quem está arrastando o NPS?
  • Análise geográfica: concentração por estado, cobertura por CEP.

Suas decisões de modelagem na IA devem começar aqui. O problema tem forma porque o produto tem forma.

Brainstorming e modelagem do problema Olist com IA

Objetivo: cada grupo sai da aula com a primeira versão de spec/00_problem.md comitada no repo — construída em diálogo com as duas IAs.

Prompt inicial sugerido (adapte com sua voz) 
> Estou modelando o problema de confiabilidade de um ETL
> Python que processa o dataset Olist Brazilian E-commerce.
> Me ajude a listar stakeholders, fluxos críticos, modos de
> falha e riscos emergentes usando a ótica do SEBoK
> (systems thinking + propriedades emergentes + SoS).
> Em seguida, critique sua própria resposta: o que ficou
> ambíguo? o que está faltando?
Roteiro da atividade
  1. Rodar o prompt inicial no opencode.
  2. Iterar: questionar suposições e pedir autocrítica.
  3. Repetir o mesmo prompt no gemini.
  4. Comparar divergências entre as duas saídas.
  5. Consolidar em spec/00_problem.md e commitar.
Regra de ouro do SDD

A IA é ferramenta; o responsável pela qualidade da spec é você. Se algo soar certo demais, desconfie. Se algo soar vago, reescreva. A spec é contrato.

Teoria · Conceitos

O que é um Data Lake

Repositório centralizado que armazena dados estruturados, semi-estruturados e não estruturados em seu formato original, separando armazenamento de computação.

Princípios fundamentais
  1. Schema-on-read — o esquema é aplicado no momento da consulta, não na escrita.
  2. Formatos abertos — CSV, JSON, Parquet, ORC, Avro; sem lock-in de fornecedor.
  3. Desacoplamento — armazenamento barato (S3) separado de engines de processamento (Athena, EMR, Spark).
  4. Escala praticamente ilimitada — petabytes com custo linear e previsível.
  5. Multi-workload — BI, ciência de dados, ML e analytics ad-hoc sobre o mesmo dado.
Teoria · Comparativo

Data Lake vs Data Warehouse

Cada arquitetura nasceu para resolver um problema diferente. Organizações modernas combinam as duas (ou adotam Lakehouse).

Aspecto Data Lake Data Warehouse
SchemaSchema-on-readSchema-on-write
Tipos de dadoEstruturado + semi + não estruturadoEstruturado tabular
Custo de armazenamentoMuito baixo (S3)Alto (storage proprietário)
Performance analíticaBoa com Parquet + partiçãoExcelente (MPP colunar)
WorkloadsML, exploração, BIBI e relatórios corporativos
Exemplo AWSS3 + Glue + AthenaRedshift
Teoria · Arquitetura

Medallion Architecture

Organização em camadas progressivas de qualidade e valor de negócio. Padrão adotado por Databricks e amplamente usado em Data Lakes modernos.

Raw / Bronze

  • Cópia fiel da fonte, imutável
  • Formato original (CSV, JSON)
  • Histórico completo preservado
  • Bucket: raw

Trusted / Silver

  • Dados limpos e deduplicados
  • Parquet particionado
  • Schema estável e documentado
  • Bucket: trusted

Refined / Gold

  • Agregações e modelos dimensionais
  • Pronto para consumo (BI, ML)
  • Alinhado a casos de uso de negócio
  • Bucket: refined
Teoria · Amazon S3

Amazon S3 — visão geral

Simple Storage Service: object storage lançado em 2006, o serviço mais antigo da AWS. É a espinha dorsal de praticamente todo Data Lake na AWS.

Conceitos centrais
  • Bucket — contêiner global, nome único mundialmente.
  • Object — arquivo + metadados + chave (key).
  • Key — caminho lógico, ex.: olist/orders/2026/04/23/file.csv.
  • Prefix — parte inicial da key; usado como pseudo-diretório.
Características
  • Durabilidade de 11 noves (99,999999999%)
  • Disponibilidade de 99,99%
  • Consistência forte read-after-write
  • Objetos de 0 bytes até 5 TB
  • Versionamento, replicação e lifecycle
Teoria · Amazon S3

Teoria · Amazon S3

Durabilidade e consistência

Dois conceitos que garantem que o Data Lake não perca dados e que pipelines não vejam estados intermediários.

Durabilidade 99,999999999%
  • Cada objeto replicado em múltiplas AZs dentro da região.
  • Perda esperada: 1 objeto em 10 milhões, a cada 10 mil anos.
  • Checksums verificados automaticamente em cada operação.
Strong read-after-write (desde 2020)
  • PUT de novo objeto: GET subsequente sempre retorna a versão final.
  • Overwrite e DELETE: consistência forte também.
  • LIST reflete imediatamente PUTs e DELETEs.
  • Elimina retries e workarounds em pipelines.

Teoria · Formatos

Row-based vs Column-based e Parquet

Consultas analíticas normalmente leem poucas colunas de muitas linhas. Formatos colunares leem só o necessário.

Row-based (CSV, JSON, Avro)
  • Registros armazenados em sequência completa.
  • Ótimo para ingestão e OLTP.
  • Para SELECT avg(price) é preciso ler o arquivo todo.
Column-based (Parquet, ORC)
  • Dados agrupados por coluna dentro de row groups.
  • Projection pushdown — Athena lê só as colunas do SELECT.
  • Predicate pushdown — estatísticas por row group filtram dados sem lê-los.
  • Compressão por coluna é muito mais eficiente.
Teoria · Otimização

Compressão: Snappy, Gzip e Zstd

Parquet suporta vários codecs. A escolha é um trade-off entre taxa de compressão, CPU e se o arquivo será splittable para paralelismo.

Codec Taxa CPU Splittable Quando usar
SnappyMédiaMuito baixoSim (Parquet)Default para Parquet em analytics
GzipAltaAltoNão (CSV) / Sim (Parquet)Arquivamento, dados frios
ZstdAltaMédioSim (Parquet)Substituto moderno de Gzip
Sem compressãoZeroZeroSimRaramente — só debug
Teoria · Otimização

Particionamento estilo Hive

Particionar é dividir dados em subdiretórios por valor de coluna. Reduz drasticamente o volume escaneado pelo Athena.

Convenção de prefixo 
s3://aluno-trusted/olist/orders/
  year=2026/month=04/day=23/part-00000.parquet
  year=2026/month=04/day=24/part-00000.parquet
  year=2026/month=05/day=01/part-00000.parquet
Boas práticas
  • Particione por colunas usadas em WHERE.
  • Data é a partição mais comum.
  • Evite cardinalidade muito alta (ex.: user_id).
Armadilhas
  • Arquivos pequenos demais (< 128 MB) degradam performance.
  • Partições vazias ocupam metadados sem benefício.
  • Nomes de partição são case-sensitive.
Teoria · Catálogo

AWS Glue Data Catalog

Metastore Hive gerenciado da AWS. Guarda a descrição dos dados (schema, localização, partições); o dado em si continua no S3.

Objetos do catálogo
  • Database — agrupador lógico (ex.: olist_raw).
  • Table — schema + localização S3 + formato.
  • Partition — combinação de valores de partição.
  • Crawler — inferência automática de schema.
Quem consome
  • Athena — executa SQL sobre tabelas do catálogo.
  • Redshift Spectrum — consulta S3 pelo catálogo.
  • EMR, Glue Jobs, Lake Formation, SageMaker.
  • Interface compatível com Apache Hive Metastore.
Teoria · Athena

Amazon Athena

Serviço de consulta SQL serverless construído sobre Trino (antigo PrestoSQL). Sem servidores para gerenciar.

Três propriedades essenciais
  1. Serverless — não há cluster para provisionar, escala automaticamente.
  2. Standard SQL — ANSI SQL com extensões Trino; joins, window functions, CTEs.
  3. Pay-per-query — cobrado pelos TBs de dados efetivamente escaneados (US$ 5,00 por TB aproximadamente).
Teoria · Athena

Como reduzir a conta do Athena

Três alavancas combinadas reduzem o custo em 90% ou mais sem trocar o engine.

1. Parquet + compressão

  • Formato colunar lê só as colunas do SELECT.
  • Snappy reduz volume em 2–5x.
  • CSV 1 GB vira Parquet de 200–300 MB.

2. Particionamento

  • Filtro por partição evita scan de dados irrelevantes.
  • WHERE year=2026 AND month=04 lê só o subdiretório.
  • Complementa projection pushdown.

3. Colunas explícitas

  • Sempre liste colunas — evite SELECT *.
  • Cada coluna não lida é economia direta.
  • Impacto amplificado em Parquet.
Teoria · Athena

Workgroups, result location e guardrails

Workgroups isolam usuários, queries e custos. Antes de rodar a primeira query é obrigatório configurar a pasta de resultados.

Workgroup
  • Grupo lógico de queries e usuários.
  • Isola histórico, métricas e settings.
  • Permite configurar limite máximo de TB por query.
  • Emite alarmes CloudWatch por uso.
Query result location
  • Bucket S3 onde o Athena grava resultados e manifests.
  • Exemplo: s3://aluno-athena-results/.
  • Obrigatório antes de rodar a primeira query.
  • Crie lifecycle rule para apagar resultados antigos.

Prática · Passo 3

Criar o database olist_raw no Glue

Database é o agrupador lógico no Glue Data Catalog. Cada camada do Data Lake terá seu próprio database.

Console
  1. Abra o console AWS e procure por AWS Glue.
  2. Menu lateral: Data Catalog → Databases.
  3. Clique em Add database.
  4. Nome: olist_raw.
  5. Location (opcional): s3://<aluno>-raw/olist/.
  6. Clique em Create database.
Prática · Passo 4

Configurar e executar um Glue Crawler

O Crawler lê amostras dos arquivos em S3, infere o schema e cria/atualiza tabelas no Glue Data Catalog.

Passo a passo
  1. Glue → Crawlers → Create crawler.
  2. Nome: olist-raw-crawler.
  3. Data source: S3, path s3://<aluno>-raw/olist/.
  4. IAM role: selecione LabRole.
  5. Target database: olist_raw.
  6. Schedule: On demand.
  7. Clique em Create crawler e depois em Run crawler.
  8. Aguarde status Ready. Devem aparecer 9 tabelas.
Prática · Passo 5

Preparar o Athena para rodar queries

Antes da primeira query, o Athena exige uma localização S3 para gravar resultados. Esse passo é esquecido com frequência.

Configuração
  1. Console: abra Amazon Athena.
  2. Query editor → Settings → Manage.
  3. Query result location: s3://<aluno>-athena-results/.
  4. Se o bucket não existir: aws s3 mb s3://<aluno>-athena-results.
  5. Salve as configurações.
  6. No editor, selecione database olist_raw.
Prática · Passo 6

Primeiras consultas SQL no Athena

Rode as consultas no editor e observe em Run details o volume escaneado (em MB) para cada uma.

Queries de exploração 
-- 1. Espiar o dataset de pedidos
SELECT * FROM olist_orders_dataset LIMIT 10;

-- 2. Pedidos por status
SELECT order_status, COUNT(*) AS total
FROM olist_orders_dataset
GROUP BY order_status
ORDER BY total DESC;

-- 3. JOIN pedidos x itens x clientes
SELECT c.customer_state,
       COUNT(DISTINCT o.order_id) AS pedidos,
       SUM(oi.price)              AS receita
FROM olist_orders_dataset o
JOIN olist_order_items_dataset oi USING (order_id)
JOIN olist_customers_dataset c    USING (customer_id)
WHERE o.order_status = 'delivered'
GROUP BY c.customer_state
ORDER BY receita DESC;
Prática · Passo 7

CTAS — converter CSV em Parquet particionado

CREATE TABLE AS SELECT materializa o resultado em S3 no formato desejado e registra a tabela no Catalog.

CTAS no Athena 
CREATE TABLE olist_raw.orders_parquet
WITH (
  format = 'PARQUET',
  parquet_compression = 'SNAPPY',
  external_location = 's3://<aluno>-trusted/olist/orders/',
  partitioned_by = ARRAY['order_purchase_year_month']
) AS
SELECT
  order_id,
  customer_id,
  order_status,
  CAST(order_purchase_timestamp AS timestamp) AS order_purchase_ts,
  order_delivered_customer_date,
  date_format(CAST(order_purchase_timestamp AS timestamp), '%Y-%m')
    AS order_purchase_year_month
FROM olist_raw.olist_orders_dataset;
Prática · Passo 8

Comparativo: CSV bruto vs Parquet particionado

A mesma consulta lógica em duas tabelas — observe o campo Data scanned em Run details.

Duas versões da mesma pergunta 
-- Sobre CSV bruto
SELECT COUNT(*) FROM olist_raw.olist_orders_dataset
WHERE order_status = 'delivered';

-- Sobre Parquet particionado por ano-mes
SELECT COUNT(*) FROM olist_raw.orders_parquet
WHERE order_status = 'delivered'
  AND order_purchase_year_month BETWEEN '2017-01' AND '2017-12';
Expectativa
  • CSV: scan do arquivo completo (dezenas de MB).
  • Parquet + partição: scan de poucos MB da coluna order_status.
  • Redução típica de 80–95% em volume escaneado.

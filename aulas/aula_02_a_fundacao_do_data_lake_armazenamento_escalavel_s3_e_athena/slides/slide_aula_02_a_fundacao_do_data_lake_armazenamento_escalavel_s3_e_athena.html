Fundacao

Por que o S3 vira a fundacao do Data Lake

Motivos principais
  • Escala praticamente ilimitada.
  • Durabilidade alta para dados brutos e historicos.
  • Separacao entre armazenamento e processamento.
  • Integracao nativa com Athena, Glue, Redshift e EMR.
Leitura de arquitetura
  • S3 armazena.
  • Athena consulta.
  • Glue cataloga.
  • Outros servicos processam sem mover tudo para um banco unico.
Um Data Lake bem desenhado comeca menos pelo volume e mais pela disciplina de organizacao desde o primeiro bucket.
Camadas

Estrutura recomendada: Raw, Trusted e Refined

Raw / Bronze

  • Recebe dado como chegou da origem.
  • Baixa transformacao.
  • Serve para auditoria e reprocessamento.

Trusted / Silver

  • Dados limpos, padronizados e tipados.
  • Qualidade melhorada.
  • Ponto de consumo analitico intermediario.

Refined / Gold

  • Dados agregados por dominio ou produto.
  • Prontos para dashboards e analises de negocio.
  • Maior semantica, menor ambiguidade.

Regra operacional

  • Nao pule camadas sem motivo claro.
  • Defina responsabilidade de cada zona.
  • Evite misturar dado bruto com dado curado.
Particionamento

Particionamento estrategico no S3

Quando faz sentido
  • Leitura por data, regiao, tenant ou dominio.
  • Consultas filtram subconjuntos recorrentes.
  • Volume cresce continuamente.
Quando atrapalha
  • Particoes demais com poucos arquivos em cada uma.
  • Estrutura de pastas sem alinhamento com consulta.
  • Explosao de pequenos arquivos.
Regra pratica: particao boa e a que reduz leitura sem tornar a escrita e a manutencao mais caras do que o ganho.
Boas praticas

Formato de arquivo, compressao e lifecycle

CSV e JSON

  • Otimos para ingestao simples e interoperabilidade.
  • Custam mais em leitura analitica.

Parquet

  • Formato colunar.
  • Menor custo e melhor performance no Athena.

Compressao

  • Reduz storage e leitura desnecessaria.
  • Deve ser combinada com um bom particionamento.

Lifecycle

  • Move dados frios para classes mais baratas.
  • Ajuda a controlar custo sem perder historico.
Athena

O que e o Amazon Athena no ecossistema analitico

Ideia central
  • Executa SQL sem servidor dedicado.
  • Le dados diretamente do S3.
  • Paga por volume lido.
Dependencias de qualidade
  • Schema coerente.
  • Particionamento util.
  • Formato eficiente.
Athena nao corrige um lake mal organizado. Ele expoe rapidamente os custos das mas decisoes de layout.
Consulta

Como reduzir custo e ganhar performance no Athena

Usar Parquet

  • Le menos dados e acelera consultas.

Filtrar por particao

  • Evita scan desnecessario de diretorios inteiros.

Evitar arquivos pequenos

  • Reduz overhead operacional e de leitura.

Separar raw e curated

  • Evita schema inconsistente e consumo indevido.

Hands-on

Passo a passo operacional

No S3
  1. Criar um bucket com nome unico.
  2. Criar prefixos para `raw/`, `trusted/` e `refined/`.
  3. Enviar um arquivo de origem para `raw/`.
No Athena
  1. Definir output location para resultados.
  2. Criar tabela externa apontando para `raw/`.
  3. Executar `SELECT * LIMIT 10` para validar leitura.
Se a consulta estiver lenta ou cara, volte ao desenho do storage. Em Data Lake, performance comeca no layout, nao na query.

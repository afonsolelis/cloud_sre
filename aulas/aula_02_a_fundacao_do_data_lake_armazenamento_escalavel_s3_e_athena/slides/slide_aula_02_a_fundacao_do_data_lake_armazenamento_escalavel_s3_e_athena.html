Fundação

Por que o S3 vira a fundação do Data Lake

Motivos principais
  • Escala praticamente ilimitada.
  • Durabilidade alta para dados brutos e históricos.
  • Separação entre armazenamento e processamento.
  • Integração nativa com Athena, Glue, Redshift e EMR.
Leitura de arquitetura
  • S3 armazena.
  • Athena consulta.
  • Glue cataloga.
  • Outros serviços processam sem mover tudo para um banco único.
Um Data Lake bem desenhado começa menos pelo volume e mais pela disciplina de organização desde o primeiro bucket.

Camadas

Estrutura recomendada: Raw, Trusted e Refined

Raw / Bronze

  • Recebe dado como chegou da origem.
  • Baixa transformação.
  • Serve para auditoria e reprocessamento.

Trusted / Silver

  • Dados limpos, padronizados e tipados.
  • Qualidade melhorada.
  • Ponto de consumo analítico intermediário.

Refined / Gold

  • Dados agregados por domínio ou produto.
  • Prontos para dashboards e análises de negócio.
  • Maior semântica, menor ambiguidade.

Regra operacional

  • Não pule camadas sem motivo claro.
  • Defina responsabilidade de cada zona.
  • Evite misturar dado bruto com dado curado.

Particionamento

Particionamento estratégico no S3

Quando faz sentido
  • Leitura por data, região, tenant ou domínio.
  • Consultas filtram subconjuntos recorrentes.
  • Volume cresce continuamente.
Quando atrapalha
  • Partições demais com poucos arquivos em cada uma.
  • Estrutura de pastas sem alinhamento com consulta.
  • Explosão de pequenos arquivos.
Regra prática: partição boa é a que reduz leitura sem tornar a escrita e a manutenção mais caras do que o ganho.

Boas práticas

Formato de arquivo, compressão e lifecycle

CSV e JSON

  • Ótimos para ingestão simples e interoperabilidade.
  • Custam mais em leitura analítica.

Parquet

  • Formato colunar.
  • Menor custo e melhor performance no Athena.

Compressão

  • Reduz storage e leitura desnecessária.
  • Deve ser combinada com um bom particionamento.

Lifecycle

  • Move dados frios para classes mais baratas.
  • Ajuda a controlar custo sem perder histórico.

Athena

O que é o Amazon Athena no ecossistema analítico

Ideia central
  • Executa SQL sem servidor dedicado.
  • Lê dados diretamente do S3.
  • Paga por volume lido.
Dependências de qualidade
  • Schema coerente.
  • Particionamento útil.
  • Formato eficiente.
Athena não corrige um lake mal organizado. Ele expõe rapidamente os custos das más decisões de layout.

Consulta

Como reduzir custo e ganhar performance no Athena

Usar Parquet

  • Lê menos dados e acelera consultas.

Filtrar por partição

  • Evita scan desnecessário de diretórios inteiros.

Evitar arquivos pequenos

  • Reduz overhead operacional e de leitura.

Separar raw e curated

  • Evita schema inconsistente e consumo indevido.

Hands-on

Passo a passo operacional

No S3
  1. Criar um bucket com nome único.
  2. Criar prefixos para `raw/`, `trusted/` e `refined/`.
  3. Enviar um arquivo de origem para `raw/`.
No Athena
  1. Definir output location para resultados.
  2. Criar tabela externa apontando para `raw/`.
  3. Executar `SELECT * LIMIT 10` para validar leitura.
Se a consulta estiver lenta ou cara, volte ao desenho do storage. Em Data Lake, performance começa no layout, não na query.
